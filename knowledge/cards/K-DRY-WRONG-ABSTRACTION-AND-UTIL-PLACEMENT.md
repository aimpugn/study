# K-DRY-WRONG-ABSTRACTION-AND-UTIL-PLACEMENT

## 1) 이 카드가 답하는 질문

실무에서 아래 질문이 반복됩니다.

"중복 구현을 발견했을 때, 언제는 공통 유틸/공통 모듈로 올려서 재사용해야 하고, 언제는 중복을 유지하는 편이 더 안전합니까? 그리고 Hexagonal/DDD처럼 경계가 있는 구조에서, 그 공통 코드는 `core`/`adapter`/`server` 중 어디에 두는 것이 정석에 가깝습니까?"

이 질문은 단순히 DRY 원칙을 지키느냐의 문제가 아닙니다. 중복을 없애는 순간 "결합도(한 번에 같이 바뀌는 범위)"가 커지고, 그 결합도가 팀의 변경 속도와 장애 형태를 바꿉니다. 따라서 이 카드는 "중복"을 선악이 아니라 "비용 교환"으로 다루고, 그 교환을 어떤 제어점으로 판단해야 하는지 고정합니다.

---

## 2) 배경: 왜 이 문제가 실무에서 비싸게 터지는가

중복 구현이 많아지면 가장 먼저 나타나는 비용은 "드리프트(drift)"입니다. 같은 이름의 함수가 여러 군데에 있고 동작이 조금씩 다르면, 어느 순간부터는 "어떤 경로에서는 통과하고 어떤 경로에서는 실패"하는 비결정성이 생깁니다. 이때 관측되는 증상은 `null`, 파라미터 바인딩 실패, 특정 DB row만 누락, 특정 요청만 실패처럼 2차 증상입니다. 원인(중복 구현의 미세한 차이)은 코드베이스 여기저기에 흩어져 있고, 디버깅은 전체 검색 + 추측에 의존하게 되어 비용이 급격히 커집니다.

반대로 중복을 빠르게 제거하면(공통화/유틸화) 또 다른 비용이 생깁니다. 공통화는 "같이 바뀌는 범위"를 넓히는 행위입니다. 즉 한 곳을 바꾸면 여러 모듈/기능/입구가 함께 영향을 받습니다. 이때 공통 코드가 "잘못된 추상화(wrong abstraction)"였다면, 중복이 남아 있었을 때보다 더 큰 폭발 반경으로 회귀/장애가 발생합니다. Sandi Metz가 강조하는 메시지도 여기에 가깝습니다. 그는 "중복 자체"보다 "잘못된 추상화"의 비용이 더 크며, 그래서 중복을 성급히 없애는 것이 항상 이득이 아니라고 말합니다.

경계가 있는 시스템(Hexagonal/DDD)에서는 비용이 한 번 더 증폭됩니다. 공통 코드의 위치를 잘못 두면, `core`가 인프라 의미(DB/Jackson/Spring)를 알게 되거나, 어댑터가 도메인 의미를 침범하는 식으로 "의존성 방향"이 깨질 수 있습니다. 이 경우에는 단순한 가독성 문제가 아니라, 테스트/배포/확장 전략 자체가 흔들립니다.

## 3) 핵심 제어점(왜 하필 이 축인가)

이 주제는 "원칙"을 나열하면 끝이 없습니다. 그래서 먼저 판단을 고정하는 제어점 3개로 축을 좁히는 것이 안전합니다. 아래 3개는 중복을 제거할지 말지, 제거한다면 어디에 둘지를 결정하는 최소 축입니다.

첫째는 **중복이 같은 의미(semantic)인가, 우연히 비슷해 보이는 구현(incidental)인가**입니다. "문자열을 trim하고 blank면 null" 같은 형태는 겉으로는 단순하지만, 그 규칙이 적용되는 맥락에 따라 의미가 달라질 수 있습니다. 예를 들어 DB 저장에서는 `""`를 `NULL`로 바꾸는 것이 "정규화"일 수 있지만, 고정 길이 프로토콜에서는 공백이 "패딩"으로 의미를 가지며 trim이 곧 데이터 손상일 수 있습니다. 의미가 같지 않다면 공통화는 잘못된 추상화가 됩니다.

둘째는 **그 의미의 소유자가 누구인가(경계/레이어 소유권)**입니다. Hexagonal에서 핵심은 "업무 의미(도메인/유스케이스)"와 "기술/전송/저장(인프라)"의 변화 주기를 분리하는 것입니다. 따라서 공통 코드가 인프라 의미(DB NULL, SQL 조립, HTTP header 등)를 담고 있다면 `core`로 올리는 순간 도메인이 인프라의 변화 주기에 묶입니다. 반대로 공통 코드가 업무 의미(예: `MessageNo`의 정규화, 멱등성 키 구성 규칙)를 담고 있다면 인프라로 내려보내는 순간 도메인 규칙이 여러 입구/출구로 흩어질 위험이 커집니다.

셋째는 **공통화가 만들어내는 결합도의 비용(폭발 반경)과 중복이 만드는 드리프트 비용 중 무엇이 더 큰가**입니다. 공통화는 "한 번의 변경이 여러 소비자에게 동시에 적용"되는 장점이 있지만, 그만큼 "한 번의 실수"도 여러 소비자에게 동시에 전파됩니다. 중복은 변경 반경이 작다는 장점이 있지만, "한 군데만 수정하고 나머지는 놓치는" 샷건 수술과 드리프트가 비용입니다. 이 둘 중 무엇이 더 큰지는 팀의 배포 단위(모놀리식/멀티모듈 vs 서비스 분리), 테스트 신뢰도, 변경 빈도에 의해 달라집니다.

## 4) 기본 결론(기본값)과 선택 이유

기본 결론은 다음처럼 닫습니다.

중복을 보면 즉시 공통화하기보다, 먼저 "이 중복이 같은 의미인가"를 확인하고, 그 의미가 안정화될 때까지는 제한적 중복을 허용하는 편이 안전합니다. 공통화를 하기로 결정했다면, 공통 코드는 "가장 많은 곳에서 쓰이기 쉬운 위치"가 아니라 "그 의미의 소유자 경계"에 둬야 합니다. 즉 도메인 의미면 `core`, 인프라 의미면 `server/outbound`, 특정 VAN/프로토콜 의미면 `adapter` 쪽이 기본값이 됩니다.

이 기본값이 정당화되는 이유는 실패 비용을 더 일찍, 더 좁은 범위에서 관측하게 만들기 때문입니다. 의미가 안정화되지 않은 상태에서 공통화를 하면, 추상화가 틀렸다는 사실을 "다음 변경"에서야 알게 되고, 그 변경은 여러 소비자에 걸쳐 동시에 깨지기 때문에 복구 비용이 큽니다. 반대로 의미가 안정화된 뒤 공통화하면, "한 번에 같이 바뀌어야 하는 것"을 실제로 같이 바꾸게 되어 드리프트 비용을 줄일 수 있습니다.

## 5) 핵심 용어 정의(이 문서 맥락에서)

이 카드에서 사용하는 용어는 다음처럼 닫습니다.

**중복(duplication)**은 "비슷한 코드가 있다"가 아니라, "같은 의미를 표현하는 로직이 여러 곳에 있다"를 뜻합니다. 단순히 코드가 비슷해 보여도 의미가 다르면 이 카드는 그것을 중복으로 취급하지 않습니다.

**드리프트(drift)**는 "원래 같은 의미였던 두 구현이 시간이 지나며 서로 다른 의미로 갈라지는 현상"입니다. 드리프트의 핵심은 실패가 즉시 드러나지 않고, 특정 경로/특정 데이터에서만 관측되는 형태가 되기 쉽다는 점입니다.

**추상화(abstraction)**는 "코드를 줄이기 위한 함수"가 아니라, "여러 소비자가 공유하는 의미를 한 이름 아래 고정하는 계약"입니다. 이 계약이 틀리면 **잘못된 추상화(wrong abstraction)**가 되고, 그 비용은 소비자 수만큼 커집니다.

**의미의 소유자(owner of meaning)**는 "그 규칙이 바뀔 때, 누구의 변경 이유가 정당한가"로 판정합니다. DB NULL 정규화는 DB/리포지토리의 요구가 소유하고, 업무 식별자 생성 규칙은 도메인/유스케이스가 소유합니다.

## 6) 메커니즘(입력 -> 변환 -> 산출물 -> 소비자)

중복 제거/공통화가 실제로 시스템을 어떻게 바꾸는지, 입력-변환-산출물-소비 흐름으로 닫아야 판단이 일관됩니다.

중복을 발견했을 때의 입력은 "동일/유사한 로직이 여러 위치에 존재한다"는 사실입니다. 여기서 변환(의사결정)은 "이 로직들이 같은 의미인지"를 확인하고, 같다면 "공유할 이름/계약"을 만들고, 그 계약을 어디에 둘지(경계)를 선택하는 과정입니다. 산출물은 "공통 함수/값 객체/정책 객체"이며, 소비자는 그 공통 코드를 호출하는 여러 모듈/유스케이스/어댑터입니다.

이때 공통화의 핵심 효과는 두 가지입니다.

첫째, **동작 일관성**입니다. 공통 코드가 단일 구현이 되면, 같은 입력에 대해 같은 출력을 보장하기 쉬워집니다.

둘째, **결합도 증가**입니다. 공통 코드를 바꾸면 소비자들이 동시에 영향을 받습니다. 이 결합도는 배포 단위가 작을수록(독립 서비스) 더 비싸고, 배포 단위가 클수록(단일 애플리케이션) 상대적으로 덜 비쌉니다. 그러나 배포 단위가 커도 "테스트가 얕으면" 결합도 비용은 운영에서 터질 수 있습니다.

따라서 이 메커니즘을 기반으로, 공통화는 "동작 일관성 이득"이 "결합도 비용"을 이길 때만 정당화됩니다.

## 7) 조건 변경 시 예측(무엇이 깨지고 어떻게 관측되는가)

이 섹션은 제어점 단위로 실패 모드와 관측을 닫습니다.

### 7.1 의미가 같지 않은데 공통화하면 무엇이 깨지나

예를 들어 "문자열을 trim하고 blank면 null"이라는 규칙을 공통화했는데, 어떤 소비자는 고정 길이 필드(공백 패딩)를 다루고 있었다고 가정하겠습니다. 이 경우 공통 유틸이 trim을 수행하는 순간, 필드 길이가 줄어들고 프로토콜 프레이밍이 깨질 수 있습니다.

관측 포인트는 단순한 "문자열 값"이 아니라, 길이/프레임/체크섬 같은 구조적 실패입니다. 예를 들어 "expected length=20, actual=18" 같은 예외, 프레임 파싱 실패, 상위 장비의 "invalid message length" 같은 오류가 증가합니다. 즉 데이터 손상이 "다음 단계의 파서"에서 관측된다는 점이 핵심입니다.

### 7.2 의미는 같은데 중복을 방치하면 무엇이 깨지나

중복이 방치되면 시간이 지나며 구현이 미세하게 달라질 가능성이 높습니다. 특히 누군가는 `trim()`을 추가하고, 누군가는 `isBlank()` 대신 `isEmpty()`를 쓰고, 누군가는 `"   "`를 빈 값으로 처리하지 않는 식의 차이가 생깁니다.

관측 포인트는 "같은 입력인데 경로별 결과가 다르다"입니다. 예를 들어 DB 저장에서는 `" a "`가 `"a"`로 정규화되지만, 다른 경로에서는 `" a "`가 그대로 저장되거나, 검색 조건에서만 trim이 적용되는 식의 불일치가 나타납니다. 이때 장애는 특정 데이터/특정 고객에서만 발생하는 형태로 나타나기 쉬워서 더 비쌉니다.

### 7.3 공통 코드를 core에 두면 무엇이 깨질 수 있나

Hexagonal에서 `core`의 목적은 "인프라에 독립적인 업무 의미"를 담는 것입니다. 따라서 공통 코드가 인프라 라이브러리(SDK, HTTP client, JDBC, 프레임워크)에 기대면, 그 코드를 `core`에 두는 순간 core가 인프라 변화에 종속됩니다.

관측 포인트는 빌드/의존성 규칙 붕괴입니다. 예를 들어 core가 특정 프레임워크 패키지를 import하게 되어 "core는 프레임워크를 몰라야 한다"는 규칙이 깨지고, 이를 막는 테스트(아키텍처 테스트)가 있다면 빌드 단계에서 실패합니다. 테스트가 없다면 운영/배포 단계에서 더 뒤늦게 비용이 나타납니다.

## 8) 코드 예시(나쁜 패턴 1 + 권장 패턴 1 이상)

아래 코드는 "중복이 왜 위험한가"와 "공통화가 왜 위험할 수 있는가"를 대비시키기 위한 최소 예시입니다.

### 8.1 같은 이름이지만 의미가 다른 중복(나쁜 패턴)

```java
final class StringsA {
    // "검색 조건"에서는 앞뒤 공백을 무시하고 싶어서 trim을 함
    static String nullIfBlank(String value) {
        if (value == null) return null;
        String trimmed = value.trim();
        return trimmed.isBlank() ? null : trimmed;
    }
}

final class StringsB {
    // "고정 길이 필드"에서는 공백이 패딩이므로 trim을 하면 안 됨
    static String nullIfBlank(String value) {
        if (value == null) return null;
        return value.isBlank() ? null : value; // intentionally no trim
    }
}
```

이 둘은 겉보기에는 같은 함수처럼 보이지만, 실제로는 의미가 다릅니다. 이 상태에서 "중복이니까 하나로 합치자"라고 공통화를 하면, 둘 중 하나는 반드시 망가집니다.

### 8.2 의미 소유자 경계에 공통화를 두는 권장 패턴

```java
// "DB 컬럼 저장" 맥락에서의 정규화 규칙이라면, persistence 경계에 두는 편이 안전합니다.
final class DbStrings {
    private DbStrings() {}

    static String nullIfBlankForDb(String value) {
        if (value == null) return null;
        String trimmed = value.trim();
        return trimmed.isBlank() ? null : trimmed;
    }
}

// "고정 길이 필드" 맥락에서는, 길이/패딩을 보존하는 규칙을 별도로 둡니다.
final class FixedLengthFields {
    private FixedLengthFields() {}

    static String nullIfAllSpacesKeepLength(String value) {
        if (value == null) return null;
        return value.chars().allMatch(ch -> ch == ' ') ? null : value; // no trim
    }
}
```

핵심은 "함수를 하나로 줄였다"가 아니라, "함수 이름/위치가 의미를 드러내고, 잘못된 공통화를 막는다"입니다.

## 9) 최소 실험(검증)

아래 실험은 특정 프로젝트 도구 없이, 자바 코드만으로 바로 재현 가능한 형태로 고정합니다. 적어도 하나는 실패를 관측하도록 설계합니다.

### 9.1 드리프트가 실제 버그가 되는지 확인(실패 관측)

```java
public class DriftLab {
    static String v1(String value) {
        if (value == null) return null;
        String trimmed = value.trim();
        return trimmed.isBlank() ? null : trimmed;
    }

    static String v2(String value) {
        if (value == null) return null;
        return value.isBlank() ? null : value; // no trim
    }

    public static void main(String[] args) {
        String input = "  a  ";
        System.out.println("v1=[" + v1(input) + "]");
        System.out.println("v2=[" + v2(input) + "]");
    }
}
```

기대 결과는 두 출력이 다르다는 것입니다. 즉 중복 구현은 시간이 지나면 "같은 입력이 다른 결과"로 수렴할 수 있으며, 이 차이가 실제 기능/정합성 버그로 이어질 수 있음을 확인합니다.

### 9.2 잘못된 공통화가 구조를 깨는지 확인(실패 관측)

```java
public class FixedLengthLab {
    static String normalizeWrong(String fixedLengthField) {
        // 고정 길이 필드에 trim을 적용하면 길이가 변할 수 있습니다.
        return fixedLengthField == null ? null : fixedLengthField.trim();
    }

    public static void main(String[] args) {
        String field = "A         "; // 길이 10 (A + padding)
        String normalized = normalizeWrong(field);

        System.out.println("raw.length=" + field.length());
        System.out.println("normalized.length=" + normalized.length());
    }
}
```

기대 결과는 `normalized.length`가 더 작아지는 것입니다. 이 실험은 "공통 유틸을 만들 때 의미 소유 경계를 무시하면, 다음 소비자(프레임/프로토콜)가 구조적 실패로 관측할 수 있다"는 메커니즘을 고정합니다.

## 10) 트레이드오프/대안/Variants

여기서는 "기본값이 항상 옳다"가 아니라, 전제 변화에 따라 결론이 바뀌는 지점을 닫습니다.

첫째 트레이드오프는 **중복(드리프트 위험)** vs **공통화(결합도 증가)**입니다. 중복은 변경 반경이 작고 모듈 독립성을 유지하기 쉽지만, 시간이 지나면 불일치가 생기기 쉽습니다. 공통화는 불일치를 줄이지만, 추상화가 틀리면 한 번의 변경으로 여러 소비자가 동시에 깨집니다. 따라서 공통화는 "의미가 안정적"이고 "동시에 같이 바뀌어야 하는 것"일 때만 선택하는 편이 안전합니다.

둘째 트레이드오프는 **core 공통화(재사용)** vs **인프라/어댑터 로컬화(경계 보호)**입니다. core에 올리면 여러 모듈에서 쉽게 재사용되지만, core가 인프라 의미를 알게 되면 Hexagonal의 이점(독립성)이 사라집니다. 반대로 인프라/어댑터에 두면 core는 깨끗해지지만, 중복이 더 남을 수 있습니다. 이때는 "중복이 남는 것"보다 "경계가 유지되는 것"이 더 가치가 큰 경우가 많습니다.

아래는 자주 등장하는 Variants입니다.

Variant A: **독립 배포(서비스 분리) 환경**에서는 공통 라이브러리 공유가 팀 간 릴리즈 결합을 만들 수 있습니다. 이 경우 "중복을 남기되, 테스트/샘플/스펙을 공유"하는 방식이 더 저렴할 수 있습니다.

Variant B: **단일 배포(멀티모듈 모노레포)** 환경에서는 공통화 비용이 상대적으로 낮습니다. 단, 이 경우에도 "의미 소유 경계"를 무시하면 core 오염이 발생하므로, 공통화의 위치는 여전히 중요합니다.

## 11) 흔한 오해

오해는 그럴듯해 보이기 때문에 위험합니다. 왜 오해인지와 어디서 깨지는지를 닫습니다.

첫째 오해는 "중복은 무조건 나쁘다"입니다. 중복은 나쁘고 좋음이 아니라 비용 교환입니다. 의미가 불안정한 상태에서 중복을 없애면, 잘못된 추상화가 만들어지고 결합도 비용이 폭발합니다. 중복을 즉시 없애는 것이 아니라 "의미가 같음을 확인하고" "의미가 안정화된 뒤"에 없애는 것이 핵심입니다.

둘째 오해는 "공통은 core에 모아야 한다"입니다. core는 "공통"이 아니라 "업무 의미"의 중심입니다. 인프라 의미(저장/전송/포맷)까지 core로 올리면, core가 인프라 변화에 끌려가고 경계가 무너집니다. 공통화의 위치는 재사용 편의가 아니라 의미 소유권으로 결정해야 합니다.

## 12) 팀 적용 질문(선택)

아래 질문은 체크리스트가 아니라, 순서가 있는 의사결정 질문입니다.

첫째, 지금 보이는 중복은 "같은 의미"입니까, 아니면 "비슷해 보이는 구현"입니까? 이 질문에 답하려면, 입력/출력 예시를 3개 이상으로 고정하고(예: `"  a  "`, `"   "`, `null`), 구현 간 결과가 동일해야 합니다.

둘째, 이 규칙의 소유자는 어디입니까? 도메인 규칙입니까, 인프라 정규화 규칙입니까, 특정 어댑터의 프로토콜 규칙입니까? 소유자를 고정하면 위치(core/server/adapter)가 자연스럽게 좁혀집니다.

셋째, 공통화의 실패를 어떻게 조기에 관측할 수 있습니까? 공통화는 결합도를 올리므로, 테스트/아키텍처 규칙/로그로 "잘못된 공통화"를 빠르게 드러내는 안전장치가 필요합니다.
