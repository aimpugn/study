# K-TYPE-BOUNDARY-ENUM-FAILFAST

## 1. 이 문서의 목적
이 문서는 "외부에서 문자열로 들어오는 식별자를 내부에서 어떤 타입으로 다뤄야 안전한가"를 실무 기준으로 설명합니다.

이 주제는 단순히 `enum` 문법 문제가 아닙니다. 장애가 언제, 어디서, 얼마나 비싸게 터지는지를 결정하는 설계 문제입니다.

이 문서를 읽고 나면 아래를 스스로 판단하실 수 있어야 합니다.
- 어떤 식별자는 `String`이 아니라 `enum`으로 고정해야 하는지
- 왜 경계에서 fail-fast 검증을 해야 하는지
- 왜 `switch` 완결성(exhaustiveness)이 운영 안정성으로 이어지는지

---

## 2. 먼저 결론
실무 기본값은 다음과 같습니다.

- 값 후보가 정해진 식별자(예: `mode`, `profile`, `operation type`)는 내부에서 `enum`으로 다룹니다.
- 외부 입력은 경계에서 `String -> enum`으로 변환하고, 잘못된 값은 즉시 실패시킵니다.
- 분기 로직은 enum `switch`로 작성하고, 가능하면 `default`를 두지 않아 누락 분기를 컴파일 단계에서 드러내도록 합니다.

이 결론의 핵심은 "문제 자체를 없애는 것"이 아니라 "문제가 터지더라도 가장 이른 시점에, 가장 싼 비용으로 발견하는 것"입니다.

---

## 3. 왜 이 주제가 어려운가
문자열 분기는 처음엔 빠르게 작성됩니다. 그래서 초기 개발에서는 매우 편해 보입니다.

하지만 시간이 지나면 다음 비용이 누적됩니다.
- 같은 의미의 문자열이 여러 파일에 흩어집니다.
- 오타가 컴파일을 통과하기 때문에, 실패가 런타임까지 숨어 있습니다.
- 새로운 값이 추가되어도 분기 누락이 빌드에서 드러나지 않습니다.

즉, 개발 초반의 편의가 운영 후반의 불확실성으로 바뀌는 구조입니다. 이 지점을 끊어내기 위해 타입화를 하는 것입니다.

---

## 4. 핵심 개념 정의 (혼동 방지)
### 4.1 닫힌 집합(Closed Set)
허용 가능한 값이 명확히 정해져 있고, 임의의 새 값이 들어오면 안 되는 경우를 말합니다.

예시:
- `BYPASS`, `TRANSLATE`
- `HECTO_300`, `KPN_200`

경계:
- 사용자가 자유롭게 입력하는 검색어, 별칭, 라벨은 닫힌 집합이 아닙니다.

### 4.2 경계(Boundary)
외부 세계의 자유로운 입력이 내부 모델로 들어오는 첫 지점입니다.

예시:
- YAML 설정 파싱
- HTTP 요청 본문 바인딩
- TCP 헤더/전문 파싱

### 4.3 Fail-Fast
잘못된 입력을 "늦게"가 아니라 "즉시" 실패시키는 전략입니다.

의도:
- 장애를 조기에 표면화
- 원인 위치를 경계로 고정
- 디버깅 범위 축소

### 4.4 분기 완결성(Exhaustiveness)
가능한 경우의 수를 빠짐없이 처리했는지 컴파일 타임에 검증받는 성질입니다.

실무 의미:
- enum 상수 추가 시 분기 누락이 빌드에서 바로 드러납니다.

---

## 5. 의사결정 프레임워크
아래 질문을 순서대로 보시면 판단이 빨라집니다.

1) 이 값은 닫힌 집합입니까?
- 예: 다음 질문으로 진행
- 아니오: 문자열 유지 + 검증 규칙 강화

2) 이 값이 분기 로직에 영향을 줍니까?
- 예: enum으로 올려야 합니다.
- 아니오: 상수화만으로 충분한지 검토합니다.

3) 잘못된 값이 운영 장애로 이어집니까?
- 예: 경계 fail-fast는 필수입니다.
- 아니오: 경고/보정 정책을 둘 수 있으나, 이유를 문서화해야 합니다.

4) 새 값이 추가될 가능성이 있습니까?
- 예: enum switch 기반으로 누락을 컴파일에 맡기는 것이 유리합니다.

---

## 6. 동작 메커니즘 (입력 -> 처리 -> 산출 -> 소비)
이 절이 가장 중요합니다. 실제로 시스템이 어떻게 안정성을 얻는지 메커니즘으로 설명합니다.

- 입력:
  - 외부 시스템에서 전달된 문자열 값
- 처리(1차):
  - raw 입력 검증(빈 값, 오타, 필수 조합 위반)
- 처리(2차):
  - `String -> enum` 변환
- 산출:
  - 내부 모델의 타입 안전한 값(enum)
- 소비자:
  - 서비스 로직의 `switch` 분기, lookup 레지스트리

이 구조를 지키면 "유효하지 않은 문자열이 내부 핵심 로직으로 들어갈 경로" 자체를 줄일 수 있습니다.

---

## 7. 코드 예제
### 7.1 안 좋은 패턴: 문자열이 내부까지 전파되는 경우
```java
String profile = request.getProfile();

if ("HECTO_300".equals(profile)) {
    return handleHecto();
} else if ("KPN_200".equals(profile)) {
    return handleKpn();
}

return handleUnknown(profile); // 런타임까지 밀림
```

왜 문제인가:
- 오타가 컴파일 단계에서 걸리지 않습니다.
- 신규 profile 추가 시 누락 분기가 조용히 남을 수 있습니다.

### 7.2 권장 패턴: 경계 변환 + 내부 enum 고정
```java
enum InboundProfile {
    HECTO_300,
    KPN_200;

    static InboundProfile from(String raw) {
        if (raw == null || raw.isBlank()) {
            throw new IllegalArgumentException("inbound-profile is required");
        }
        return InboundProfile.valueOf(raw.trim().toUpperCase());
    }
}
```

```java
InboundProfile profile = InboundProfile.from(rawProfile);

switch (profile) {
    case HECTO_300 -> handleHecto();
    case KPN_200 -> handleKpn();
}
```

개선 포인트:
- 경계에서 잘못된 값을 즉시 차단합니다.
- 내부 분기는 컴파일 타임 완결성 혜택을 받습니다.

### 7.3 경계 1차 검증 예시
```java
String mode = readKey(listenerMap, "mode");
String profile = readKey(listenerMap, "inbound-profile");

if ("TRANSLATE".equalsIgnoreCase(mode) && (profile == null || profile.isBlank())) {
    errors.add("inbound-profile is required for TRANSLATE mode");
}
```

이 검증은 "바인더 기본값이 오류를 흡수"하는 상황을 예방하는 안전장치입니다.

## 8. 단점과 트레이드오프
### 8.1 단점
- 초기 설계와 코드 정리가 늘어납니다.
- enum/validator가 늘어나면 코드량이 증가합니다.
- 외부 표준이 매우 자주 바뀌면 변경 빈도가 높아질 수 있습니다.

### 8.2 그럼에도 채택하는 이유
운영비용 관점에서 장점이 큽니다.
- 장애가 늦게 터지는 구조를 줄일 수 있습니다.
- 원인 추적 범위를 경계로 고정할 수 있습니다.
- 신규 인력도 "허용 가능한 값"을 코드 타입에서 즉시 파악할 수 있습니다.

### 8.3 완화 전략
- 닫힌 집합 식별자에만 enum을 적용합니다.
- 열린 집합 데이터는 문자열로 두되 validator 계약을 강화합니다.
- 예외 판단은 반드시 문서에 남겨 팀 규칙을 흔들리지 않게 합니다.

---

## 9. 상황별 선택 예시
| 상황 | 권장 선택 | 이유 |
|---|---|---|
| 모드/프로파일처럼 허용값이 고정된 식별자 | enum + switch | 누락 분기와 오타를 조기 검출하기 쉽습니다. |
| 사용자 자유 입력(검색어, 별칭) | String + validator | 값 집합이 열려 있어 enum 고정이 과도합니다. |
| 외부 표준이 잦게 바뀌는 매핑 값 | registry/map + 계약 테스트 | 코드 배포 주기보다 데이터 변경 주기가 빠를 수 있습니다. |

이 표의 핵심은 "항상 enum"이 아니라, "닫힌 집합이면 enum"입니다.

---

## 10. 흔한 오해
- 오해 1: "enum 쓰면 모든 검증이 자동으로 끝난다"
  - 사실: 외부 입력은 여전히 문자열이므로 경계 검증이 필요합니다.

- 오해 2: "default를 두면 안전하다"
  - 사실: default는 누락 분기를 숨겨서 조기 검출 기회를 줄일 수 있습니다.

- 오해 3: "처음엔 문자열이 빠르니 계속 문자열이 낫다"
  - 사실: 초기 개발 속도와 장기 운영 안정성은 다른 축입니다.

---

## 11. 팀 적용 체크리스트
- 이 식별자는 닫힌 집합입니까?
- 경계에서 잘못된 값을 fail-fast 합니까?
- 내부 API는 enum 타입만 받습니까?
- 분기 누락을 컴파일 단계에서 검출할 수 있습니까?
- 예외 규칙(문자열 유지)은 문서화되어 있습니까?

다섯 항목 중 네 항목 이상이 "예"라면, 방향은 올바르게 잡으신 것입니다.
