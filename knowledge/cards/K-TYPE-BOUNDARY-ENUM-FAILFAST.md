# K-TYPE-BOUNDARY-ENUM-FAILFAST

## 1) 이 카드가 답하는 질문
이 카드가 답하려는 질문은 다음입니다.

"외부에서 문자열로 들어오는 식별자(mode/profile/operation type)를 내부에서도 계속 `String`으로 들고 가면 왜 위험해지고, 어디에서 어떤 타입(enum/값 객체)으로 바꿔야 안전합니까? 그리고 fail-fast 검증은 정확히 어느 지점에서 어떤 형태로 구현해야 합니까?"

여기서 핵심은 `enum` 문법이 아니라, 장애를 언제 발견하고(시점), 어디에서 원인을 고정하고(범위), 어느 정도 비용으로 복구할지(운영 비용)를 설계로 통제하는 것입니다.

---

## 2) 배경: 왜 이 문제가 실무에서 비싸게 터지는가
문자열 분기는 처음에는 빠르게 만들 수 있습니다.
하지만 시간이 지나면 같은 의미의 문자열이 여러 파일에 흩어지고, 오타가 컴파일을 통과하기 때문에 실패가 런타임까지 숨어 있게 됩니다.
문제는 그 실패가 "입력이 잘못됐다"라는 형태로 바로 관측되지 않는다는 점입니다.
예를 들어 profile 문자열이 한 글자 다르면, 어떤 곳에서는 `null`이 되고 어떤 곳에서는 default 분기로 흘러가며, 어떤 곳에서는 전혀 다른 핸들러가 선택될 수 있습니다.
이때 관측되는 증상은 NPE, 잘못된 라우팅, 알 수 없는 정책 적용처럼 2차 증상이고, 원인(문자열 값)은 경계(설정/요청 파싱) 근처에 숨어 있어 디버깅 비용이 커집니다.

또 하나의 비용은 "새 값 추가"에서 발생합니다.
mode/profile은 보통 닫힌 집합인데, 문자열 분기에서는 새 값을 추가해도 분기 누락이 빌드에서 강제되지 않습니다.
즉 기능이 확장되었는데도 일부 경로는 과거 로직을 그대로 타고, 그 불일치가 운영에서만 드러날 수 있습니다.
이 비용을 줄이려면, 실패를 더 이른 시점(경계/컴파일)으로 당기는 설계가 필요합니다.

## 3) 핵심 제어점(왜 하필 이 축인가)
이 주제에서 모든 디테일을 한 번에 다루면, 결국 "좋은 말"만 많아지고 실제 설계 선택이 흐려집니다.
그래서 먼저 실패를 가장 직접적으로 설명하는 제어점을 3개로 고정합니다.

이 카드의 핵심 제어점은 세 가지이고, 각각은 서로 다른 실패 비용을 설명하는 "최소 축"입니다.
즉 이 세 축을 먼저 고정해 두면, 뒤에서 용어 정의(닫힌 집합/경계/표준화)와 메커니즘(입력 -> 변환 -> 산출물 -> 소비자), 조건 변경 시 예측이 같은 뼈대 위에서 자연스럽게 연결됩니다.

첫째는 "이 식별자가 닫힌 집합인지 열린 집합인지"를 먼저 판정하는가입니다.
닫힌 집합은 허용 후보가 정해져 있으므로, 새로운 값이 들어오면 "새 기능"이 아니라 "오류"가 되어야 합니다.
이 판정이 틀리면 양쪽 모두 비용이 발생합니다. 열린 집합을 닫힌 집합으로 착각해 enum으로 고정하면 외부가 값을 확장하는 순간 거절이 폭증하고, 반대로 닫힌 집합을 열린 집합처럼 문자열로 방치하면 오타/부분 수정/분기 누락이 런타임까지 흘러가 장애가 됩니다.

둘째는 경계에서 `String -> enum` 변환과 fail-fast를 함께 수행하는가입니다.
변환이 경계가 아니라 내부 여기저기에 흩어지면, 어떤 경로는 표준화를 하고 어떤 경로는 하지 않으며, 어떤 경로는 결측을 허용하는 식으로 동작이 갈라집니다.
그 결과 같은 입력이 어떤 경우에는 통과하고 어떤 경우에는 실패하는 비결정성이 생기고, 이 비결정성은 로그/스택트레이스만으로 원인을 좁히기 어렵게 만들어 운영 비용을 키웁니다.

셋째는 내부 API와 분기 로직을 enum-only + exhaustiveness로 고정하는가입니다.
내부가 다시 `String`을 받는 순간, "검증을 거쳤는가"가 호출자 책임으로 퍼지면서 설계의 일관성이 깨집니다.
또한 `default`가 분기 누락을 숨기면, 새 값 추가가 빌드에서 드러나지 않고 운영에서만 관측될 수 있습니다.
반대로 enum-only API와 `switch`의 완결성을 결합하면, 누락을 컴파일 단계로 끌어올려 "운영에서만 발견되는 비용"을 구조적으로 줄일 수 있습니다.

왜 하필 이 세 가지를 먼저 보느냐는 질문에 대한 답은, 실무에서 관측되는 실패가 대체로 이 세 축으로 설명되기 때문입니다.
첫째는 "값 후보가 닫혀 있는데도 문자열로 흩어져서 오타/누락이 난다"(닫힌 집합 판정 실패),
둘째는 "잘못된 입력이 내부 깊은 곳까지 들어가 간헐적으로 터진다"(경계 fail-fast 부재),
셋째는 "새 값이 추가됐는데 일부 분기가 누락된다"(exhaustiveness 부재)입니다.
이 세 축은 모두 코드 레벨에서 즉시 실험 가능하고(파싱/스위치/예외), 관측 지점을 고정할 수 있어 학습의 출발점으로 적합합니다.

## 4) 기본 결론(기본값)과 선택 이유
기본 결론은 다음처럼 한 묶음으로 닫습니다.

닫힌 집합 식별자는 외부 입력으로 들어오는 순간 경계에서 `String -> enum`으로 변환하고, 잘못된 값은 즉시 실패(fail-fast)시켜야 합니다.
변환 이후 내부 로직은 `String`을 받지 말고 enum만 받도록 API를 고정해야 합니다.
그리고 분기 로직은 enum `switch`로 작성하되, 가능하면 `default`를 두지 않아 누락 분기가 빌드 단계에서 드러나게 만들어야 합니다.

이 선택이 정당화되는 이유는 "타입이 멋져서"가 아니라, 실패 비용을 옮기기 때문입니다.
문자열 오타/누락은 런타임에서만 터지고 원인과 관측이 멀어지지만, 경계 변환과 enum-only API는 "유효하지 않은 값이 내부 핵심 로직으로 들어갈 통로"를 줄입니다.
또한 exhaustiveness는 새 값 추가 시 분기 누락을 컴파일 단계로 끌어올려, 운영에서 발견되는 비용을 크게 낮춥니다.

## 5) 용어 정의 (이 카드 기준)
### 5.1 닫힌 집합(Closed Set)
허용 가능한 값 후보가 미리 정해져 있고, 그 외의 값이 들어오면 "새 기능"이 아니라 "오류"로 취급해야 하는 식별자입니다.
이 카드에서 닫힌 집합에는 `mode`, `profile`, `operation type`, 프로토콜 메시지 타입 같은 분기 키가 포함됩니다.

반대로 사용자 입력 문자열, 임의의 외부 ID처럼 값 후보가 열려 있는 데이터는 닫힌 집합에서 제외합니다.
닫힌 집합의 본질은 "새 값이 추가되면 코드도 함께 바뀌어야 한다"는 점이므로, 코드를 바꾸지 않았는데도 새 값이 조용히 들어와 통과하는 것은 안전하지 않습니다.

### 5.2 경계(Boundary)
외부 세계의 입력이 내부 도메인 모델로 들어오는 첫 지점입니다.
설정 파일(YAML/JSON)을 읽어 객체로 만드는 지점, HTTP 요청 본문을 DTO로 바인딩하는 지점, TCP 헤더/전문을 파싱하는 지점이 모두 경계입니다.

### 5.3 표준화(normalization, canonicalization)
외부 입력의 표현 차이(공백/대소문자 등)를 하나의 대표 형태로 맞추는 과정입니다.
표준화는 "편의"가 아니라 "계약"입니다.
어떤 표현 차이를 허용할지(예: `trim()`만 할지, `toUpperCase()`까지 할지)를 팀이 고정하지 않으면, 어떤 입력은 통과하고 어떤 입력은 실패하는 애매한 상태가 됩니다.

### 5.4 Fail-Fast
잘못된 입력을 "나중"이 아니라 "지금"(경계)에서 실패시키는 전략입니다.
fail-fast의 목적은 엄격함이 아니라, 장애 원인 위치를 경계로 고정해 디버깅 범위를 줄이는 것입니다.

### 5.5 분기 완결성(Exhaustiveness)
가능한 모든 경우의 수를 빠짐없이 처리했는지 컴파일 타임에 검증받는 성질입니다.
이 카드에서 분기 완결성은 "새 enum 상수가 추가되었을 때 분기 누락이 빌드에서 바로 드러나는가"라는 운영 의미로 사용합니다.

## 6) 메커니즘(입력 -> 변환 규칙 -> 산출물 -> 소비자)
이 절은 "enum이 왜 운영 안정성으로 이어지는지"를 실제 데이터 흐름으로 닫는 부분입니다.

입력은 외부에서 들어오는 raw 문자열입니다. 예를 들어 `profile="KPN_200"` 같은 값은 처음에는 단지 텍스트일 뿐이고, 그 텍스트가 유효한지 여부는 아직 확정되지 않았습니다.

경계에서의 변환 규칙은 보통 세 단계가 한 묶음으로 움직입니다.
먼저 `null/blank` 같은 결측을 잡고, 다음으로 표현 차이를 허용할지 정책을 고정한 뒤(예: 공백 허용 여부, 대소문자 허용 여부), 허용한다면 표준화를 적용합니다.
마지막으로 표준화된 값을 enum으로 변환합니다. 변환이 실패하면 그 시점에서 "입력이 잘못됐다"가 확정되므로, 내부 로직으로 들어가기 전에 실패를 고정할 수 있습니다.

산출물은 enum입니다.
이때 중요한 설계는 내부 API가 enum만 받도록 고정하는 것입니다. 내부가 다시 `String`을 받기 시작하면, 어떤 호출자는 검증을 거치고 어떤 호출자는 검증을 건너뛰는 상태가 되어 fail-fast의 효과가 사라집니다.

소비자는 내부 분기 로직입니다.
분기 로직을 enum `switch`로 고정하면, 새 값 추가 시 누락 분기를 빌드에서 드러내는 형태(특히 switch expression, default 없음)로 설계할 수 있습니다.

## 7) 조건 변경 시 예측(무엇이 깨지고 어떻게 관측되는가)
이 절은 위에서 고정한 제어점을 일부러 흔들어 보면서, 무엇이 깨지고 어떤 증상으로 관측되는지 설명합니다.

### 7.1 닫힌 집합 판정을 잘못하면 무엇이 깨지나
닫힌 집합/열린 집합 판정이 틀리면, 실패는 단순히 "더 자주"가 아니라 "다른 형태"로 나타납니다.
왜냐하면 닫힌 집합에서는 "알 수 없는 값"이 곧 오류여야 하고, 열린 집합에서는 "알 수 없는 값"이더라도 검증 규칙을 통과하는 한 데이터로 취급될 수 있기 때문입니다.

열린 집합을 닫힌 집합으로 착각해 enum으로 고정하면, 외부가 새 값을 보내는 순간 경계에서 즉시 거절이 발생합니다.
보통은 `Enum.valueOf`에서 `IllegalArgumentException`이 터지면서 `"No enum constant ..."` 같은 메시지가 쌓이고, 에러율이 입력 분포 변화와 1:1로 연결됩니다.
이 상황에서 중요한 관측 신호는 "거절이 늘었다"가 아니라, 거절이 어느 값에서 터지는지까지 고정되어 있는가입니다.
예를 들어 `"unknown inbound profile: <raw>"`처럼 원인 값을 포함한 메시지나, `unknown_profile_total` 같은 카운터가 있으면 외부 계약 변화(새 값 유입)를 조기에 확정할 수 있습니다.

반대로 닫힌 집합을 열린 집합처럼 취급해 문자열로 방치하면, 새 값 추가 시 분기 누락이 빌드에서 강제되지 않습니다.
즉 코드는 배포되지만 일부 경로는 과거 분기표를 그대로 타고, 결과는 운영에서 "특정 모드/프로파일에서만 이상하게 동작한다"는 편차로 관측됩니다.
이때 관측 신호는 실패가 NPE 같은 2차 증상으로만 보이는지, 아니면 `"unhandled profile: <raw>"`처럼 분기 키 자체를 경계나 라우터에서 직접 드러내는지입니다.
후자가 없으면, 원인은 문자열 하나인데도 장애 복원 범위는 시스템 전체로 퍼집니다.

### 7.2 경계에서 변환하지 않고 내부에서 String 비교를 시작하면 무엇이 깨지나
경계에서 변환하지 않으면, 내부 여기저기에서 문자열 비교가 시작되고 검증이 분산됩니다.
이 상태에서는 어떤 경로는 표준화를 하고, 어떤 경로는 하지 않으며, 어떤 경로는 결측을 허용하는 식으로 동작이 갈라집니다.
따라서 관측 포인트는 같은 입력이 어떤 경로에서는 통과하고 어떤 경로에서는 실패하는 비결정성입니다. 예를 들어 `raw="kpn_200 "` 같은 입력이 어떤 API에서는 통과하고 어떤 API에서는 거절된다면, 표준화/검증이 한 곳(경계)에 모여 있지 않다는 신호입니다.

또한 실패가 경계가 아니라 내부 깊은 곳에서만 관측되므로, 스택트레이스/로그가 원인을 직접 가리키지 못하게 됩니다.
따라서 경계에서 fail-fast로 원인을 고정하는 것이 디버깅 비용을 줄입니다.

### 7.3 표준화 정책을 합의하지 않으면 무엇이 깨지나
표준화는 "대충" 하면 안 됩니다.
예를 들어 `toUpperCase()`를 적용하면 대소문자 차이를 허용하는 대신, 대소문자를 의미로 사용하는 프로토콜에서는 잘못된 동작이 됩니다.
반대로 표준화를 하지 않으면 외부가 소문자를 보내는 순간 `valueOf("bypass")` 같은 입력이 실패합니다.
관측 포인트는 특정 케이스에서만 발생하는 `IllegalArgumentException`과, "왜 이 값이 거절되는지"에 대한 팀 내부 해석 불일치입니다.
즉 한 팀은 `"bypass"`가 당연히 `"BYPASS"`로 취급될 거라고 가정하고, 다른 팀은 "대소문자도 계약의 일부"라고 가정하는 순간, 동일 입력이 환경/경로에 따라 다르게 처리될 수 있습니다.
이 불일치는 테스트로도 드러나지만, 운영에서는 "특정 파트너/특정 시간대에만" 거절이 늘어나는 형태로 나타나기 쉬워서 더 비쌉니다.

### 7.4 default가 분기 누락을 숨기면 무엇이 깨지나
`default`는 당장 코드를 간단하게 만들지만, 새 enum 상수가 추가될 때 누락 분기를 숨길 수 있습니다.
즉 빌드는 통과하고, 새 값은 default로 흘러가며, 그 결과는 운영에서만 관측될 수 있습니다.
따라서 관측은 "default가 실행됐다"를 조용히 삼키지 않고, 그 사실이 명시적으로 남는지로 고정해야 합니다.
예를 들어 default에서 `"unhandled mode: <enumName>"`을 로그로 남기거나, `unhandled_mode_total` 같은 카운터를 올리면, 새 값 추가가 분기표를 깨뜨렸음을 운영에서 즉시 확정할 수 있습니다.
다만 이 관측은 최후의 안전망일 뿐이고, 기본값은 default를 두지 않아 누락을 빌드 단계에서 드러내는 것입니다.

## 8) 코드 예시(나쁜 패턴 1 + 권장 패턴 1 이상)
### 8.1 내부까지 String이 전파되는 나쁜 패턴
```java
class BadRouter {
    void route(String rawProfile) {
        if ("HECTO_300".equals(rawProfile)) {
            handleHecto();
            return;
        }
        if ("KPN_200".equals(rawProfile)) {
            handleKpn();
            return;
        }

        // 오타/누락은 컴파일에서 잡히지 않습니다.
        // rawProfile의 원인 위치(경계)가 멀어질수록, 장애 추적 비용이 급격히 증가합니다.
        handleUnknown(rawProfile);
    }

    void handleHecto() {}
    void handleKpn() {}
    void handleUnknown(String raw) {}
}
```

### 8.2 경계에서 enum으로 변환하고 내부를 enum-only로 고정하는 권장 패턴
```java
enum InboundProfile {
    HECTO_300,
    KPN_200;

    static InboundProfile fromExternal(String raw) {
        if (raw == null || raw.isBlank()) {
            throw new IllegalArgumentException("inbound-profile is required");
        }

        String normalized = raw.trim().toUpperCase();
        return InboundProfile.valueOf(normalized);
    }
}

class GoodRouter {
    void route(String rawProfile) {
        InboundProfile profile = InboundProfile.fromExternal(rawProfile);

        switch (profile) {
            case HECTO_300 -> handleHecto();
            case KPN_200 -> handleKpn();
        }
    }

    void handleHecto() {}
    void handleKpn() {}
}
```

## 9) 최소 실험(검증)
아래 실험은 프로젝트에 종속되지 않고, 자바 코드만으로 확인할 수 있는 수준으로 고정합니다.

### 9.1 switch expression이 누락 분기를 컴파일 단계에서 막는지 확인
아래 코드는 일부러 `TRANSLATE` 분기를 빼 둔 예시입니다.

```java
enum Mode { BYPASS, TRANSLATE }

class ExhaustiveLab {
    static int handle(Mode mode) {
        return switch (mode) {
            case BYPASS -> 0;
            // case TRANSLATE -> 1;  // 일부러 누락
        };
    }
}
```

기대 결과는 "컴파일 실패"입니다.
즉 새 enum 상수가 추가되거나(case가 누락되면) 빌드 단계에서 바로 드러나게 만들 수 있습니다.

### 9.2 default가 누락을 숨기는지 확인
아래 코드는 일부러 `TRANSLATE`를 처리하지 않되, `default`를 둔 예시입니다.

```java
enum Mode { BYPASS, TRANSLATE }

class DefaultHidesLab {
    static int handle(Mode mode) {
        return switch (mode) {
            case BYPASS -> 0;
            default -> -1;
        };
    }
}
```

기대 결과는 "컴파일 성공"입니다.
하지만 `TRANSLATE`는 `default`로 흘러 `-1`이 되고, 이후 enum 상수가 더 추가되어도(예: `NEW`) 동일하게 `default`로 흘러갈 수 있습니다.
즉 `default`는 "컴파일 타임에 누락을 강제하는 힘"을 약하게 만들 수 있으므로, 누락을 빌드에서 잡고 싶다면 기본값으로 두지 않는 편이 안전합니다.

### 9.3 Enum.valueOf가 어떤 입력에서 어떻게 실패하는지 확인
```java
enum Mode { BYPASS, TRANSLATE }

public class ValueOfLab {
    public static void main(String[] args) {
        try {
            System.out.println(Mode.valueOf("BYPASS"));
        } catch (Exception e) {
            System.out.println("unexpected: " + e);
        }

        try {
            System.out.println(Mode.valueOf("bypass"));
        } catch (Exception e) {
            System.out.println(e.getClass().getSimpleName());
            // 기대 결과: IllegalArgumentException
        }

        try {
            System.out.println(Mode.valueOf(null));
        } catch (Exception e) {
            System.out.println(e.getClass().getSimpleName());
            // 기대 결과: NullPointerException
        }
    }
}
```

이 실험은 "왜 경계에서 결측/표준화를 먼저 해야 하는지"를 메커니즘으로 확인하게 해 줍니다.

## 10) 트레이드오프/대안/Variants
fail-fast와 enum-only는 강한 기본값이지만, 외부 환경 조건에 따라 다른 선택이 필요할 수 있습니다.

외부가 예고 없이 새 값을 보내는 환경에서 fail-fast는 거절 폭증으로 보일 수 있습니다.
이 경우에는 enum에 `UNKNOWN`을 두거나, 파싱 결과를 별도 타입으로 분리해 fail-soft로 전환하는 선택지가 있습니다.
다만 이 선택은 문제를 숨기기 쉬우므로, `UNKNOWN` 발생을 반드시 로그/메트릭으로 집계해 "외부 계약 변화"를 조기에 관측해야 합니다.

값 후보는 닫혀 있지만 변경 주기가 코드 배포 주기보다 빠르면 enum만으로 운영이 불가능할 수 있습니다.
이 경우는 레지스트리/설정 기반 매핑 + 계약 테스트로 통제하는 방향을 고려해야 합니다.
즉 "코드로 닫힌 집합을 고정"하는 대신, "데이터로 닫힌 집합을 관리"하되 테스트로 드리프트를 잡는 방식입니다.

## 11) 흔한 오해
"enum을 쓰면 검증이 자동으로 끝난다"는 오해가 대표적입니다.
외부 입력은 여전히 문자열이므로, enum은 내부 모델의 안전성을 높이지만 경계 검증은 별개로 필요합니다.

"default를 두면 안전하다"는 오해도 자주 나옵니다.
default는 새 값 추가/분기 누락을 숨길 수 있습니다. 누락을 빌드에서 잡고 싶다면 default를 두지 않는 방향(특히 switch expression)이 더 안전합니다.

"fail-fast는 항상 예외를 던진다"는 오해도 있습니다.
fail-fast의 본질은 경계에서 실패시키는 것이고, 예외를 던질 수도 있으며 검증 에러를 모아 요청을 거절하는 방식으로도 구현할 수 있습니다.

## 12) 팀 적용 질문(선택)
이 패턴은 "enum을 쓴다" 하나로 끝나지 않고, 경계 변환과 내부 API 고정, 분기 방식이 함께 맞물려야 효과가 납니다.
따라서 팀 적용 시에는 아래 질문을 같은 순서로 확인하시는 편이 좋습니다. 앞 질문이 닫히지 않으면, 뒤의 질문은 사실상 의미가 없어지기 때문입니다.

- `이 식별자는 닫힌 집합입니까(허용 후보가 정해져 있습니까)?`: 닫힌 집합이 아니면 enum 고정이 과도해지고, 대신 validator 계약이 핵심이 됩니다.
- `외부 입력은 경계에서 String -> enum으로 변환됩니까?`: 내부 어디선가 문자열 비교가 시작되면, 검증이 분산되고 누락 경로가 생깁니다.
- `잘못된 값은 내부 로직으로 들어가기 전에 실패합니까(fail-fast)?`: 장애 원인 위치를 경계로 고정해야 디버깅 범위가 줄어듭니다.
- `내부 핵심 로직은 enum-only로 고정되어 있습니까?`: 내부가 `String`을 받는 순간, 검증 완료 여부가 호출자에게 달라붙어 재현성이 떨어집니다.
- `분기 누락을 컴파일 단계에서 드러내는 구조(switch expression 등)입니까?`: 누락 분기를 운영까지 보내지 않기 위해 빌드에서 깨지도록 설계해야 합니다.
