# K-STRING-CONSTANTS-SCOPE-AND-SQL

## 1. 이 문서의 목적
이 문서는 "문자열 상수화를 어디까지 해야 하는가"를 실무적으로 설명합니다.

실무에서는 두 극단이 반복됩니다.
- 극단 A: 문자열을 거의 그대로 둔다 -> 오타/수정 누락 증가
- 극단 B: 모든 문자열을 상수화한다 -> 읽기 흐름 단절, 인지 부하 증가

이 문서의 목표는 두 극단을 피하고, 팀이 같은 기준으로 판단할 수 있도록 기준선을 제공하는 것입니다.

---

## 2. 문제 맥락
문자열은 개발 속도가 빠르지만, 시간이 지나면 드리프트(같은 의미의 서로 다른 표현) 문제가 생깁니다.

예를 들어 아래 세 줄은 사람이 보기엔 비슷하지만, 시스템 관점에서는 서로 다른 키입니다.
- `inbound-profile`
- `inboundProfile`
- `inbound_profile`

이 차이는 설정 파싱/검증/분기 로직에서 장애로 연결됩니다.

반대로, SQL 본문까지 과하게 상수화하면 읽을 때 문맥이 끊겨 의도를 파악하기 어려워집니다. 쿼리는 "한 덩어리 문장"으로 읽을 때 정확히 이해되는 경우가 많기 때문입니다.

따라서 핵심은 "상수화를 많이/적게"가 아니라, "무엇을 의미 단위로 통제할 것인가"입니다.

---

## 3. 기본 의사결정
### 3.1 결정
- 같은 의미를 여러 위치에서 공유하는 문자열은 상수 또는 enum으로 승격합니다.
- 한 위치에서 문맥적으로 읽어야 하는 표현 문자열은 인라인 유지를 기본으로 합니다.
- SQL은 아래처럼 구분합니다.
  - 정적 본문: 읽기 흐름 우선
  - 동적 조합 키/파라미터명: 상수로 통제

### 3.2 왜 이 결정이 정당한가
이 결정은 "변경 안정성"과 "읽기 효율"의 균형을 맞춥니다.
- 변경 안정성: 의미 키를 한 곳에서 관리해 수정 누락을 줄입니다.
- 읽기 효율: SQL/메시지 본문은 현장에서 바로 읽히게 유지합니다.

즉, "찾기 쉬움"과 "읽기 쉬움"을 동시에 확보하는 전략입니다.

---

## 4. 결정 프레임워크 (순서대로 판단)
1) 이 문자열이 도메인 식별자인가요?
- 예: `mode`, `profile`, `operation id`, 정책 키
- 판단: 상수/enum 우선

2) 이 문자열이 표현 문장인가요?
- 예: SQL 본문, 사용자에게 보여줄 텍스트
- 판단: 인라인 유지 우선

3) 이 문자열이 보안/정합성 경계에 있나요?
- 예: SQL 파라미터 키, 동적 조합 토큰
- 판단: 상수화 + 바인딩/검증 필수

4) 재사용 횟수가 늘어날 가능성이 높나요?
- 판단: 현재 1회라도 승격을 고려합니다.

## 5. 코드 예제
### 5.1 안 좋은 패턴: 의미 키를 인라인으로 반복
```java
if ("TRANSLATE".equals(mode)) {
    String profile = readValue(map, "inbound-profile");
    if (profile == null || profile.isBlank()) {
        errors.add("inbound-profile is required");
    }
}
```

문제:
- 오타와 표기 드리프트에 취약합니다.
- 변경 시 영향 범위 파악이 어렵습니다.

### 5.2 권장 패턴: 의미 키 승격
```java
private static final String MODE_TRANSLATE = "TRANSLATE";
private static final String FIELD_INBOUND_PROFILE = "inbound-profile";

if (MODE_TRANSLATE.equalsIgnoreCase(mode)) {
    String profile = readValue(map, FIELD_INBOUND_PROFILE);
    if (profile == null || profile.isBlank()) {
        errors.add(FIELD_INBOUND_PROFILE + " is required");
    }
}
```

개선 포인트:
- 의미 단위를 코드 구조로 드러냅니다.
- 수정/검색/리뷰 비용을 줄입니다.

### 5.3 SQL 균형 패턴: 본문은 읽기 좋게, 주입은 안전하게
```java
public static final String SELECT_RECENT = """
    SELECT transaction_id, status, result_code
    FROM bridge_transactions
    WHERE operation = :operation
      AND idempotency_key = :idempotency_key
    ORDER BY created_at DESC
    LIMIT 1
    """;
```

```java
Map<String, Object> params = Map.of(
    "operation", operation,
    "idempotency_key", idempotencyKey
);
```

핵심:
- 쿼리 본문은 사람이 한 번에 이해할 수 있게 유지합니다.
- 값 주입은 문자열 결합이 아니라 바인딩으로 통제합니다.

## 6. 단점과 트레이드오프
### 6.1 단점
- 기준이 없으면 팀원마다 상수화 범위가 달라져 일관성이 깨집니다.
- 상수가 너무 많아지면 "정의 파일 왕복"이 잦아집니다.
- 동적 SQL이 늘면 복잡도가 빠르게 증가합니다.

### 6.2 그럼에도 채택하는 이유
이 기준을 적용하면 실수의 종류가 줄어듭니다.
- 키 오타/드리프트 같은 반복 실수가 감소합니다.
- SQL은 읽기 쉬운 형태를 유지해 코드 리뷰 속도가 올라갑니다.
- 신규 구성원도 "이 문자열이 정책 키인지, 단순 표현인지"를 빠르게 구분할 수 있습니다.

### 6.3 완화 전략
- 상수 승격 기준을 명시합니다. (예: 의미 동일 + 재사용 가능성)
- SQL은 바인딩을 기본 원칙으로 고정합니다.
- 리뷰 시 "읽기 흐름 손상" 여부를 반드시 점검합니다.

## 7. 상황별 선택 예시
| 상황 | 권장 선택 | 이유 |
|---|---|---|
| 분기/검증 키 | 상수 또는 enum | 같은 의미를 여러 곳에서 공유하므로 통제가 필요합니다. |
| 단일 문맥 설명 문자열 | 인라인 유지 | 상수화 이득보다 읽기 손실이 큽니다. |
| 정적 SQL 본문 | 본문 유지 | 한 번에 읽는 것이 정확도와 속도에 유리합니다. |
| 동적 SQL 조합이 많은 영역 | DSL/조립 규칙 강화 | 문자열 결합 실수와 보안 위험을 줄여야 합니다. |

이 표는 "표현 방식 취향"이 아니라 "실패 비용 최적화" 기준입니다.

## 8. 흔한 오해
- 오해 1: "재사용 2회 이상이면 무조건 상수화"
  - 사실: 의미가 다르면 상수화하면 안 됩니다. "같은 의미"가 먼저입니다.

- 오해 2: "SQL도 다 상수로 쪼개면 안전하다"
  - 사실: 안전성 핵심은 바인딩입니다. 과도한 분해는 가독성만 해칩니다.

- 오해 3: "상수화는 리팩터링 마지막에 해도 된다"
  - 사실: 분기 키/정책 키는 초기에 잡아야 드리프트 누적을 막을 수 있습니다.

## 9. 팀 적용 체크리스트
- 이 문자열은 "의미 단위"입니까, "표현 단위"입니까?
- 의미 단위라면 한 곳에서 관리되고 있습니까?
- SQL 값 주입이 바인딩으로 통제되고 있습니까?
- 상수화 후에도 코드 읽기 흐름이 유지됩니까?
- 팀원이 같은 결론을 낼 수 있도록 기준이 문서화되어 있습니까?

다섯 항목 중 네 항목 이상이 "예"이면, 상수화 범위를 건강하게 유지하고 계신 상태입니다.
