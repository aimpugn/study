# K-STRING-CONSTANTS-SCOPE-AND-SQL

## 1. 이 카드가 답하는 질문
실무에서 아래 질문은 계속 다시 등장합니다.

"코드에 흩어진 문자열 리터럴을 어디까지 상수/enum으로 올려야 하고, 어디부터는 인라인으로 두는 것이 더 낫습니까? 특히 SQL 문자열은 가독성을 해치지 않으면서도 안전하게 유지하려면 어떻게 해야 합니까?"

이 질문이 어려운 이유는, 문자열을 상수화하면 오타/수정 누락은 줄지만 읽기 흐름이 끊길 수 있고, 반대로 인라인을 유지하면 처음엔 빠르지만 시간이 지날수록 드리프트(같은 의미의 서로 다른 표기)와 부분 수정(한 군데만 바꾸고 다른 곳은 놓침)이 누적되기 때문입니다.
이 카드는 이 균형을 "취향"이 아니라, 실패 비용을 어디에서 줄일지라는 관점에서 설명합니다.

---

## 2. 적용 범위와 비범위
이 카드의 결론은 문자열이 코드에서 "식별자(identifier)"로 소비되는 경우에 바로 적용됩니다.
여기서 식별자란, 그 문자열 자체가 분기/검증/매핑의 기준이 되는 것을 뜻합니다. 예를 들어 설정 키(YAML/JSON), 프로토콜 필드명(TCP 헤더/전문 키), 정책 키/모드/프로파일, SQL의 named parameter 이름, 그리고 동적으로 조립되는 `ORDER BY` 컬럼 같은 것들이 모두 식별자입니다.
이런 문자열은 한 글자만 달라도 프로그램이 다른 키로 취급하기 때문에, 오타나 표기 변화가 런타임 장애로 직결되기 쉽습니다.

반대로 이 카드의 기본 결론을 그대로 적용하면 오히려 읽기/유지보수성이 나빠질 수 있는 경우도 있습니다.
대표적으로 사용자에게 보여주는 문구나, 한 블록에서 문장으로 읽히는 것이 중요한 정적 SQL 본문은 "표현(expression)"이 목적입니다.
이런 문자열까지 무리하게 상수로 쪼개면, 읽는 사람이 문맥을 머릿속에서 다시 조립해야 하므로 리뷰/디버깅 속도가 떨어질 수 있습니다.

## 3. 기본 결론(기본값)과 선택 이유
이 카드의 기본 결론은 하나의 문장으로 요약할 수 있습니다.

문자열을 먼저 "의미 단위(식별자)"와 "표현 단위(문장)"로 나눈 뒤, 의미 단위는 상수/enum으로 통제하고 표현 단위는 읽기 흐름을 우선해 인라인으로 둡니다.

이 결론이 정당화되는 이유는 다음 메커니즘으로 닫힙니다.
의미 단위 문자열은 컴파일러가 오타를 잡아주지 않기 때문에, 실패가 런타임까지 숨어 있다가 더 깊은 곳에서 `null`, 분기 누락, 잘못된 라우팅 같은 형태로 터집니다.
반면 표현 단위 문자열(특히 정적 SQL 본문)은 사람이 한 덩어리로 읽을 때 이해 속도가 가장 빠르므로, 과도한 상수화는 안전성보다 인지 비용을 더 크게 올릴 수 있습니다.

SQL은 여기서 한 번 더 분해가 필요합니다.
SQL에서 "값(value)"은 반드시 바인딩(PreparedStatement 바인딩 또는 named parameter 바인딩)으로 주입해야 하며, 문자열 결합으로 값을 끼워 넣으면 입력이 SQL 문법으로 해석될 수 있어 SQL injection과 같은 취약점이 생깁니다.
하지만 SQL에서 동적으로 바뀌는 것이 "값"이 아니라 "식별자"(정렬 컬럼/테이블명/컬럼명)라면 바인딩으로 해결되지 않습니다. 이때는 허용 가능한 후보를 enum(allow-list)으로 고정해 두고 그 enum만으로 SQL 조립을 허용하는 것이 안전합니다.

## 4. 용어 정의 (이 카드 기준)
이 카드에서 쓰는 용어는 아래 의미로 고정합니다.

### 4.1 문자열 리터럴(String literal)
코드에 직접 박혀 있는 문자열 값입니다. 예: `"inbound-profile"`, `"SELECT ..."`.

### 4.2 의미 단위 문자열(semantic identifier string)
프로그램이 "식별자"로 해석하는 문자열입니다. 이 문자열 자체가 분기/검증/매핑의 기준이 됩니다.
이 카드에서 의미 단위 문자열에는 설정 키, 프로토콜 필드명, 모드/프로파일, 정책 키, SQL named parameter 이름, 동적 SQL 식별자(컬럼/정렬 키)가 포함됩니다.
반대로 한 블록 안에서 문장으로 읽히는 SQL 본문이나 사용자 안내 문구 같은 것은 의미 단위 문자열에서 제외합니다.

의미 단위 문자열이 위험한 이유는 단순히 "오타가 난다"가 아니라, 오타가 컴파일을 통과해 런타임까지 전파되고, 그 결과가 종종 "멀리서"(완전히 다른 위치에서) 관측된다는 점입니다.
예를 들어 설정 키가 한 글자 달라져 `null`이 되면, 그 `null`은 훨씬 뒤의 분기나 객체 생성 단계에서 NPE로 증폭될 수 있습니다.

### 4.3 표현 단위 문자열(human-readable expression string)
사람이 한 번에 읽어야 의미가 살아나는 문자열입니다.
정적 SQL 본문이 대표적입니다. SQL을 조각 상수로 과분해하면 "한 문장"이 깨져서, 읽는 사람이 머릿속에서 다시 조립해야 합니다.

### 4.4 드리프트(drift)
"같은 의미"가 시간이 지나며 "서로 다른 표기"로 분화되는 현상입니다.
예를 들어 `inbound-profile`, `inboundProfile`, `inbound_profile`는 사람의 눈에는 비슷하지만 프로그램 관점에서는 서로 다른 키입니다.

### 4.5 바인딩(binding)
SQL 문자열에 값을 직접 붙이지 않고, 드라이버/프레임워크가 제공하는 파라미터 자리에 값을 안전하게 전달하는 방식입니다.
바인딩의 핵심은 값이 SQL 문법의 일부로 해석되지 않고 데이터로 취급되도록 "경계"를 유지하는 것입니다.
따라서 바인딩은 값(value)에만 적용할 수 있고, 컬럼명/정렬키 같은 식별자(identifier)에는 적용할 수 없습니다.

## 5. 메커니즘: 상수/enum 승격이 실제로 줄이는 오류의 종류
"상수로 바꾸면 오타가 줄어듭니다"는 결과만 말하는 설명이라, 실무 의사결정에 도움이 되지 않습니다.
상수/enum 승격이 실제로 어떤 실패를 어떻게 줄이는지, 그리고 무엇은 줄이지 못하는지를 데이터 흐름으로 닫아 보겠습니다.

먼저 상수화는 의미 단위 문자열을 "한 곳에서" 관리하도록 강제합니다.
입력은 여러 곳에 흩어진 동일 의미의 문자열 리터럴이고, 변환 규칙은 그 문자열에 이름을 붙여 한 곳에 모으는 것입니다. 산출물은 "동일 의미는 동일 상수"라는 구조이고, 소비자는 설정 읽기/검증, 프로토콜 파싱, 분기 로직, 에러 메시지 조립입니다.
이 구조에서 줄어드는 실패는 주로 부분 수정(partial update)과 표기 드리프트입니다. 즉, 키 이름을 바꾸어야 할 때 A 파일만 고치고 B 파일은 놓치는 문제, 그리고 `inbound-profile`과 `inboundProfile` 같은 표기 차이가 서로 다른 키로 동작해 장애가 되는 문제가 줄어듭니다.

다만 상수화만으로는 외부 입력 자체가 잘못된 경우를 해결하지 못합니다.
외부에서 `inboundProfile`이라는 키로 값이 들어오면, 내부 상수(`INBOUND_PROFILE`)는 그 입력을 "자동으로" 바꿔주지 않습니다.
이 경우에는 경계에서 "허용 키 집합"을 검증하거나 필수 키가 존재하는지 확인하는 fail-fast 검증이 필요합니다.

enum은 상수화보다 한 단계 더 강한 통제입니다.
의미 단위 문자열 중에서도 값 후보가 정해진 닫힌 집합(예: mode/profile/operation type)은, 상수로만 두면 "새 값 추가"가 분기 누락으로 이어질 수 있습니다.
반면 enum으로 올리고 enum `switch`로 분기하면, 새 값 추가 시 누락 분기를 컴파일 단계에서 더 쉽게 드러낼 수 있습니다.

## 6. 의사결정: 무엇을 상수/enum으로 올리고 무엇을 인라인으로 둘 것인가
판단은 단순히 "몇 번 쓰였는가"가 아니라, 그 문자열이 시스템에서 어떤 역할을 맡는가로 시작해야 합니다.

먼저 이 문자열이 프로그램에서 식별자로 쓰이는지, 사람이 읽기 위한 문장(표현)인지부터 구분합니다.
식별자라면 의미 단위 문자열이므로 상수/enum 승격을 기본값으로 검토하고, 문장이라면 인라인 유지가 기본값입니다.

여기서 식별자라고 해서 모두 enum은 아닙니다.
값 후보가 닫힌 집합이면 enum이 유리하지만, 값 후보가 열린 집합(사용자 자유 입력, 임의 ID 등)이면 enum 고정은 오히려 변경을 불가능하게 만들기 때문에 문자열을 유지하고 validator 계약을 강화해야 합니다.

SQL은 반드시 "값"과 "식별자"를 분리해서 봐야 합니다.
값은 바인딩으로만 주입하고(문자열 결합 금지), 식별자는 바인딩이 불가능하므로 allow-list(enum)로만 조립되도록 API 형태 자체를 제한해야 합니다.

## 7. 코드 패턴 예시
아래 예시는 특정 프로젝트 파일에 의존하지 않도록, 카드 안에서 학습 가능한 최소 단위로 구성했습니다.

### 7.1 안 좋은 패턴: 의미 키를 인라인으로 반복(드리프트와 부분 수정을 부릅니다)
```java
import java.util.Map;

class BadConfigReader {
    String readProfile(Map<String, String> listener) {
        // 같은 의미의 키가 서로 다른 표기로 흩어지면, 한쪽은 항상 null이 됩니다.
        String profile = listener.get("inbound-profile");
        if (profile == null) {
            profile = listener.get("inboundProfile"); // 드리프트: 비슷해 보이지만 완전히 다른 키
        }
        return profile;
    }
}
```

이 코드는 "지금은 우연히 동작"할 수 있다는 점이 더 위험합니다.
설정/입력 데이터가 어떤 표기를 쓰느냐에 따라 결과가 바뀌고, 장애는 간헐적이 되며, 원인 위치(키 표기)가 관측 위치(NPE 등)와 멀어지기 쉽습니다.

### 7.2 권장 패턴: 의미 단위 문자열을 상수로 승격하고, 경계에서 검증합니다
```java
import java.util.Map;

final class ListenerKeys {
    static final String INBOUND_PROFILE = "inbound-profile";

    private ListenerKeys() {}
}

class GoodConfigReader {
    String readRequired(Map<String, String> map, String key) {
        String value = map.get(key);
        if (value == null || value.isBlank()) {
            // 관측 가능한 실패: 어떤 키가 비었는지 메시지로 고정됩니다.
            throw new IllegalArgumentException("required key is missing: " + key);
        }
        return value;
    }

    String readProfile(Map<String, String> listener) {
        // 키는 상수로 고정하고, 잘못된 입력은 경계에서 즉시 실패시킵니다.
        return readRequired(listener, ListenerKeys.INBOUND_PROFILE);
    }
}
```

이 예시에서 상수화와 검증은 역할이 다릅니다.
상수화는 내부 코드에서 부분 수정/오타/의미 충돌을 줄이고, 검증은 외부 입력이 틀렸을 때 실패를 내부 깊은 곳이 아니라 경계에서 터뜨려 원인 추적 비용을 낮춥니다.

### 7.3 SQL 패턴: 정적 본문은 읽기 좋게 유지하고, 값은 바인딩으로만 주입합니다
```java
String sql = """
    SELECT transaction_id, status, result_code
    FROM bridge_transactions
    WHERE operation = :operation
      AND idempotency_key = :idempotency_key
    ORDER BY created_at DESC
    LIMIT 1
    """;

// named parameter 바인딩(예: NamedParameterJdbcTemplate)의 입력 형태를 가정한 예시입니다.
Map<String, Object> params = Map.of(
    "operation", operation,
    "idempotency_key", idempotencyKey
);
```

여기서 안전성의 핵심은 SQL 본문을 상수로 쪼갰는지가 아니라, `operation`/`idempotencyKey` 같은 값이 문자열 결합으로 SQL 문장에 섞이지 않도록 바인딩 경계를 유지하는지입니다.

### 7.4 (중요) 동적 SQL 식별자: 바인딩이 불가능하므로 allow-list(enum)로만 조립합니다
정렬 컬럼/정렬 방향 같은 식별자는 바인딩할 수 없습니다.
따라서 외부 입력을 그대로 붙이면 SQL injection과 동일한 성격의 취약점이 됩니다.

```java
enum SortKey {
    CREATED_AT("created_at"),
    AMOUNT("amount");

    private final String column;

    SortKey(String column) {
        this.column = column;
    }

    String column() {
        return column;
    }
}

class SafeSqlBuilder {
    String orderBy(SortKey key) {
        // 식별자는 바인딩할 수 없으므로, 외부 입력 String을 직접 받지 않습니다.
        return " ORDER BY " + key.column() + " DESC";
    }
}
```

이 예시는 "동적 SQL이 늘어날수록 타입 시스템의 가치가 커진다"는 점을 보여줍니다.
값은 바인딩으로, 식별자는 allow-list로 통제하지 않으면 동적 조립 지점이 곧 취약점이 되기 때문입니다.

## 8. 단점과 트레이드오프 (그리고 왜 그래도 하는가)
상수화/타입화는 만능이 아닙니다. 단점이 있기 때문에, 단점을 모른 채 적용하면 "무지성 상수화"로 흐르기 쉽습니다.

가장 흔한 단점은 읽기 흐름이 끊기는 것입니다.
특히 의미가 약한 문자열까지 모두 상수화하면, 코드를 읽는 사람은 계속 정의 위치로 점프해야 하고, 작은 변경의 인지 비용이 커집니다.

두 번째 단점은 `Constants` 같은 거대한 중앙 파일이 생기기 쉽다는 점입니다.
의미가 다른 문자열이 한 파일에 모이면, 상수명 네이밍 경쟁이 생기고, 결국 "이 파일을 열어봐야 하는" 상황이 늘어 오히려 탐색성이 떨어질 수 있습니다.

세 번째 단점은 SQL을 조각 상수로 분해해 문자열 결합으로 조립하는 방식이 늘면, 쿼리 자체를 한 번에 읽기 어려워진다는 점입니다.
이 경우는 가독성 저하뿐 아니라 조립 실수로 인한 정합성 문제, 로그에서 실제 실행 SQL을 추적하기 어려운 문제가 함께 나타납니다.

그럼에도 이 패턴을 기본값으로 두는 정당화는 실패 비용 구조에 있습니다.
의미 단위 문자열은 오타가 컴파일에서 잡히지 않기 때문에, 가장 자주 운영에서 비싸게 터지고, 또한 관측 지점이 원인과 멀어지기 쉽습니다.
상수/enum 승격은 이 실수를 "코드 구조"로 끌어올려 리뷰/리팩터링에서 발견될 확률을 크게 올립니다.
그리고 SQL은 정적 본문 가독성을 유지하되 값 주입을 바인딩으로 고정하는 것만으로도 보안/정합성 위험을 크게 낮출 수 있습니다.

---

## 9. 다르게 판단해야 하는 경우(Variants)
기본 결론과 다르게 판단해야 하는 케이스를 조건으로 고정해두면, 팀이 예외를 감정이 아니라 논리로 다룰 수 있습니다.

첫째, 값 후보가 열린 집합(open set)이라면 enum으로 고정하지 않습니다.
이때는 문자열을 유지하되 길이/포맷/금지 문자 같은 validator 계약을 강화합니다. 예를 들어 사용자 검색어, 자유 입력 라벨, 외부 시스템의 임의 ID는 닫힌 집합으로 만들 수 없기 때문에 enum이 아니라 검증 규칙이 중심이 되어야 합니다.

둘째, 재사용이 1회라도 실패 비용이 매우 큰 식별자라면 상수/enum 승격을 고려합니다.
예를 들어 동적 SQL 식별자(정렬 컬럼)나 보안 경계에서의 키는 "횟수"보다 "실패 비용"이 기준입니다.

셋째, SQL이 동적으로 복잡해진다면(필드 선택/조건 조합이 급증) 문자열만으로 관리하지 않습니다.
이때 선택지는 enum allow-list 강화, 조립 규칙을 별도 모듈로 격리, SQL DSL 도입 등으로 갈리지만, 핵심은 "문자열 결합이 유일한 경로"가 되지 않게 만드는 것입니다.

## 10. 최소 실험(검증): 이 결론을 직접 확인하기
아래 실험은 특정 프로젝트 도구 없이, 자바 코드만으로 확인할 수 있는 수준으로 고정합니다.

### 10.1 드리프트가 실제 버그가 되는지 확인
```java
import java.util.Map;

final class Keys {
    static final String INBOUND_PROFILE = "inbound-profile";
    private Keys() {}
}

public class DriftLab {
    static String readProfileBad(Map<String, String> m) {
        // 사람이 보기엔 비슷하지만, 프로그램에겐 완전히 다른 키입니다.
        return m.get("inboundProfile");
    }

    static String readProfileGood(Map<String, String> m) {
        return m.get(Keys.INBOUND_PROFILE);
    }

    public static void main(String[] args) {
        Map<String, String> ok = Map.of(Keys.INBOUND_PROFILE, "KPN_200");

        System.out.println(readProfileBad(ok));  // 기대 결과: null
        System.out.println(readProfileGood(ok)); // 기대 결과: KPN_200
    }
}
```

이 실험이 보여주는 것은 단순합니다.
키 문자열은 한 글자만 달라도 즉시 `null`이 되고, 그 `null`은 종종 더 깊은 로직에서 NPE/분기 오류로 증폭됩니다.

### 10.2 값 바인딩이 왜 필요한지(문자열 결합의 위험 확인)
```java
public class SqlInjectionLab {
    public static void main(String[] args) {
        String userInput = "x' OR '1'='1";

        String unsafe = "SELECT * FROM users WHERE name = '" + userInput + "'";
        System.out.println(unsafe);
        // 기대 결과:
        // SELECT * FROM users WHERE name = 'x' OR '1'='1'
        // (값이 SQL 문법으로 해석될 여지가 생깁니다)

        String safeTemplate = "SELECT * FROM users WHERE name = ?";
        System.out.println(safeTemplate);
        // 기대 결과:
        // SELECT * FROM users WHERE name = ?
        // (값은 바인딩으로 전달되어 SQL 문법으로 해석되지 않습니다)
    }
}
```

여기서 핵심은 "SQL 문자열을 상수로 뺐는가"가 아니라, 값이 SQL 문법으로 합쳐지는 경로가 있는가/없는가입니다.

### 10.3 동적 식별자(정렬 컬럼)는 바인딩할 수 없음을 확인
값은 바인딩할 수 있지만, 컬럼명은 바인딩할 수 없습니다.
따라서 외부 입력 String을 `ORDER BY <입력>`에 직접 붙이지 말고, allow-list(enum)로만 통과시키셔야 합니다.
이 결론은 위의 `SortKey` 예시처럼 "외부 String을 받지 않는" 형태의 API로 가장 쉽게 강제됩니다.

## 11. 흔한 오해
오해는 지식 부족이 아니라, 경계가 닫혀 있지 않을 때 생기는 자연스러운 결과입니다.
그래서 오해를 정정할 때는 "왜 오해가 그럴듯해 보이는지"와 "어디서 깨지는지"를 함께 닫아야 합니다.

"2번 이상 쓰이면 무조건 상수화"가 정답이라는 오해가 대표적입니다.
재사용 횟수는 힌트일 뿐이고, 핵심은 같은 의미를 공유하는지와 실패 비용이 얼마나 큰지입니다.

"상수화하면 입력 검증도 끝"이라는 오해도 자주 나옵니다.
상수화는 내부의 부분 수정/드리프트를 줄이지만, 외부 입력이 잘못된 경우는 경계 검증이 담당해야 합니다.

마지막으로 "SQL은 다 상수로 쪼개면 안전"이라는 오해가 있습니다.
안전성의 핵심은 바인딩 경계를 유지하는 것이고, 문자열 조각 상수는 안전을 보장하지 않으며 가독성만 해칠 수 있습니다.

## 12. 팀 적용 체크리스트
이 기준을 팀 규칙으로 쓰려면, 질문을 "각각 따로" 체크하기보다 "순서대로" 통과시키는 형태가 더 안전합니다.
첫 질문에서 의미 단위/표현 단위가 갈리면, 이후 판단(상수화/인라인 유지)의 방향이 바뀌고, SQL은 마지막에서 값/식별자 경계로 다시 한 번 갈립니다.
즉 아래 질문들은 독립적인 체크박스가 아니라, 앞의 답이 뒤의 선택지를 제한하는 하나의 논리 흐름입니다.

- `이 문자열은 의미 단위입니까, 표현 단위입니까?`: 의미 단위(식별자)라면 상수/enum 쪽으로, 표현 단위(문장)라면 읽기 흐름을 지키는 쪽으로 출발점을 고정합니다.
- `의미 단위라면, 한 곳에서 이름을 가지고 관리되고 있습니까(상수/enum/값 객체)?`: 여기서 "한 곳"이 흔들리면 부분 수정과 표기 드리프트가 다시 생깁니다. 상수화의 목적은 값 공유가 아니라 의미(의도)의 단일화를 강제하는 데 있습니다.
- `외부 입력이 이 값을 만든다면, 경계에서 fail-fast 검증이 있습니까?`: 상수화만으로는 잘못된 입력을 고칠 수 없습니다. 외부 입력이 틀리면 내부에서는 멀리서 NPE/분기 오류로 관측될 수 있으므로, 실패를 경계로 당겨 원인을 고정해야 합니다.
- `SQL 값 주입이 문자열 결합이 아니라 바인딩으로 통제되고 있습니까?`: SQL에서 값은 바인딩 경계를 깨는 순간 보안/정합성 문제가 생깁니다. 쿼리 본문의 상수화 여부보다 이 경계가 먼저입니다.
- `동적 SQL 식별자(정렬/컬럼)는 allow-list(enum)로만 조립되도록 막혀 있습니까?`: 식별자는 바인딩할 수 없으므로, 외부 String을 그대로 받는 API는 곧 취약점이 됩니다. API 형태를 enum-only로 만들면 팀 규칙이 코드로 강제됩니다.
