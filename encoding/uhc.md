# UHC (CP949 / MS949)와 한글 mojibake

UHC(Unified Hangul Code)는 Windows 한국어 환경의 문자 코드페이지다. 코드페이지 번호 949라서 CP949, Microsoft 확장이라 MS949라고도 부른다. 여기서 실무에 가장 자주 부딪히는 문제는 한 가지다 — **Windows 한국어 로캘의 *기본 문자셋*이 UTF-8이 아니라 이 CP949**라서, UTF-8로 저장된 한글 데이터를 문자셋 명시 없이 읽으면 깨진다(mojibake). 같은 코드가 UTF-8이 기본인 리눅스/맥에서는 멀쩡하다가 Windows에서만 깨지는 전형적 크로스플랫폼 버그가 여기서 나온다.

이 노트는 CP949가 무엇인지, UTF-8 데이터가 왜·어떻게 깨지는지, 깨짐이 *어느 계층*에서 일어났는지 진단하는 법, 그리고 계층별로 고정하는 법을 정리한다.

## 1. CP949/UHC가 무엇인가

CP949는 EUC-KR을 확장해 현대 한글 음절 11,172자를 모두 담은 가변폭 인코딩이다. ASCII 영역은 1바이트, 한글·한자는 2바이트로 표현한다. Windows에서 한국어 로캘의 "ANSI 코드페이지"가 949이며, Java에서는 `MS949`(또는 `x-windows-949`)라는 charset 이름으로 부른다. EUC-KR과 거의 호환되지만 CP949가 더 넓다(EUC-KR이 못 담는 음절을 포함).

대비되는 것이 UTF-8이다. UTF-8은 유니코드를 1~4바이트로 표현하며 한글 음절은 보통 3바이트다. 오늘날 리눅스·macOS·웹·대부분의 소스 파일은 UTF-8이 기본이다. 문제는 두 인코딩이 같은 바이트열을 *다르게* 해석한다는 데서 출발한다.

## 2. 왜 깨지나: 기본 문자셋의 함정

파일이나 스트림을 읽을 때 charset을 명시하지 않으면, JVM과 많은 도구는 **플랫폼 기본 문자셋**을 쓴다. Windows 한국어에서는 그 기본이 CP949(MS949)다. 그래서 UTF-8로 저장된 한글 파일을 명시 없이 읽으면, 도구는 그 바이트를 CP949로 해석해 엉뚱한 글자를 만든다.

예를 들어 Maven 빌드 로그의 환경 정보에 `platform encoding: MS949`가 찍히는 Windows에서, UTF-8 리소스를 기본 charset으로 읽는 코드는 한글을 깨뜨린다. 같은 코드가 CI(리눅스, UTF-8 기본)에서는 멀쩡하다. "내 PC에서만/Windows에서만 한글이 깨진다"는 증상의 거의 대부분이 이 기본 문자셋 불일치다.

## 3. mojibake가 생기는 바이트 수준 그림

핵심은 **글자당 바이트 수가 다르다**는 것이다. UTF-8 한글은 보통 3바이트, CP949 한글·한자는 2바이트다. UTF-8 바이트열을 CP949로 읽으면 경계가 어긋나 전혀 다른 CJK 글자나 깨짐 문자(`�`)가 나온다.

실제로 관측한 변환(대상 프로젝트의 H2 적재 실패에서 그대로 캡처한 값):

```text
정상            → �젙�긽
빗썸            → 鍮쀬뜽
수취인계좌없음   → �닔痍⑥씤怨꾩쥖�뾾�쓬
```

여기서 `鍮쀬뜽`처럼 멀쩡한 한자/한글이 나오는 건 어긋난 2바이트 쌍이 우연히 유효한 CP949 글자에 맞은 경우이고, `�`은 유효한 CP949 글자에 맞지 않아 디코딩이 실패한 지점이다. 둘이 섞여 나오는 게 UTF-8→CP949 오독의 특징적 무늬다.

(반대 방향, 즉 CP949 데이터를 UTF-8로 읽으면 대부분 `�`만 잔뜩 나온다. 무늬가 다르다.)

## 4. 깨짐의 세 계층

한글이 깨졌을 때, *어디서* 깨졌는지를 세 계층으로 나누면 원인을 빨리 좁힌다.

1. **소스 컴파일 인코딩** — `javac`가 `.java`를 읽는 charset. Maven은 `project.build.sourceEncoding`으로 정한다. 이게 틀리면 소스에 박힌 한글 *문자열 리터럴* 전부가 컴파일 시점에 깨진다.
2. **JVM 런타임 기본 인코딩** — `file.encoding`(과 `sun.jnu.encoding`). charset을 명시하지 않은 모든 파일·스트림 읽기에 적용된다. Java 17까지는 플랫폼 로캘을 따르므로 Windows 한국어에서 MS949, Java 18부터는 기본이 UTF-8이다.
3. **특정 리소스 읽기의 명시 인코딩** — 개별 API가 자기 charset을 받는 경우. 예: Spring `ResourceDatabasePopulator.setSqlScriptEncoding(...)`. 이걸 지정하지 않으면 그 리소스만 2번(런타임 기본)으로 읽혀 깨진다.

## 5. 어느 계층인지 진단하기

대조쌍 하나가 계층을 가른다. 단언 실패에서 **"expected 리터럴은 멀쩡한데 actual 값만 깨졌다"**면 1번(소스 컴파일)은 범인이 아니다. 소스가 잘못된 charset으로 컴파일됐다면 expected 한글 리터럴도 똑같이 깨졌을 텐데 그렇지 않기 때문이다. 따라서 범인은 *런타임에 데이터를 읽는 경계*(2번 또는 3번)다 — 파일·DB·네트워크에서 들어온 값이 기본 charset으로 잘못 읽힌 것이다.

반대로 소스의 한글이 *전부* 깨져 보이면 1번(컴파일 인코딩)을 의심한다.

이 추론은 큰 실패 집합을 가를 때 그대로 쓰인다([`../thinking/failure_triage.md`](../thinking/failure_triage.md)).

## 6. 계층별 고정법

원칙은 하나다 — **기본값에 기대지 말고 인코딩을 명시한다.**

- 소스 컴파일: Maven `pom.xml`에 `<project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>`. (대상 프로젝트는 이게 이미 설정돼 있어 리터럴은 안 깨졌다.)
- JVM 런타임 기본: 테스트·실행 JVM에 `-Dfile.encoding=UTF-8`. Java 18+는 기본이 UTF-8이라 불필요. 단 surefire 등 fork된 JVM에는 그 플러그인 설정(argLine)으로 넘겨야 부모 `-D`가 새 JVM에 전달된다.
- 특정 리소스 읽기: 그 API의 charset 인자를 쓴다. 실제 사례에서, H2에 UTF-8 `schema.sql`(한글 시드 포함)을 적재할 때 `populator.setSqlScriptEncoding("UTF-8")` 한 줄을 추가하자 Windows(MS949 기본)에서 깨지던 한글이 정상화됐다. 즉 런타임 기본을 바꾸지 않고 그 리소스만 UTF-8로 명시했다.

## 7. 표시 문제와 데이터 문제는 다르다

콘솔에 한글이 깨져 *보이는* 것과, 메모리의 String이 실제로 깨진 것은 다른 문제다.

- **표시 문제**: 데이터(String)는 멀쩡한데 터미널이 다른 코드페이지로 출력한다. Windows 콘솔은 `chcp 65001`(UTF-8)로 바꾸거나, PowerShell에서 `[Console]::OutputEncoding = [Text.Encoding]::UTF8`로 출력 인코딩만 맞추면 된다. 로그가 깨져 보여도 프로그램 동작·테스트 단언은 멀쩡할 수 있다.
- **데이터 문제**: String 자체가 이미 깨진 바이트에서 만들어졌다. 이건 출력 설정으로 못 고치고, 읽는 지점의 charset을 고쳐야 한다(6번).

진단에서 이 둘을 섞으면, 멀쩡한 데이터를 두고 콘솔만 만지거나, 반대로 데이터가 깨졌는데 출력 탓만 한다.

## 8. 스스로 확인하기

- Windows 한국어에서 charset을 명시하지 않고 UTF-8 파일을 읽으면 무슨 일이 일어나나? 같은 코드가 리눅스 CI에서 멀쩡한 이유는?
- UTF-8 한글이 CP949로 읽힐 때 `�`과 `鍮쀬뜽`이 섞여 나오는 이유는?
- "expected는 멀쩡, actual만 깨짐"이면 소스 컴파일 인코딩은 왜 범인이 아닌가?
- `-Dfile.encoding=UTF-8`을 줬는데도 surefire가 fork한 테스트 JVM에서 안 먹을 수 있는 이유는?
- 콘솔에 깨져 보이는데 테스트는 통과한다. 무엇이 깨졌고 무엇은 멀쩡한 것인가?

## 9. 확인한 사실과 추론

직접 확인한 사실(한 펌뱅킹 작업, Windows): Maven이 `platform encoding: MS949`로 동작한 점, UTF-8 `schema.sql`의 한글 시드가 H2 적재 시 위 표처럼 깨진 점, `setSqlScriptEncoding("UTF-8")` 한 줄로 정상화되고 의존 테스트가 통과한 점, expected 리터럴은 멀쩡하고 actual만 깨져 컴파일이 아닌 데이터 읽기 경계가 범인으로 특정된 점.

일반 원리·추론: CP949/UTF-8의 바이트 폭과 기본 charset 동작은 표준 사실이다. Java 18부터 `file.encoding` 기본이 UTF-8이라는 것도 사실(JEP 400)이다. 다만 특정 글자쌍이 어떤 CJK 글자로 보일지는 코드포인트별 매핑에 달려 있어, 위 표의 글자 대응은 *관측값*이지 일반 규칙으로 외울 것은 아니다.
