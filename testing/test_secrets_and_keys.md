# 운영은 HSM, 테스트는? — 암호화 키·비밀을 테스트에서 결정적으로 다루기

운영 환경의 암호화 키는 안전한 출처에서 온다. 물리 보안장비(HSM), 클라우드 KMS, 비밀 저장소(Vault) 같은 곳이다. 이런 키는 테스트에서 그대로 못 쓴다. 망·장비에 의존하고, 값이 공개되지 않으며, 환경마다 다르기 때문이다. 그런데 테스트가 *진짜 키의 출력*에 의존하면 결정적으로 돌릴 수가 없다. 핵심 해법은 단순하다 — **테스트는 진짜 키를 쓰지 않고, 알려진 키를 결정적으로 주입해 자기완결적으로 동작하게 만든다.**

이 노트의 목표는 단위 테스트와 통합(E2E) 테스트가 각각 키를 어떻게 다뤄야 하는지, 그리고 왜 "절대 암호문"을 단언하면 안 되는지를 스스로 설명할 수 있게 하는 것이다.

## 1. 직답

테스트는 진짜 키를 쓰지 않는다. 두 갈래로 나뉜다.

- **단위 테스트**: 키를 제공하는 컴포넌트를 mock해서 *알려진 상수*를 돌려준다. 같은 상수로 암호화하고 같은 상수로 검증하므로 자기완결이다 — 키 값이 무엇이든 상관없다.
- **E2E(스프링 컨텍스트) 테스트**: 진짜 키 컴포넌트를 그대로 쓰되, 테스트 프로파일에서 `@Primary` 빈으로 *알려진 키*를 주입해 HSM 호출 경로를 우회한다. 결정적이고 재현 가능하다.

공통 원칙: 암호화한 키와 복호화·검증하는 키가 같으면(자기완결) 테스트는 진짜 비밀과 무관하게 통과한다.

## 2. 왜 어려운가

암호화 의존 테스트가 환경을 타는 이유는 *키의 출처*다. HSM/KMS는 물리·망 의존이고 값이 비공개이며 환경마다 다르다. 그래서 어떤 입력을 암호화한 *절대 결과값(암호문)*을 테스트에 박아 두면, 키가 다른 환경에서는 그 단언이 깨진다.

실제로, 한 펌뱅킹 서비스의 사내망 전체 테스트 실행에서 암호화에 의존하는 통합 테스트 다수가 실패로 잡혔는데, 이는 그 환경의 HSM 키로 만든 값에 기댄 결과였다. 같은 테스트가 로컬에서는 *주입한 알려진 키*로 자기완결적으로 통과했다. 즉 "사내망에서 실패한 암호화 테스트"의 상당수는 진짜 결함이 아니라 키-환경 차이였다. (이런 가르기는 [`../thinking/failure_triage.md`](../thinking/failure_triage.md) 참고.)

## 3. 단위 테스트: 키 컴포넌트를 mock

단위 테스트는 스프링 컨텍스트 없이 SUT를 직접 만든다. 키 컴포넌트는 Mockito로 mock해서 정해 둔 상수를 돌려주게 한다.

```java
var key = mock(KpnEncryptionKey.class);
when(key.getEncKey()).thenReturn(ENC_KEY);   // 알려진 32자리 상수
when(key.getEncIv()).thenReturn(ENC_IV);     // 알려진 16자리 IV
// SUT에 이 mock을 주입하고, fixture도 같은 ENC_KEY/ENC_IV로 암호화한다.
```

여기서 fixture를 만들 때 쓰는 키와 SUT가 복호화에 쓰는 키가 *같은 상수*이므로, 암복호화가 자기완결로 맞아떨어진다. 키 값 자체는 의미가 없다 — 테스트마다 아무 32자리 값이나 써도 된다.

상수는 한곳에 모아 두는 편이 좋다. 같은 키 리터럴이 수십 개 테스트 파일에 흩어지면 관리가 어렵고 드리프트가 생긴다. 공용 상수 홀더 하나를 만들고 `static import`로 참조하면 중복이 사라진다.

## 4. E2E: 알려진 키를 @Primary로 주입

E2E 테스트는 실제 스프링 컨텍스트를 띄워 진짜 흐름을 검증한다. 여기서는 키 컴포넌트를 mock하지 않고 *진짜 빈*을 쓰되, 테스트 프로파일에서 알려진 키를 주입한 빈을 `@Primary`로 올려 운영 빈(HSM 조회)을 가린다.

```java
@Profile("junit")
@Primary
@Bean
public KpnEncryptionKey kpnEncryptionKeyForJunit() {
    var key = new KpnEncryptionKey();
    ReflectionTestUtils.setField(key, "encKey", JUNIT_ENCRYPTION_KEY); // 알려진 값
    ReflectionTestUtils.setField(key, "encIv",  JUNIT_ENCRYPTION_IV);
    return key;
}
```

이러면 컨텍스트 안의 모든 코드가 결정적 키를 쓰므로, HSM 없이도 같은 암복호화 결과가 재현된다. 운영 프로파일에서는 이 빈이 활성화되지 않으니 운영 동작은 그대로다.

리플렉션으로 필드를 직접 세팅하는 이유는, 회사 키 컴포넌트의 인터페이스를 바꾸지 않고(테스트용 생성자·setter를 추가하지 않고) 값만 덮기 위해서다. 회사 라이브러리 표면을 테스트 편의로 늘리면 그게 또 다른 드리프트가 된다.

## 5. 정합성 함정과 round-trip 단언

단위에서 쓰는 상수 키와 E2E에서 주입하는 키가 *서로 다른 값*일 수 있다. 각자 자기완결이면 그 자체는 문제가 아니다. 다만 한 가지 계약은 지켜야 한다 — **E2E fixture는 반드시 "E2E가 주입한 키"로 암호화**해야 SUT의 복호화와 맞는다. 단위 상수 키로 E2E fixture를 암호화해 넣으면, 컨텍스트의 빈 키와 달라 복호화가 어긋난다.

가장 견고한 단언은 *절대 암호문 비교*가 아니라 **round-trip**이다.

- 약한 방식: `assertThat(encrypted).isEqualTo("Zm9v...")` — 키가 바뀌면 깨진다(키 종속).
- 강한 방식: `assertThat(decrypt(encrypt(plain))).isEqualTo(plain)` — 같은 키로 왕복하므로 키 값과 무관하게 성립(키 비종속).

암호문 자체를 꼭 단언해야 한다면, 결정적 키에 고정한 상태에서만 의미가 있다.

## 6. 하지 말 것

- **키 컴포넌트를 상속해 키를 박지 말 것** → 회사 인터페이스를 훼손하고, 시험 대상·핵심 컴포넌트를 상속하는 안티패턴과 같다. 실제 컴포넌트 + mock/주입으로 푼다. (같은 원리: [`./characterization_testing.md`](./characterization_testing.md)의 SUT 경계.)
- **외부에서 못 받는 보안 라이브러리를 빌드에서 빼지 말 것** → 좌표는 계약이다. 대신 shim으로 채운다. 단 암호화 같은 순수 계산은 출력이 진짜와 byte 단위로 같아야 테스트가 맞으므로, shim에서 알고리즘을 충실히 구현해야 한다. ([`../jvm/maven_local_shim_repo.md`](../jvm/maven_local_shim_repo.md)의 동작 충실도.)
- **절대 암호문을 환경 무관하게 단언하지 말 것** → 키 종속이라 환경이 바뀌면 깨진다. round-trip이나 결정적 키 고정으로 바꾼다.
- **가짜 성공값을 만들어 두지 말 것** → 키가 없다고 "성공한 것처럼" 더미를 돌려주면, 테스트가 그 허구를 정답으로 굳힌다.

## 7. 스스로 확인하기

- 단위 테스트에서 암호화 키 상수가 "아무 값이나 괜찮은" 이유는? 무엇이 보장되기 때문인가?
- E2E에서 키 컴포넌트를 mock하지 않고 `@Primary` 빈으로 진짜를 주입하는 이유는?
- 단위 상수 키와 E2E 주입 키가 달라도 되는데, E2E fixture 암호화에서 꼭 지켜야 하는 한 가지는?
- `encrypt→decrypt==원문`(round-trip) 단언이 절대 암호문 단언보다 환경에 견고한 이유는?
- HSM 키로 만든 암호문을 그대로 테스트에 박으면 다른 환경에서 무엇이 일어나나?

## 8. 확인한 사실과 추론

직접 확인한 사실(한 펌뱅킹 작업): 단위 테스트가 키 컴포넌트 mock으로 알려진 상수를 주입한 점, E2E가 `@Primary`·`@Profile("junit")` 빈에서 리플렉션으로 알려진 키를 세팅해 HSM을 우회한 점, 사내망(HSM)에서 실패하던 암호화 의존 테스트가 로컬(주입 키)에서는 통과해 그 실패들이 환경 산물이었음이 드러난 점, 단위 상수 키와 E2E 주입 키 값이 실제로 달랐던 점.

일반 원리·추론: "round-trip 단언이 더 견고하다", "키 컴포넌트 상속을 피한다"는 일반 설계 원칙이다. 구체 주입 방식(리플렉션 vs 전용 테스트 빈 vs 환경변수)은 프레임워크·코드베이스마다 더 깔끔한 대안이 있을 수 있다.
