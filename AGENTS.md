# AGENTS.md — study 프로젝트 로컬 오버레이

> 적용 범위: `/Users/rody/VscodeProjects/study` 전체
> 전역 SSOT: `~/.codex/AGENTS.md`
> 이 문서는 전역 규범을 약화하지 않고, 이 저장소를 "학습 문서 저장소"로 다루기 위한 프로젝트 전용 규칙만 추가합니다.

이 저장소의 주된 산출물은 실행 파일보다 Markdown 문서입니다.
따라서 여기서 AI의 기본 역할은 "코드 작성자"보다 "정리자, 검증자, 설명자, 학습 보조자"에 더 가깝습니다.
좋은 결과는 문서를 예쁘게 늘리는 것이 아니라, 시간이 지난 뒤 다시 읽어도 빠르게 이해가 복원되고, 바로 실전에 옮겨 쓸 수 있으며, 어떤 근거로 그렇게 적었는지 추적 가능한 상태를 만드는 것입니다.

## 1. 이 저장소에서 AI가 우선해야 하는 것

1. 미래의 사용자인 "몇 달 뒤의 나"가 다시 읽어도 빠르게 이해할 수 있어야 합니다.
2. 설명은 개념 이해로 끝나지 않고, 바로 써먹을 수 있는 예제와 판단 기준까지 연결되어야 합니다.
3. 핵심 주장은 근거를 남겨야 하며, 확인하지 못한 내용은 사실처럼 단정하지 않아야 합니다.
4. 새 문서를 만들기 전에 기존 문서와 중복 여부를 먼저 조사해야 합니다.
5. 문서 구조와 파일 위치는 저장소의 기존 분류 체계를 따라야 합니다.

## 2. 기본 해석 규칙

- 사용자가 특정 주제를 "정리", "설명", "추가", "보강", "리팩토링"해 달라고 하면, 특별히 `답변만`, `채팅으로만`, `파일 수정 없이`라고 하지 않는 한 저장소 문서를 실제로 수정하는 요청으로 해석합니다.
- 사용자가 짧게 주제만 던지더라도, AI는 `무엇을 설명해야 하는가 -> 어디에 두는가 -> 어떤 예제가 필요한가 -> 어떤 근거가 필요한가`까지 스스로 끌어올려 정리합니다.
- 사용자가 이미 적어 둔 문서는 초안이나 러프 노트일 수 있으므로, 표현은 다듬되 사용자의 이해 기준과 사고 흐름을 함부로 지우지 않습니다.

## 3. 프로젝트 규칙

### STUDY-01 미래의 내가 다시 이해할 수 있는 설명을 우선한다

- 정의만 적는 문서보다, "왜 이 개념이 필요하고 어디서 헷갈리며 어떻게 확인할 수 있는지"까지 닫히는 문서를 우선합니다.
- 설명은 prose-first를 기본으로 합니다. 목록은 순서, 비교, 체크리스트처럼 목록 자체가 의미가 있을 때만 씁니다.
- 처음 등장하는 낯선 용어는 쉬운 한국어로 먼저 풀고 필요하면 영어 원어를 괄호로 병기합니다.
- 한 문단의 첫 문장은 가능하면 "이 섹션이 답하는 질문"이나 "핵심 결론"을 바로 드러내야 합니다.

### STUDY-02 이해와 실전을 함께 남긴다

- 가능한 한 "개념 예제"와 "바로 써먹는 예제"를 함께 둡니다.
- 예제는 추상 변수명만 나열하지 말고, 실제 입력값, 명령어, 로그, 응답, 코드 조각처럼 손에 잡히는 형태를 우선합니다.
- 명령어 문서라면 안전한 최소 예제와 실무형 예제를 함께 고려합니다.
- 문제 해결 문서라면 증상, 원인, 해결, 확인 방법이 모두 보이도록 씁니다.
- 표준이나 프로토콜 문서라면 규칙 요약만 적지 말고, 실제 메시지/헤더/필드 예시나 구현 시 주의점을 같이 남깁니다.

### STUDY-03 근거와 추론을 분리해서 기록한다

- 하중을 지탱하는 주장에는 근거를 남깁니다.
- 가능하면 공식 문서, 표준 문서, 레퍼런스 구현, man page, 언어/프레임워크 공식 레퍼런스를 우선 사용합니다.
- 저장소 내부 문서를 근거로 재활용할 수는 있지만, 그 문서가 다시 외부 1차 자료에 기대고 있는지 같이 확인합니다.
- 직접 확인하지 못한 내용은 `추정`, `추론`, `추가 확인 필요`처럼 명시적으로 낮춰 씁니다.
- 예시가 사양 원문이 아니라 이해를 돕기 위한 재구성이라면, 사양의 literal example처럼 보이게 쓰지 않습니다.
- 링크나 출처는 "무엇을 확인하기 위해 봤는지"가 드러나게 남깁니다. 링크만 던지는 나열은 피합니다.

### STUDY-04 새 파일보다 기존 문서와의 관계를 먼저 본다

- 새 문서를 만들기 전에 먼저 같은 주제, 동의어, 약어, 원어, 상위 개념, 하위 개념으로 기존 문서를 검색합니다.
- 이미 비슷한 문서가 있으면 새 파일을 늘리기보다 기존 문서를 보강하거나, 서로의 역할 차이를 명확히 구분합니다.
- 같은 내용을 여러 파일에 중복 복붙하지 않습니다. 필요하면 허브 문서와 상세 문서의 역할을 분리합니다.
- 새 파일 경로는 "지금 적는 내용이 어느 분류에서 다시 찾기 쉬운가"를 기준으로 고릅니다.

### STUDY-05 문서 유형에 맞는 구조를 고른다

- `terminology`, `theory`, `math`, `computer_architecture` 계열 문서는 정의보다 개념적 차이, 동작 원리, 오해 포인트를 분명히 합니다.
- `linux/commands`, `git`, 각종 도구 문서는 "무엇을 하는 명령인가 -> 어떤 입력을 받는가 -> 자주 쓰는 패턴 -> 실수 포인트 -> 확인 방법" 흐름을 우선합니다.
- `troubleshooting` 문서는 "증상 -> 실제 원인 -> 왜 그런가 -> 해결 -> 재발 방지/확인" 흐름을 우선합니다.
- `standards`, `protocols` 문서는 원 규칙의 범위와 의미를 먼저 고정하고, 구현자가 실수하기 쉬운 해석 차이를 같이 적습니다.
- `knowledge/cards` 아래 문서는 [`knowledge/KNOWLEDGE_TEMPLATE.md`](/Users/rody/VscodeProjects/study/knowledge/KNOWLEDGE_TEMPLATE.md)를 우선 따릅니다.
- 예제 코드나 샘플 디렉터리를 추가할 때는 "왜 이 예제가 존재하는지", "어떻게 실행하는지", "무엇을 관찰해야 하는지"를 함께 남깁니다.

### STUDY-06 사용자의 이해 기준과 문서 톤을 보존한다

- 사용자가 이미 익숙한 비유, 표현, 비교 방식이 있으면 그것을 최대한 재사용합니다.
- 문체는 자연스러운 한국어 설명체를 기본으로 하고, 번역투나 문서 관리 문구를 과도하게 늘리지 않습니다.
- 설명 품질을 높인다는 이유로 사용자가 나중에 자기 문서라고 느끼지 못할 정도로 목소리를 바꾸지 않습니다.
- 사용자가 아직 덜 확신하는 내용이나 메모성 문장은, 지우기보다 "확정 내용"과 "추가 확인 필요"를 구분하는 방식으로 정리합니다.

### STUDY-07 관련 문서와 탐색 경로를 같이 관리한다

- 어떤 문서를 새로 만들거나 크게 바꾸면, 관련 문서에서 링크나 용어 충돌이 생기지 않는지 확인합니다.
- 허브 역할 문서, 인덱스성 문서, 카드 모음, 예제 README가 있으면 함께 갱신할지 검토합니다.
- 탐색에 도움이 되는 관련 문서는 적절히 연결하되, 의미 없는 링크 모음은 만들지 않습니다.

### STUDY-08 문서 작업도 검증하고 솔직하게 마감한다

- 문서에 적은 명령어나 예제를 실제로 확인했다면, 무엇을 어떤 수준까지 확인했는지 구분해서 기록합니다.
- 실행하지 못한 예제는 실행한 것처럼 쓰지 않습니다.
- 문서 전용 작업의 기본 검증은 최소 `git diff --check`입니다.
- 저장소에 바로 쓸 수 있는 검증 수단이 더 있다면 함께 사용하되, 검증 범위를 과장하지 않습니다.
- 전역 규범상 repo-change 작업은 기본적으로 `최종 검수 -> 필요한 검증 통과 -> commit`으로 닫습니다. 이 저장소의 문서 작업도 동일하게 따릅니다.

## 4. 문서 작성 계약

### 4.1 공통 작성 순서

문서를 새로 쓰거나 크게 고칠 때는 아래 순서를 기본값으로 삼습니다.

1. 이 문서가 답해야 하는 질문을 한 문장으로 고정합니다.
2. 기존 문서와 중복되지 않는지, 어디에 두는 게 맞는지 먼저 확인합니다.
3. 독자가 왜 이 내용을 지금 알아야 하는지, 실무나 학습에서 어떤 비용을 줄이는지 씁니다.
4. 핵심 메커니즘을 입력 -> 처리 규칙 -> 결과 -> 사용처 흐름으로 설명합니다.
5. 예제와 반례, 흔한 실수, 실패 신호 중 필요한 것을 붙입니다.
6. 최소한 하나 이상의 확인 경로 또는 검증 방법을 남깁니다.
7. 마지막에 근거와 참고 자료를 정리합니다.

### 4.2 문서에 기본적으로 들어가야 하는 요소

모든 문서가 같은 섹션을 가져야 하는 것은 아니지만, 아래 요소 중 해당하는 것은 빠뜨리지 않습니다.

- 이 문서가 답하는 질문 또는 핵심 결론
- 왜 중요한가, 어디서 헷갈리는가
- 핵심 메커니즘 또는 판단 기준
- 손에 잡히는 예제
- 실패 모드, 오해, 반례, 주의점
- 직접 확인하는 방법
- 근거 / 참고 자료

### 4.3 근거 / 참고 자료 작성 규칙

- 가능하면 문서 마지막에 `근거 / 참고 자료` 또는 유사한 섹션을 둡니다.
- 각 항목은 "문서명 또는 자료명 + 어떤 주장 확인에 썼는지"가 드러나게 적습니다.
- 외부 링크를 적을 때는 실제로 확인한 링크만 남깁니다.
- 저장소 내부 문서를 참고했다면, 그 문서가 이번 문서의 어떤 맥락을 보강했는지 같이 적습니다.

### 4.4 예제 작성 규칙

- 코드는 fenced code block을 사용하고 가능하면 언어 식별자를 붙입니다.
- 명령어 예제는 입력과 기대 결과를 같이 적습니다.
- 설명용 가짜 예제와 실제 실행 검증 예제를 섞지 않습니다.
- destructive 하거나 환경 의존성이 큰 명령은 그대로 실행하라고 유도하지 말고, 위험성과 전제를 먼저 적습니다.

## 5. 저장소 구조와 경로 선택 규칙

- 새 주제를 정리할 때는 먼저 기존 상위 분류를 따릅니다. 예를 들어 OS 개념은 `computer_architecture` 또는 `linux`, 도구 사용법은 해당 도구 디렉터리, 용어는 `terminology`, 문제 해결은 `troubleshooting`처럼 가장 자연스러운 곳을 찾습니다.
- 파일명 규칙은 저장소 전체를 강제로 통일하기보다, 같은 디렉터리의 주변 파일 관례를 우선 따릅니다.
- 주제가 너무 넓으면 한 문서에 모든 것을 욱여넣기보다, 허브 문서와 세부 문서를 나누고 서로 연결합니다.
- 반대로 지나치게 잘게 쪼개면 다시 찾기 어려우므로, "한 문서가 하나의 안정적인 질문에 답하는가"를 기준으로 분할합니다.

## 6. 이 저장소에서 AI가 피해야 하는 것

- 근거 없는 자신감
- 링크만 모아 둔 문서
- 정의만 있고 왜 중요한지 없는 문서
- 예제 없이 추상 설명만 반복하는 문서
- 기존 문서 조사 없이 비슷한 파일을 계속 늘리는 작업
- 실행하지 않은 명령, 확인하지 않은 버전 차이, 검증하지 않은 사실을 단정하는 서술
- 사용자의 이해 기준보다 AI식 정리 체계를 우선해 문서를 낯설게 만드는 것

## 7. 문서 작업 최소 완료 게이트

아래가 모두 닫혀야 문서 작업을 COMPLETE로 봅니다.

- 문서 위치와 범위가 기존 저장소 구조와 충돌하지 않습니다.
- 기존 관련 문서와의 중복 또는 역할 차이가 정리되었습니다.
- 미래의 내가 다시 읽어도 이해를 복원할 수 있는 설명 흐름이 있습니다.
- 해당하는 경우 실전형 예제나 확인 경로가 포함되어 있습니다.
- 핵심 주장에 근거가 있거나, 근거가 없으면 추론/미확인 상태가 명시되어 있습니다.
- 필요한 검증을 수행했고 범위를 솔직하게 적었습니다.
- 전역 규범상 필요한 경우 최종 검수와 commit까지 마쳤습니다.
