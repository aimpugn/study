# AGENTS.md — study 로컬 실행 규범

> 적용 범위: 이 저장소 전체
> 전역 SSOT: `~/.codex/AGENTS.md`
> 로컬 WORK 템플릿: [`AGENTS_WORK_TEMPLATE.md`](AGENTS_WORK_TEMPLATE.md)

이 저장소는 `strict-io-task-request` 스타일의 입력 형식을 강제하는 프로젝트가 아닙니다.
사용자는 자연어로 요청하면 되고, AI는 내부적으로 `rigorous-task`에 가까운 집행 규율을 적용합니다.

이 저장소의 목적은 "배운 것을 모아두는 것"이 아니라, 시간이 지난 뒤에도 다시 이해를 복원하고, 실제 문제에 적용하고, 더 나은 개발자의 사고 방식을 체득할 수 있는 지식 자산을 만드는 것입니다.
좋은 결과는 그럴듯한 요약이 아니라, 근거와 논리가 닫혀 있고, 반례에 견디며, 다시 검증할 수 있고, 다른 도메인에도 전이 가능한 문서와 예제를 남기는 것입니다.

## 1. 최상위 목표

1. 객관적 탁월함: 주장보다 근거와 검증을 우선합니다.
2. 개념적 완결성: 국소 수정이 아니라 영향 범위를 동기화합니다.
3. 간결·직관: 코드는 단순하게, 설명은 재현 가능하게 작성합니다.
4. 전역 실패-폐쇄: 규칙 미충족이면 출력, 완료, 커밋을 금지합니다.

위 네 가지는 취향이 아니라 판정 기준입니다.
우열은 문장 수나 인상 비평이 아니라, 근거 강도, 논리 도약의 적음, 반례 내성, 재현 가능성, 검증 가능성으로 판단합니다.

### 1.1 기준선은 현재 평균이 아니라 최고 수준이다

- 이 저장소의 현재 문서 평균 품질은 기준선이 아닙니다.
- 지금 잘 안 써진 문서, 근거가 약한 문서, 설명이 얕은 문서는 "현재 관행"이 아니라 "개선해야 할 부채"로 취급합니다.
- 로컬 관행보다 더 중요한 기준은 다음 두 가지입니다.
  - 저장소 안에서 가장 잘 쓰인 exemplar 문서
  - 해당 주제에서 객관적으로 더 강한 외부 1차 자료, 표준, 레퍼런스, 탁월한 전문가 수준의 설명
- 따라서 "이 저장소에서는 원래 이렇게 쓴다"는 이유만으로 낮은 기준을 유지하면 안 됩니다.
- 문서를 새로 쓰거나 고칠 때는 현재 상태를 맞추는 것이 아니라, 가장 잘 된 문서 수준으로 끌어올리는 쪽을 기본값으로 둡니다.
- 이 저장소에서 속도는 1차 목표가 아닙니다. 더 오래 걸리더라도 더 정확하고, 더 논리적이고, 더 근거가 튼튼하고, 더 반례에 강한 결과를 우선합니다.
- 이 저장소의 일반 문서 기준선은 우선 `algorithms`, `computer_architecture`, `linux`, `git` 축에서 찾습니다. 이 디렉터리들은 질문에서 시작해 배경, 원리, 예시, 비교, 실무 감각까지 길게 연결하는 기본 장문 설명 스타일을 가장 잘 드러냅니다.
- `knowledge/cards`는 일반 문서의 기본 형식이 아닙니다. 이 영역은 AI가 안정적으로 다시 참고할 수 있는 지식 베이스이자, 한 질문 또는 한 판단을 강하게 닫아야 할 때 쓰는 카드형 포맷으로 봅니다.
- 현재 canonical exemplar는 우선 아래 파일들을 기준으로 삼습니다.
  - `computer_architecture/ostep/xv6-riscv/kernel/entry.S`
  - `algorithms/dynamic_programming.md`
  - `algorithms/optimal_solution.md`
  - `git/git_rebase.md`
  - `computer_architecture/threads/threads.md`
  - `jvm/java/java_synchronized.md`
- `.tmp/interview*.md`, `.tmp/interviews*.md`, `interviews/interview_questions*.md`처럼 크고 거친 수집 문서는 중요한 원재료일 수 있지만, 기본 exemplar나 문체 기준선으로 바로 승격하지 않습니다. 이런 문서는 source reservoir로 보고, 질문 cluster를 뽑아 정식 문서로 promotion 하는 쪽을 기본값으로 둡니다.

## 2. 저장소 정체성과 기본 해석

- 이 저장소의 주 산출물은 Markdown 문서, 예제 코드, 작은 실험, 비교 메모, 지식 카드입니다.
- 대부분의 작업은 `research`, `analysis`, `design`, `explain`, `audit`, `execute`의 혼합으로 봅니다.
- 사용자가 특정 주제를 "정리", "설명", "보강", "검증", "리팩토링"해 달라고 하면, 특별히 `채팅으로만`, `파일 수정 없이`, `report only`, `no commit`이라고 하지 않는 한 저장소 자산을 실제로 수정하는 작업으로 해석합니다.
- 사용자가 구조화 필드를 주지 않아도 AI가 내부적으로 `goal`, `refs`, `scope`, `mode`, `run_mode`, `finish`, `must_keep`, `extra_checks`를 복원합니다.
- 이 저장소의 작업은 "답을 대신 써 주는 일"이 아니라, 사용자가 탁월한 개발자의 판단 과정을 내재화하도록 돕는 일입니다.
- 따라서 현재 사용자의 지식 수준이나 기존 문서의 미완성 상태를 기준선으로 삼지 않습니다. 목표는 "지금 부족한 부분을 그대로 반영하는 문서"가 아니라 "배우려는 대상의 수준에 맞는 문서"입니다.

## 3. 이 저장소에서 AI가 흉내 내야 하는 탁월한 개발자의 사고 패턴

AI는 결과만 가져오지 말고, 아래 사고 패턴을 문서와 예제 안에 남겨야 합니다.

- 먼저 근본 문제를 고정하고, 그 다음 구현이나 세부 설명으로 내려갑니다.
- 하나의 주제를 설명할 때 애플리케이션, 프레임워크, 런타임, OS, CPU처럼 여러 추상화 계층을 필요할 만큼 연결합니다.
- 경계와 불변식을 먼저 드러내고, 무엇이 포함되고 제외되는지 분명히 합니다.
- 사실, 추론, 불확실성을 섞지 않습니다.
- 첫 번째 떠오른 설명이나 설계를 그대로 채택하지 말고, 반례와 대안을 검토합니다.
- "왜 이게 맞는가"만이 아니라 "언제 틀릴 수 있는가"와 "어떻게 확인하는가"까지 닫습니다.
- 헷갈리기 쉬운 쌍은 대조해서 설명합니다. 예: blocking vs non-blocking, sync vs async, current vs incoming, base branch vs rebasing branch
- 국소 사례를 일반 원리로 연결해, 다른 언어·프레임워크·도메인으로 옮겨 생각할 수 있게 합니다.
- 멋있어 보이는 복잡한 구조보다, 더 단순하고 일관되며 검증하기 쉬운 모델을 선호합니다.
- 오타, 용어 불일치, 계약 드리프트 같은 사람 실수를 구조적으로 줄이는 방향을 선호합니다.
- 현재 저장소의 기존 평균 수준이 아니라, 가장 잘 된 exemplar와 외부 최고 수준 설명을 목표점으로 삼습니다.
- 더 짧거나 더 빨리 끝나는 경로가 있더라도, 그 경로가 근거 강도, 논리적 완결성, 재현 가능성, 검증 밀도를 낮추면 채택하지 않습니다.

## 4. 작업 깊이와 기본 실행 루프

이 저장소의 작업은 크게 `minimal`, `standard`, `full` 깊이로 봅니다.

- `minimal`: 단순 오탈자 수정, 아주 작은 문장 다듬기, 위험이 거의 없는 기계적 정리
- `standard`: 단일 주제 보강, 몇 개 문서와 예제를 함께 손보는 작업, 중간 정도의 검증 필요 작업
- `full`: 개념 문서 신설, 표준/프로토콜 정리, 아키텍처 비교, 광범위한 문서 재구성, 용어 정리, 근거 재검증, 여러 자산이 함께 움직여야 하는 작업

다음 중 하나라도 해당하면 기본값을 `full`로 둡니다.

- 주제가 저장소 전체 학습 품질에 영향을 줍니다.
- 설명 품질, 근거 강도, 반례 내성이 핵심입니다.
- 여러 문서, 예제, 허브 문서, 용어 문서가 같이 움직여야 합니다.
- 사용자가 엄격함, 엄밀함, 근거, 검증, 객관적 탁월함을 명시했습니다.
- 잘못 정리하면 장기적으로 학습 비용이 커집니다.

`full` 작업의 기본 루프는 아래 순서를 따릅니다.

1. 요청 정규화와 근본 문제 고정
2. 주제 분석
3. 분석 비판 + 보강
4. 근거 수집
5. 근거 비판 + 보강
6. design
7. 설계 비판 + 보강
8. 전체 계획
9. 계획 비판 + 보강
10. 상세 작업 계획
11. 상세 계획 비판 + 보강
12. execute
13. verification
14. 최종 감사와 마감

새 근거가 앞선 가정을 깨면 가장 이른 영향을 받은 단계로 되돌아가 다시 닫습니다.

속도는 위 루프가 모두 닫힌 뒤에만 부차적으로 고려합니다.
근거 수집, 반례 검토, 설계 비교, 검증, 최종 감사 중 하나라도 줄여야만 빨라지는 경로라면 그 속도 이점은 채택 근거가 아닙니다.

## 5. WORK 문서와 집행 기록

- 규모가 있는 작업은 기본적으로 [`AGENTS_WORK_TEMPLATE.md`](AGENTS_WORK_TEMPLATE.md)를 사용해 집행 기록을 남깁니다.
- 사용자가 별도 WORK 파일 생성을 원하지 않거나 작업이 충분히 작으면, 최소한 그에 준하는 내부 ledger를 유지합니다.
- WORK의 목적은 양식 채우기가 아니라, 근거, 결정, 반박, 검증, 남은 리스크를 재구성 가능하게 남기는 것입니다.
- 체크리스트는 실행 전에 초안 작성 -> 품질 검수 -> freeze -> 실행 후 재판정 순서를 따라야 합니다.
- 사용자 체크리스트나 명시 요구는 최소 바닥선이며, 삭제하거나 약화할 수 없습니다.

## 6. 근거와 소스 계약

### 6.1 소스 우선순위

하중을 지탱하는 주장은 가능하면 아래 우선순위로 지지합니다.

1. 저장소 내부 실험, 예제 실행 결과, 코드, 테스트, 명령 결과
2. 공식 문서, 표준 문서, 언어/플랫폼 레퍼런스, man page
3. 신뢰할 수 있는 1차 자료 또는 레퍼런스 구현
4. 고신뢰 전문가 자료
5. 명시적 추론

### 6.2 사실 / 추론 / 미확인 구분

- 직접 확인한 내용은 사실로 씁니다.
- 아직 실험하지 못했거나 1차 자료로 닫지 못한 내용은 `추론`, `가정`, `추가 확인 필요`로 낮춰 씁니다.
- 서로 다른 소스가 충돌하면 충돌 사실과 각각의 출처를 함께 적습니다.
- 저장소 내부 문서를 다시 근거로 재사용할 때는, 그 문서가 어떤 1차 자료나 실험에 기대는지 확인합니다.

### 6.3 검증 경로 의무

핵심 주장에는 최소한 하나의 검증 경로가 있어야 합니다.

- 무엇을 실행하거나 확인할지
- PASS가 무엇인지
- FAIL이 무엇인지

`확인해 보면 된다`는 검증 경로가 아닙니다.

## 7. 범위 확장과 영향 범위 동기화

새 문서를 만들거나 기존 문서를 크게 바꿀 때는 보이는 파일 하나만 보지 않습니다.

- 검색은 원문 용어에서 시작하되 동의어, 약어, 원어, 구명칭, 관련 명령어, 표준 번호, 에러 문구, 상위 개념, 하위 개념까지 확장합니다.
- `영향 없음`, `관련 없음`, `N/A`는 근거가 있는 판정이어야 합니다.
- 같은 계약이나 같은 어휘를 공유하는 자산은 함께 점검합니다.

특히 아래 표면을 같이 봅니다.

- 관련 Markdown 문서
- 허브 문서, README, index 성격 문서
- 예제 코드, 샘플 디렉터리, 실험 파일
- troubleshooting 문서
- terminology 문서
- knowledge card
- 코드/테스트/스크립트가 설명의 근거로 쓰이는 경우 그 자산

한쪽만 바꾸면 어떤 오해나 드리프트가 생기는지도 적어야 합니다.

## 8. 설명과 학습 품질 계약

이 저장소는 "읽고 고개를 끄덕이게 만드는 문서"보다 "나중에 혼자 다시 설명하고 검증하게 만드는 문서"를 목표로 합니다.

### 8.1 Narrative First

- 설명은 prose-first를 기본으로 합니다.
- 결론과 이유는 같은 흐름 안에 있어야 합니다.
- 목록은 순서, 비교, 체크리스트처럼 목록 자체가 의미 있을 때만 씁니다.
- 내부 품질 프로토콜은 독자에게 억지 라벨로 노출하지 않습니다.
- 길이가 있고 구조가 중요한 학습 문서는 간단한 목차를 두는 편을 기본값으로 둡니다. 독자가 목차만 보고도 "이 문서가 무엇을 어디까지 설명하는지" 감을 잡을 수 있어야 합니다.
- 긴 학습 문서는 가능하면 `개요` 또는 동등한 서두를 두고, 이 문서가 정확히 무엇을 설명하는지 바로 고정합니다.
- `가장 먼저 머릿속에 고정해야 할 것은`, `실무에서 자주 언급되는`, `갈립니다`처럼 설명자의 진행 멘트가 전면에 오는 표현보다, 주제를 곧바로 설명하는 직접 진술형 문장을 우선합니다.
- 이후 설명이 어떤 규약, 스펙, 파일 구조, 실제 산출물에 기대고 있다면, 그 load-bearing artifact는 가능한 한 초반에 실제 예시와 함께 보여 줍니다. 예: `META-INF/MANIFEST.MF`, `BOOT-INF/lib/`, 실행 파일 포맷, 실제 로그, 실제 명령 출력.
- 구조 예시, 디렉터리 트리, manifest 조각, 로그 조각, 패킷, 설정 블록을 보여 준 뒤에는 "구조가 보인다" 같은 감상형 문장으로 넘기지 않습니다. 먼저 보인 구성요소를 하나씩 설명하고, 그 다음 그 구성요소들이 어떻게 엮여 상위 메커니즘을 만드는지 설명합니다.

### 8.2 설명이 기본적으로 닫아야 하는 것

표면 라벨은 강제하지 않지만, 아래 실질 요소는 해당하는 만큼 닫아야 합니다.

- 지금 무엇을 말하는가
- 왜 지금 중요한가
- 어떤 사실과 제약이 이 결론을 지지하는가
- 어떻게 직접 확인할 수 있는가
- 어떤 조건에서 깨지는가
- 어떤 대안이 있었고 왜 이번 선택이 더 나은가
- 다른 도메인으로 옮겨 생각할 수 있는 원리가 무엇인가

### 8.3 질문 중심, 계층형, 비교형 설명을 적극 활용한다

- 좋은 문서는 정의를 던지는 문서보다 "이 질문에 답한다"는 형태로 시작하는 문서인 경우가 많습니다.
- 질문형 제목, 짧은 직답, 이어지는 상세 설명 구조를 허용하고, 학습 효과가 높다면 오히려 권장합니다.
- 한 층위의 설명만으로 오해가 남는 주제는 CPU, 메모리, OS, 런타임, 프레임워크, 애플리케이션 순으로 계층을 오가며 닫습니다.
- 자주 혼동되는 개념은 별도 비교 절이나 대칭 구조로 정리합니다.
- 대조 구조를 만들 때는 양쪽 모두에 대해 "정의 -> 동작 -> 관측 -> 실패/주의점"의 밀도를 비슷하게 맞춥니다.
- 막 소개한 차이, 구분, 대조쌍을 이해하는 데 꼭 필요한 설명은 뒤로 미루지 않습니다. 현재 문단에서 이해가 막히는 지점이라면 같은 문단이나 바로 다음 문장에서 즉시 풀어 씁니다.
- `그 사이`, `이 과정`, `위 구조`, `이것`처럼 핵심 메커니즘을 가리키는 지시어는 그대로 두지 말고, 무엇과 무엇 사이인지, 어떤 구조를 말하는지 바로 풀어 씁니다.
- 작은 수의 닫힌 비교 항목을 설명할 때는 필요하면 번호 목록을 사용하고, 각 항목 아래 문장을 4칸 들여쓴 prose로 이어 써서 항목 의미와 설명이 함께 보이게 합니다.
- 독자가 자연스럽게 "운영 체제가 기계인데 어떻게 실행하지?", "`META-INF/MANIFEST.MF`는 누가 정했지?", "`libjvm`은 뭐지?" 같은 질문을 할 만한 지점이라면, 한 단계 더 아래층으로 내려가 최소 설명을 제공합니다. 단, 현재 질문을 닫는 데 필요한 범위까지만 내려갑니다.

### 8.4 기초부터 쌓는 Deep Study Monograph를 기본값으로 둔다

- 이 저장소의 일반 학습 문서는 얕은 요약문보다 `deep study monograph`에 가깝게 작성하는 쪽을 기본값으로 둡니다.
- 좋은 문서는 "이미 아는 사람의 압축 메모"가 아니라, 아직 모르는 사람이 가장 아래 기초부터 차근차근 벽돌을 쌓듯 따라 올라갈 수 있는 구조를 가집니다.
- 따라서 설명은 가능하면 아래 흐름을 따릅니다.
  - 지금 답하려는 질문 또는 직답
  - 이 개념이 왜 생겼는지, 무엇을 해결하려는지, 어떤 맥락과 역사에서 등장했는지
  - 가장 작은 작동 단위 또는 가장 낮은 전제
  - 작은 예제, 최소 실험, line-by-line walkthrough 중 최소 하나
  - 실제 동작 메커니즘
  - 더 큰 구조 또는 실무 맥락으로의 확장
  - 혼동하기 쉬운 인접 개념과의 비교
  - 실패 모드, 한계, 검증 경로
- 넓은 주제를 설명할 때도 곧바로 고급 추상화부터 던지지 않습니다. 먼저 baseline, 가장 작은 예, 가장 단순한 상태 전이, 또는 첫 명령부터 시작해 독자의 mental model을 세운 뒤 확장합니다.
- 설명이 구조나 구성요소를 다룰 때는 가능하면 아래 순서를 따릅니다.
  - 작은 구성요소를 보여 준다
  - 각 구성요소의 역할을 설명한다
  - 구성요소들이 서로 어떻게 연결되는지 설명한다
  - 그 연결이 왜 상위 동작으로 이어지는지 설명한다
- 코드가 핵심 근거인 주제는 line-by-line 또는 block-by-block 설명을 허용하고, 특히 부팅 코드, low-level runtime, 동기화, 메모리 모델처럼 숨은 전제가 많은 주제는 첫 상태와 첫 가정을 먼저 고정해야 합니다.
- 최종 목적은 "정보를 전달했다"가 아니라, 사용자가 나중에 이 주제를 자기 말로 다시 설명하고 자기 손으로 다시 검증할 수 있게 만드는 것입니다.

### 8.5 학습 전이성과 기억 anchor

- 사용자가 나중에 스스로 설명할 수 있게, 필요하면 작은 예제, before/after, 짧은 비유, active recall 질문, 최소 실험을 넣습니다.
- 다만 학습 장치를 의식적으로 남기는 것이 실제 이해에 도움이 될 때만 사용합니다.
- 설명용 예제와 실전형 예제를 구분해서 배치합니다.

### 8.6 공식 용어와 쉬운 한국어의 병행

- 먼저 쉬운 한국어로 개념을 고정하고, 필요하면 공식 용어를 괄호로 병기합니다.
- 독자가 공식 문서까지 역추적할 수 있도록 용어 연결은 남기되, 설명 자체를 전문 용어의 낭독으로 만들지 않습니다.
- 사용자가 특정 표기, 연결 방식, 말투, 목록 형식에 대해 명시적으로 피드백하면, 그 피드백은 이후 같은 유형의 문서에서 우선 반영할 설명 계약으로 취급합니다. 다만 표현 그 자체보다 독자의 오해를 줄이고 재구성을 쉽게 만드는 구조적 규칙으로 일반화하는 편을 우선합니다.
- 현재 실행 환경에서 직접 확인한 사실과 다른 플랫폼까지 포함하는 일반 원리를 함께 다룰 때는 둘을 섞지 않습니다. 예: "이 macOS 머신에서는 Mach-O가 보인다. Linux에서는 보통 ELF를 본다."처럼 현재 관측과 일반 규칙을 분리해 적습니다.

## 9. 예제와 실험 계약

- 예제는 추상 변수명보다 실제 입력값, 명령, 로그, 데이터 모양, before/after를 우선합니다.
- 가능하면 "개념을 이해시키는 예제"와 "실무에서 바로 써먹는 예제"를 함께 둡니다.
- 명령어나 코드 예제를 넣을 때는 목적, 전제, 기대 결과, 실패 신호를 같이 적습니다.
- 샘플 코드나 실험 디렉터리를 추가하면 "왜 존재하는지", "어떻게 실행하는지", "무엇을 관찰해야 하는지"를 남깁니다.
- 실행하지 못한 예제는 실행한 것처럼 쓰지 않습니다.

## 10. 문서 배치와 구조 선택

- 새 파일을 만들기 전에 기존 분류 체계 안에서 가장 자연스러운 위치를 먼저 찾습니다.
- 같은 질문에 이미 답하는 문서가 있다면, 새 파일을 늘리기보다 기존 문서를 보강하거나 역할 차이를 명확히 분리합니다.
- 너무 넓은 주제는 허브 문서와 상세 문서로 나누고, 너무 잘게 쪼개진 문서는 재통합을 고려합니다.
- `knowledge/cards` 아래 문서는 [`knowledge/KNOWLEDGE_TEMPLATE.md`](knowledge/KNOWLEDGE_TEMPLATE.md)를 우선 따릅니다.
- `knowledge/cards`는 AI 지식 베이스와 판단형 카드의 역할을 우선합니다. 일반 개념 문서, 명령/도구 문서, troubleshooting, 장문 비교 문서는 기본적으로 `algorithms`, `computer_architecture`, `linux`, `git` 축의 설명 감각을 기준으로 삼습니다.
- 기존 문서가 있다 해도, 그 문서의 현재 수준이 낮다면 그것을 기준으로 삼지 않습니다. 기존 문서는 출발점일 뿐이며, 목표 수준은 항상 더 높은 exemplar 쪽입니다.

문서 유형에 따라 기본 흐름은 달라집니다.

다만 이 저장소의 일반 학습 문서는 가능한 한 `deep study monograph` 감각을 유지해야 합니다.
즉, 주제를 넓게 다루더라도 "배경/역사 -> 가장 작은 작동 단위 -> 최소 예제 -> 메커니즘 -> 확장 -> 비교 -> 검증"의 뼈대가 보여야 합니다.

- 개념 문서: 질문 -> 왜 중요한가 -> 핵심 메커니즘 -> 예제 -> 오해/반례 -> 검증 -> 근거
- 명령/도구 문서: 무엇을 하는가 -> 입력/옵션 감각 -> 안전한 예제 -> 실무 예제 -> 실패/주의점 -> 검증 -> 근거
- troubleshooting 문서: 증상 -> 직접 원인 -> 근본 원인 -> 해결 -> 재발 방지/확인 -> 근거
- 표준/프로토콜 문서: 규칙 범위 -> 원문 의미 -> 구현 해석 포인트 -> 예시 -> 실패/주의점 -> 검증 -> 근거
- 인터뷰/질문형 문서: 질문 -> 짧은 직답 -> 계층별 설명 -> 구체 예시 -> 흔한 꼬리 질문 또는 반례 -> 검증/근거

- `.tmp/interview*.md`, `.tmp/interviews*.md`, `interviews/interview_questions*.md` 같은 대형 수집 문서는 기본 문서 형식으로 그대로 다듬기보다, 먼저 질문 cluster와 개념 cluster를 분리해 정식 문서 후보로 승격합니다. 이런 파일은 기본 exemplar가 아니라 원재료 풀로 봅니다.

## 11. 금지 패턴

- 근거 없는 단정
- 링크만 모아 둔 문서
- 정의만 있고 왜 중요한지 없는 문서
- 예제 없이 추상 설명만 반복하는 문서
- 첫 검색 결과만 보고 범위를 닫는 작업
- 반례 검토 없이 첫 설계를 곧바로 정답처럼 확정하는 일
- 실행하지 않은 검증을 실행한 것처럼 쓰는 일
- 사용자 명시 요구를 일부만 수행하고 완료로 선언하는 일
- 설명을 읽는 사람이 다시 조립해야 하는 라벨식 나열
- 지나친 축약 때문에 추상화 계층 사이 연결이 끊긴 설명
- 현재 저장소의 낮은 품질 문서를 암묵적 표준처럼 취급하는 일

## 12. 검증, 완료, 커밋

### 12.1 기본 완료 조건

아래가 모두 닫혀야 COMPLETE입니다.

- 사용자 명시 요구 전부 PASS
- frozen checklist 전부 PASS
- 핵심 주장에 근거 또는 명시적 추론 경로 존재
- 필요한 범위 조사와 영향 범위 동기화 완료
- 필요한 검증 완료
- 최종 감사에서 목적 적합성, 설명 품질, 검증 정직성 PASS

### 12.2 repo 변경 작업의 기본 closure

- 이 저장소에서 파일을 바꾸는 실행형 작업은 기본적으로 `최종 감사 -> 필요한 검증 통과 -> commit`으로 닫습니다.
- 사용자가 `no commit`, `report only`, `커밋하지 마세요`라고 명시하면 그때만 commit을 제외합니다.
- push는 명시 요청이 있을 때만 합니다.

### 12.3 정직한 상태 모델

- `COMPLETE`: 필수 게이트가 모두 닫힘
- `PARTIAL`: 방향은 맞지만 필수 게이트 일부가 열려 있음
- `BLOCKED`: 접근 권한, 부족한 근거, 외부 의존성 등으로 신뢰 가능한 closure 불가

열린 게이트가 하나라도 남아 있으면 낙관적 표현으로 숨기지 않습니다.
