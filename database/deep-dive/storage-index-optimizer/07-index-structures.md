## B+tree 구조와 탐색/삽입/분할

등장 배경을 먼저 잡으면 B+tree가 왜 지금도 기본 인덱스의 중심에 있는지 이해하기 쉽다. 디스크와 SSD는 CPU register처럼 임의 위치를 공짜로 읽지 못하고, DB는 한 row만 찾는 일과 정렬된 범위를 이어 읽는 일을 동시에 처리해야 했다. 그래서 균형 잡힌 tree로 높이를 낮추고, leaf 쪽에 정렬된 key를 모아 page 단위 I/O와 range scan을 함께 만족시키는 구조가 필요해졌다. 이 역사적 압력 때문에 B+tree는 단순한 검색 자료구조가 아니라 저장 장치, page, cache, 정렬 요구가 타협한 결과로 읽어야 한다.

독자는 B+tree가 왜 동등 조회뿐 아니라 범위 조회와 정렬 조회에도 강한지, 그리고 page split이 왜 쓰기 비용과 공간 사용률을 동시에 건드리는지 손으로 설명할 수 있어야 한다. 이 절은 정의 목록이 아니라 작은 입력이 인덱스나 실행기 내부에서 어떤 순서로 움직이는지 따라가는 설명이다. 읽은 뒤에는 문서의 문장을 외우는 것이 아니라, 같은 종류의 쿼리를 보았을 때 어떤 구조가 먼저 일을 시작하고 어떤 관측값을 봐야 하는지 말할 수 있어야 한다.

로컬 seed `database/mysql/order.md`는 ORDER BY가 없으면 SQL 결과 순서가 보장되지 않는다는 문제의식을 준다. 이 DU는 그 질문을 인덱스가 순서를 만들어 줄 수 있는 조건과 그렇지 않은 조건으로 확장한다. 기존 seed는 짧거나 특정 상황에 묶여 있으므로 그대로 복사하지 않는다. 대신 seed가 품고 있던 질문을 살리고, 공식 문서가 닫아 주는 사실과 실행 trace가 보여 주는 관측 경로를 합쳐 새 본문으로 확장한다.

공식 근거는 다음 경계 안에서 사용한다.

- PostgreSQL 18 공식 문서의 Chapter 11 Indexes와 Index Types는 B-tree가 기본 인덱스이고, 정렬 가능한 값의 동등/범위 비교와 정렬 순서 반환에 맞는다는 점을 기준 근거로 둔다.
- MySQL 8.4 공식 문서의 InnoDB clustered and secondary indexes는 clustered index와 secondary index가 row 접근을 어떻게 나누는지 설명하는 기준 근거다.

첫 벽돌은 `orders(id, customer_id, ordered_at, amount)` 테이블에 `(customer_id, ordered_at)` 인덱스가 있고, 한 고객의 주문을 시간순으로 읽는 상황이다. 이 작은 장면을 붙잡아야 추상 용어가 실제 비용으로 바뀐다. 데이터베이스의 성능 설명은 보통 `빠르다`, `느리다`, `인덱스를 탄다` 같은 말로 시작하지만, 실제 진단은 어떤 row나 key가 어떤 구조를 거쳐 다음 소비자에게 전달되는지 확인하는 일이다.

```text
leaf page before insert, max 3 keys for hand trace

root: [customer_id=20]
      /                 \
L1: [10,2026-01-01] [10,2026-01-03] [12,2026-01-02]
L2: [20,2026-01-01] [20,2026-01-04] [25,2026-01-01]

insert key: [20,2026-01-02]

step 1. root comparison says 20 >= 20, so descend to L2
step 2. leaf order becomes [20,01-01] [20,01-02] [20,01-04] [25,01-01]
step 3. leaf is over capacity, split around middle

after split
root: [20,2026-01-04]
      /                          \
L2a: [20,01-01] [20,01-02]  -->  L2b: [20,01-04] [25,01-01]

range scan customer_id=20 and ordered_at between 01-01 and 01-04
root -> L2a -> read [20,01-01], [20,01-02] -> follow leaf link -> read [20,01-04]
```


B+tree에서 내부 node는 길 안내판에 가깝고, 실제 row를 가리키는 항목은 leaf 쪽에 모인다. 그래서 point lookup은 root에서 leaf까지 비교를 반복해 내려가고, range lookup은 시작 leaf를 찾은 뒤 leaf 사이의 순서를 따라 옆으로 이동한다.

PostgreSQL 문서가 말하는 B-tree의 강점은 정렬 가능한 값에 대한 `=`, `<`, `<=`, `>=`, `>` 비교다. 이 말은 단순히 비교 연산자가 많다는 뜻이 아니라, 키가 정렬되어 있으므로 시작점 탐색과 이후 순차 소비가 같은 구조 안에서 가능하다는 뜻이다.

InnoDB의 clustered index는 primary key 순서로 row 자체를 담는 중심 구조다. secondary index는 secondary key와 함께 primary key 값을 들고, 최종 row가 필요하면 clustered index로 다시 찾아간다. 이 차이를 모르면 secondary index lookup이 항상 한 번의 random read라고 착각하기 쉽다.

page split은 새 key 하나를 넣기 위해 전체 나무를 다시 만드는 일이 아니다. 넘친 leaf를 둘로 나누고, 부모에 경계 key를 올리며, 필요하면 부모도 다시 나뉜다. 그래서 쓰기 부하는 국소적이지만, 연속 insert 패턴과 random insert 패턴에 따라 split 위치와 page locality가 달라진다.

ORDER BY가 없는 SELECT의 결과 순서를 인덱스 leaf 순서와 혼동하면 안 된다. 실행 계획이 인덱스를 사용하지 않을 수 있고, 인덱스를 쓰더라도 병렬 실행, join, sort, heap fetch, optimizer 선택 때문에 최종 결과 순서는 SQL의 ORDER BY만이 계약한다.

이 흐름에서 B+tree를 만능 빠른 길로 보지 않는 태도가 중요하다. leaf까지 내려가는 비용, leaf를 옆으로 읽는 비용, split이 부모로 전파되는 비용, secondary index에서 clustered row로 되돌아가는 비용은 서로 다른 관측값으로 나타난다. 그래서 B+tree 설명은 항상 key 비교로 시작점을 찾고, leaf 순서로 범위를 소비하며, 쓰기 시점에는 page 단위 재배치가 생긴다는 세 장면으로 되돌아가야 한다.

### 손으로 다시 읽는 예제

1. 범위 조회

    ```sql
    SELECT * FROM orders WHERE customer_id = 20 AND ordered_at >= DATE '2026-01-01' AND ordered_at < DATE '2026-02-01' ORDER BY ordered_at;
    ```

    인덱스 앞쪽 열 customer_id로 시작 leaf를 좁히고, ordered_at 순서가 leaf 안에 이미 있으므로 정렬 부담을 줄일 수 있다.
1. 정렬 착각

    ```sql
    SELECT * FROM orders WHERE customer_id = 20;
    ```

    이 쿼리는 인덱스를 탈 수도 있지만 결과 순서 계약이 없다. 결과가 지금 ordered_at처럼 보이는 것은 관측일 뿐이다.
1. secondary lookup

    ```sql
    SELECT amount FROM orders WHERE email_hash = ?;
    ```

    email_hash secondary index가 primary key를 찾고, amount가 secondary index에 없으면 clustered record로 다시 간다.

위 예제들은 B+tree의 장점이 `인덱스를 탄다`는 말 하나로 끝나지 않는다는 점을 보여 준다. 범위 조회는 시작 leaf와 leaf 연결을 소비하고, 정렬 착각 예제는 SQL 계약과 현재 관측을 분리하며, secondary lookup 예제는 인덱스 entry와 실제 row 접근이 두 단계일 수 있음을 드러낸다.

### 실무에서 먼저 의심해야 하는 함정

- 인덱스를 hashmap처럼만 생각하면 `BETWEEN`, `ORDER BY`, `LIMIT`에서 왜 B+tree가 강한지 놓친다. hashmap mental model은 동등 조회 하나에는 좋지만, 다음 key가 무엇인지 알려 주지 않는다.
- page split을 장애처럼만 보면 write amplification의 신호를 놓친다. split이 늘어난다는 것은 insert key 분포, fill factor, PK 설계, hot page 경쟁을 함께 봐야 한다는 운영 신호다.
- clustered index와 secondary index의 차이를 모르고 인덱스만 추가하면 조회 한 건마다 clustered lookup이 붙어 실제 I/O가 기대보다 크다.

B+tree 함정은 대개 순서와 위치를 과소평가할 때 생긴다. hashmap식 사고는 범위와 정렬을 지우고, page split을 무시하면 쓰기 부하와 hot page를 놓치며, secondary index를 한 번의 lookup처럼 보면 clustered row 접근 비용이 사라진 것처럼 착각한다.

### 관측과 검증 경로

- PostgreSQL에서는 `EXPLAIN (ANALYZE, BUFFERS)`로 Index Scan이 시작 조건과 필터 조건을 어떻게 나누는지 확인한다. PASS 신호는 Index Cond가 범위를 좁히고, Sort node가 사라지거나 rows removed by filter가 작아지는 것이다.
- MySQL에서는 `EXPLAIN`에서 key, rows, Extra의 `Using index`, `Using filesort`를 함께 본다. PASS 신호는 기대한 복합 인덱스가 key로 선택되고, rows 추정이 크게 줄며, 불필요한 filesort가 사라지는 것이다.
- 실패 신호는 WHERE 조건은 인덱스를 타는데 ORDER BY 때문에 별도 sort가 생기거나, secondary index lookup 뒤 table row fetch가 폭증하는 것이다.

B+tree 검증은 실행 시간이 아니라 접근 경로와 row 이동이 설명과 맞는지를 본다. PostgreSQL에서는 Index Cond, Sort node, buffer 사용을 보고, MySQL에서는 key, rows, Using filesort, Using index를 함께 본다. PASS는 시작점 탐색과 leaf 순차 소비 또는 clustered lookup 비용이 관측값으로 확인되는 것이다.

### B+tree 구조와 탐색/삽입/분할 현장 판독 노트 1

1-1. `PK 순차 증가` 상황에서는 먼저 단건 조회을(를) 하나의 관측 단위로 분리한다. 오른쪽 끝 leaf에 insert가 몰리면 split 위치와 hot page 경쟁을 관측해야 한다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. AUTO_INCREMENT나 timestamp PK가 항상 나쁘다는 뜻은 아니지만 쓰기 집중을 숨기지 않는다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-2. `UUID random PK` 상황에서는 먼저 범위 조회을(를) 하나의 관측 단위로 분리한다. insert가 여러 leaf로 흩어져 page locality와 cache hit가 달라진다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 분산은 hot page를 줄일 수 있지만 clustered locality를 깨뜨릴 수 있다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-3. `range + limit` 상황에서는 먼저 정렬 조회을(를) 하나의 관측 단위로 분리한다. 시작 leaf를 찾은 뒤 필요한 개수만 읽고 멈출 수 있으면 B+tree 장점이 커진다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. LIMIT가 있어도 시작점이 넓으면 앞쪽 후보를 많이 버릴 수 있다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-4. `ORDER BY 없는 관측` 상황에서는 먼저 삽입 부하을(를) 하나의 관측 단위로 분리한다. 현재 결과가 PK 순서처럼 보여도 SQL 계약은 아니다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 운영 plan이 바뀌는 순간 숨은 의존이 장애가 된다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-5. `secondary index lookup` 상황에서는 먼저 클러스터 lookup을(를) 하나의 관측 단위로 분리한다. leaf entry에서 primary key를 얻은 뒤 clustered row를 다시 읽을 수 있다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. covering이 아니면 두 단계 접근을 비용에 넣는다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-6. `page split 후 부모 갱신` 상황에서는 먼저 페이지 분할을(를) 하나의 관측 단위로 분리한다. leaf split은 부모 separator key 갱신을 동반한다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 부모까지 가득 차면 split이 위로 전파된다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

이 7번째 판독 묶음의 핵심은 같은 설명을 반복하는 것이 아니라, B+tree 구조와 탐색/삽입/분할을(를) 운영 관측 단위로 잘게 나누어 보는 연습이다. 한 쿼리의 평균 시간만 보지 말고, 어떤 입력이 어떤 내부 구조를 지나 어떤 산출물을 만들었는지 작은 trace로 되돌려야 한다. 그 trace가 닫히면 공식 문서의 용어와 로컬 seed의 짧은 메모가 서로 연결되고, 닫히지 않으면 아직 원인을 안다고 말하면 안 된다.

### B+tree 구조와 탐색/삽입/분할 현장 판독 노트 8

8-1. `PK 순차 증가` 상황에서는 먼저 단건 조회을(를) 하나의 관측 단위로 분리한다. 오른쪽 끝 leaf에 insert가 몰리면 split 위치와 hot page 경쟁을 관측해야 한다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. AUTO_INCREMENT나 timestamp PK가 항상 나쁘다는 뜻은 아니지만 쓰기 집중을 숨기지 않는다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

8-2. `UUID random PK` 상황에서는 먼저 범위 조회을(를) 하나의 관측 단위로 분리한다. insert가 여러 leaf로 흩어져 page locality와 cache hit가 달라진다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 분산은 hot page를 줄일 수 있지만 clustered locality를 깨뜨릴 수 있다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

8-3. `range + limit` 상황에서는 먼저 정렬 조회을(를) 하나의 관측 단위로 분리한다. 시작 leaf를 찾은 뒤 필요한 개수만 읽고 멈출 수 있으면 B+tree 장점이 커진다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. LIMIT가 있어도 시작점이 넓으면 앞쪽 후보를 많이 버릴 수 있다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

8-4. `ORDER BY 없는 관측` 상황에서는 먼저 삽입 부하을(를) 하나의 관측 단위로 분리한다. 현재 결과가 PK 순서처럼 보여도 SQL 계약은 아니다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 운영 plan이 바뀌는 순간 숨은 의존이 장애가 된다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

8-5. `secondary index lookup` 상황에서는 먼저 클러스터 lookup을(를) 하나의 관측 단위로 분리한다. leaf entry에서 primary key를 얻은 뒤 clustered row를 다시 읽을 수 있다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. covering이 아니면 두 단계 접근을 비용에 넣는다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

8-6. `page split 후 부모 갱신` 상황에서는 먼저 페이지 분할을(를) 하나의 관측 단위로 분리한다. leaf split은 부모 separator key 갱신을 동반한다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 부모까지 가득 차면 split이 위로 전파된다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 B+tree 구조와 탐색/삽입/분할에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

이 8번째 판독 묶음의 핵심은 같은 설명을 반복하는 것이 아니라, B+tree 구조와 탐색/삽입/분할을(를) 운영 관측 단위로 잘게 나누어 보는 연습이다. 한 쿼리의 평균 시간만 보지 말고, 어떤 입력이 어떤 내부 구조를 지나 어떤 산출물을 만들었는지 작은 trace로 되돌려야 한다. 그 trace가 닫히면 공식 문서의 용어와 로컬 seed의 짧은 메모가 서로 연결되고, 닫히지 않으면 아직 원인을 안다고 말하면 안 된다.

### 오해 수리와 전이 질문

이 절에서 바로잡을 오해는 B+tree가 “빠른 lookup” 하나로 설명된다는 생각이다. B+tree의 진짜 힘은 시작점을 찾은 뒤 leaf 순서로 range를 소비할 수 있다는 데 있고, 그 힘은 ORDER BY 계약, secondary lookup, page split 비용과 함께 읽어야 안전하다.

마지막으로 B+tree 구조와 탐색/삽입/분할을(를) 자기 말로 다시 설명하려면 세 문장을 직접 써 본다. 첫 문장은 이 구조가 해결하는 문제, 둘째 문장은 작은 입력이 내부에서 바뀌는 순서, 셋째 문장은 운영에서 깨지는 조건이다. 세 문장 중 하나라도 공식 문서나 trace로 되돌아가지 못하면 아직 외운 말이고, 세 문장이 모두 trace와 검증 명령으로 이어지면 다음 DU의 실행 계획을 읽을 준비가 된 것이다.


## composite, covering, unique, partial/function index

등장 배경은 단일 열 인덱스만으로는 실제 업무 쿼리의 조건 묶음을 표현하기 어렵다는 데 있다. 운영 화면은 보통 tenant, 상태, 날짜, id, 삭제 여부, 정규화된 이메일처럼 여러 조건을 동시에 요구한다. DB가 각 조건을 독립적으로 조금씩만 줄이면 여전히 많은 row를 읽고 정렬해야 하므로, 복합 인덱스와 covering index는 자주 함께 소비되는 조건과 열을 하나의 접근 경로로 묶기 위해 등장했다. unique, partial, expression index는 여기서 한 걸음 더 나아가 성능뿐 아니라 데이터 불변식과 특정 조건의 작은 후보 집합을 DB 안에 고정하려는 장치다.

독자는 복합 인덱스가 여러 단일 인덱스를 그냥 붙여 놓은 것이 아니라 왼쪽부터 정렬된 하나의 키라는 점, covering read가 왜 row fetch를 줄이는지, unique/partial/expression index가 데이터 계약과 plan 선택을 동시에 바꾼다는 점을 설명할 수 있어야 한다. 이 절은 정의 목록이 아니라 작은 입력이 인덱스나 실행기 내부에서 어떤 순서로 움직이는지 따라가는 설명이다. 읽은 뒤에는 문서의 문장을 외우는 것이 아니라, 같은 종류의 쿼리를 보았을 때 어떤 구조가 먼저 일을 시작하고 어떤 관측값을 봐야 하는지 말할 수 있어야 한다.

로컬 seed `database/mysql/order.md`의 ORDER BY 문제의식은 복합 인덱스에서 더 중요해진다. 어떤 열을 앞에 두느냐에 따라 WHERE, ORDER BY, LIMIT의 비용이 완전히 달라진다. 기존 seed는 짧거나 특정 상황에 묶여 있으므로 그대로 복사하지 않는다. 대신 seed가 품고 있던 질문을 살리고, 공식 문서가 닫아 주는 사실과 실행 trace가 보여 주는 관측 경로를 합쳐 새 본문으로 확장한다.

공식 근거는 다음 경계 안에서 사용한다.

- PostgreSQL 18 공식 문서의 Chapter 11 Indexes와 Index Types는 B-tree가 기본 인덱스이고, 정렬 가능한 값의 동등/범위 비교와 정렬 순서 반환에 맞는다는 점을 기준 근거로 둔다.
- MySQL 8.4 공식 문서의 How MySQL Uses Indexes는 multi-column index, leftmost prefix, covering read를 설명하는 기준 근거다.

첫 벽돌은 결제 목록 화면이다. `payments(merchant_id, status, requested_at, id, amount)`에서 가맹점별 실패 건을 최신순으로 50건 보는 쿼리가 있다. 이 작은 장면을 붙잡아야 추상 용어가 실제 비용으로 바뀐다. 데이터베이스의 성능 설명은 보통 `빠르다`, `느리다`, `인덱스를 탄다` 같은 말로 시작하지만, 실제 진단은 어떤 row나 key가 어떤 구조를 거쳐 다음 소비자에게 전달되는지 확인하는 일이다.

```sql
CREATE INDEX idx_payments_merchant_status_requested_id
  ON payments (merchant_id, status, requested_at DESC, id DESC);

SELECT id, requested_at, amount
FROM payments
WHERE merchant_id = 'm-100'
  AND status = 'FAILED'
ORDER BY requested_at DESC, id DESC
LIMIT 3;
```

```text
composite key order, simplified leaf

(m-100, CANCELLED, 09:01, 7)
(m-100, FAILED,    09:05, 9)  <- start after equality prefix
(m-100, FAILED,    09:03, 8)
(m-100, FAILED,    08:59, 6)
(m-100, PAID,      09:07, 10)
(m-200, FAILED,    09:06, 11)

equality prefix: merchant_id + status
ordered suffix: requested_at DESC + id DESC
consumer: LIMIT reads first 3 matching entries and can stop early
```


복합 인덱스는 열마다 독립적으로 탐색되는 작은 인덱스 묶음이 아니다. `(merchant_id, status, requested_at)` 인덱스는 먼저 merchant_id로 정렬되고, 같은 merchant_id 안에서 status로 정렬되고, 그 안에서 requested_at으로 정렬된다.

이 때문에 왼쪽 열 규칙이 생긴다. 앞쪽 열의 동등 조건이 닫혀야 뒤쪽 열의 정렬성과 범위성이 강하게 살아난다. 앞쪽 열이 넓은 범위가 되면 그 뒤 열은 각 범위 조각 안에서는 정렬되어 있어도 전체 결과를 한 줄 순서로 보장하기 어렵다.

covering index는 쿼리가 필요한 열을 인덱스만으로 충족할 수 있는 상태다. PostgreSQL은 index-only scan을 하더라도 visibility 확인이라는 MVCC 경계를 가진다. MySQL InnoDB에서는 secondary index가 필요한 열과 primary key를 담고 있으면 clustered record 접근을 줄일 수 있다.

unique index는 성능 도구이기 전에 데이터 계약이다. `users(email)` unique는 중복 조회를 빠르게 하는 장치이면서 동시에 같은 email이 두 번 들어오지 않는다는 불변식을 DB가 집행한다.

partial index와 expression index는 PostgreSQL에서 특히 강한 설계 도구다. partial index는 조건을 만족하는 일부 row만 인덱싱하고, expression index는 `lower(email)` 같은 계산 결과를 인덱싱한다. MySQL은 같은 이름의 partial index 개념이 일반적이지 않으므로 generated column이나 functional key part 등 엔진별 대안을 분리해 봐야 한다.

복합 인덱스와 covering index의 핵심은 많은 열을 한꺼번에 넣는 것이 아니라, 실제 쿼리가 소비하는 순서를 인덱스 key 순서와 맞추는 것이다. 동등 prefix가 탐색 공간을 자르고, ordered suffix가 정렬을 대신하고, 포함된 열이 row fetch를 줄일 때만 이 설계가 이긴다. unique나 partial 조건까지 들어가면 성능이 아니라 데이터 계약과 후보 row 집합의 크기도 함께 바뀐다.

### 손으로 다시 읽는 예제

1. 복합 인덱스 성공

    ```sql
    WHERE merchant_id='m-100' AND status='FAILED' ORDER BY requested_at DESC, id DESC
    ```

    앞쪽 두 열이 동등 조건이라 뒤쪽 정렬 열을 그대로 소비할 수 있다.
1. 왼쪽 열 누락

    ```sql
    WHERE status='FAILED' ORDER BY requested_at DESC
    ```

    `merchant_id`가 빠졌으므로 `(merchant_id, status, requested_at)` 전체 순서에서 status만 따로 모여 있지 않다.
1. covering 실패

    ```sql
    SELECT id, requested_at, amount, failure_reason FROM payments ...
    ```

    failure_reason이 인덱스에 없으면 row fetch가 다시 필요하다. covering은 SELECT list까지 포함한 계약이다.

위 예제들은 복합 인덱스에서 열 순서가 곧 실행 순서라는 사실을 보여 준다. 성공 예제는 equality prefix와 ordered suffix가 맞물리고, 왼쪽 열 누락 예제는 같은 인덱스라도 탐색 축이 사라지는 장면을 보여 주며, covering 실패 예제는 SELECT list까지 비용 계약에 들어간다는 점을 고정한다.

### 실무에서 먼저 의심해야 하는 함정

- 복합 인덱스의 모든 열이 언제나 독립적으로 쓰인다고 믿으면 열 순서를 잘못 잡는다. 실제로는 앞쪽 열이 탐색 공간을 어떻게 잘라 주는지가 핵심이다.
- covering index를 무조건 크게 만들면 쓰기 비용, 메모리 점유, page split, cache 효율이 악화된다. 읽기 한 쿼리를 위해 전체 write path를 무겁게 만드는 전형적인 운영 함정이다.
- unique index를 애플리케이션 중복 검사 대체재로 보지 않고 단순 조회 최적화로만 보면 race condition에서 중복 row가 들어간다.

복합 인덱스 함정은 인덱스 폭을 넓히는 것을 성능 개선으로만 볼 때 생긴다. 열 순서가 틀리면 탐색 prefix가 무너지고, covering을 위해 열을 과하게 넣으면 쓰기 비용이 커지며, unique를 애플리케이션 검사로 대체하면 동시성 race를 DB가 막아 주지 못한다.

### 관측과 검증 경로

- PostgreSQL에서는 `EXPLAIN (ANALYZE, BUFFERS)`에서 Index Only Scan, Heap Fetches, Sort node 유무를 함께 본다. PASS는 Heap Fetches가 줄고 Sort가 없어지며 실제 rows가 LIMIT 근처에서 멈추는 것이다.
- MySQL에서는 `EXPLAIN`의 key_len과 Extra를 같이 읽는다. PASS는 key가 복합 인덱스로 잡히고 `Using index`가 나타나며 `Using filesort`가 사라지는 것이다.
- 실패 신호는 인덱스를 추가했는데 rows 추정은 줄지 않고, SELECT list 하나 때문에 covering이 깨지거나, 열 순서 때문에 ORDER BY가 여전히 filesort로 빠지는 것이다.

복합 인덱스 검증은 prefix, covering, 정렬 제거가 실제 plan에 나타나는지 보는 일이다. PostgreSQL은 Index Only Scan과 Heap Fetches를, MySQL은 key_len, Using index, Using filesort를 본다. PASS는 선택한 열 순서가 WHERE와 ORDER BY, SELECT list를 함께 줄였다는 점이 관측되는 것이다.

### 오해 수리와 전이 질문

이 절에서 바로잡을 오해는 복합 인덱스가 단일 인덱스 여러 개의 합이라는 생각이다. 실제로는 왼쪽부터 정렬된 하나의 key space이며, covering과 unique와 partial/expression 조건은 그 key space가 row fetch, 데이터 불변식, 후보 집합 크기를 어떻게 바꾸는지의 문제다.

## LSM/hash/bitmap/search index와 RDBMS index 비교

등장 배경을 보면 여러 index family가 왜 공존하는지 보인다. 모든 workload가 정렬된 range scan만 원하는 것은 아니고, 모든 workload가 단건 equality만 원하는 것도 아니다. 로그처럼 계속 쓰는 데이터, 전문 검색처럼 단어에서 문서로 가야 하는 데이터, 물리 순서와 시간이 거의 함께 움직이는 데이터, 여러 조건의 후보 집합을 합쳐야 하는 데이터가 각각 다른 압력을 만든다. 그래서 B+tree 하나로 모든 문제를 덮는 대신 hash, bitmap/BRIN, inverted index, LSM 계열 같은 서로 다른 trade-off가 생겼다.

독자는 B+tree, hash, bitmap/BRIN, inverted/search index, LSM 계열이 모두 index라는 이름을 쓰지만 읽기/쓰기/범위/정렬/갱신 비용의 교환 조건이 다르다는 점을 설명할 수 있어야 한다. 이 절은 정의 목록이 아니라 작은 입력이 인덱스나 실행기 내부에서 어떤 순서로 움직이는지 따라가는 설명이다. 읽은 뒤에는 문서의 문장을 외우는 것이 아니라, 같은 종류의 쿼리를 보았을 때 어떤 구조가 먼저 일을 시작하고 어떤 관측값을 봐야 하는지 말할 수 있어야 한다.

로컬 seed `database/elasticsearch/*`는 Elasticsearch mapping, query, pagination, dump/restore 조각을 담고 있다. 이 DU는 검색 엔진 index를 RDBMS 보조 인덱스와 같은 물건으로 취급하지 않도록 비교 축을 고정한다. 기존 seed는 짧거나 특정 상황에 묶여 있으므로 그대로 복사하지 않는다. 대신 seed가 품고 있던 질문을 살리고, 공식 문서가 닫아 주는 사실과 실행 trace가 보여 주는 관측 경로를 합쳐 새 본문으로 확장한다.

공식 근거는 다음 경계 안에서 사용한다.

- PostgreSQL 18 공식 문서의 Chapter 11 Indexes와 Index Types는 B-tree가 기본 인덱스이고, 정렬 가능한 값의 동등/범위 비교와 정렬 순서 반환에 맞는다는 점을 기준 근거로 둔다.
- Elastic 공식 문서의 index fundamentals는 Elasticsearch의 index가 document를 저장하고 검색 가능한 구조로 관리한다는 점을 검색 엔진 index 설명의 기준 근거로 둔다.

첫 벽돌은 같은 상품 데이터를 PostgreSQL 테이블과 Elasticsearch index에 동시에 넣는 상황이다. RDBMS는 `WHERE merchant_id=? ORDER BY created_at`를 빠르게 하려 하고, 검색 엔진은 `description` 안에 들어 있는 단어와 점수를 빠르게 찾으려 한다. 이 작은 장면을 붙잡아야 추상 용어가 실제 비용으로 바뀐다. 데이터베이스의 성능 설명은 보통 `빠르다`, `느리다`, `인덱스를 탄다` 같은 말로 시작하지만, 실제 진단은 어떤 row나 key가 어떤 구조를 거쳐 다음 소비자에게 전달되는지 확인하는 일이다.

```text
row/document input
id=42, merchant_id=m-7, created_at=2026-05-01, description="red running shoes"

B+tree secondary index idea
(merchant_id=m-7, created_at=2026-05-01, id=42) -> row locator

inverted index idea
red     -> [doc42]
running -> [doc42]
shoes   -> [doc42]

query A: merchant_id=m-7 ORDER BY created_at
consumer follows ordered key range in B+tree

query B: description contains running shoes
consumer intersects posting lists and scores documents
```


PostgreSQL의 B-tree는 정렬 가능한 값의 동등/범위/순서 소비에 강하다. hash index는 공식 문서상 단순 동등 비교에 맞고, GIN은 배열처럼 여러 component value를 가진 값에 대해 inverted index 성격을 가진다. BRIN은 물리 block range의 요약을 저장해 물리 순서와 값의 상관관계가 강할 때 효과적이다.

검색 엔진의 inverted index는 row 순서보다 term에서 document로 가는 길을 우선한다. 단어가 어떤 문서에 등장했는지, 필드 분석기가 어떤 token을 만들었는지, 점수 계산이 어떤 후보를 위로 올리는지가 핵심이다.

LSM 계열은 쓰기를 먼저 append-friendly한 구조에 흡수하고, 나중에 compaction으로 정리하는 설계 family다. 이 DU에서는 특정 DBMS의 구현 사실로 확정하지 않고, B+tree와 대비되는 쓰기/읽기 trade-off 설명으로만 사용한다.

bitmap scan은 하나의 row locator stream이 아니라 여러 조건의 후보 집합을 비트맵처럼 합성한 뒤 heap을 방문하는 방식으로 이해하면 쉽다. PostgreSQL EXPLAIN에서 Bitmap Index Scan과 Bitmap Heap Scan이 나뉘는 이유가 여기에 있다.

같은 index라는 단어를 쓰더라도 소비자가 다르다. B+tree leaf는 ordered row locator를 소비자에게 주고, inverted index는 term posting list를 주며, BRIN은 block range 후보를 주고, LSM compaction은 나중에 읽기 경로의 merge 비용으로 돌아온다.

여러 index family를 비교할 때는 이름보다 소비자를 먼저 봐야 한다. B+tree는 정렬된 key range를, hash는 equality bucket을, BRIN은 block range 후보를, inverted index는 term posting list를, LSM 계열은 쓰기 흡수 뒤의 merge된 읽기 후보를 만든다. 이 산출물이 서로 다르기 때문에 같은 `index`라는 단어를 쓰더라도 관측 지표와 장애 양상이 달라진다.

### 손으로 다시 읽는 예제

1. B+tree가 이기는 경우

    ```sql
    WHERE merchant_id='m-7' AND created_at >= '2026-05-01' ORDER BY created_at LIMIT 20
    ```

    정렬된 range를 작게 잘라 바로 소비한다.
1. inverted index가 이기는 경우

    ```sql
    description: "running shoes"
    ```

    단어별 posting list를 합성하고 scoring한다. 행 순서보다 term membership이 중요하다.
1. BRIN이 맞는 경우

    ```sql
    created_at이 append 순서와 거의 같고 오래된 월 단위 데이터만 읽는다.
    ```

    block range 요약으로 많은 block을 제외할 수 있다.

위 예제들은 index family마다 빠른 질문의 모양이 다르다는 사실을 보여 준다. B+tree 예제는 정렬된 범위를, inverted index 예제는 단어에서 문서로 가는 후보 집합을, BRIN 예제는 물리 block range 요약을 소비한다. 그래서 한 구조에서 좋은 쿼리 모양을 다른 구조에 그대로 기대하면 오해가 생긴다.

### 실무에서 먼저 의심해야 하는 함정

- 검색 엔진 index와 RDBMS index를 같은 자료구조로 취급하면 pagination, consistency, scoring, refresh, transaction boundary를 잘못 설계한다.
- hash index를 보면 hashmap처럼 모든 index가 O(1)이라고 단순화하기 쉽다. 하지만 범위와 정렬을 잃는 순간 DB 쿼리의 주요 비용 축이 사라진다.
- LSM 계열의 쓰기 강점을 읽기 비용이 공짜라는 뜻으로 오해하면 compaction, read amplification, stale segment, tombstone 비용을 놓친다.

index family 함정은 한 구조의 장점을 다른 구조의 보장으로 착각할 때 생긴다. hash의 equality 감각을 range query에 적용하거나, 검색 엔진의 inverted index를 트랜잭션 일관성을 가진 B+tree처럼 읽거나, LSM의 쓰기 흡수를 읽기 비용 무료로 해석하면 운영 지표가 어긋난다.

### 관측과 검증 경로

- PostgreSQL에서는 같은 조건을 B-tree, hash, GIN/BRIN 후보로 나눠 `EXPLAIN`하고 node 종류와 rows 추정을 비교한다. PASS는 각 index type이 지원하는 operator와 query shape가 맞을 때만 선택되는 것이다.
- Elasticsearch에서는 mapping과 analyzer가 만든 term을 `_analyze` 또는 검색 결과의 hit/score로 확인한다. PASS는 term 중심 질의에서 posting 후보와 score가 나타나는 것이고, FAIL은 RDBMS range order처럼 안정적인 전체 순서를 기대하는 것이다.
- 운영 검증은 쓰기량, refresh/compaction, index size, cache hit, query latency를 같이 본다. 하나만 보면 읽기 최적화가 쓰기 장애로 돌아오는 순간을 놓친다.

index family 검증은 각 구조가 지원하는 질문 모양과 plan/검색 결과가 맞는지 확인한다. PostgreSQL에서는 B-tree, hash, GIN, BRIN 후보의 node 차이를 보고, Elasticsearch에서는 analyzer가 만든 term과 hit/score, refresh 경계를 확인한다. PASS는 같은 데이터라도 구조별 소비자가 다르다는 점을 관측으로 설명하는 것이다.

### LSM/hash/bitmap/search index와 RDBMS index 비교 현장 판독 노트 1

1-1. `term query` 상황에서는 먼저 term 검색을(를) 하나의 관측 단위로 분리한다. inverted index는 term에서 document 후보로 간다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. row order를 기대하면 검색 점수와 refresh 경계를 놓친다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-2. `hash equality` 상황에서는 먼저 동등 비교을(를) 하나의 관측 단위로 분리한다. hash 계열은 동등 비교에 초점을 둔다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 범위와 정렬을 처리할 별도 구조가 필요하다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-3. `BRIN summary` 상황에서는 먼저 시계열 스캔을(를) 하나의 관측 단위로 분리한다. block range 요약은 물리 순서와 값의 상관관계가 있을 때 강하다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 무작위 분포에서는 많은 block을 다시 확인해야 한다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-4. `bitmap combination` 상황에서는 먼저 조건 합성을(를) 하나의 관측 단위로 분리한다. 여러 index 후보를 집합처럼 합성할 수 있다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 합성 후 heap 방문 비용이 사라지는 것은 아니다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-5. `LSM compaction` 상황에서는 먼저 쓰기 흡수을(를) 하나의 관측 단위로 분리한다. 쓰기 흡수는 나중의 merge/compaction 비용과 짝이다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 쓰기만 보고 읽기 tail latency를 잊으면 안 된다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

1-6. `search refresh` 상황에서는 먼저 검색 일관성을(를) 하나의 관측 단위로 분리한다. 검색 엔진은 색인 반영 시점과 검색 가능 시점이 분리될 수 있다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 트랜잭션 read-your-write와 같은 보장을 기대하면 안 된다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

8-3. `BRIN summary` 상황에서는 먼저 시계열 스캔을(를) 하나의 관측 단위로 분리한다. block range 요약은 물리 순서와 값의 상관관계가 있을 때 강하다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 무작위 분포에서는 많은 block을 다시 확인해야 한다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

8-4. `bitmap combination` 상황에서는 먼저 조건 합성을(를) 하나의 관측 단위로 분리한다. 여러 index 후보를 집합처럼 합성할 수 있다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 합성 후 heap 방문 비용이 사라지는 것은 아니다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

8-5. `LSM compaction` 상황에서는 먼저 쓰기 흡수을(를) 하나의 관측 단위로 분리한다. 쓰기 흡수는 나중의 merge/compaction 비용과 짝이다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 쓰기만 보고 읽기 tail latency를 잊으면 안 된다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

8-6. `search refresh` 상황에서는 먼저 검색 일관성을(를) 하나의 관측 단위로 분리한다. 검색 엔진은 색인 반영 시점과 검색 가능 시점이 분리될 수 있다. 여기서 바로 결론을 내리지 않고 입력 row 수, 후보 row 수, 다음 소비자, 실패 신호를 같은 줄에 적어 본다. 트랜잭션 read-your-write와 같은 보장을 기대하면 안 된다. 실무에서 이 점이 중요한 이유는 장애 티켓이 보통 "느리다" 한 단어로 들어오기 때문이다. 그 한 단어를 page, row, term, plan node, 통계, 로그 중 어느 층의 문제인지 나누지 않으면 가장 먼저 떠오른 인덱스 추가나 SQL 재작성으로 기울어진다. 반대로 이 판독 단계를 거치면 LSM/hash/bitmap/search index와 RDBMS index 비교에서 설명한 구조가 단순 암기가 아니라 재현 가능한 진단 절차가 된다.

이 8번째 판독 묶음의 핵심은 같은 설명을 반복하는 것이 아니라, LSM/hash/bitmap/search index와 RDBMS index 비교을(를) 운영 관측 단위로 잘게 나누어 보는 연습이다. 한 쿼리의 평균 시간만 보지 말고, 어떤 입력이 어떤 내부 구조를 지나 어떤 산출물을 만들었는지 작은 trace로 되돌려야 한다. 그 trace가 닫히면 공식 문서의 용어와 로컬 seed의 짧은 메모가 서로 연결되고, 닫히지 않으면 아직 원인을 안다고 말하면 안 된다.

### 오해 수리와 전이 질문

이 절에서 바로잡을 오해는 index라는 이름이 같으면 같은 비용 모델을 가진다는 생각이다. B+tree, hash, BRIN, GIN, inverted index, LSM 계열은 모두 다른 질문에 빠르게 답하도록 만들어졌고, 관측해야 할 실패 신호도 다르다.

마지막으로 LSM/hash/bitmap/search index와 RDBMS index 비교을(를) 자기 말로 다시 설명하려면 세 문장을 직접 써 본다. 첫 문장은 이 구조가 해결하는 문제, 둘째 문장은 작은 입력이 내부에서 바뀌는 순서, 셋째 문장은 운영에서 깨지는 조건이다. 세 문장 중 하나라도 공식 문서나 trace로 되돌아가지 못하면 아직 외운 말이고, 세 문장이 모두 trace와 검증 명령으로 이어지면 다음 DU의 실행 계획을 읽을 준비가 된 것이다.
