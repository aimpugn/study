목표는 개발 자동화의 속도를 얻되, 내부망에서 연구/개발망 또는 업무망으로 흘러가는 데이터 흐름을 기본 차단하는 것입니다.

권장 구조 연구/개발망 GitLab은 source/build 후보까지만 담당하고, 내부 배포 판단과 결과 로그는 내부망에 둡니다.
금지 기본값 내부망 Runner/Agent가 연구/개발망 GitLab 또는 업무망 Jenkins로 로그, 상태, 메타데이터를 되돌리는 구조입니다.
DMZ 역할 DMZ는 명령권자가 아니라 검증된 artifact 후보를 받는 quarantine/inbox로 시작하는 편이 안전합니다.
핵심 관점 배포 성공/실패, 로그, token, 서버 목록, 에러 메시지, artifact metadata도 모두 데이터 흐름입니다.

권장 구조

GitLab은 연구/개발망의 source와 build 후보 관리에 둡니다. DMZ는 webhook inbox와 artifact quarantine으로 제한하고, 내부망 deploy orchestrator와 pull agent가 최종 배포와 rollback을 소유합니다.

비권장 구조

내부망 GitLab Runner가 연구/개발망 GitLab에 등록되어 job trace, 상태, 로그를 되돌리는 구조는 내부망 데이터 역류 경로가 생깁니다. Jenkins controller가 외부/업무망에 있고 내부 agent가 붙는 구조도 같은 문제가 있습니다.

숨은 데이터 흐름

실패 메시지, stack trace, 배포 대상 호스트명, agent 식별자, 환경 변수명, artifact download URL, runner token, SBOM의 내부 dependency명도 데이터입니다. 기본값은 내부망 밖으로 내보내지 않는 것입니다.

Artifact 이동 경로

  1. GitLab merge 후 build runner가 artifact, checksum, SBOM, provenance를 생성합니다.
  2. DMZ quarantine repo가 artifact와 manifest를 받되 내부 대상 서버 정보는 받지 않습니다.
  3. 내부망 mirror가 artifact를 가져와 signature와 digest를 다시 확인한 뒤 배포 후보로 승격합니다.

감사 로그 위치

GitLab에는 merge, pipeline, release evidence까지만 남깁니다. 내부망에는 deployment ledger를 별도로 두고, 서버별 결과와 rollback 기록을 내부망 밖으로 내보내지 않습니다.

장애/복구 흐름

Webhook 중복은 idempotency key로 제거하고, 일부 서버 실패는 서버별 state machine으로 재시도합니다. Rollback은 이전 artifact digest를 기준으로 수행하며, raw log는 내부 감사 저장소에만 남깁니다.

개발자 사용 흐름

  1. 개발자는 연구/개발망 GitLab에 merge request를 올리고 AI 개발 agent는 코드/테스트/문서 보조만 수행합니다.
  2. merge 후 pipeline이 test, build, scan, SBOM, signing을 수행하고 DMZ에 배포 후보 manifest를 남깁니다.
  3. 운영 승인자는 내부망 배포 포털에서 artifact digest와 검증 결과를 보고 승인합니다.
  4. 내부 pull agent가 대상 서버에 배포하고 결과를 내부 audit log에 기록합니다.
  5. 개발자가 외부에서 볼 수 있는 정보는 보안팀이 허용한 후보 생성 상태까지로 제한합니다.

주요 판단 근거

  • NIST SP 800-207은 네트워크 위치만으로 신뢰하지 말고 resource 중심으로 인증과 권한을 판단하라고 설명합니다.
  • GitLab Runner는 GitLab 서버를 polling해 job을 받고 trace/status를 되돌리는 구조이므로, 내부망 runner가 연구/개발망 GitLab에 붙으면 내부 데이터 역류 경로가 생깁니다.
  • GitLab webhook은 secret header와 payload template을 제공하지만, payload 자체가 어떤 정보를 싣는지 별도로 최소화해야 합니다.
  • Jenkins credentials와 GitLab CI/CD variables 모두 masking이 완전한 비밀 유출 방지책은 아니므로, production secret은 내부망 secret store에 두는 편이 안전합니다.
  • SLSA와 Sigstore의 핵심은 artifact를 소비하는 지점에서 provenance/signature/digest를 검증하는 것입니다.
