# 학습 문서 작성 가이드

이 한 파일은 그 자체로 완결된 학습 문서 작성 지침입니다. 특정 프로젝트의 다른 파일이나 특정 도구에 의존하지 않으므로, 어떤 저장소에든 그대로 복사해 쓸 수 있습니다.

## 무엇을 위한 가이드인가

"읽고 고개를 끄덕이는 글"이 아니라, 나중에 작성자 자신이나 다른 사람이 **다시 이해를 복원하고, 직접 검증하고, 남에게 다시 설명할 수 있는** 학습 문서를 만들기 위한 지침입니다. 개념 정리, 비교 문서, 트러블슈팅 노트, 도구 사용기, 실험 기록 같은 설명형 자산에 두루 적용됩니다.

좋은 학습 문서의 기준은 그럴듯한 요약이 아니라, **근거·메커니즘·예시·검증 경로가 닫혀 있는 설명**입니다. 직답은 짧게 두되, 그 뒤에 배경·동작 원리·예제·확인 방법이 이어져 있어야 합니다. 이 가이드는 얕은 요약이 아니라 **깊고 넓은 설명**을 기본값으로 잡습니다. 그 깊이와 넓이의 기준은 3장에서 따로 정의합니다.

이 가이드의 장치들은 "하면 좋다"는 권고가 아니라 **요구**입니다. 분량이나 시간을 이유로 어느 하나를 빼면, 그만큼 얕은 요약으로 후퇴한 것입니다.

### 한 가지 전제 — 형식과 사실은 다르다

이 가이드는 문서의 **형식**(어떤 장치를 넣고 어떻게 배치할지)을 정의합니다. 그러나 각 장치를 채울 **사실**은 가이드가 주지 않습니다. 작성자가 1차 출처(코드 실행 결과, 표준·스펙, man page, 논문, 실제 측정값)에서 직접 확보해야 합니다.

형식만 흉내 내고 사실이 빈약하면 — 비교표 축이 2개뿐, 역사 절이 "예전엔 락을 썼다" 한 줄, 관측값이 "대략 빠르다" 수준 — 그것은 깊은 문서가 아니라 **틀만 갖춘 얕은 요약**입니다. 표·역사 절·trace의 빈 칸을 추측으로 채우는 것은 이 가이드를 따른 것이 아닙니다. 깊이는 형식이 아니라 그 칸에 들어간 확인된 사실의 밀도에서 나옵니다.

## 어떻게 적용하나

- 이 파일을 프로젝트 루트나 문서 디렉토리에 둡니다.
- 에이전트나 기여자가 자동으로 참조하게 하려면, 프로젝트의 관례 파일(예: `AGENTS.md`, `CLAUDE.md`, `CONTRIBUTING.md`)에서 이 파일을 한 줄로 가리키면 됩니다. 그런 파일이 없어도 이 가이드는 단독으로 동작합니다.
- 새 학습 문서를 시작할 때는 부록 A의 스켈레톤을 복사하고, 마무리 전에 부록 B로 점검합니다. 부록 B는 통과시켜 주는 형식 점검이 아니라, 깊이 미달을 걸러 내는 게이트입니다.

## 1. 잘 된 학습 문서의 공통 DNA

각 패턴은 `원리 -> 판정 질문 -> 예시(좋음 / 피함)` 순서로 정리합니다. 예시는 따라야 할 규칙이 아니라, 좋고 나쁨의 감각을 맞추는 calibration 용도입니다. 아래 여덟 가지는 선택지가 아니라, 주제에 해당하면 채워야 하는 요구 항목입니다.

### 1.1 질문형 또는 직답으로 연다

- 원리: 정의를 던지기 전에 "이 문서가 답하는 질문"과 짧은 답을 먼저 고정합니다.
- 판정 질문: 첫 문단만 읽고도 무엇을 왜 다루는지 잡히는가.
- 좋음: `짧게 답하면, 표준이 강제해서가 아니라 낮은 계층의 기본 단위가 연결이 아니라 소켓이기 때문입니다.`
- 피함: `OOO란 ~를 의미하는 기술이다`로 시작해 한참 정의만 나열.

### 1.2 실제 artifact에서 출발한다

- 원리: 추상 설명보다 먼저, 독자가 붙잡을 구체물을 보여 줍니다. 실제 명령, 코드, 로그, 에러 메시지, 입력값. 그 구체물을 어떻게 해부하는지는 1.8에서 다룹니다.
- 판정 질문: 첫 번째 벽돌이 추상 개념인가, 손에 잡히는 artifact인가.
- 좋음: 실제 빌드 에러 로그를 먼저 보여 주고 그 원인을 한 줄씩 추적 / 실제 코드를 놓고 줄 단위로 짚기.
- 피함: 동작하는 예가 하나도 없이 "일반적으로 ~한다"만 반복.

### 1.3 배경과 역사, "왜 생겼나"를 닫는다

- 원리: 핵심 개념은 등장 배경과 역사도 이해의 일부입니다. 독자가 "왜 생겼지, 그 전에는 어떻게 했지"를 물을 지점에서 답합니다. 핵심 개념이라면 연도·인물·사건 수준의 역사와, 설계자가 어떤 문제와 대안을 거쳐 그 해법에 도달했는지(설계 사고)까지 재구성합니다. 이 깊이는 3.1에서 강제합니다.
- 판정 질문: 이 개념이 풀려던 문제와 이전 방식의 한계가 보이는가. 누가 언제 왜 만들었는지 추적되는가.
- 좋음: 기존 방식의 한계 -> 누가 언제 그 한계를 인식 -> 어떤 대안을 거쳐 새 개념에 도달 -> 실제 채택으로 잇는 흐름.
- 피함: 현재 동작만 적고 "왜 이런 모양이 됐는지"가 비어 있음.

### 1.4 단계별 trace로, 같은 대상을 여러 시점에서 본다

- 원리: 입력 -> 규칙 -> 상태 변화 -> 산출물 -> 다음 소비자로 이어지는 흐름을 시간 순서로 따라갑니다. 한 번 흐르고 끝나지 않고, **같은 대상을 시점·세션·상태를 바꿔 가며 반복 관측**해, 각 시점에서 무엇이 보이고 무엇이 안 보이는지 대조합니다.
- 판정 질문: 독자가 종이에 같은 단계를 따라 그릴 수 있는가. 같은 객체를 여러 시점에서 다시 들여다봤는가.
- 좋음: 동일 데이터를 `T0 -> T1(세션 열림) -> T2(수정) -> T3(다른 세션)`로 다시 그려 각 시점의 가시성을 비교 / 같은 주소를 초기 상태와 동작 중에 반복 관측해 값 변화를 대조.
- 피함: 결과만 제시하고 중간 상태 전이를 생략. 한 시점만 보고 끝냄.

### 1.5 계층을 오르내리고, 혼동쌍은 대조한다

- 원리: 한 층위 설명으로 오해가 남는 주제는 위아래 추상화 계층(예: 애플리케이션·런타임·OS·하드웨어)을 필요한 만큼 오갑니다. 헷갈리는 개념 쌍은 나란히 놓습니다.
- 판정 질문: 인접 개념과의 경계가 분명한가. 대조쌍의 양쪽이 같은 밀도로 설명되는가.
- 좋음: `동기 vs 비동기`, `블로킹 vs 논블로킹` 같은 쌍을 정의·동작·관측·주의점까지 대칭으로.
- 피함: 한쪽만 자세하고 반대쪽은 한 줄로 흘림.

### 1.6 비유, 다이어그램, 비교표로 직관을 만든다

- 원리: 추상 구조에는 실생활 비유 하나, 흐름에는 간단한 다이어그램, 여러 대상 비교에는 표를 붙입니다. 비유를 쓸 때는 어디까지 맞고 어디서 깨지는지(적용 한계)를 함께 적습니다.
- 판정 질문: 이 장치가 실제 이해를 돕는가, 장식인가. 비유가 깨지는 지점을 독자가 아는가.
- 좋음: 핵심 개념을 일상 사물에 빗대되 "이 비유는 X까지만 맞고 Y에서 깨진다"를 명시 / 대상별 차이를 여러 축으로 쪼갠 비교표 / 흐름을 보여 주는 ASCII 그림.
- 피함: 비유가 원개념과 어긋나거나, 적용 한계 없이 비유만 던지거나, 표가 본문 설명과 그대로 중복.

### 1.7 출처와 직접 검증 경로를 남긴다

- 원리: 하중을 지탱하는 주장에는 1차 출처를, 핵심 주장에는 직접 확인할 경로(무엇을 실행하면 PASS이고 무엇이 FAIL인지)를 남깁니다.
- 판정 질문: 독자가 문서를 덮고 스스로 확인할 수단이 있는가. `확인해 보면 된다`로 끝나지 않는가.
- 좋음: 끝에 `직접 확인 포인트`(명령·관측값)와 `출처`(공식 문서·표준·레퍼런스) 절. 예상값이 아니라 실제 관측값까지 적기.
- 피함: 링크만 모아 두거나, 검증 경로 없이 단정.

### 1.8 원자료는 통째로 붙이지 말고 줄 단위로 해부한다

- 원리: 코드·로그·덤프·설정·표 같은 원자료는 길게 통째로 붙여 놓고 넘어가지 않습니다. 하중을 지는 줄마다 인라인 주석으로 ① 그 값이 무엇인지 ② 왜 그 값인지(계산·근거) ③ 다른 값이면 무슨 일이 나는지를 붙입니다. 긴 원자료 전체가 꼭 필요하면 접어 두고, 본문에는 해부한 핵심 줄만 남깁니다.
- 판정 질문: 원자료가 "보면 알겠지" 상태로 방치됐는가, 줄마다 의미가 박혔는가.
- 좋음: 설정 옵션마다 "이게 없으면 무슨 에러가 난다"를 붙이기 / 덤프의 주소 줄에 "이 길이가 8바이트이므로 레지스터 간격은 0, 그래서 …" 계산을 붙이기 / 코드 시그니처의 타입이 어떻게 흐르는지 줄마다 짚기.
- 피함: 30줄 로그를 붙이고 "위와 같이 동작한다" 한 줄로 마무리. 구조를 보여 준 뒤 "구조가 보인다"로 넘어가기.

## 2. 기본 전개 흐름

일반 학습 문서는 얕은 요약보다 깊은 학습 문서(deep study)에 가깝게 씁니다. 기본 골격은 아래 순서입니다.

1. 직답 또는 질문 고정
2. 왜 생겼나 / 배경 / 역사
3. 가장 작은 작동 단위와 먼저 알아야 할 공통 메타데이터·artifact
4. (필요하면) 실무에서 왜 중요한지 짧게 연결
5. 최소 예제 또는 단계별 trace
6. 실제 동작 메커니즘
7. 더 큰 구조·실무 맥락으로 확장
8. 혼동하기 쉬운 인접 개념과 비교
9. 실패 모드·한계·검증 경로·근거

이 골격은 라벨로 노출하지 않고 자연스러운 흐름으로 녹입니다. 문서 유형에 따라 앞부분을 바꿉니다.

- 개념 문서: 질문 -> 왜 중요한가 -> 핵심 메커니즘 -> 예제 -> 오해·반례 -> 검증 -> 근거
- 명령·도구 문서: 무엇을 하는가 -> 입력·옵션 감각 -> 안전한 예제 -> 실무 예제 -> 실패·주의점 -> 검증 -> 근거
- 트러블슈팅 문서: 증상 -> 직접 원인 -> 근본 원인 -> 해결 -> 재발 방지·확인 -> 근거
- 표준·프로토콜 문서: 규칙 범위 -> 원문 의미 -> 구현 해석 포인트 -> 예시 -> 실패·주의점 -> 검증 -> 근거
- 질문·인터뷰형 문서: 질문 -> 짧은 직답 -> 계층별 설명 -> 구체 예시 -> 흔한 꼬리 질문·반례 -> 검증·근거

길고 구조가 있는 문서는 상단에 목차를 두고, 서두에서 "이 문서가 정확히 무엇을 다루는지"를 바로 고정합니다.

## 3. 깊이와 넓이를 어디까지 가져갈까

이 가이드의 목표는 짧은 치트시트가 아니라, 나중에 혼자 다시 이해를 복원할 수 있을 만큼 **깊고 넓은** 학습 문서입니다. "어디까지"는 주제마다 다르지만, 깊이와 넓이를 우연에 맡기지 않고 의식적으로 결정합니다. 아래는 권고가 아니라, 주제에 해당하면 충족해야 하는 기준입니다.

### 3.1 깊이 — 한 단계 아래까지 내려간다

- 독자가 그 자리에서 자연스럽게 떠올릴 "한 단계 아래" 질문까지 답합니다. *왜 하필 이 범위인가, 내부적으로 무엇이 일어나는가, 저 숫자는 어디서 나왔나.*
- 멈추는 기준은 "지금 다루는 질문을 닫는 데 필요한 만큼"입니다. 무한히 파고들지는 않되, "그건 범위 밖"이라고 얼버무려 이해의 사슬을 끊지 않습니다.
- 핵심 개념의 등장은 연도·인물·사건 수준의 역사와, 설계자의 문제 해결 흐름까지 재구성합니다. 이를 생략하면 깊이 미달로 봅니다. 설계 사고는 다음 질문으로 풉니다.
    1. 설계자가 인식한 구체적 문제는 무엇이었나.
    2. 기존 방식의 어떤 가정을 버렸나.
    3. 어떤 대안을 검토했고 왜 이것을 택했나.
    4. 어떤 부작용을 미리 예측하고 대비했나.
    5. 왜 이것이 맞는지 어떻게 정당화했나.
- 판정 질문: 독자가 "그런데 그건 왜 그래?"라고 한 번 더 물었을 때, 문서 안에 답이 있는가.

### 3.2 넓이 — 한 주제를 여러 면으로 본다

- 한 개념을 여러 면에서 봅니다: 정의·원리 / 등장 역사 / 내부 구현 / 대안과 비교 / 성능·비용 / 한계·실패 모드 / 실무 적용.
- 모든 면이 항상 필요하지는 않습니다. 그러나 어떤 면을 왜 넣고 왜 뺐는지는 의식합니다. `정의 -> 장점`만 있고 한계·대안·실패가 비어 있으면 넓이가 부족하다는 신호입니다.
- 판정 질문: 이 문서는 한 면만 비추는가, 주제를 입체로 두르는가.

### 3.3 다대상 비교는 여러 축으로 쪼갠다

- 둘 이상을 비교할 때는 한 줄 요약("A가 더 빠르다")에 그치지 않고, 의미 있는 여러 축으로 쪼개 표로 정리합니다. 축은 식별자·저장 위치·동작 시점·비용·한계처럼 주제마다 다릅니다.
- 축이 몇 개여야 하는지에 절대 수치는 없습니다. 주제가 요구하는 만큼이 기준이되, 축이 둘셋에 그치면 더 쪼갤 수 있는지 의심합니다. 큰 비교 주제의 천장은 3.6에 있습니다(거기 적힌 9축 수준까지 의미 있게 갈 수 있습니다).
- 비교 대상이 N개, 축이 M개면 그 `N x M` 표 자체가 깊이를 만듭니다. 각 칸은 한 단어라도 추측이 아니라 확인된 근거가 있어야 합니다.
- 판정 질문: 비교가 인상 수준인가, 축별로 검증 가능한 수준인가.

### 3.4 같은 목표의 naive와 개선을 대비한다

- 같은 일을 하는 단순한(naive) 버전과 개선된 버전을 before/after로 나란히 놓으면 "왜 이 설계인가"가 드러납니다.
- 1.5의 대조쌍이 *서로 다른 개념*을 가르는 것이라면, 이것은 *같은 목표의 두 구현*을 비교하는 것입니다. 둘은 다른 장치입니다.
- 판정 질문: 개선안만 보여 주고 끝나는가, 무엇을 개선했는지가 대비로 보이는가.

### 3.5 넓어지면 허브와 상세로 나눈다

- 한 문서가 너무 넓거나 길어지면, 전체 흐름과 지도를 잡는 허브 문서와 각 부분을 깊이 파는 상세 문서로 나눕니다.
- 허브는 "무엇이 어디에 있고 어떻게 이어지는지"를 맡고, 상세는 깊이를 맡습니다. 한 파일에 다 넣어 깊이를 희생하느니, 역할을 나눠 양쪽을 다 살립니다.
- 판정 질문: 깊이를 지키려고 분할을 검토했는가, 분량 때문에 깊이를 줄였는가.

### 3.6 깊이의 상한 — 여기까지 갈 수 있다

깊이를 과소 추정하지 않도록, 도달 가능한 천장을 알아 둡니다. 아래 수치는 의무 할당량이 아니라, "여기까지 의미 있게 내려갈 수 있다"는 상한 예시입니다. 부록 B의 게이트는 이 천장을 기준으로 삼습니다 — 큰 주제인데 이 천장에 한참 못 미치면 깊이 미달입니다.

중요한 것은 **모든 천장을 모든 주제에 적용하지 않는다**는 점입니다. 주제 유형마다 적용되는 천장이 다릅니다. 코드 사용기에 역사 8연도를 억지로 채우거나, 순수 개념 문서에 디버거 관측값을 끼워 넣지 않습니다. 자기 주제 유형에 해당하는 천장만 목표로 삼습니다.

- 이론·개념 주제(역사가 있는 것): 연도·인물·사건 8개 이상의 타임라인, 설계자의 문제 해결 사고(3.1의 5질문), 의미 있는 축 9개까지의 다축 비교.
- 코드·라이브러리 주제: 시그니처와 각 줄의 타입·값 흐름까지 줄 단위 해부, 같은 목표의 naive vs 개선 대비. (역사 타임라인은 짧거나 없어도 됩니다.)
- 트러블슈팅 주제: 증상 로그 -> 원인 -> 비트·주소 범위 같은 수치 계산 -> 실제 디버거·도구로 찍은 관측값까지.
- 공통(유형 불문): 비유는 깨지는 지점까지, 핵심 주장은 실행 가능한 검증 경로까지, 메커니즘은 비유가 아니라 실제 메모리·레지스터·패킷·테이블 상태 수준까지.

### 3.7 칸은 1차 출처로 채운다

위 장치들(표, 역사 절, 관측값, trace)은 형식일 뿐입니다. 각 칸은 코드 실행·표준·man page·논문·실제 측정 같은 1차 출처로 채웁니다. 비교표의 한 칸, 역사의 한 연도, 관측값 하나까지 추측이 아니라 확인된 사실이어야 합니다. 사실을 확보하지 못한 칸은 비워 두고 `추가 확인 필요`로 표시할지언정, 그럴듯한 값으로 메우지 않습니다.

## 4. 문장과 서식 규칙

- 한 가지 문체(예: 존댓말 `~합니다`)를 정해 일관되게 씁니다.
- 쉬운 말로 개념을 먼저 고정하고, 공식 용어는 괄호로 병기합니다. 예: `매니페스트(manifest)는 아카이브 전체에 공통으로 적용되는 메타데이터를 적는 파일입니다.`
- 학습 문서 본문에서는 진행 중계 문장을 피하고 주제를 곧바로 말합니다. `이제 X 안으로 들어가 보겠습니다`류보다 `X는 Y를 위해 ~합니다`. (이 가이드 같은 규범·안내 문서에서는 `다음 순서로 정리한다` 같은 절차 안내가 일부 불가피하지만, 학습 문서 본문에서는 그런 진행 중계도 덜어 냅니다.)
- `그 사이`, `이 과정`, `위 구조`, `이것` 같은 지시어는 무엇을 가리키는지 즉시 풀어 씁니다.
- 순차·인과·시간 순서는 번호 목록, 동등한 비교축은 불릿 목록을 씁니다. 짧은 항목은 한 줄로 닫고, 두세 문장 이상은 들여쓴 prose로 잇습니다.
- 코드나 로그가 핵심 근거면 인라인 주석으로 중요한 줄을 짚습니다(1.8).
- 직접 관측한 사실과 일반 규칙을 섞지 않습니다. 예: `이 환경에서는 A가 보인다. 일반적으로는 보통 B다.`처럼 분리해 적습니다.
- 핵심 차이를 이해하는 데 꼭 필요한 설명은 뒤로 미루지 않고, 그 차이를 처음 꺼낸 문단에서 바로 풉니다.
- 교차 참조는 평문(`3.1 참고`, `1.8에서 다룸`)으로 적습니다. 자동 생성 앵커가 플랫폼마다 달라 깨지기 쉬운 인라인 링크는 본문에서 피하고, 목차 정도로 제한합니다.

## 5. 근거와 검증

- 소스 우선순위: ① 프로젝트 내부 실험·실행 결과·코드·테스트 > ② 공식 문서·표준·레퍼런스·man page > ③ 신뢰할 수 있는 1차 자료 또는 레퍼런스 구현 > ④ 고신뢰 전문가 자료 > ⑤ 명시적 추론.
- 깊이를 채우는 사실은 가능한 한 위 ①~③의 1차 출처까지 내려가 확보합니다. 2차 요약이나 기억에만 기댄 값은 검증 대상으로 남깁니다(3.7).
- 사실·추론·미확인을 섞지 않습니다. 1차 자료로 닫지 못한 것은 `추론`, `가정`, `추가 확인 필요`로 낮춰 적습니다.
- 핵심 주장에는 검증 경로를 답니다: 무엇을 실행·확인하는가, PASS는 무엇인가, FAIL은 무엇인가. `확인해 보면 된다`는 검증 경로가 아닙니다.
- 공식 자료를 인용할 때는 `문서가 말한다`로 끝내지 않고, `자료에 따르면 X다 -> 그래서 Y가 필요하다 -> 그 결과 Z 구조가 등장한다`처럼 인과로 잇습니다.
- 서로 다른 자료가 충돌하면, 충돌 사실과 각각의 출처를 함께 적습니다.

## 6. 본받을 문서를 고르고 쓰는 법

이 가이드는 특정 모범 문서를 지정하지 않습니다(그래야 어떤 프로젝트에든 그대로 쓸 수 있습니다). 대신 당신 프로젝트 안에서 본받을 문서를 직접 고르고, 거기서 **무엇을 따르고 무엇을 거를지** 갈라 쓰는 방법을 정의합니다.

### 6.1 고르는 법

- 프로젝트에서 가장 잘 된 학습 문서 1개(필요하면 보조 1개)를 고릅니다. 분량이 많은 문서가 아니라, 위 1장 공통 DNA와 3장 깊이·넓이 기준을 가장 잘 만족하는 문서를 고릅니다.
- 새 문서를 쓸 때, 주제에 가장 가까운 문서를 그 기준점으로 삼습니다.
- 본받을 문서가 아직 없으면(가이드를 처음 도입하는 프로젝트라면), 이 가이드의 1장 DNA·3장 깊이 기준·부록 B 게이트가 곧 기준선입니다. 본받을 문서가 없다는 이유로 깊이를 낮추지 않습니다.

### 6.2 본받을 원리와 따라 하지 말 표면을 나눈다

모범 문서를 통째로 흉내 내지 않습니다. 두 가지를 갈라 적습니다.

- **본받을 설명 원리**: 다른 주제에도 옮길 수 있는, 이해를 돕는 구조(예: 역사로 배경을 닫는 방식, 단계별 trace, 다축 비교, 줄 단위 해부, 대조쌍 설명).
- **따라 하지 말 표면 습관(visible trait)**: 그 문서에만 있는 장식이거나, 오히려 품질을 떨어뜨리는 형태(예: 이모지 헤더 남발, 헤딩 과다 중첩, 원자료 통째 첨부, 도입부의 `정의 -> 특징 -> 장점` 라벨 나열).

### 6.3 흔한 함정

가장 잦은 실수는 **좋아 보이는 표면 습관을 본받을 원리로 착각**하는 것입니다. "잘 됐다"고 느낀 문서라도, 그 인상이 어디서 오는지 분해해야 합니다. 예컨대 풍부한 역사 서술과 비유는 본받을 원리지만, 같은 문서의 도입부가 라벨식 정의 나열이라면 그건 거를 표면입니다. "이 문서가 좋다"가 아니라 "이 문서의 무엇이 왜 좋다"까지 내려가야 기준으로 쓸 수 있습니다.

## 7. 금지 패턴

- 근거 없는 단정.
- 링크만 모아 둔 문서.
- 정의만 있고 왜 중요한지 없는 문서.
- 예제 없이 추상 설명만 반복하는 문서.
- 원자료를 통째로 붙여 놓고 줄 단위 해부 없이 넘어가는 것.
- 실행하지 않은 검증을 실행한 것처럼 적는 일.
- 표·역사 절·trace의 틀만 만들고 칸을 추측이나 빈약한 값으로 채우는 것(형식 모방).
- 독자가 다시 조립해야 하는 라벨식 나열(`정의 -> 특징 -> 장점`만 채우기).
- 추상화 계층 사이 연결이 끊긴 지나친 축약.
- 핵심 차이를 뒤 문단으로 미루고 현재 문단에서 닫지 않는 것.
- 분량이나 시간을 이유로 깊이·넓이·근거·검증을 줄이고 얕은 요약으로 타협하는 것.
- 모범 문서의 표면 습관을 원리로 착각해 복제하는 것.

## 부록 B. 발행 전 자가 점검

이 게이트의 목적은 통과시키는 것이 아니라 **깊이 미달을 걸러 내는 것**입니다. 그래서 두 단으로 나눕니다. 먼저 주제 규모를 정하고, hard gate부터 통과시킨 뒤 soft check로 다듬습니다.

### B.0 먼저 주제 규모를 정한다

- 이 주제는 역사·여러 구현·여러 계층·여러 비교 대상을 가진 **큰 주제**인가, 한 가지 동작·한 가지 답에 가까운 **좁은 주제**인가. 다음 중 하나라도 참이면 큰 주제로 봅니다(자의로 좁다고 낮추지 않습니다): 연도·인물이 있는 역사가 존재한다 / 구현·방식이 둘 이상이다 / 비교 대상이 둘 이상이다 / 여러 추상화 계층에 걸친다.
- 큰 주제일수록 3.6의 천장에 가깝게 가야 합니다. 좁은 주제는 그에 맞게 얕아도 되지만, 좁다는 핑계로 hard gate를 건너뛰지는 않습니다.
- 주제 유형(이론·코드·트러블슈팅)에 따라 어떤 천장이 적용되는지는 3.6을 따릅니다. 코드 사용기에 역사 8연도를 억지로 넣지 않습니다.

### B.1 Hard gate — 하나라도 걸리면 깊이 미달 (고치기 전에는 발행하지 않는다)

- [ ] 원자료(코드·로그·덤프)를 붙였는데 줄 단위 해부가 없다 (1.8)
- [ ] 동작·메커니즘을 다루는데 예상값이 아닌 실제 관측값이 하나도 없다 (1.7)
- [ ] 표·역사·trace의 칸을 1차 출처 없이 추측으로 채웠다 (3.7)
- [ ] 다대상 비교가 "A가 더 낫다"는 인상 한 줄로 끝났다(축별로 안 쪼갬) (3.3)
- [ ] 역사가 있는 주제인데 연도·인물·사건이 하나도 없다 (1.3, 3.1)
- [ ] **큰 주제인데 3.6 천장에 한참 못 미친다** — 예: 해당 유형인데 역사 한두 줄, 비교 축 두셋, "한 단계 아래" 질문 무응답, 설계 사고 5질문 거의 미응답. 이러면 더 팔 수 있는지 재작업한다. (3.1, 3.6)

### B.2 품질 점검 — 발행을 즉시 막지는 않지만 여전히 요구된다

여기 항목은 "있으면 좋은" 권고가 아닙니다. hard gate처럼 발행을 곧바로 막지 않을 뿐, 미충족분은 그만큼 얕은 요약으로 후퇴한 것입니다(이 가이드의 장치는 권고가 아니라 요구입니다 — 1장 도입 참조).

기본기

- [ ] 목차나 서두만 보고 다루는 범위가 예측되는가
- [ ] 초반에 질문에 대한 직답이 있는가
- [ ] 추상 설명보다 실제 artifact에서 출발했는가
- [ ] 단계별 trace로 메커니즘을 보였고, 같은 대상을 여러 시점에서 관측했는가 (1.4)
- [ ] 필요한 추상화 계층이 연결되고, 혼동쌍이 대칭으로 설명됐는가
- [ ] 사실 / 추론 / 미확인이 구분됐는가
- [ ] 메타 진행 멘트·미해소 지시어가 남지 않았는가
- [ ] 순차에는 번호 목록, 동등 비교에는 불릿 목록을 썼는가

깊이·넓이 (해당 유형에서)

- [ ] 설계 사고 5질문(문제·버린 가정·대안·부작용·정당화) 중 해당분을 답했는가 (3.1)
- [ ] 한 주제를 여러 면(역사·구현·비교·성능·한계·실무)에서 봤고, 뺀 면은 의식적으로 뺀 것인가 (3.2)
- [ ] (해당 시) 같은 목표의 naive vs 개선을 before/after로 대비했는가 (3.4)
- [ ] (넓은 주제) 깊이를 지키려 허브·상세 분리를 검토했는가 (3.5)

전이

- [ ] 독자가 이 문서로 가까운 문제를 하나 더 풀거나 틀린 설명을 반박할 수 있는가
- [ ] (모범 문서 참고 시) 본받을 원리만 취하고 표면 습관은 걸렀는가
