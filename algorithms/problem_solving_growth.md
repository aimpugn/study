## 이 문서가 답하려는 질문

알고리즘 문제 해결 능력은 정답 코드를 많이 외운다고 바로 쌓이지 않습니다.
반대로 매번 처음부터 원리를 새로 발견하려고 해도 오래 버티기 어렵습니다.
실제로 필요한 능력은 문제 문장을 읽고, 그 안에서 상태와 사건을 찾아내고, 어떤 순간에 답이 확정되는지 판단한 뒤, 그 판단을 코드의 자료구조, 루프, 조건문, 재귀, 테이블로 옮기는 능력입니다.

이 문서는 그 과정을 한 번에 다시 꺼낼 수 있게 만드는 기준 문서입니다.
문서를 덮은 뒤에는 최소한 아래 문장을 자기 말로 다시 설명할 수 있어야 합니다.

```text
자료구조는 먼저 고르는 것이 아니라,
문제에서 반복해서 답해야 하는 질문을 빠르게 처리하기 위해 붙인다.

루프는 후보나 시간이 한 번 움직이는 단위이고,
조건문은 그 후보가 지금 확정되는지, 보류되는지, 버려지는지를 가르는 질문이다.
```

여기서 말하는 능력은 `Queue를 외운다`, `DP 점화식을 외운다`, `다익스트라 코드를 외운다`보다 한 단계 앞에 있습니다.
물론 대표 알고리즘의 이름과 형태는 알아야 합니다.
다만 외워야 할 것은 코드의 줄 수가 아니라, 그 알고리즘이 어떤 사건을 언제 확정하고 어떤 상태를 계속 들고 가는지입니다.

## 왜 문제를 풀 때는 해설처럼 생각이 정리되지 않는가

해설은 대개 정리된 뒤의 사고를 보여 줍니다.
그래서 이런 문장으로 시작합니다.

```text
이 문제는 큐를 사용하면 됩니다.
이 문제는 다익스트라 알고리즘입니다.
이 문제는 0/1 배낭 문제이므로 DP로 풀 수 있습니다.
```

하지만 실제 풀이자의 머릿속은 보통 이렇게 시작합니다.

```text
무엇을 들고 있어야 하지?
큐를 쓴다 해도 큐에 무엇을 넣지?
언제 count를 올리지?
현재 반복문 한 바퀴가 무엇을 확정하지?
DP라면 dp[i]가 정확히 무엇을 의미하지?
```

이 간극이 알고리즘을 어렵게 만듭니다.
해설은 이미 답이 된 상태를 보여 주고, 풀이자는 아직 답이 되기 전의 흐릿한 상태에서 출발하기 때문입니다.

그러므로 문제를 풀 때의 목표는 곧바로 `Queue`, `PriorityQueue`, `DP`, `Dijkstra` 같은 이름을 맞히는 것이 아닙니다.
먼저 문제를 아래 세 가지로 다시 읽어야 합니다.

1. 무엇이 움직이는가

    시간, 인덱스, 사람, 프로세스, 트럭, 정점, 간선, 물건, 용량, 문자열의 위치처럼 문제 안에서 한 단계씩 바뀌는 대상이 있습니다.

2. 무엇이 아직 확정되지 않았는가

    아직 실행되지 않은 프로세스, 아직 다리를 건너지 않은 트럭, 아직 방문하지 않은 정점, 아직 고르지 않은 물건처럼 나중에 판단해야 할 대상이 남습니다.

3. 어떤 순간에 답의 일부가 확정되는가

    프로세스가 실제로 실행된 순간, 트럭이 다리를 완전히 지난 순간, 어떤 정점까지의 최단 거리가 확정된 순간, 어떤 물건까지 고려한 최댓값이 테이블에 저장된 순간이 있습니다.

이 세 가지가 보이면 자료구조는 뒤따라옵니다.
반대로 이 세 가지가 흐린 상태에서 자료구조 이름부터 고르면, 코드는 돌아가는 것 같지만 왜 맞는지 설명하기 어렵고 테스트에 끼워 맞춘 풀이가 되기 쉽습니다.

## 첫 번째 벽돌: `cnt++`는 언제 일어나는가

`42587 프로세스` 문제를 기준으로 보겠습니다.
이 문제에서 `cnt`는 반복문이 몇 번 돌았는지가 아니라, 프로세스가 실제로 몇 개 실행되었는지를 세는 값입니다.
그러므로 `cnt++`가 일어나는 위치는 "큐에서 하나를 꺼낸 순간"이 아니라 "그 프로세스가 실행된 순간"입니다.

작은 입력을 하나 놓고 따라가면 차이가 바로 보입니다.

```text
priorities = [2, 1, 3, 2]
location   = 2

대기 순서: 0번(2), 1번(1), 2번(3), 3번(2)
현재 최고 우선순위: 3
```

처음 큐 앞의 0번 프로세스를 꺼냅니다.
하지만 아직 우선순위 3인 프로세스가 남아 있으므로 0번은 실행되지 않고 뒤로 갑니다.
이때 큐에서 꺼냈다는 사건은 일어났지만, 프로세스 실행이라는 답의 단위는 확정되지 않았습니다.
따라서 `cnt++`를 하면 안 됩니다.

```text
step 1
front: 0번(2)
highest remaining priority: 3

2 == 3 ? 아니요

실행: 안 함
cnt: 0 유지
queue: 1번(1), 2번(3), 3번(2), 0번(2)
```

그다음 1번도 마찬가지입니다.

```text
step 2
front: 1번(1)
highest remaining priority: 3

1 == 3 ? 아니요

실행: 안 함
cnt: 0 유지
queue: 2번(3), 3번(2), 0번(2), 1번(1)
```

이제 2번 프로세스가 큐 앞에 옵니다.
현재 남은 프로세스 중 가장 높은 우선순위가 3이고, 2번의 우선순위도 3입니다.
이 순간에는 실행이 확정됩니다.
그래서 여기서만 `cnt++`가 맞습니다.

```text
step 3
front: 2번(3)
highest remaining priority: 3

3 == 3 ? 예

실행: 함
cnt: 1
location == 2 이므로 답: 1
```

이 trace에서 배워야 할 것은 `PriorityQueue` 사용법이 아닙니다.
더 중요한 문장은 이것입니다.

```text
반복문이 한 번 돌았다는 사실과, 문제에서 세려는 사건이 한 번 확정되었다는 사실은 다를 수 있다.
```

이 차이를 놓치면 `poll`을 실행으로 착각하고, `cnt++`를 잘못된 곳에 둡니다.
반대로 이 차이를 잡으면 루프와 조건문이 자연스럽게 나뉩니다.

```java
// 후보를 하나 본다.
int idx = queue.poll();

// 지금 실행 가능한지 확인한다.
if (priorities[idx] == highestPriority) {
    // 이때만 문제에서 세려는 사건이 확정된다.
    executedCount++;
} else {
    // 아직 확정되지 않았으므로 다시 대기 상태로 돌린다.
    queue.offer(idx);
}
```

## 문제는 자료구조 이름이 아니라 사건의 흐름으로 시작한다

문제를 처음 읽을 때 `이건 Queue인가?`, `DP인가?`, `Heap인가?`부터 묻기 쉽습니다.
하지만 더 안정적인 출발점은 사건입니다.
문제 문장에는 보통 동사가 있습니다.

```text
실행된다
대기한다
뒤로 간다
다리를 지난다
도착한다
선택한다
연결한다
방문한다
최단 거리를 구한다
가치를 최대로 만든다
```

이 동사들은 코드에서 상태 전이로 바뀝니다.
상태 전이는 "무엇이 어떤 규칙으로 바뀌는가"입니다.

```text
프로세스가 실행된다
-> 대기 큐에서 제거되고, 실행 횟수가 1 증가한다.

트럭이 다리에 올라간다
-> 다리 위 트럭 목록에 들어가고, 현재 무게가 증가한다.

트럭이 다리를 지난다
-> 다리 위 트럭 목록에서 빠지고, 현재 무게가 감소한다.

정점까지의 더 짧은 경로를 찾는다
-> dist[정점] 값이 더 작은 값으로 갱신되고, 후보 큐에 들어간다.

물건을 선택한다
-> 사용한 용량이 늘고, 총 가치가 늘어난다.
```

이렇게 바꾸면 코드 재료가 보이기 시작합니다.
자료구조는 이 상태 전이를 보존하거나 빠르게 판단하기 위한 도구입니다.

`42587 프로세스`에서는 대기 순서를 그대로 보존해야 하므로 `Queue`가 필요합니다.
동시에 매번 "현재 남은 프로세스 중 가장 높은 우선순위가 무엇인가"도 빨리 알아야 합니다.
이 질문은 대기 순서와 다른 관점이므로 `PriorityQueue`나 우선순위 개수 배열을 따로 둘 수 있습니다.

```text
대기 순서 관점:
queue = [0번, 1번, 2번, 3번]

판단 기준 관점:
남은 우선순위 = [3, 2, 2, 1]

두 상태는 같은 남은 프로세스 집합을 서로 다른 관점으로 들고 있다.
```

이 문장이 중요합니다.
처음부터 "Queue와 PriorityQueue를 써야지"가 아니라, "대기 순서와 우선순위 판단이 서로 다르구나"를 먼저 봅니다.
그 다음에야 상태를 둘로 나눌 이유가 생깁니다.

## 풀이 전 5분: 문제를 코드 재료로 바꾸는 질문

문제를 읽고 바로 코드를 쓰기 전에 5분만 아래 질문을 적습니다.
처음에는 느리게 느껴지지만, 이 단계가 없으면 코드 중간에서 루프와 조건이 흔들립니다.

1. 답은 무엇을 세거나, 고르거나, 최소화하거나, 최대화하는가

    `프로세스`에서는 "몇 번째로 실행되는가"가 답입니다.
    `다리를 지나는 트럭`에서는 "모든 트럭이 건너는 데 걸리는 시간"이 답입니다.
    `입국심사`에서는 "모든 사람이 심사를 끝내는 최소 시간"이 답입니다.
    이 질문이 닫혀야 `answer`, `count`, `time`, `dist`, `dp` 같은 핵심 변수가 의미를 얻습니다.

2. 답의 일부가 확정되는 순간은 언제인가

    프로세스는 실행될 때, 트럭은 다리를 완전히 빠져나갈 때, BFS 최단거리는 해당 칸을 처음 방문할 때, 다익스트라는 가장 작은 비용 후보가 현재 거리와 일치해서 꺼내질 때 확정됩니다.
    이 순간이 보이면 `cnt++`, `answer.add(...)`, `visited = true`, `dist 갱신`, `return` 위치가 흔들리지 않습니다.

3. 루프 한 바퀴는 무엇을 하는가

    루프 한 바퀴가 시간 1초인지, 큐 앞 후보 하나를 보는 것인지, 배열 원소 하나를 처리하는 것인지, 상태 하나를 확장하는 것인지 정해야 합니다.
    이 단위가 흐리면 반복문 안에서 너무 많은 일을 하거나, 반대로 필요한 상태 갱신을 빠뜨립니다.

4. 아직 확정되지 않은 것은 무엇인가

    아직 실행되지 않은 프로세스, 아직 다리 위에 있는 트럭, 아직 방문하지 않은 칸, 아직 고려하지 않은 물건, 아직 확정되지 않은 최단거리 후보가 여기에 해당합니다.
    대부분의 자료구조는 이 "아직 확정되지 않은 것"을 담습니다.

5. 매번 비싸게 묻는 질문은 무엇인가

    "남은 것 중 최댓값은?", "이미 본 적 있는가?", "이 값이 몇 개 남았는가?", "가장 가까운 후보는?", "이 시간이 가능한가?" 같은 질문입니다.
    이 반복 질문이 자료구조 선택으로 이어집니다.

6. 가장 느린 풀이는 무엇인가

    완전 탐색, 매번 전체 스캔, 모든 조합 생성, 직접 시뮬레이션 같은 기준선을 먼저 생각합니다.
    기준선은 틀린 풀이가 아니라, 더 빠른 풀이가 왜 필요한지 알려 주는 출발점입니다.

7. 작은 입력에서 상태가 어떻게 움직이는가

    최소 하나의 입력을 손으로 끝까지 따라갑니다.
    이때 반드시 "답이 바뀌는 순간"과 "답이 바뀌면 안 되는 순간"을 둘 다 봅니다.

이 7개 질문은 문제를 푸는 동안 계속 되돌아갈 수 있는 기준입니다.
문제가 어려워질수록 질문이 사라지는 것이 아니라, 오히려 질문의 중요도가 올라갑니다.

## 자료구조는 반복 질문을 빠르게 답하기 위해 붙인다

자료구조 선택은 감으로만 하는 일이 아닙니다.
문제에서 반복해서 묻는 질문을 먼저 찾고, 그 질문에 맞는 도구를 붙입니다.

| 반복해서 묻는 질문 | 자연스러운 도구 | 대표 상황 |
| --- | --- | --- |
| 먼저 들어온 순서대로 봐야 하는가 | `Queue`, 배열 인덱스 | 대기열, BFS, 시간 순서 시뮬레이션 |
| 최근에 넣은 것부터 취소하거나 짝을 맞추는가 | `Stack` | 괄호, 뒤로가기, 이전 상태 복원 |
| 가장 작은 값이나 큰 값을 계속 골라야 하는가 | `PriorityQueue`, 정렬, 개수 배열 | 힙, 작업 스케줄링, 다익스트라 후보 |
| 이미 본 값인지 빠르게 알아야 하는가 | `Set`, `boolean[]` | 방문 체크, 중복 제거 |
| 값별 개수나 그룹 상태가 필요한가 | `Map`, count 배열 | 완주하지 못한 선수, 의상, 우선순위 개수 |
| 연결된 이웃으로 퍼져 나가는가 | 그래프, `Queue`, 재귀, `visited` | DFS, BFS, 네트워크, 미로 |
| 같은 작은 문제가 반복되는가 | 메모이제이션, DP 테이블 | 피보나치, 정수 삼각형, 배낭 |
| 답을 직접 만들기 어렵지만 가능 여부는 판정할 수 있는가 | 이분 탐색 | 입국심사, 예산, 거리 최적화 |

중요한 점은 자료구조가 문제의 "본질"을 대신 찾아 주지 않는다는 것입니다.
`PriorityQueue`를 안다고 해서 `프로세스` 문제가 바로 풀리지는 않습니다.
먼저 "대기 순서"와 "현재 최고 우선순위 판단"이 분리된다는 것을 봐야 합니다.
그때 `Queue`와 `PriorityQueue`의 조합이 의미를 가집니다.

반대로 어떤 문제에서는 같은 질문을 더 단순한 도구로 답할 수 있습니다.
우선순위가 1부터 9까지처럼 작게 제한된다면 `PriorityQueue` 대신 `int[10]` 개수 배열로 현재 최고 우선순위를 찾을 수 있습니다.
이 선택은 구현 취향이 아니라, 반복 질문의 범위가 작기 때문에 가능한 선택입니다.

```text
질문: 아직 남은 우선순위 중 가장 큰 값은?

우선순위 범위가 작다:
count[priority]를 쓰고 9부터 내려오며 찾는다.

우선순위 범위가 크거나 일반적이다:
PriorityQueue를 쓴다.
```

이렇게 생각하면 자료구조 선택이 외운 정답이 아니라 근거 있는 판단이 됩니다.

## 루프와 조건문은 어떻게 정하는가

루프는 "한 번에 무엇이 움직이는가"를 코드로 만든 것입니다.
조건문은 "그 움직임이 답을 확정하는가, 보류하는가, 버리는가, 확장하는가"를 가르는 질문입니다.

`while`과 `for`를 정할 때는 먼저 루프 단위를 말로 씁니다.

```text
배열을 왼쪽부터 한 원소씩 처리한다.
-> for (int i = 0; i < n; i++)

큐에서 아직 처리할 상태가 남아 있는 동안 하나씩 꺼낸다.
-> while (!queue.isEmpty())

시간을 1초씩 흘려 보낸다.
-> while (아직 끝나지 않았다) { time++; ... }

가능한 답의 범위를 절반씩 줄인다.
-> while (left <= right)
```

그 다음 조건문을 한국어 질문으로 바꿉니다.
좋은 `if`는 보통 한 문장으로 읽힙니다.

```text
현재 후보가 지금 실행 가능한가?
현재 칸이 범위 안이고 아직 방문하지 않았는가?
새 경로가 기존 거리보다 더 짧은가?
현재 물건을 이 용량 안에 넣을 수 있는가?
이 시간 안에 모든 사람을 처리할 수 있는가?
```

조건문을 만들 때 가장 중요한 것은 true일 때와 false일 때 상태가 어떻게 달라지는지입니다.
`프로세스` 문제의 핵심 조건은 다음처럼 닫힙니다.

```text
질문:
현재 프로세스의 우선순위가 남은 프로세스 중 최고 우선순위인가?

true:
실행한다.
실행 횟수를 1 증가시킨다.
남은 우선순위 상태에서도 이 프로세스를 제거한다.
목표 프로세스라면 현재 실행 횟수가 답이다.

false:
실행하지 않는다.
실행 횟수는 증가하지 않는다.
대기 큐 뒤에 다시 넣는다.
남은 우선순위 상태는 그대로 둔다.
```

이런 방식으로 쓰면 조건문은 테스트에 맞춘 분기가 아니라 문제 규칙의 번역이 됩니다.

특히 아래 네 가지 값은 위치를 잘못 잡기 쉽습니다.

1. `count++`

    문제에서 세는 사건이 실제로 확정된 뒤에만 증가합니다.
    후보를 봤다는 이유만으로 증가하지 않습니다.

2. `visited = true`

    다시 방문하면 안 되는 상태가 확정되는 순간에 표시합니다.
    BFS에서는 큐에 넣는 순간 방문 처리하는 편이 중복 삽입을 막기 쉽습니다.

3. `dist[next] = newCost`

    더 좋은 경로를 발견한 순간에 갱신합니다.
    다만 다익스트라에서는 나중에 더 낡은 후보가 큐에서 나올 수 있으므로, 꺼냈을 때 `cost > dist[node]`이면 버립니다.

4. `answer.add(...)`

    하나의 배포 묶음, 하나의 경로, 하나의 조합, 하나의 결과가 더 이상 바뀌지 않는 순간에 추가합니다.
    아직 뒤 원소를 더 봐야 묶음 크기가 정해지는 경우에는 너무 일찍 추가하면 안 됩니다.

루프와 조건문이 어렵게 느껴지는 이유는 문법 때문이 아닙니다.
대부분은 "이 반복이 무엇을 확정하는지"와 "이 조건의 양쪽에서 상태가 어떻게 바뀌는지"가 아직 말로 닫히지 않았기 때문입니다.

## 재귀, DFS, BFS, DP는 같은 질문에서 갈라진다

재귀를 쓸지, BFS를 쓸지, DP를 쓸지는 따로 떨어진 감각이 아닙니다.
모두 "상태 하나가 무엇을 의미하는가"에서 출발합니다.

재귀는 한 호출이 하나의 작은 문제나 상태를 대표할 수 있을 때 자연스럽습니다.

```text
dfs(node)
-> node에서 시작해 갈 수 있는 곳을 모두 방문한다.

search(index, sum)
-> index번째 선택까지 고려했고 현재 합이 sum인 상태에서 가능한 결과를 탐색한다.

solve(i, w)
-> i번째 물건부터 고려할 때, 남은 용량이 w이면 얻을 수 있는 최대 가치
```

재귀를 쓰기 전에는 아래 질문을 먼저 답해야 합니다.

```text
이 함수 호출 하나는 어떤 상태를 뜻하는가?
언제 멈추는가?
다음 호출로 갈 때 무엇이 줄어들거나 바뀌는가?
호출이 끝난 뒤 되돌려야 하는 상태가 있는가?
같은 상태를 다시 만나면 메모나 방문 체크가 필요한가?
```

DFS와 BFS의 차이는 "어떤 순서로 상태를 꺼내 보는가"입니다.
DFS는 한 갈래를 깊게 들어가며, BFS는 가까운 상태부터 넓게 퍼집니다.
간선 비용이 모두 같고 최단 이동 횟수를 묻는 문제에서는 BFS가 자연스럽습니다.
처음 방문한 순간이 곧 최단 거리 확정 순간이 되기 때문입니다.

```text
미로에서 시작점 S부터 이동 횟수를 구한다.

queue: [S]
visited[S] = true

큐에서 하나를 꺼낸다.
인접한 칸 중 범위 안이고 벽이 아니며 아직 방문하지 않은 칸을 큐에 넣는다.
처음 방문한 칸의 거리는 현재 거리 + 1로 확정된다.
```

DP는 같은 상태를 여러 경로에서 다시 만나고, 그 상태의 답을 재사용할 수 있을 때 등장합니다.
여기서 `dp[i]`나 `dp[i][w]`는 배열 이름이 아니라 문장입니다.
문장으로 말할 수 없으면 아직 DP 상태가 아닙니다.

```text
dp[i] = i번째 칸까지 왔을 때 얻을 수 있는 최대 점수
dp[i][w] = 앞에서 i개의 물건을 고려했고, 용량이 w일 때 얻을 수 있는 최대 가치
dp[count] = N을 count개 사용해서 만들 수 있는 수들의 집합
```

DP가 어렵게 느껴질 때는 점화식부터 쓰지 말고, 상태 문장부터 씁니다.
그 다음 "마지막 선택이 무엇인가"를 묻습니다.

```text
배낭 문제:
마지막으로 i번째 물건을 볼 때 선택지는 두 개다.

선택하지 않는다:
dp[i][w] = dp[i - 1][w]

선택한다:
dp[i][w] = dp[i - 1][w - weight[i]] + value[i]

둘 중 더 큰 값이 현재 상태의 답이다.
```

이 흐름은 재귀와 같습니다.
재귀가 상태 공간을 직접 탐색한다면, DP는 반복되는 상태의 답을 저장하고 재사용합니다.

## 어려운 알고리즘도 같은 뼈대 위에 올린다

다익스트라나 배낭 문제를 매번 처음부터 재발명할 필요는 없습니다.
대표 알고리즘은 반드시 배워야 합니다.
다만 외울 때도 코드가 아니라 "상태, 반복 질문, 확정 순간, 깨지는 조건"을 함께 외워야 오래 남습니다.

### 다리를 지나는 트럭

`42583 다리를 지나는 트럭`은 시간 시뮬레이션 문제입니다.
처음부터 "큐 문제"라고 외우기보다, 사건을 먼저 봅니다.

```text
시간이 1초 흐른다.
다리 위 트럭은 한 칸씩 앞으로 간다.
맨 앞 트럭이 다리 길이만큼 이동했으면 다리에서 빠진다.
새 트럭을 올려도 무게 제한을 넘지 않으면 올린다.
모든 트럭이 빠져나간 시간이 답이다.
```

여기서 상태는 세 가지로 나뉩니다.

```text
아직 대기 중인 트럭 순서
다리 위 트럭과 각 트럭이 들어온 시간 또는 남은 거리
현재 다리 위 무게
```

이 문제의 `while`은 보통 "아직 모든 트럭이 다리를 지나지 않았다"입니다.
`if`는 "지금 새 트럭을 올릴 수 있는가"와 "지금 빠져나갈 트럭이 있는가"입니다.
답을 증가시키는 단위는 트럭 수가 아니라 시간입니다.
따라서 이 문제에서는 `time++`이 루프의 핵심 움직임이고, `answer`는 마지막 트럭이 빠져나간 시간으로 확정됩니다.

### 다익스트라 알고리즘

다익스트라는 가중치가 음수가 아닌 그래프에서 시작점으로부터 각 정점까지의 최단 거리를 구할 때 쓰는 알고리즘입니다.
외워야 할 핵심은 "가장 짧아 보이는 후보부터 확정한다"입니다.

```text
상태:
dist[v] = 지금까지 발견한 시작점 -> v의 최단 후보 거리

반복 질문:
아직 확정되지 않은 후보 중 비용이 가장 작은 것은 무엇인가?

자료구조:
PriorityQueue

확정 순간:
우선순위 큐에서 꺼낸 비용이 현재 dist[v]와 같고,
모든 간선 가중치가 음수가 아니라면,
그 v까지의 최단 거리는 더 줄어들 수 없다.

깨지는 조건:
음수 간선이 있으면 나중에 더 싼 경로가 뒤늦게 나타날 수 있으므로 이 확정 논리가 깨진다.
```

작은 trace로 보면 이렇습니다.

```text
간선:
A -> B (2)
A -> C (5)
B -> C (1)

초기:
dist[A]=0, dist[B]=무한, dist[C]=무한
pq = (A,0)

1. (A,0)을 꺼낸다.
   A의 거리는 확정된다.
   B 후보 2, C 후보 5를 넣는다.

   dist = A:0, B:2, C:5
   pq = (B,2), (C,5)

2. (B,2)를 꺼낸다.
   B의 거리는 확정된다.
   B -> C를 보면 C 후보가 2 + 1 = 3으로 줄어든다.

   dist = A:0, B:2, C:3
   pq = (C,3), (C,5)

3. (C,3)을 꺼낸다.
   C의 거리는 확정된다.

4. 나중에 (C,5)가 나오면 이미 dist[C]=3이므로 버린다.
```

여기서 `PriorityQueue`는 알고리즘의 이름이 아니라 반복 질문을 빠르게 답하는 도구입니다.
반복 질문은 "현재 가장 싼 후보가 무엇인가"입니다.

### 배낭 문제

0/1 배낭 문제는 각 물건을 한 번씩만 선택하거나 선택하지 않을 수 있고, 용량 제한 안에서 가치를 최대화하는 문제입니다.
여기서 시작 질문은 "최대 가치"가 아니라 "선택이 쌓이는가"입니다.

```text
사건:
각 물건을 볼 때 선택하거나 선택하지 않는다.

상태:
앞에서 i개의 물건을 고려했다.
현재 사용할 수 있는 용량은 w다.
이때 얻을 수 있는 최대 가치를 알고 싶다.

상태 문장:
dp[i][w] = 앞에서 i개의 물건을 고려하고 용량이 w일 때 얻을 수 있는 최대 가치

확정 순간:
i번째 물건까지 고려한 뒤,
각 용량 w에 대해 선택/비선택 중 더 나은 값이 dp[i][w]로 확정된다.
```

이 상태 문장이 있으면 점화식은 선택지 비교로 나옵니다.

```text
i번째 물건을 넣지 않는다:
dp[i][w] = dp[i - 1][w]

i번째 물건을 넣는다:
dp[i][w] = dp[i - 1][w - weight[i]] + value[i]

둘 중 큰 값:
dp[i][w] = max(넣지 않는 경우, 넣는 경우)
```

DP를 외운다는 것은 `for`문 모양을 외우는 일이 아닙니다.
`dp[i][w]`가 어떤 질문의 답인지 문장으로 고정하는 일입니다.

### 이분 탐색으로 답을 찾는 문제

이분 탐색은 정렬 배열에서 값을 찾는 데만 쓰이지 않습니다.
답 자체가 숫자이고, 어떤 값이 가능한지 판정할 수 있으며, 가능/불가능이 한쪽 방향으로 이어지면 답의 범위를 줄일 수 있습니다.

`입국심사`를 예로 들면 질문은 이렇습니다.

```text
시간 T가 주어졌을 때,
심사관들이 T분 동안 처리할 수 있는 사람 수가 n명 이상인가?
```

이 질문에 답할 수 있으면 `T`를 직접 하나씩 늘릴 필요가 없습니다.

```text
T가 충분하다면:
더 큰 시간도 충분하다.
답은 왼쪽에 더 작게 있을 수 있다.

T가 부족하다면:
더 작은 시간도 부족하다.
답은 오른쪽에 있다.
```

여기서 루프 한 바퀴는 시간을 실제로 흘리는 것이 아닙니다.
후보 답 `mid` 하나를 검사하고, 답의 범위를 절반으로 줄이는 것입니다.

## 문제를 푼 뒤 지식으로 남기는 방법

문제를 맞혔다고 해서 자동으로 실력이 쌓이지는 않습니다.
맞힌 뒤에 "다음에 무엇을 떠올릴 것인가"를 남겨야 합니다.
한 문제에서 남길 것은 긴 해설이 아니라, 다시 꺼낼 수 있는 짧은 판단 경로입니다.

풀이 후에는 아래 6줄만 적어도 충분합니다.

```text
문제:

문제 신호:
어떤 문장을 보고 이 패턴을 의심했는가?

상태:
무엇을 몇 개의 관점으로 들고 갔는가?

반복 질문:
루프 한 바퀴마다 무엇을 물었는가?

확정 순간:
언제 count, answer, visited, dist, dp가 바뀌었는가?

기억 문장:
다음 문제에서 가져갈 한 문장은 무엇인가?
```

`42587 프로세스`라면 이렇게 남길 수 있습니다.

```text
문제:
프로그래머스 42587 프로세스

문제 신호:
대기 순서는 앞에서 꺼내 뒤로 보내지만, 실행 여부는 남은 우선순위의 최댓값으로 판단한다.

상태:
대기 순서는 Queue, 남은 우선순위 판단은 PriorityQueue 또는 count 배열로 분리한다.

반복 질문:
큐 앞 프로세스가 현재 남은 최고 우선순위인가?

확정 순간:
그 질문이 true일 때만 프로세스가 실행되고 cnt가 증가한다.

기억 문장:
실행 순서와 판단 기준이 다르면, 같은 남은 대상을 두 관점의 상태로 나누어 들고 간다.
```

이 기록이 쌓이면 "많이 풀었는데 쉬면 잊어버리는 상태"에서 조금씩 벗어납니다.
까먹지 않는 사람이 되는 것이 목표가 아닙니다.
까먹어도 다시 복원할 수 있는 경로를 남기는 것이 목표입니다.

반복은 네 번으로 나눕니다.

1. R1: 처음 풀이와 피드백

    지금처럼 풀고, 어디서 상태와 확정 순간이 흔들렸는지 찾습니다.

2. R2: 2-3일 뒤 빈 파일에서 다시 구현

    코드를 외우는지 보는 것이 아니라, 상태와 조건을 다시 만들 수 있는지 봅니다.

3. R3: 일주일 뒤 코드 없이 설명

    "상태는 무엇이고, 루프는 무엇을 확정하고, 조건문은 어떤 질문인가"를 말로 설명합니다.

4. R4: 2-3주 뒤 제한 시간 안에 재풀이

    실전 속도에서 다시 꺼낼 수 있는지 확인합니다.

R1은 이해, R2는 구현 복원, R3는 개념 인출, R4는 실전 속도를 훈련합니다.
같은 문제를 네 번 푸는 것이 아니라, 같은 문제로 네 가지 능력을 따로 훈련하는 것입니다.

## 막혔을 때 되돌아갈 위치

막혔을 때는 "나는 이 알고리즘을 모른다"로 바로 가지 않는 편이 좋습니다.
대부분은 더 앞 단계의 질문이 닫히지 않은 상태입니다.

아래 순서로 되돌아갑니다.

1. 답의 단위를 다시 확인합니다.

    무엇을 반환해야 하는지, 무엇을 세야 하는지, 무엇을 최소화해야 하는지 다시 씁니다.
    `count`가 반복 횟수인지, 성공 사건 수인지, 묶음 수인지 분리합니다.

2. 확정 순간을 다시 찾습니다.

    지금 이 후보를 봤다는 사실만으로 답이 바뀌는지, 아니면 더 기다려야 하는지 확인합니다.
    이 질문은 `cnt++`, `answer.add`, `visited`, `return` 위치를 잡아 줍니다.

3. 아주 작은 입력을 손으로 끝까지 따라갑니다.

    길이가 3-5 정도인 입력으로 상태를 표처럼 그립니다.
    정답이 바뀌는 순간과 바뀌면 안 되는 순간을 표시합니다.

4. 느린 풀이를 먼저 말합니다.

    매번 전체를 스캔해도 되는지, 모든 조합을 만들어도 되는지, 직접 시뮬레이션하면 어떻게 되는지 말합니다.
    느린 풀이가 없으면 빠른 풀이도 근거 없이 떠다닙니다.

5. 반복해서 비싼 질문을 찾습니다.

    느린 풀이에서 매번 오래 걸리는 질문을 찾습니다.
    그 질문이 최댓값이면 힙이나 정렬, 존재 여부면 Set, 개수면 Map, 가능 여부면 이분 탐색 후보가 됩니다.

6. 조건문의 양쪽 상태 변화를 씁니다.

    `if`가 true일 때 어떤 상태가 바뀌고, false일 때 어떤 상태가 바뀌는지 말로 씁니다.
    한쪽이 비어 있거나 둘 다 같은 일을 한다면 조건이 잘못 잡혔을 가능성이 있습니다.

7. 대표 알고리즘과 맞춰 봅니다.

    그래도 막히면 그때 알고리즘 이름을 가져옵니다.
    단, 이름만 가져오지 말고 그 알고리즘의 상태 문장과 확정 순간을 같이 가져옵니다.

이 순서는 문제를 쉽게 만드는 마법은 아닙니다.
다만 막힘을 "나는 못한다"가 아니라 "어느 질문이 아직 비어 있는가"로 바꿔 줍니다.
이렇게 바뀌면 복구가 가능합니다.

## 이 문서를 쓰는 방법

새 문제를 풀 때 이 문서를 처음부터 끝까지 다시 읽을 필요는 없습니다.
대신 아래 짧은 루틴을 씁니다.

풀이 전에는 이 5줄을 적습니다.

```text
답의 단위:
확정 순간:
루프 한 바퀴:
아직 남는 상태:
반복해서 비싼 질문:
```

풀이 중에는 작은 trace를 하나 그립니다.

```text
초기 상태:

step 1:
무엇을 봤는가:
어떤 질문을 했는가:
무엇이 바뀌었는가:
무엇은 아직 바뀌면 안 되는가:

step 2:
...
```

풀이 후에는 기억 문장을 하나 남깁니다.

```text
다음에 비슷한 문제를 보면 떠올릴 한 문장:
```

이 문서의 핵심을 한 문장으로 줄이면 다음과 같습니다.

```text
알고리즘 문제 해결 능력은 정답 코드를 외우는 능력이 아니라,
문제 문장을 상태 변화와 확정 순간으로 다시 읽고,
그 반복 질문에 맞는 자료구조와 제어 흐름을 붙이는 능력이다.
```

이 문장을 매 문제마다 조금씩 더 구체적인 기억 문장으로 바꾸면 됩니다.
`프로세스`에서는 "실행 순서와 판단 기준이 다르면 상태를 나눈다"가 되고, `다리를 지나는 트럭`에서는 "시간, 다리 위 상태, 현재 무게가 함께 움직인다"가 되고, 다익스트라에서는 "가장 싼 미확정 후보가 현재 거리와 일치할 때 최단 거리가 확정된다"가 됩니다.

그 문장들이 쌓이면 문제 수만 늘어나는 것이 아니라, 다음 문제를 읽을 때 다시 꺼낼 수 있는 판단 경로가 늘어납니다.
