## 어디서 막혔나 — 실제 실수의 기록

이 절은 자기비난이 아니라 재발 방지 장치입니다. 이번 막힘은 조금 특별한데, **틀린 답이 아니라 "머릿속에 안 그려짐"이 막힘이었습니다.** 스택에 값을 쌓아 꺼내는 풀이는 실제로 통과했지만, 풀이법이 바로 떠오르지 않았고 그 과정이 그림으로 잡히지 않았습니다. 안 그려지는 막힘은 다음에 더 어려운 포인터 문제(사이클, 재배열)에서 그대로 재발하므로, 그 형태를 정확히 적어 둡니다.

첫 번째는 **두 변수의 역할 혼동**이었습니다. 스택 풀이에서 새 리스트를 만들 때 `reversed`(돌려줄 머리)와 `currNode`(끝에 붙여 가며 걷는 손, 처음엔 `tmp`라 이름 붙였던 것)를 동시에 들고 있어야 했는데, 어느 것을 반환하고 어느 것이 움직이는지가 헷갈렸습니다. 핵심을 한 문장으로 줄이면 이렇습니다. **리스트를 만들 땐 손이 둘 필요하다 — 머리는 가만히 둬야 반환할 수 있고, 끝을 가리키는 손만 붙일 때마다 전진한다.** 손이 하나뿐이면 다 만든 뒤 끝에 가 있어 머리를 잃습니다.

두 번째이자 더 본질적인 것은 **힙 참조 모델이 안 그려진 것**입니다. `currNode = currNode.next`와 `currNode.next = ...`이 무엇을 바꾸는지가 흐릿했습니다. 막힘의 뿌리는 변수를 "노드 그 자체"로 본 데 있습니다. 변수는 노드가 아니라 **힙에 있는 노드를 가리키는 화살표**입니다. 그래서 `currNode = currNode.next`는 currNode 화살표만 다음 노드로 옮기는 일이고, `currNode.next = X`는 currNode가 가리키는 그 노드의 next 칸을 바꾸는 일입니다. 화살표를 옮기는 것과 화살표가 가리키는 노드를 고치는 것은 다릅니다 — 이 구분이 안 서면 연결 리스트의 모든 줄이 흐려집니다.

## 핵심 사고 — 왜 이렇게 푸는가

카드가 떠오르지 않을 때 이 절만 읽고 같은 결론에 도달하도록, 두 갈래를 사고 순서대로 적습니다.

**값으로 우회하기(스택 — 처음 떠올린 길).** 스택은 LIFO라, 값을 다 넣고 꺼내면 꺼내는 순서가 곧 뒤집힌 순서입니다. 포인터를 직접 뒤집는 어려움을 값 복사로 피해 가는 길이고, 대가는 스택과 새 노드로 쓰는 O(n) 공간입니다. 여기서 들고 가야 하는 상태가 위에서 말한 "머리 손 + 끝 손" 둘입니다.

**제자리에서 뒤집기(in-place — 면접 기본값).** 더 나은 길은 새 리스트를 만들지 않고 기존 노드들의 화살표를 그 자리에서 거꾸로 잇는 것입니다. 전부 한 씨앗에서 따라 나옵니다 — **"뒤집기 = 각 노드가 [다음] 대신 [앞] 노드를 가리키게 한다."** 각 노드가 앞을 가리켜야 하니 걸어가며 "내 앞 노드"를 들고 있어야 하고(그래서 `prev`), 첫 노드는 뒤집으면 꼬리가 되어 그 앞이 없으니(`prev = null`로 시작), `cur.next = prev`로 화살표를 돌리는 순간 원래 다음 노드를 잃으니 그 전에 붙잡아 두고(`next = cur.next`), 다음 노드의 앞은 지금 cur이라 둘 다 한 칸 밀고(`prev = cur; cur = next`), cur이 null이면 끝나 그때 prev가 새 머리입니다(`return prev`). 초기값도 순서도 종료도 외운 게 아니라 이 씨앗에서 도출됩니다.

세션에서 직접 막혔던 질문 — "힙에 있는 걸 참조한다는 게 안 그려진다" — 의 답이 이 두 갈래를 관통합니다. 변수는 노드를 가리키는 화살표이고, 뒤집기란 결국 그 화살표들의 방향을 바꾸는 일입니다.

## 손으로 따라가기 — `[1,2,3]`의 in-place 뒤집기

화살표 방향이 곧 각 노드의 next입니다. 한 단계가 노드 하나를 뒤집습니다.

```
시작:    null    1 -> 2 -> 3 -> null
         ^       ^
         prev    cur

cur=1:   next=2 붙잡고 / 1.next=prev(null) / prev=1 / cur=2
         null <- 1    2 -> 3 -> null            // 1의 화살표가 뒤로 돌았다. 1과 2 사이는 끊겼다
                 ^    ^
                 prev cur

cur=2:   next=3 / 2.next=1 / prev=2 / cur=3
         null <- 1 <- 2    3 -> null
                     ^    ^
                     prev cur

cur=3:   next=null / 3.next=2 / prev=3 / cur=null
         null <- 1 <- 2 <- 3    null
                          ^    ^
                          prev cur(=null) -> 루프 끝

끝 → return prev (=3) -> [3,2,1]
```

`cur=1` 단계가 이 문제의 전부입니다. `cur.next = prev`로 화살표를 돌리는 순간 1과 2 사이가 끊겨, next로 2를 먼저 붙잡아 두지 않았다면 나머지 리스트를 통째로 잃습니다.

## 뼈대 압축

```
prev = null, cur = head
while (cur != null):
    next = cur.next      # (1) 다음을 먼저 붙잡는다 -- 끊기기 전에
    cur.next = prev      # (2) 화살표를 뒤로 돌린다
    prev = cur           # (3) prev 한 칸
    cur  = next          # (4) cur  한 칸
return prev              # cur이 null이 되면 prev가 새 머리

시간 O(n) -- 노드를 한 번씩 본다. 공간 O(1) -- 새 노드 없이 화살표만 다시 잇는다.
스택 우회는 같은 O(n) 시간이지만 공간 O(n). 재귀도 가능하나 호출 스택 O(n).
```

## 패턴 전이 — 이 뼈대가 다시 나오는 곳

이 문제의 뼈대는 **"다음을 붙잡고, 화살표를 다시 잇고, 한 칸 전진"**입니다. 연결 리스트를 손대는 거의 모든 문제가 이 동작을 공유합니다. Reorder List(LC 143)는 뒤 절반을 이 방식으로 뒤집은 뒤 병합하고, Linked List Cycle(LC 141)은 화살표를 따라 빠른 손과 느린 손이 만나는지를 보고, Remove Nth Node(LC 19)는 두 손의 간격을 n으로 벌려 걷습니다. 리스트를 새로 만드는 쪽 문제 — Merge Two Lists(LC 21) — 는 "머리는 dummy.next, 커서 하나로 끝에 붙인다"는 위의 빌드 패턴을 그대로 씁니다.

한 층 더 내려가면 공통 원리는 **"노드를 옮기기 전에 다음을 먼저 붙잡는다"**입니다. 포인터를 덮어쓰면 원래 가리키던 곳을 잃는다는 것 — 이 한 문장이 배열의 in-place 이동, 그래프의 부모 포인터 끊기까지 관통합니다.

## 코드에서 배운 것

스택 우회와 in-place의 차이는 차수가 아니라 공간입니다. 둘 다 노드를 한 번씩 보는 O(n) 시간이지만, 스택은 값을 복제해 O(n) 공간을 쓰고 in-place는 기존 화살표만 다시 이어 O(1)입니다. 면접 기본값이 in-place인 이유가 여기 있습니다.

리스트를 만들 땐 dummy(가짜 머리) 노드 하나가 머리/커서 혼동을 없앱니다. 진짜 머리가 늘 dummy.next로 고정돼, 첫 노드를 특수 처리할 필요도 빈/한 개 입력을 거르는 if 가드도 사라집니다.
