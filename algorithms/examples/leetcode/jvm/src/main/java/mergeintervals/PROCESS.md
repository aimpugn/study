# Merge Intervals (LeetCode 56) — 풀이 회고

> **이 문서를 쓰는 법.** 이 문서의 일은 두 가지다. 다음에 비슷한 문제를 만났을 때 핵심을 빠르게 깨우는 것(인출 단서), 그리고 깨우기에 실패했을 때 사고를 처음부터 다시 쌓아 주는 것(복원 경로). 그래서 위에서부터 읽지 않는다. **먼저 바로 아래 30초 카드만 보고 코드를 머릿속으로 떠올려 본다.** 떠오르면 통과. 안 떠오르면 실패가 아니라 정상이다 — 막힌 칸만 골라 아래 산문을 펼치면 된다.

## 30초 카드

```
문제 신호 : "구간 배열에서 겹치는 구간을 모두 병합하라"
한 줄     : start로 정렬 → '지금까지 만든 마지막 구간'을 들고 가며, 겹치면 끝을 max로 늘린다
키워드    : ① 정렬 = 정규화   ② 비교 상대는 '결과의 마지막'   ③ 겹치면 새 칸 추가 X, 끝만 max
함정 단서 : 정렬 키를 end로 잡지 않았나? / 비교 상대를 intervals[i-1]로 잡지 않았나?
```

## 어디서 막혔나 — 실제 실수의 기록

이 절은 자기비난이 아니라 재발 방지 장치다. 같은 자리에서 또 막혔을 때 "아, 그 함정"이라고 5초 만에 빠져나오려면, 실수를 일반론("정렬을 잘못했다")이 아니라 그날 실제로 한 형태 그대로 적어야 한다.

첫 번째 실수는 **정렬 키를 end(`a[1]`)로 잡은 것**이다. end로 정렬하면 `[2,4]`가 `[1,8]`보다 앞에 와서, 뒤 구간의 시작이 앞 구간의 시작보다 작아지는 역전이 생긴다. 그 상태에서는 "겹치나?"라는 질문 자체가 성립하지 않아 그다음 설계가 전부 흔들렸다. 더 뼈아픈 사실은, 주석에는 "start가 작은 것을 시작으로"라고 **스스로 적어 놓고** 코드는 end 정렬로 짰다는 점이다. p86491 최소직사각형에서 "긴 쪽/짧은 쪽으로 생각하자"를 글로 남기고도 한 달 뒤 width/height 분기로 돌아간 것과 정확히 같은 패턴이다. 통찰은 글이 아니라 **코드의 첫 동작**(여기서는 정렬 키)에 박아야 다음 풀이에서 재현된다.

두 번째이자 더 본질적인 실수는 **비교 상대를 `intervals[i-1]`(원본 배열의 바로 앞 칸)으로 잡은 것**이다. 이러면 두 개까지는 합쳐져도 세 개 연속 병합이 깨진다. `[1,4],[2,6],[5,10]`에서 앞 둘을 `[1,6]`으로 합쳤다면 세 번째 `[5,10]`은 그 `[1,6]`과 비교해야 하는데, `intervals[i-1]`은 합치기 전의 원본 `[2,6]`을 가리키고 있기 때문이다. 막힘의 본질을 한 문장으로 줄이면 이렇다. **누적된 상태를 원본 인덱스에서 찾으려 했다.** 병합 결과는 원본 배열 어디에도 없다. 내가 만들어 가는 결과 리스트의 마지막에만 있다.

## 핵심 사고 — 왜 이렇게 푸는가

카드가 떠오르지 않을 때, 이 절만 읽고 같은 결론에 다시 도달할 수 있도록 사고 순서대로 적는다.

이 문제를 "이웃한 두 구간을 비교하는 문제"로 보면 막힌다. 병합은 연쇄되기 때문이다. 하나를 합치면 더 커진 구간이 생기고, 그 커진 구간이 다음 구간과 또 겹칠 수 있다. 그러니 매 단계에서 필요한 것은 원본의 이전 칸이 아니라 **"지금까지 병합해서 만든 마지막 구간"** 하나다. 이것이 이 문제에서 들고 가야 하는 상태의 전부다.

start로 정렬하는 이유는 단순한 정리가 아니라 **정규화**다. 시작이 작은 순으로 줄을 세우면, 어떤 구간과 겹칠 수 있는 후보는 반드시 그 바로 뒤에 이웃해 온다. 그래서 "직전 결과 구간과 겹치는가?"라는 질문 하나만 반복하면 되고, 멀리 떨어진 구간을 되돌아볼 필요가 사라진다. 판단하기 전에 표준형으로 줄 세워 문제의 자유도를 없애는, p86491과 같은 수법이다.

겹쳤을 때 무엇을 하는가가 마지막 조각이다. 풀다가 "겹친 `[2,4]`를 어떻게 처리해서 결과에 넣지?"라는 질문에서 막혔는데, 질문 자체가 함정이었다. 답은 **넣지 않는다**이다. 겹친 구간은 마지막 구간의 끝을 `max(last.end, current.end)`로 늘리는 행위 그 자체로 흡수되어 사라진다. `max`가 필요한 이유는 `[1,8]` 다음의 `[2,4]`처럼 뒤 구간이 통째로 안에 들어가는 경우 때문이다. 이때 끝을 `current.end`(4)로 덮어쓰면 구간이 줄어드는 버그가 되고, `max(8,4)=8`이라 아무 일도 일어나지 않는 것이 정확한 처리다.

경계 하나. 겹침 판정은 `<`가 아니라 `<=`다. `[1,4]`와 `[4,5]`처럼 끝과 시작이 닿기만 해도 병합 대상이고(결과는 `[1,5]`), `<`로 쓰면 이 접점 케이스가 빠진다.

## 손으로 따라가기 — 실제로 막혔던 입력

`[[1,7],[1,8],[2,4]]`는 풀다가 "정렬해도 결국 `[1,8]`이 되는 과정이 안 떠오른다"며 실제로 막혔던 입력이다. start 정렬을 해도 순서가 그대로라 더 헷갈렸다. 한 단계씩 따라가면 이렇다.

```
시작: 첫 구간을 결과에 그냥 넣는다.
  merged = [[1,7]]                          마지막 = [1,7]

step 1:  current = [1,8]
  current.start(1) <= 마지막.end(7) ?  예 → 겹침
  → 마지막.end = max(7, 8) = 8              (새 칸 추가 없음, 끝만 늘어남)
  merged = [[1,8]]                          마지막 = [1,8]

step 2:  current = [2,4]
  current.start(2) <= 마지막.end(8) ?  예 → 겹침
  → 마지막.end = max(8, 4) = 8              (4로 줄면 안 되니까 max)
  merged = [[1,8]]                          마지막 = [1,8]  ← [2,4]는 흡수되어 사라짐

끝 → [[1,8]]
```

step 2가 이 문제의 전부다. `[2,4]`는 결과에 들어가지 않았고, `max` 덕분에 "아무 일도 일어나지 않는 것"이 정확한 처리였다.

## 뼈대 압축

```
start 오름차순 정렬. 첫 구간을 결과에 깔고 시작.

각 current마다 두 갈래뿐:
  current.start <= last.end  →  겹침   →  last.end = max(last.end, current.end)
                              아니면    →  결과에 current 추가 (새 구간 시작)

last = 결과의 마지막 구간. 끝이 닿기만 해도(<=) 병합. 시간 O(n log n), 공간 O(n).
```

## 패턴 전이 — 이 뼈대가 다시 나오는 곳

이 문제의 뼈대는 **"정렬 후 누적 스위핑"**이다. 정렬로 줄을 세운 뒤, 결과의 마지막 하나를 들고 가며 조건에 따라 갱신하거나 새로 시작한다. 구간 문제 상당수가 이 틀을 공유한다. Insert Interval(LC 57)은 새 구간 하나를 끼워 넣으며 같은 병합을 하고, Non-overlapping Intervals(LC 435)는 겹칠 때 "끝이 작은 쪽을 남기고 하나를 버리는" 변형이며, Meeting Rooms 계열은 "겹침의 최대 깊이"를 세는 변형이다. 셋 다 첫 동작이 정렬이고, 들고 가는 상태가 "마지막(또는 현재 활성) 구간"이라는 점이 같다.

한 층 더 내려가면 **정렬은 정규화**라는 공통점이 보인다. p86491에서 회전이라는 자유도를 (긴 변, 짧은 변) 정규화로 없앤 것, 무방향 간선 `(u,v)`/`(v,u)`를 `(min,max)`로 접어 중복을 없애는 것, 애너그램을 글자 정렬로 표준형으로 만드는 것이 모두 같은 사고다. 판단하기 전에 표준형으로 줄 세운다 — 이 한 문장이 여러 문제를 관통한다.

## 코드에서 배운 것

배열 반환은 `merged.toArray(new int[0][])`로 한다. `new int[merged.size()][2]`처럼 안쪽 배열까지 미리 만들면, `toArray`가 각 칸을 리스트 원소의 참조로 덮어쓰기 때문에 미리 만든 `int[2]`들은 채워지기도 전에 전부 버려진다. 빈 껍데기 `new int[0][]`를 넘겨 JVM이 정확한 크기로 한 번에 만들게 하는 것이 표준 관용구다.

정렬 비교자는 `Comparator.comparingInt(interval -> interval[0])`로 쓴다. `(a, b) -> Integer.compare(a[0], b[0])`도 동작은 같지만, 전자는 "무엇으로 정렬하는가"라는 의도가 코드 표면에 바로 드러나고 후자는 읽는 사람이 비교 절차에서 의도를 역산해야 한다.

누적 마지막은 `merged.getLast()`(Java 21+)가 `merged.get(merged.size() - 1)`보다 의도가 분명하다. 그리고 `getLast()`가 돌려주는 것은 리스트 안 배열을 그대로 가리키는 참조라서, `last[1] = ...`로 고치면 결과 리스트도 함께 갱신된다. 꺼내서 고치고 다시 넣을 필요가 없다.

## 복습 계획 (R1~R4)

시작 전에 하나만 짚어 둔다. 이 문제는 기본 패턴인데도 막혔다. 그것은 연차나 재능의 문제가 아니라 **최근 노출(recency)의 문제**다. 한동안 손을 놓았다 복귀하면 누구든 기본 동작에서 삐끗한다. 막힌 지점이 구체적으로 기록되어 있으니, 복습은 그 지점만 정조준하면 된다.

- **R1 (2026-06-10, 완료):** 막힌 두 지점(end 정렬, `intervals[i-1]` 비교)을 통과해 풀이 완성. 이 문서에 기록.
- **R2 (2~3일 뒤):** 빈 파일에서 다시 구현. 코드 암기가 아니라 "마지막 구간을 들고 간다"는 상태 설계가 다시 나오는지 본다.
- **R3 (1주 뒤):** 코드 없이 말로 설명. "정렬은 왜 start인가, 비교 상대는 무엇인가, 겹치면 무엇을 하는가" 세 질문에 답한다.
- **R4 (2~3주 뒤):** 30초 카드만 보고 제한 시간 안에 재풀이. 변형(Insert Interval)에 도전해 전이를 확인한다.

막히면 30초 카드 → 핵심 사고 → 손 트레이스 순서로 한 칸씩만 펼친다.
