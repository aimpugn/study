# Group Anagrams (LeetCode 49) - 풀이 회고

> 이 문서는 위에서부터 읽는 글이 아니다. 먼저 바로 아래 30초 카드만 보고 코드를 머릿속에 떠올려 본다. 떠오르면 통과다. 안 떠오르면 실패가 아니라 정상이니, 막힌 칸만 골라 아래 산문을 펼친다. 코드 줄별 설명은 같은 폴더 Solution.java 첨삭에 있다 - 이 문서는 "사고를 꺼내는 카드", 첨삭은 "막혔을 때 펼치는 본문"이다.

## 30초 카드

```
문제 신호 : "문자열 여러 개를 애너그램끼리 묶어라"
한 줄     : 각 단어를 표준형(글자를 정렬한 문자열)으로 바꿔 맵의 키로 쓴다. 같은 그룹은 같은 키로 저절로 모인다.
키워드    : 1) 관계(둘 비교) -> 속성(각자의 키)  2) 표준형 = 정렬 결과  3) computeIfAbsent
함정 단서 : 애너그램을 "두 단어가 같은 부류인가?"로 보고 모든 쌍을 맞대보지 않았나? -> 그게 TLE다.
```

## 어디서 막혔나 - 실제 사고의 갈림길

이 절은 자기비난이 아니라 재발 방지 장치다. 같은 자리에서 또 막혔을 때 "아, 그 갈림길"이라고 5초 만에 빠져나오려고, 그날 실제로 한 사고 그대로 적는다.

1차 풀이는 애너그램을 "두 단어 A와 B가 같은 부류인가?"라는 관계로 봤다. 관계로 보면 모든 쌍을 맞대보는 이중 루프가 자연히 따라온다 - isAnagram으로 둘을 비교하고, bitmap으로 이미 묶인 걸 건너뛰는 구조. 이게 TLE(시간 초과)의 원인이었다. isAnagram이 느려서가 아니라, "쌍 비교"라는 구조 자체가 단어 수 n에 대해 O(n^2)라서다.

뼈아픈 건, 정답을 이미 스쳤다는 점이다. isAnagram 안에 "차라리 정렬하는 게 낫나?"라고 직접 적어 뒀다. 그 정렬이 바로 표준형(canonical form)이다. 다만 그 정렬을 "두 단어를 맞대보는 도구"로만 쓰고, "한 단어를 대표하는 키"로 승격시키지 못했다. 도구에서 키로 한 걸음이었는데 그 걸음을 못 디뎠다.

왜 못 디뎠나. 문제를 관계로 본 프레임 때문이다. 관계로 보면 "둘을 어떻게 비교하나"가 머릿속을 채워서, "각자를 하나의 값으로 바꿔 묶는다"는 발상이 자리를 못 잡는다. 그래서 다음에 잡을 신호 하나: "A와 B가 같은 부류인가?"라는 관계 질문이 보이면 멈추고 물어라 - "각 원소를 하나의 대표값으로 바꿔서, 그 값으로 묶을 수 있나?" 묶이면 쌍 비교 O(n^2)가 키 조회 O(n)로 내려간다.

## 핵심 사고 - 왜 이렇게 푸는가

카드가 안 떠오를 때 이 절만 읽고 같은 결론에 다시 도달하도록 사고 순서대로 적는다.

애너그램은 "글자 구성이 같고 순서만 다른 단어"다. 그러면 글자를 정렬해 버리면 순서 차이가 지워져, 같은 그룹의 단어는 모두 똑같은 문자열로 수렴한다 - "eat", "tea", "ate"가 전부 "aet"가 된다. 이 "정렬한 문자열"이 표준형이고, 그룹의 이름표 노릇을 한다.

이름표가 있으면 그룹핑은 쉽다. 맵을 하나 두고, 각 단어의 표준형을 키로 그 단어를 담으면 된다. 같은 그룹은 키가 같으니 한 칸에 모이고, 다른 그룹은 키가 다르니 갈라진다. "A와 B가 같은 그룹인가?"라는 쌍 질문이 "이 단어의 키가 맵에 있나?"라는 조회 한 번으로 바뀐다 - 비교로 찾지 말고 키를 계산해 한 번에 도착하라는 해시의 사고다.

불변식 한 문장으로 들고 간다: i번째 단어를 처리한 직후, 맵에는 (표준형 키 -> 그 키의 단어 전부)가 들어 있다. 코드는 이 문장의 받아쓰기다.

## 손으로 따라가기 - ["eat","tea","tan","ate","nat","bat"]

맵이 단어를 하나씩 받아 자라는 과정. 바뀐 그룹만 적는다.

```
"eat" -> 키 "aet" -> {aet=[eat]}                새 그룹
"tea" -> 키 "aet" -> {aet=[eat, tea]}           기존 그룹 합류 (비교 없이 키로)
"tan" -> 키 "ant" -> {..., ant=[tan]}           새 그룹
"ate" -> 키 "aet" -> {aet=[eat, tea, ate], ...} 기존 그룹 합류
"nat" -> 키 "ant" -> {..., ant=[tan, nat]}      기존 그룹 합류
"bat" -> 키 "abt" -> {..., abt=[bat]}           새 그룹

맵의 값들이 곧 답: [[eat, tea, ate], [tan, nat], [bat]]
```

TLE를 낸 적대 입력은 단일 문자를 1~100번 반복한 단어 1만 개("a", "bb", "ccc", ...)였다. 1차 풀이는 같은 길이 쌍마다 무거운 isAnagram이 돌아 수 초가 걸렸고, 표준형 키 풀이는 단어마다 한 번 정렬하고 끝이라 7ms에 통과했다.

## 뼈대 압축

```
각 단어 word:
  키 = sort(word)              // 표준형: 글자를 정렬한 문자열
  map[키].add(word)            // computeIfAbsent로 없으면 새 리스트
답 = map의 값들
시간 O(n * L log L), 공간 O(n * L)   (n = 단어 수, L = 단어 길이)
```

## 패턴 전이 - 이 뼈대가 다시 나오는 곳

이 문제의 뼈대는 "관계를 속성으로 바꿔 키로 묶기"다. "둘이 같은 부류인가?"를 묻고 싶을 때, 각 원소를 변하지 않는 대표값(표준형)으로 바꿔 그 값을 맵의 키로 쓰면, 쌍 비교 O(n^2)가 그룹핑 O(n)로 내려간다. 같은 사고가 나오는 곳:

- 중복/유사 묶기 일반: 무방향 간선 (u,v)를 (min,max)로 정규화해 중복 제거, 좌표쌍 정렬, 분수를 기약분수로 - 전부 "각자를 표준형으로 바꿔 키로".
- 빈도 기반 그룹핑: 표준형을 "글자 정렬" 대신 "26칸 등장 횟수"로 바꾸면 정렬 없이 O(L). 같은 뼈대, 키 만드는 법만 교체.
- 한 층 아래 원리: p86491(최소 직사각형)에서 회전 자유도를 (긴 변, 짧은 변)으로 정규화한 것과 같은 사고 - 판단하기 전에 표준형으로 줄 세워 자유도를 없앤다.

## 코드에서 배운 것

- 표준형 만들기는 `toCharArray() -> Arrays.sort() -> new String()` 3단이다. String이 불변이라 제자리 정렬이 안 되니 char[]로 복사해 정렬하고, char[]는 내용이 같아도 참조로만 비교돼 맵 키가 못 되니 다시 String으로 되돌린다.
- `computeIfAbsent(key, k -> new ArrayList<>()).add(word)` 한 줄이 "키 없으면 새 리스트, 있으면 그 리스트에 추가"를 닫는다.
- Integer를 `!=`로 비교하면 값이 아니라 참조 비교다 - 1차 풀이 isAnagram에서 우연히만 통과하던 지뢰였다(등장 횟수 L <= 100이 Integer 캐시 -128~127 안이라 같은 객체가 재사용돼서). 값 비교는 `.equals()`. (자세한 메커니즘은 Solution.java 첨삭 참조.)
