# Daily Temperatures (LC 739) — 회고

## 사용법

위에서부터 다 읽는 문서가 아닙니다. 30초 카드를 먼저 보고, 막히는 절만 펼치세요. 스택이 어떻게 쌓였다 빠지는지가 머릿속에 안 그려지면, 같은 폴더 `PROCESS.html`을 브라우저로 열어 한 칸씩 넘겨 보세요 — 이 회고(설명)와 그 시각화(움직임)가 한 쌍입니다.

## 30초 카드

> 문제 신호: 각 원소에서 "다음으로 더 큰 값까지의 거리"를 구한다 (오른쪽에서 처음으로 더 따뜻한 날).
> 한 줄: 아직 더 따뜻한 날을 못 만난 인덱스를 스택에 쌓고, 더 따뜻한 날이 오면 꺼내며 answer = i - idx로 채운다.
> 키워드: 단조 스택(monotonic stack), 각 원소 한 번 push, 한 번 pop, O(n).
> 함정: 중첩 스캔은 같은 구간을 다시 훑어 O(n^2) — "전부 같은 값 뒤 봉우리"([99]*99999 + [100]) 입력에서 TLE.

## 어디서 막혔나

이 세션의 실제 막힘은 셋이었습니다.

1) 처음 떠오른 중첩 스캔(각 i에서 오른쪽을 한 칸씩)을 그대로 제출했는데, 그 비용을 적대적 입력으로 따져 보지 않았습니다. `[99]`를 99999개 늘어놓고 끝에 `[100]`을 둔 입력에서 각 99가 자기 오른쪽의 99들을 매번 다시 훑어 O(n^2)가 됐고, 47/48만 통과한 채 TLE가 났습니다(실측 n=100000에서 926ms).

2) 풀이 중 "이런 경우 스택을 썼던 거 같다"며 정답 도구를 스쳤지만, "왜 중첩 루프보다 나은지 모르겠다"며 접어 두고 다른 방법을 찾았습니다. 그 "왜"(각 날을 두 번 보지 않는다)가 바로 핵심이었는데, 모르겠다는 이유로 버린 게 가장 큰 막힘이었습니다.

3) 스택 코드를 적고도 동작이 머릿속에 안 그려져 `println`으로 스택을 찍어 봤습니다. 정적 출력으로는 "쌓였다 한 번에 빠지는" 움직임이 안 잡혀, 시각화가 필요했습니다.

## 핵심 사고

왜 스택인가는 비용에서 거꾸로 풀면 길이 보입니다. 중첩 스캔이 느린 진짜 이유는 "같은 칸을 여러 번 다시 본다"는 것입니다. `[99,99,99,100]`에서 첫 99도, 둘째 99도, 셋째 99도 각자 오른쪽 99들을 다시 훑습니다. 그 "다시 보기"만 없애면 한 번 훑기로 끝납니다.

그걸 없애는 장치가 스택입니다. 왼쪽에서 오른쪽으로 한 번 가면서, 아직 더 따뜻한 날을 못 만난 인덱스를 스택에 쌓아 둡니다. 새 날 i가 오면 스택 맨 위(가장 최근에 기다리기 시작한 날)부터 봅니다. i가 그 날보다 따뜻하면 그 날은 드디어 답을 찾은 것이라 꺼내고 answer[idx] = i - idx로 채웁니다. i가 더 따뜻한 동안 계속 꺼냅니다. 더는 못 꺼내면 i도 스택에 쌓아 기다립니다.

스택이 단조(monotonic)인 이유: 위로 갈수록 온도가 작아집니다. 더 따뜻한 날이 왔으면 이미 꺼냈을 테니, 스택에 남는 건 늘 "아직 안 따뜻한, 위로 갈수록 더 최근이고 더 차가운" 인덱스뿐입니다. 그래서 각 인덱스는 딱 한 번 들어가고 딱 한 번 나옵니다 — 전체 연산이 2n번 안쪽, 곧 O(n)입니다. "왜 빠른가"가 이 한 줄로 닫힙니다.

## 손으로 따라가기

막혔던 자리(스택이 쌓였다 한 번에 빠지는 순간)를 `[99, 99, 99, 100]`으로 따라가 봅니다. 같은 값이 쌓이다가 봉우리에서 한꺼번에 비는 게 핵심입니다.

```
i=0 (99): 스택 비어 있음 -> push 0          stack=[0]        answer=[0,0,0,0]
i=1 (99): 99 > 99? 아니오 -> push 1          stack=[0,1]      answer=[0,0,0,0]
i=2 (99): 99 > 99? 아니오 -> push 2          stack=[0,1,2]    answer=[0,0,0,0]   <- 99들이 쌓임
i=3 (100): 100 > 99(top=2)? 예 -> pop 2, answer[2]=3-2=1
           100 > 99(top=1)? 예 -> pop 1, answer[1]=3-1=2
           100 > 99(top=0)? 예 -> pop 0, answer[0]=3-0=3
           스택 비어 있음 -> push 3            stack=[3]        answer=[3,2,1,0]   <- 한 번에 드레인
결과: [3, 2, 1, 0]
```

i=0,1,2에서 스택이 차오르고, i=3(봉우리)에서 기다리던 셋이 줄줄이 빠지며 각자 답을 받습니다. 중첩 스캔이 99들을 매번 다시 훑던 그 일을, 스택은 각 99를 한 번 쌓고 한 번 꺼내 끝냅니다. `PROCESS.html`의 "같은 값+봉우리" 프리셋이 바로 이 장면입니다.

## 왜 항상 그런가 — 숫자를 문자로 (기호 시뮬)

위 트레이스는 `[99,99,99,100]` 한 입력에서 본 것입니다. "스택이 늘 내려가게 쌓인다(불변식)"가 *모든* 입력에서 참인지는, 숫자를 문자로 바꿔 규칙이 무엇을 금지하는지 물으면 증명됩니다. 구체 시뮬은 발견이고, 이게 증명입니다.

```text
스택에 두 인덱스 a(먼저, 온도 A), b(나중, 온도 B)가 함께 있다고 하자.
규칙: 새 날은 자기보다 차가운 대기자를 모두 꺼낸다.

b가 올 때 a는 대기 중이었다. 만약 B > A였다면 b가 들어오며 a를 꺼냈을 것이다.
그런데 a가 아직 남아 있다 -> B > A는 불가능 -> B <= A.
즉 나중에 온 b는 먼저 온 a보다 따뜻할 수 없다. 그래서 위로 갈수록 차갑다 (항상 참).
```

숫자가 하나도 없습니다. 문자 둘 + 규칙 하나로 모든 입력을 한 번에 덮은 게 증명입니다. 이 "공존하는 둘을 집고, 규칙이 둘 사이에 무엇을 금지하나 묻기"가 불변식을 세우는 일반 도구입니다 — 다른 문제에도 같은 손놀림을 씁니다. 자세히는 algorithms 루트의 `problem_solving_growth.md`, "구체 시뮬은 발견, 기호 시뮬은 증명" 절을 보세요.

## 뼈대 압축

```
for i in 0..n-1:
    while 스택이 비지 않음 and t[i] > t[stack.top]:
        idx = stack.pop();  answer[idx] = i - idx   // i가 idx의 더 따뜻한 날
    stack.push(i)
return answer
```

## 패턴 전이

같은 뼈대(단조 스택)가 나오는 곳: 다음으로 더 큰 값(Next Greater Element, LC 496 / 503), 다음으로 더 작은 값, 히스토그램 최대 직사각형(LC 84), 빗물 가두기(LC 42).

한 층 아래 공통 원리: "각 원소가 자기 답을 줄 상대를 딱 한 번 만난다"를 보장하면, 중첩처럼 보이는 탐색도 한 번 훑기로 압축된다. 신호는 "각 원소에 대해 다음으로 더 크거나 작은 값을 찾으라"는 요구다 — 이게 보이면 단조 스택을 꺼낸다.

## 코드에서 배운 것

- 통과 개수가 복잡도를 보증하지 않는다. 47/48은 적대적 입력이 없었을 뿐, O(n^2)는 그대로였다. 큰 입력에서 같은 구간을 다시 훑는지 의심하는 습관이 비용 분석의 시작이다.
- "왜 더 나은지 모르겠다"는 정답 도구를 버리는 이유가 아니라, 바로 그 자리에서 풀어야 할 질문이다. 여기선 그 "왜"가 "각 날을 두 번 보지 않는다" 한 줄이었다.
- 디버그 `println`(`dailyTemperatures2`의 `"stack: "` 출력)은 스택을 보려는 시도였지만 정적이라 움직임을 못 준다. 같은 폴더 `PROCESS.html`이 그 자리를 대신하니, 제출 전 `println`은 지운다.
