method 주석에 문제를 분석하고, 바로 문제 해결을 시작했습니다.
그러다 중간에 멈추게 됐는데,
1. 30 x 70 -> 70 x 30으로 뒤집어서 더 작은 높이를 찾는 로직을 어떻게 해야 할지
2. 그리고 가로 기준으로 했을 때 세로를 뒤집어 본다면, 세로 기준으로 해서 가로도 뒤집어 봐야 하는지

이런 정제되지 않은 생각들이 떠오르기 시작했기 때문입니다.

    각 명함은 가로 또는 세로로 놓을 수 있습니다.

- width, height 라는 용어의 한계

    사실 너비와 높이, 가로와 세로라고 생각하지만, 사실 항상 그렇게 고정되어 있는 것은 아닙니다.

    ```sh
     ┌───┐   ┌──────┐
    2│   │  1│      │
     │   │   └──────┘
     └───┘      2
       1
    ```

    이렇게 보면 좌측은 가로가 짧지만 우측은 가로가 깁니다. 하지만 둘 다 크기는 2입니다.
    따라서 가로, 세로라는 개념 대신 "긴 쪽"과 "짧은 쪽"이라고 생각했으면 더 좋았을 거 같습니다.

    NOTE: 문제에 주어진 단어나 개념에 사고가 제한되는 경향이 있는데, 주의해야겠습니다.

- 모두 포함할 수 있는 최소 크기의 직사각형

    앞서 가로, 세로 대신 긴 쪽과 짧은 쪽으로 개념을 바꿔서 생각하기로 했습니다.

    따라서, "모두 포함"해야 하므로
    - ~~가로든 세로든 한 쪽을 기준으로 잡았을 때, 최대값 이상이어야 합니다~~ 라고 생각하는 대신
    - 긴 쪽은 긴 값들 중에 최대값 이상이고, 짧은 쪽은 짧은 값들 중에 최대값 이상이어야 합니다.

```java
if (sizes.length == 1) {
    return sizes[0][0] * sizes[0][1];
}

// 가장 짧은 가로 길이,
// 가잘 짧은 세로 길이
int maxLong = Integer.MIN_VALUE;
int maxShort = Integer.MIN_VALUE;

for (int[] size : sizes) {
    int longSide = Math.max(size[0], size[1]);
    int shortSide = Math.min(size[0], size[1]);

    maxLong = Math.max(maxLong, longSide);
    maxShort = Math.max(maxShort, shortSide);
}

return maxLong * maxShort;
```

## 재풀이 회고 — 2026-06-09 (페어 리뷰)

위 1차 풀이에서 "가로/세로가 아니라 긴 쪽/짧은 쪽으로 생각했어야 한다"는 깨달음을 적어 두었다. 그런데 한 달 뒤 `attempt260603`에서 다시 풀 때, 변수 이름을 또 `width`와 `height`로 두는 순간 그 깨달음이 잊혔고, 손은 `if (height < width)`로 가로와 세로 중 누가 긴지를 명함마다 가르는 분기로 돌아갔다. 정답이긴 했지만, 한 달 전 자신이 명문화한 통찰을 재현하지 못한 것이다. 이 절은 그 간극을 메우는 정리이고, 다음에 같은 함정을 피하기 위한 기록이다.

### 회전 가능은 가로와 세로의 대칭이고, 그래서 먼저 정규화한다

문제가 "명함은 회전할 수 있다"고 말하는 순간, `[60, 50]`과 `[50, 60]`은 같은 명함이 된다. 즉 가로와 세로라는 두 이름은 서로 맞바꿔도 결과가 변하지 않는, 구분이 의미 없는 이름표가 된다. 수학에서 이렇게 맞바꿔도 같은 관계를 대칭(symmetric)이라 부르고, 두 값을 자유롭게 교환할 수 있는 이 여지를 자유도라 부른다. `if (height < width)`는 이 대칭을 명함마다 손으로 처리하려는 시도였다.

더 나은 길은 회전해도 변하지 않는 값을 기준으로 삼는 것이다. 긴 변과 짧은 변은 어떻게 돌려도 바뀌지 않는 불변량(invariant)이다. 그래서 각 명함을 받자마자 `(긴 변, 짧은 변)`으로 바꿔 적는다. 여러 모양을 하나의 표준형으로 통일하는 이 작업을 정규화(normalize)라 부르는데, 한번 정규화하고 나면 회전이라는 골칫거리가 처음부터 없었던 것처럼 사라진다.

```text
입력(가로, 세로)        정규화(긴 변, 짧은 변)      누적(maxLong, maxShort)
[60, 50]          ->   (60, 50)              ->   (60, 50)
[30, 70]          ->   (70, 30)  ← 자리 바뀜  ->   (70, 50)
[60, 30]          ->   (60, 30)              ->   (70, 50)
[80, 40]          ->   (80, 40)              ->   (80, 50)
답 = 80 * 50 = 4000
```

모든 명함을 같은 형태로 줄 세우고 나면 "이 명함은 어떻게 놓지?"라는 경우의 차이가 사라지고, 경우를 가르는 `if`도 존재할 이유를 잃는다. 그래서 정규화 풀이에는 분기가 없다. 분기를 없앤 것이 아니라, 정규화라는 더 앞 단계로 끌어올려 한 번에 흡수한 것이다.

### if 분기와 max·min, 그리고 비교 횟수 트레이드오프

재풀이 과정에서 정당한 의문이 하나 나왔다. `Math.max`와 `Math.min`을 둘 다 쓰면 같은 두 값을 두 번 비교하는 셈인데, `if`로 한 번만 비교하면 긴 쪽과 짧은 쪽을 동시에 알 수 있지 않은가. 맞는 관찰이다. 명함 하나를 처리할 때 비교 횟수를 세어 보면 차이가 드러난다.

```text
if/else 방식:
  height < width ?            1번  (한 번 비교로 긴/짧은 판정)
  Math.max(resultWidth,  ..)  1번
  Math.max(resultHeight, ..)  1번
  = 명함당 3번

max/min 방식(현재 채택 코드):
  Math.max(size0, size1)      1번  (긴 변)
  Math.min(size0, size1)      1번  (같은 두 값을 또 비교 — 중복)
  Math.max(longestWidth,  ..) 1번
  Math.max(longestHeight, ..) 1번
  = 명함당 4번
```

여기서 중요한 분리가 있다. 거부할 대상은 "max/min 구현의 중복 비교"이지 "정규화라는 사고"가 아니다. 정규화는 `if`로도 할 수 있다. 한 번만 비교하면서 긴 변과 짧은 변을 정규화하려면 이렇게 쓴다.

```java
int longSide, shortSide;
if (size[0] >= size[1]) {   // 단 한 번 비교로 긴 변과 짧은 변을 동시에 확정
    longSide  = size[0];
    shortSide = size[1];
} else {
    longSide  = size[1];
    shortSide = size[0];
}
```

이 버전은 비교가 3번이면서 `longSide`/`shortSide`라는 이름으로 정규화 의도까지 드러낸다. 비교를 아끼려는 본능과 의도를 드러내려는 목표는 충돌하지 않는다. 세 방식 모두 정답이고, 남는 차이는 "비교 한 번이냐 코드 한 줄이냐"의 취향에 가깝다.

### 비교 횟수는 곧 속도가 아니다

비교를 아끼려는 감각은 큰 데이터나 뜨거운 반복문(hot loop)에서는 옳다. 다만 두 가지를 같이 기억해야 공정하다. 첫째, 이 문제는 명함이 최대 10,000개라 명함당 비교 1번 차이는 통틀어 1만 번, 수마이크로초로 사람이 체감할 수도 채점에 영향을 줄 수도 없다. 둘째, 더 미묘한데, `Math.max`와 `Math.min`은 JIT 컴파일러가 분기 없는 명령(branchless, 조건 이동 CMOV)으로 바꾸는 경우가 많다. 반면 `if`는 진짜 분기(branch)라서, CPU가 어느 쪽으로 갈지 예측에 실패하면(branch misprediction) 수십 사이클을 버린다. 그래서 "비교 4번인데 분기 없음"이 "비교 3번인데 분기 1개"보다 빠른 역전도 실제로 일어난다. 연산 횟수를 머릿속으로 세는 직관은 이 수준의 최적화에서 자주 틀리고, 진짜로 알려면 측정(벤치마크)해야 한다. 세 방식 모두 시간 복잡도는 O(n), 공간은 O(1)로 같으므로, 이 문제에서 갈림의 기준은 속도가 아니라 어느 코드가 의도를 더 잘 드러내는가다.

### 같은 수법이 쓰이는 다른 자리 — 정규화의 전이

"서로 바꿔도 같은 자유도를 표준형으로 접은 뒤에 판단한다"는 이 수법은 최소직사각형만의 것이 아니다. 다른 문제에서도 같은 모양으로 되풀이된다. 무방향 그래프에서 간선 `(3,1)`과 `(1,3)`은 같은 간선인데, 각 간선을 `(min, max)`로 접으면 둘 다 `"1,3"`이 되어 중복이 저절로 합쳐진다.

```java
int u = Math.min(e[0], e[1]);
int v = Math.max(e[0], e[1]);
seen.add(u + "," + v);   // (3,1)도 (1,3)도 "1,3" → 중복 간선 제거
```

두 단어가 애너그램인지 볼 때도 글자를 정렬해 표준형으로 만들면 순서라는 자유도가 사라져 비교가 한 번에 끝난다.

```java
char[] a = "listen".toCharArray(); Arrays.sort(a);  // ['e','i','l','n','s','t']
char[] b = "silent".toCharArray(); Arrays.sort(b);  // ['e','i','l','n','s','t']
boolean same = Arrays.equals(a, b);                 // true
```

분수 `2/4`와 `1/2`가 같은지 볼 때는 최대공약수로 약분해 기약분수라는 표준형으로 맞추고, 좌표쌍이나 선분을 키로 쓸 때는 두 끝점을 `(작은 점, 큰 점)` 순서로 고정해 끝점 순서라는 자유도를 없앤다. 전부 원리가 같다. 정규화를 먼저 하고, 판단은 나중이다. 앞으로 `if`가 갈래마다 비슷한 일을 반복하거나 같은 것을 두 번 등록·비교하게 되면, "내가 정규화를 건너뛰고 있나?"를 떠올리는 것이 이 도구를 꺼내는 신호다.

### 이 문제에서 가져갈 교훈

진짜로 남길 것은 정답 코드가 아니라 세 문장이다. 첫째, 서로 바꿔도 결과가 같은 값이 보이면 비교하거나 분기하기 전에 표준형으로 정규화해 그 대칭을 없앤다. 둘째, 통찰은 글로 적어 두는 것만으로는 다음 풀이에서 자동으로 꺼내지지 않는다. 변수 이름을 `longSide`/`shortSide`로 짓는 그 첫 동작에 통찰을 박아 넣어야 손이 기억한다. 이름이 사고를 가둘 수도 있고(`width`/`height`는 분기를 부른다) 열 수도 있다(`longSide`/`shortSide`는 정규화를 부른다). 셋째, 정리(머리에 넣기)와 인출(꺼내기)은 다른 능력이라, 두 번째 만남부터는 전체 해설을 다시 읽지 말고 키워드 한 줄로 복원을 시도해야 한다.

```text
최소직사각형 → "회전 = 가로/세로 대칭 → 긴 변/짧은 변 정규화 → 분기 소멸"
```
