## 개요

DFS와 백트래킹은 같은 말이 아니다. DFS는 깊이 우선으로 내려가는 **탐색 순서**이고, 백트래킹은 선택을 하나 해 보고, 조건을 검사하고, 막히면 그 선택을 되돌린 뒤 다른 선택을 시도하는 **문제 풀이 구조**다.

알고리즘 문제에서 `DFS/백트래킹으로 풀자`라는 말은 보통 다음 뜻이다.

1. 가능한 선택들을 트리처럼 펼쳐 본다.
2. 한 선택을 고른 뒤 다음 선택으로 깊게 내려간다.
3. 중간에 조건을 만족할 수 없으면 그 아래 경우는 더 보지 않는다.
4. 내려가며 바꾼 상태를 되돌리고, 같은 깊이의 다른 선택을 시도한다.
5. 끝까지 도달한 상태가 문제 조건을 만족하면 답을 세거나 기록한다.

즉 DFS는 아래로 내려가는 엔진이고, 백트래킹은 그 엔진을 이용해 `선택 -> 검사 -> 진입 -> 복원`을 반복하는 방식이다.

## 왜 DFS와 백트래킹이 같이 등장하는가

많은 문제는 처음부터 그래프가 주어지지 않는다. 대신 선택을 할 때마다 다음 상태가 생긴다. 예를 들어 숫자 `1, 2, 3`으로 길이 2짜리 순열을 만든다고 하면, 실제 입력에는 아래 트리가 들어 있지 않지만 머릿속에서는 이런 결정 트리가 생긴다.

```text
시작
├── 1
│   ├── 12
│   └── 13
├── 2
│   ├── 21
│   └── 23
└── 3
    ├── 31
    └── 32
```

이 트리는 코드에 직접 저장되어 있지 않다. 재귀 함수가 현재 선택 상태를 들고 내려가면서 그때그때 만들어 낸다. 이런 트리를 보통 **결정 공간** 또는 **탐색 공간**이라고 부른다. 문제에서 가능한 답 후보들이 선택의 연속으로 만들어진다면, 그 후보들은 대개 이런 보이지 않는 트리 위에 놓인다.

DFS가 자연스러운 이유는 한 후보를 끝까지 만들어 봐야 그 후보가 답인지 알 수 있는 경우가 많기 때문이다. 백트래킹이 필요한 이유는 한 후보를 만들기 위해 바꾼 상태를 다음 후보에도 그대로 남겨 두면 안 되기 때문이다. `1 -> 2`를 만들고 난 뒤에는 `2` 선택을 해제해야 `1 -> 3`을 만들 수 있다.

## 가장 작은 예: 길이 2 순열 만들기

입력 `[1, 2, 3]`에서 길이 2 순열을 만들면 답은 `12, 13, 21, 23, 31, 32`다. 이 문제에서 상태는 두 가지면 충분하다.

- `path`: 지금까지 고른 숫자들
- `used`: 각 숫자를 이미 썼는지 여부

처음 상태는 비어 있다.

```text
path = []
used = [false, false, false]
```

첫 번째 숫자로 `1`을 고르면 상태가 바뀐다.

```text
path = [1]
used = [true, false, false]
```

이제 두 번째 숫자로 `2`를 고른다.

```text
path = [1, 2]
used = [true, true, false]
```

길이 2가 되었으므로 `12`를 답에 추가한다. 여기서 중요한 동작은 그 다음이다. `12`를 만들었다고 탐색이 끝난 것이 아니다. `1`로 시작하는 다른 후보인 `13`도 만들어야 한다. 그래서 마지막 선택 `2`를 되돌린다.

```text
path = [1]
used = [true, false, false]
```

그 다음 같은 깊이에서 `3`을 고른다.

```text
path = [1, 3]
used = [true, false, true]
```

이 과정이 백트래킹의 핵심이다. 코드는 깊게 내려가지만, 각 깊이의 선택을 끝낸 뒤에는 반드시 이전 상태로 돌아와야 한다. 이 복원이 빠지면 다음 가지가 이전 가지의 흔적을 들고 시작한다.

## DFS와 백트래킹은 무엇이 다른가

DFS와 백트래킹은 자주 함께 쓰이지만 바라보는 대상이 다르다.

1. DFS

    DFS는 그래프나 트리에서 한 방향으로 끝까지 내려간 뒤 돌아오는 탐색 순서다. 그래프가 이미 주어져 있고 모든 정점을 방문하는 문제라면 `visited`를 한 번 true로 만든 뒤 끝까지 유지하는 경우가 많다. 이때 시간 복잡도는 보통 `O(V + E)`다. `V`는 정점 수, `E`는 간선 수다.

2. 백트래킹

    백트래킹은 답 후보를 만드는 선택 과정을 탐색한다. 그래프가 입력으로 주어지지 않아도, 선택이 이어지면서 보이지 않는 결정 트리가 만들어진다. 한 가지를 내려가며 상태를 바꾸고, 돌아올 때 그 상태를 복원한다. 시간 복잡도는 보통 `분기 수^깊이` 형태에서 가지치기한 만큼 줄어든다.

3. DFS 기반 백트래킹

    실전에서 가장 흔한 형태는 DFS 재귀 함수로 결정 트리를 내려가고, 재귀 호출 전후에 상태 변경과 복원을 배치하는 방식이다. 그래서 코드 모양은 DFS처럼 보이지만, 문제의 본질은 그래프 방문이 아니라 가능한 답 후보를 구성하고 검증하는 데 있다.

이 차이를 놓치면 `visited`를 언제 되돌려야 하는지 헷갈린다. 그래프의 연결 요소를 세는 DFS라면 방문한 정점을 다시 방문하지 않도록 `visited`를 유지한다. 순열, 조합, 퍼즐, 배치 수 세기처럼 한 후보 안에서만 선택을 막아야 하는 문제라면 재귀가 돌아올 때 `used`나 배치 상태를 되돌려야 한다.

## 백트래킹 문제를 볼 때 먼저 정해야 하는 것

백트래킹은 코드를 쓰기 전에 상태 설계를 먼저 해야 한다. 보통 아래 여섯 가지가 닫히면 구현이 크게 흔들리지 않는다.

1. 한 단계의 선택이 무엇인가

    숫자 하나를 고르는 것인지, 칸 하나에 값을 넣는 것인지, 현재 위치에서 다음 방향을 고르는 것인지 먼저 정한다. 선택 단위가 애매하면 재귀 깊이도 애매해진다.

2. 상태가 무엇인가

    현재까지의 선택을 설명하기 위해 필요한 값이다. 예를 들어 `현재 위치`, `현재 방향`, `path`, `used`, `board`, `남은 개수`, `현재 합` 같은 값이 상태가 된다.

3. 후보는 어떻게 만들어지는가

    매 단계에서 가능한 선택 목록을 만들어야 한다. 순열이면 아직 쓰지 않은 숫자, 격자 이동이면 네 방향, 선로 문제라면 현재 칸과 이웃 칸이 서로 연결될 수 있는 방향과 선로가 후보가 된다.

4. 언제 가지치기하는가

    현재 상태에서 이미 조건을 위반했다면 더 깊게 내려가지 않는다. 이 검사는 가능하면 재귀 호출 전에 한다. 그래야 불가능한 하위 트리를 만들지 않는다.

5. 언제 답으로 세는가

    깊이가 끝났다고 항상 답은 아니다. 끝 상태가 문제의 모든 조건을 만족하는지 확인해야 한다. 예를 들어 목적지에 도착했더라도 반드시 지나야 할 선로를 지나지 않았다면 답이 아니다.

6. 무엇을 복원해야 하는가

    재귀 호출 전에 바꾼 값은 재귀 호출 뒤에 원래대로 되돌려야 한다. `used[i] = true`를 했다면 `used[i] = false`가 필요하고, 보드에 값을 넣었다면 빈칸으로 되돌려야 한다.

## 기본 코드 골격

백트래킹 코드는 보통 아래 모양에서 크게 벗어나지 않는다.

```kotlin
fun dfs(state: State) {
    if (isTerminal(state)) {
        if (isValidAnswer(state)) {
            answer++
        }
        return
    }

    for (choice in candidates(state)) {
        if (!canChoose(state, choice)) {
            continue
        }

        apply(state, choice)
        dfs(state)
        undo(state, choice)
    }
}
```

여기서 각 줄의 의미가 중요하다.

1. `isTerminal(state)`

    더 이상 선택할 단계가 없는지 본다. 순열에서는 `path.size == targetLength`, 격자 경로에서는 목적지 도착, 배치 문제에서는 모든 칸 처리 같은 조건이 들어간다.

2. `isValidAnswer(state)`

    끝까지 왔을 때 진짜 답인지 확인한다. 목적지에 도착했지만 추가 조건을 만족하지 못하면 답으로 세면 안 된다.

3. `candidates(state)`

    지금 상태에서 시도할 수 있는 선택 목록이다. 후보를 잘 줄이면 탐색량이 크게 줄어든다.

4. `canChoose(state, choice)`

    지금 선택이 조건을 깨는지 본다. 여기서 false가 나오면 그 아래 경우를 전부 버릴 수 있다. 이것이 가지치기다.

5. `apply`와 `undo`

    백트래킹의 안전장치다. `apply`로 상태를 바꾸고 내려갔다면, 반드시 `undo`로 되돌린다. `undo`가 빠지면 다음 후보가 오염된 상태에서 시작한다.

순열 예제는 이 골격을 거의 그대로 보여 준다.

```kotlin
fun permutations(numbers: IntArray, targetLength: Int): List<List<Int>> {
    val result = mutableListOf<List<Int>>()
    val path = mutableListOf<Int>()
    val used = BooleanArray(numbers.size)

    fun dfs() {
        if (path.size == targetLength) {
            result += path.toList()
            return
        }

        for (i in numbers.indices) {
            if (used[i]) continue

            used[i] = true
            path += numbers[i]

            dfs()

            path.removeAt(path.lastIndex)
            used[i] = false
        }
    }

    dfs()
    return result
}
```

이 코드에서 `path += numbers[i]`와 `used[i] = true`는 현재 선택을 상태에 반영한다. 재귀 호출 뒤의 `path.removeAt(...)`와 `used[i] = false`는 그 선택을 없던 일로 만든다. 이 네 줄이 백트래킹의 근육이다.

## 가지치기는 언제 하는가

가지치기는 답을 놓치지 않는 범위에서만 해야 한다. 조건을 너무 약하게 잡으면 탐색이 느리고, 너무 강하게 잡으면 가능한 답을 버린다.

예를 들어 목표 합이 7이고 양수 배열에서 부분집합을 고르는 문제를 생각해 보자.

```text
numbers = [2, 3, 5]
target = 7
```

현재 합이 8이 되었다면 더 내려가도 목표 7이 될 수 없다. 모든 수가 양수이기 때문이다. 이때 `currentSum > target`은 안전한 가지치기 조건이다.

```kotlin
if (currentSum > target) return
```

하지만 음수가 섞이면 이 조건은 안전하지 않다.

```text
numbers = [8, -1]
target = 7
```

처음에 8을 골랐다고 바로 버리면 `8 + (-1) = 7`이라는 답을 놓친다. 그래서 가지치기 조건은 항상 문제의 전제와 함께 판단해야 한다.

백트래킹에서 좋은 가지치기는 보통 이런 질문으로 찾는다.

1. 지금 상태가 이미 문제 조건을 어겼는가
2. 앞으로 어떤 선택을 해도 조건을 회복할 수 없는가
3. 현재 선택이 나중 선택지를 구조적으로 막는가
4. 이미 같은 상태를 더 좋은 조건으로 방문한 적이 있는가

네 번째 질문은 DP나 메모이제이션으로 이어질 수 있다. 같은 상태가 반복되고, 그 상태 이후의 답이 이전 경로와 독립적이라면 백트래킹에 캐시를 붙일 수 있다.

## BFS, DFS, 백트래킹, DP 중 무엇을 떠올릴까

문제를 처음 봤을 때는 알고리즘 이름보다 답의 형태를 먼저 본다.

1. 최단 거리나 최소 이동 횟수를 묻는가

    간선 비용이 모두 같고 상태 전이가 명확하면 BFS가 먼저 떠오른다. BFS는 가까운 상태부터 보기 때문에 처음 도착한 순간이 최단 거리인 경우가 많다.

2. 가능한 배치, 순서, 조합, 경로의 수를 세는가

    DFS/백트래킹이 먼저 떠오른다. 하나의 후보를 끝까지 만들어 보고, 조건을 만족하면 세고, 아니면 되돌아가야 하기 때문이다.

3. 그래프가 이미 주어졌고 모든 노드나 연결 요소를 봐야 하는가

    일반 DFS 또는 BFS가 먼저 떠오른다. 이 경우 `visited`는 보통 전역 방문 기록으로 유지된다.

4. 같은 상태가 여러 경로에서 반복되고, 그 상태 이후의 답이 이전 경로에 의존하지 않는가

    DP나 메모이제이션을 생각한다. 백트래킹으로 먼저 상태를 정의한 뒤, 중복이 보이면 캐시를 붙이는 흐름도 자연스럽다.

5. 현재 선택이 항상 전체 최선으로 이어진다는 증명이 가능한가

    그리디를 검토할 수 있다. 하지만 그 증명이 없으면 먼저 작은 반례를 찾아야 한다.

이 기준으로 보면 `DFS냐 BFS냐`는 마지막 질문이 아니다. 먼저 이 문제가 `도달 가능성`, `최단 거리`, `모든 후보 세기`, `최적값`, `상태 중복` 중 무엇을 묻는지 봐야 한다.

## 기차선로 문제에서 떠올려야 하는 방향

프로그래머스 `기차선로` 문제처럼 격자 위에 선로를 놓는 방법의 수를 세는 문제는 고정 그래프를 단순히 탐색하는 문제가 아니다. 빈칸에 어떤 선로를 놓느냐에 따라 그래프 자체가 달라진다. 그래서 BFS로 이미 만들어진 그래프를 도는 느낌보다, DFS/백트래킹으로 가능한 배치를 만들어 가며 조건을 확인하는 느낌이 더 자연스럽다.

이 문제에서 초반에 잘 잡은 부분은 선로를 방향 집합으로 보는 것이다.

```kotlin
val tracks = mapOf(
    1 to setOf(Direction.LEFT, Direction.RIGHT),
    2 to setOf(Direction.UP, Direction.DOWN),
    3 to setOf(Direction.UP, Direction.DOWN, Direction.LEFT, Direction.RIGHT),
    4 to setOf(Direction.LEFT, Direction.UP),
    5 to setOf(Direction.RIGHT, Direction.UP),
    6 to setOf(Direction.RIGHT, Direction.DOWN),
    7 to setOf(Direction.LEFT, Direction.DOWN),
)
```

이 맵은 `이 선로에서 다음에 갈 수 있는 선로 번호`를 말하는 표가 아니다. 더 안전한 해석은 `이 선로가 어느 방향으로 열려 있는가`다. 실제 이동은 두 칸이 서로를 향해 열려 있을 때만 가능하다.

```text
현재 칸의 선로가 RIGHT를 열고 있고,
오른쪽 칸의 선로가 LEFT를 열고 있으면,
두 칸은 연결된다.
```

따라서 상태를 생각할 때는 적어도 아래가 후보가 된다.

- 현재 위치
- 현재 칸으로 들어온 방향 또는 현재 칸에서 나갈 방향
- 빈칸에 이미 확정한 선로
- 이미 지난 선로 또는 반드시 지나야 하는 고정 선로의 처리 상태
- 3번 선로처럼 모든 방향 연결이 필요한 특수 조건

여기서 `n * m <= 20`은 완전탐색을 완전히 금지하는 크기는 아니다. 하지만 빈칸마다 `0..7`을 전부 넣으면 `8^k`가 되어 금방 커진다. 그래서 모든 빈칸을 먼저 채운 뒤 검사하기보다, 기차의 진행과 연결 조건이 허용하는 선택만 만들어 가는 쪽으로 탐색 공간을 줄이는 생각이 필요하다.

이 문제를 볼 때 자연스럽게 떠올라야 하는 문장은 다음에 가깝다.

```text
빈칸 전체를 마음대로 채운 뒤 검사하지 말고,
현재 선로와 다음 칸이 서로 연결될 수 있는 선택만 만들며 내려가자.
끝에서는 도착 조건뿐 아니라 모든 기존 선로를 지났는지, 3번 선로 연결 조건이 닫혔는지도 확인하자.
```

이 문장이 잡히면 `DFS 또는 BFS?`라는 질문이 조금 더 구체적으로 바뀐다.

```text
고정 그래프에서 최단 도달을 찾는가? 아니다.
선택으로 만들어지는 배치 수를 세는가? 그렇다.
그러면 DFS/백트래킹을 먼저 생각한다.
```

## 자주 틀리는 지점

1. `visited`를 무조건 전역으로 유지한다

    연결 요소 DFS에서는 한 번 방문한 노드를 다시 볼 필요가 없는 경우가 많다. 하지만 백트래킹에서는 한 후보 안에서만 방문을 막아야 하는 경우가 많다. 다른 후보를 시도할 때는 방문 상태가 되돌아가야 할 수 있다.

2. 끝 조건을 너무 빨리 답으로 센다

    목적지에 도착했다고 항상 답은 아니다. 문제에 추가 조건이 있으면 끝 상태에서 다시 확인해야 한다. `기차선로` 문제라면 도착 여부만이 아니라 모든 선로 통과 조건과 3번 선로 연결 조건까지 봐야 한다.

3. 후보 생성과 유효성 검사를 섞어 흐린다

    후보를 만드는 일과 그 후보가 가능한지 검사하는 일을 분리해서 생각하면 디버깅이 쉬워진다. `candidate`는 시도할 선택이고, `canChoose`는 문제 조건을 깨는지 보는 문이다.

4. 상태 복원을 빼먹는다

    재귀 호출 전에 바꾼 값은 호출 뒤에 복원해야 한다. 보드, 방문 배열, 현재 합, 남은 개수, 경로 리스트 중 하나라도 복원하지 않으면 다음 가지가 오염된다.

5. 가지치기 조건을 전제 없이 넣는다

    `현재 합이 목표보다 크면 중단` 같은 조건은 모든 수가 양수일 때만 안전하다. 입력에 음수나 나중에 회복 가능한 선택이 있으면 답을 버릴 수 있다.

6. BFS와 백트래킹을 알고리즘 이름만 보고 고른다

    BFS는 가까운 상태부터 보는 데 강하고, 백트래킹은 후보를 끝까지 구성하고 세는 데 강하다. 문제의 출력이 `최단 거리`인지 `경우의 수`인지 먼저 확인해야 한다.

## 검증 방법

백트래킹 코드는 작은 입력을 손으로 따라가며 검증하는 편이 가장 확실하다.

1. 가장 작은 정상 입력을 만든다

    순열 문제라면 `[1, 2]`, 격자 문제라면 `2 x 2`처럼 손으로 모든 경우를 셀 수 있는 크기를 고른다.

2. 재귀 깊이마다 상태를 출력한다

    `depth`, `choice`, `path`, `used`, `currentPosition` 같은 값을 출력하면 상태 복원이 맞는지 바로 보인다.

3. 재귀 호출 전후가 대칭인지 본다

    아래 두 줄처럼 호출 전 변경과 호출 후 복원이 짝을 이루는지 확인한다.

    ```kotlin
    used[i] = true
    path += numbers[i]

    dfs()

    path.removeAt(path.lastIndex)
    used[i] = false
    ```

4. 실패해야 하는 입력을 넣는다

    답이 0이어야 하는 입력, 제약 조건을 일부러 깨는 입력, 이미 완성된 입력을 넣는다. 성공 예제만 있으면 가지치기가 너무 강해서 답을 버리는지 확인하기 어렵다.

5. 완전탐색 기준선과 비교한다

    입력이 아주 작을 때는 느린 완전탐색을 따로 만들어 답 개수를 비교할 수 있다. 빠른 백트래킹 풀이가 작은 입력에서 기준선과 다르면 상태 정의나 가지치기가 틀렸을 가능성이 크다.

## 다시 떠올릴 질문

DFS/백트래킹 문제를 만났을 때는 아래 질문을 먼저 적어 보면 좋다.

1. 이 문제의 한 단계 선택은 무엇인가
2. 현재 상태를 설명하는 데 꼭 필요한 값은 무엇인가
3. 지금 선택 가능한 후보는 어디서 오는가
4. 어떤 조건이면 더 내려가도 절대 답이 될 수 없는가
5. 언제 답으로 세어야 하는가
6. 재귀 호출 전에 바꾼 상태 중 호출 뒤에 되돌릴 것은 무엇인가
7. 같은 상태가 여러 경로에서 반복되는가, 그렇다면 DP나 메모이제이션으로 줄일 수 있는가

짧게 외운다면 이렇게 잡아도 된다.

```text
DFS는 내려가는 순서다.
백트래킹은 선택하고, 안 되면 되돌리는 구조다.
실전에서는 상태, 후보, 가지치기, 종료 조건, 복원 대상을 먼저 정한다.
```
