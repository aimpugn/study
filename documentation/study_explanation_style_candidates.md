# Study Explanation Style Candidates

## 개요

이 문서는 `study` 저장소의 exemplar 문서와 현재 대화에서 나온 피드백을 바탕으로, 앞으로 `study-explanation`이 따라야 할 설명 스타일 후보를 비교하기 위해 작성했습니다.

여기서 말하는 스타일은 단순한 말투 취향이 아닙니다.
어떤 순서로 설명을 시작하는지, 작은 요소를 어떻게 먼저 보여 주는지, 공식 자료를 어떻게 현재 메커니즘으로 연결하는지, 번호 목록과 불릿 목록을 언제 쓰는지 같은 설명 구조 전체를 뜻합니다.

지금까지 드러난 방향을 먼저 한 문장으로 요약하면 이렇습니다.

작은 구성요소를 먼저 보여 주고, 각 요소의 역할을 설명한 뒤, 그것들이 왜 그렇게 연결되어 상위 메커니즘이 되는지 자연스러운 한국어 문장 흐름으로 이어 주는 설명이 가장 가깝습니다.

## 공통적으로 드러난 선호

1. 설명자는 앞에 나서지 않고, 주제를 바로 설명합니다.

    `이 문서는 질문을 닫는다`, `이제 위로 올라가 보겠다` 같은 메타 진행 멘트보다, 지금 무엇을 설명하는지 바로 말하는 문장을 더 선호합니다.

2. 가장 작은 벽돌부터 쌓아 올리는 설명을 선호합니다.

    구조를 먼저 보여 주었다면, 각 구성요소가 무엇인지 설명하고, 그다음 그 요소들이 어떻게 연결되어 상위 동작으로 이어지는지 설명해야 합니다.

3. 뒤 설명을 떠받치는 핵심 artifact는 초반에 실제 예시로 보여 주는 편이 좋습니다.

    `META-INF/MANIFEST.MF`, `BOOT-INF/lib/`, 실제 로그, 실제 디렉터리 구조처럼 뒤 설명이 기대는 대상은 먼저 눈으로 보게 해 두는 편이 학습에 유리합니다.

4. 공식 자료는 인용에서 끝나면 안 됩니다.

    `어디에 그렇게 적혀 있는가`도 중요하지만, 그보다 더 중요한 것은 `왜 그런 규약이 생겼고`, `그 규약이 현재 구조에 어떤 흔적으로 남았고`, `그래서 지금 설명하는 메커니즘에서 어디에 참여하는가`입니다.

5. 설명을 뒤로 미루지 않는 편을 선호합니다.

    지금 문장을 이해하는 데 필요한 구분이라면 같은 문단이나 바로 다음 문장에서 풀어 써야 합니다.
    `그 사이`, `이 구조`, `이 과정`처럼 지시어만 남겨 두는 방식은 잘 맞지 않습니다.

6. 번호 목록과 불릿 목록의 역할을 분명히 구분합니다.

    순차 흐름, 시간 순서, 인과 순서에는 번호 목록이 맞고, 서로 동등한 비교 항목에는 불릿 목록이 맞습니다.
    짧게 끝나는 항목은 한 줄로 닫고, 정말 설명이 더 필요한 항목만 들여쓴 prose로 이어 가는 편이 자연스럽습니다.

7. 역사와 등장 맥락이 중요합니다.

    `이게 무엇인가`만이 아니라 `왜 등장했는가`, `그 전에는 어떤 감각이었는가`, `무엇을 해결하려고 이런 구조가 생겼는가`가 같이 들어가야 더 잘 이해되고 오래 남습니다.

8. 친절함은 말투보다 구조와 장치에서 나와야 합니다.

    실제 예시를 먼저 보여 주는 것, 작은 단위부터 연결하는 것, 공식 링크를 바로 옆에 두는 것, 혼동하기 쉬운 쌍을 비교해 주는 것, 현재 머신 관측과 일반 규칙을 분리하는 것 같은 장치가 중요합니다.

## 특히 피해야 하는 패턴

- 내부 작업용 표현이 그대로 본문에 새어 나오는 문장
- `이제 ... 들어가 보겠습니다` 같은 stage-direction 문장
- 구조를 보여 준 직후 `이 다섯 줄만 봐도 구조가 보인다`처럼 감상으로 한 문장을 쓰는 방식
- `그 사이`, `이것`, `이 구조`처럼 지시어만 남기고 핵심 구분을 뒤로 미루는 방식
- 공식 문서를 링크만 걸고 현재 주제의 인과와 구조로 연결하지 않는 방식
- 실제 예시를 너무 뒤에 배치해서, 독자가 앞 문단을 빈손으로 읽게 만드는 방식
- 순차 흐름과 동등 비교를 같은 목록 형식으로 뒤섞는 방식

## 스타일 후보

아래 후보들은 서로 완전히 배타적인 문체라기보다, 설명의 출발점과 전개 방식이 다른 뼈대라고 보면 됩니다.

샘플 주제는 모두 같습니다.

`META-INF/MANIFEST.MF`는 무엇이고 왜 필요한가?

### 후보 A. 직진형 계층 누적형

- 핵심 감각:
  - 사실을 바로 말합니다.
  - 가장 작은 구성요소부터 시작해 상위 메커니즘까지 차례대로 쌓아 올립니다.
  - 설명자의 동선보다는 주제 자체가 앞에 옵니다.
- 잘 맞는 주제:
  - 컴퓨터 아키텍처
  - JVM
  - 프로토콜
  - 런타임과 부팅 과정
  - 일반적인 깊은 학습 문서 전반
- 강점:
  - 읽는 사람이 어디서부터 이해를 시작해야 하는지 분명합니다.
  - 작은 단위에서 큰 구조로 올라갈 때 논리 도약이 적습니다.
- 주의점:
  - 역사와 등장 맥락이 빠지면 다소 건조한 교재처럼 느껴질 수 있습니다.

샘플:

보통은 `jar` 파일을 클래스와 리소스를 묶은 압축 파일 정도로 이해하기 때문에 `META-INF/MANIFEST.MF`를 자세히 보지 않습니다. 하지만 `java -jar`는 이 파일을 먼저 읽습니다. 이 파일에는 JAR 전체에 공통으로 적용되는 메타데이터가 들어 있고, 그중 `Main-Class`는 어떤 클래스를 시작할지 알려 줍니다. 즉 manifest는 archive 안의 개별 파일 설명이 아니라, 이 JAR 전체를 어떻게 해석하고 실행해야 하는지 적어 두는 표준 위치입니다.

### 후보 B. 원인-등장-구조형

- 핵심 감각:
  - 이 구조가 왜 생겼는지를 먼저 설명합니다.
  - 문제, 제약, 역사적 배경이 현재 구조를 어떻게 만들었는지 보여 줍니다.
  - 공식 자료와 현재 구조를 인과로 연결하기 좋습니다.
- 잘 맞는 주제:
  - 포맷
  - 스펙
  - 표준 문서
  - 아키텍처 선택 이유
  - 오래된 규약이나 관례가 남아 있는 대상
- 강점:
  - 독자가 `왜 이런 파일이 있어야 하지?`라는 의문을 빨리 해소합니다.
  - 구조를 외워서가 아니라 납득해서 기억하게 만들기 좋습니다.
- 주의점:
  - 배경 설명이 길어지면 현재 메커니즘으로 들어가는 속도가 늦어질 수 있습니다.

샘플:

JAR 형식이 등장한 배경에는 여러 클래스와 리소스를 하나의 archive로 묶고, 압축과 서명까지 한꺼번에 다루려는 요구가 있었습니다. 이렇게 많은 파일을 하나로 묶으면 archive 전체에 대한 공통 설명이 필요해집니다. 무엇이 시작 클래스인지, 어떤 버전 정보가 들어 있는지, 무엇이 서명되었는지 같은 정보를 파일마다 흩어 적을 수는 없기 때문입니다. `META-INF/MANIFEST.MF`는 바로 그 공통 메타데이터를 담는 표준 위치이고, `java -jar`는 이 파일을 읽어 JAR 전체를 어떻게 시작해야 하는지 판단합니다.

### 후보 C. 실행 경로 추적형

- 핵심 감각:
  - 사용자가 입력한 명령이나 외부 이벤트에서 시작합니다.
  - 누가 무엇을 언제 읽는지 시간 순서대로 따라갑니다.
  - `입력 -> 해석 -> 다음 단계`가 또렷합니다.
- 잘 맞는 주제:
  - 부팅 과정
  - 요청 처리 흐름
  - 네트워크 핸드셰이크
  - 스레드 전이
  - 런타임 초기화
- 강점:
  - `실제로 무슨 순서로 움직이는가`를 이해하기 쉽습니다.
  - 주체가 바뀌는 지점을 잘 드러낼 수 있습니다.
- 주의점:
  - 시간 순서에만 집중하면 각 artifact의 정체와 등장 이유가 약해질 수 있습니다.

샘플:

`java -jar app.jar`를 실행하면 Java launcher는 먼저 JAR 안의 `META-INF/MANIFEST.MF`를 엽니다. 여기서 `Main-Class`를 읽고 첫 시작 클래스를 결정합니다. 즉 manifest는 단순한 설명용 파일이 아니라, 실행 경로를 다음 단계로 넘겨 주는 기준점입니다. 이 파일이 없거나 필요한 속성이 없으면 launcher는 어디로 진입해야 할지 판단할 수 없습니다.

### 후보 D. 질문-직답 확장형

- 핵심 감각:
  - 먼저 질문을 짧게 세웁니다.
  - 바로 직답을 준 뒤, 그 답을 계층별로 확장합니다.
  - 인터뷰형 원재료나 FAQ성 문서를 정식 문서로 승격할 때 유용합니다.
- 잘 맞는 주제:
  - 인터뷰 질문 승격
  - FAQ
  - 정의가 자주 흔들리는 개념 문서
  - 허브 문서의 진입 섹션
- 강점:
  - 독자가 지금 어떤 의문을 해소하고 있는지 놓치지 않습니다.
  - 긴 문서에서도 각 절의 목적을 선명하게 유지하기 쉽습니다.
- 주의점:
  - 모든 문단을 질문형으로 만들면 문서가 잘게 쪼개져서 장문 흐름이 약해질 수 있습니다.

샘플:

`META-INF/MANIFEST.MF`는 왜 필요할까요? `java -jar`가 JAR를 실행할 때 어디서 시작해야 하는지 알려 주는 공통 메타데이터 파일이기 때문입니다. JAR 안에는 클래스와 리소스가 많이 들어 있지만, archive 전체를 대표해 설명하는 위치가 따로 없으면 시작 클래스나 버전 정보 같은 공통 규칙을 묶어 둘 수 없습니다. manifest는 바로 그 공통 규칙을 담는 자리입니다.

### 후보 E. 실무 시나리오 연결형

- 핵심 감각:
  - 실제 장애나 운영 상황에서 시작합니다.
  - 왜 이 구조를 알아야 하는지 먼저 체감시키고, 그다음 메커니즘으로 들어갑니다.
  - systemd, infra, troubleshooting 문서에서 특히 강합니다.
- 잘 맞는 주제:
  - 운영 문서
  - troubleshooting
  - 배포
  - 프로세스 관리
  - 시스템 설정
- 강점:
  - 이 개념이 왜 중요한지 바로 체감됩니다.
  - 학습 내용이 실무 판단으로 연결되기 쉽습니다.
- 주의점:
  - 기초 개념 문서에 과하게 쓰면 상황 설명이 본론을 가릴 수 있습니다.

샘플:

실무에서 `java -jar`로 실행한 애플리케이션이 기대한 클래스가 아니라 다른 클래스에서 시작되는 이유를 추적할 때 가장 먼저 보는 파일이 `META-INF/MANIFEST.MF`입니다. 이 파일에는 JAR 전체의 시작 규칙이 들어 있기 때문입니다. `Main-Class`가 무엇으로 잡혀 있는지에 따라 launcher가 실제로 진입하는 클래스가 달라지고, Spring Boot executable jar라면 여기서 곧바로 애플리케이션 `main()`으로 가지 않고 `JarLauncher`를 먼저 타게 됩니다. 즉 manifest는 설명용 부속 파일이 아니라 실행 경로를 결정하는 실전용 메타데이터입니다.

## 이번 피드백으로 좁혀진 선택

이번 라운드 피드백까지 반영하면, 사용자가 실제로 원하는 것은 후보 하나를 고르는 방식보다 다음 순서로 설명을 쌓는 혼합형에 더 가깝습니다.

1. 시작은 후보 A `직진형 계층 누적형`과 후보 B `원인-등장-구조형`을 함께 사용합니다.

    먼저 지금 설명할 대상을 직접 말하고, 그다음 그 구조가 왜 생겼는지와 어떤 배경에서 등장했는지를 붙입니다.
    이 단계에서는 독자가 구조를 이해할 최소한의 맥락과 공통 메타데이터를 먼저 잡아야 합니다.

2. 필요하면 중간에 후보 E `실무 시나리오 연결형`을 섞습니다.

    이 구조를 왜 알아야 하는지, 실무에서 어떤 문제를 만나면 이 지식이 실제로 필요해지는지 연결합니다.
    다만 운영 상황 설명이 본론보다 길어지면 안 되므로, 현재 주제를 밀어 주는 범위까지만 사용합니다.

3. 그다음 후보 C `실행 경로 추적형`으로 더 깊게 내려갑니다.

    여기서부터는 `누가`, `무엇을`, `언제`, `어떤 순서로` 읽고 실행하는지 시간 순서대로 따라갑니다.
    즉 앞에서 구조와 맥락을 잡고, 뒤에서 실제 동작 경로를 추적하며 깊이를 더하는 형태입니다.

이 선택은 `후보 A+B로 문서를 연 뒤, 필요하면 E로 중요성을 연결하고, 이후 C로 실제 실행 경로를 깊게 파고드는 방식`으로 요약할 수 있습니다.

### 왜 이 조합이 더 정확한가

- 후보 A만으로 시작하면 구조는 선명하지만 등장 이유가 약해질 수 있습니다.
- 후보 B만으로 시작하면 배경은 강하지만 현재 메커니즘으로 들어가는 속도가 느려질 수 있습니다.
- 후보 C를 처음부터 전면에 두면 실행 순서는 잘 보이지만, 그 전에 알아야 하는 구조와 공통 메타데이터가 충분히 고정되지 않을 수 있습니다.
- 후보 E는 문서를 실무 감각과 연결하는 데 강하지만, 기초 설명 문서 전체를 E로만 밀면 상황 설명이 본론을 덮을 수 있습니다.

즉 지금 기준으로 가장 가까운 기본값은 `A+B로 문서를 시작하고, 필요하면 E를 짧게 연결한 뒤, C로 실제 실행 경로를 깊게 추적하는 방식`입니다.

`spring_boot_jar_startup.md` 같은 문서는 이 조합이 특히 잘 맞습니다.
처음에는 `META-INF/MANIFEST.MF`와 JAR 공통 메타데이터를 설명하면서 구조와 등장 배경을 함께 잡아야 하고, 그다음에는 `Main-Class -> JarLauncher -> Start-Class -> SpringApplication.run()`처럼 실제 실행 경로를 순서대로 따라가야 하기 때문입니다.

    독립 기본 문체라기보다, 현재 선택된 A+B+C 뼈대에 필요할 때 끼워 넣는 보조 축으로 보는 편이 더 정확합니다.
