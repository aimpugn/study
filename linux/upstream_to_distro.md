## 개요 — 이 문서가 답하는 질문

이 문서는 다음 의문에서 출발합니다.

1. nginx 공식 사이트의 latest와 우분투 저장소의 nginx 버전은 왜 다른가.
2. 업스트림이 "호환된다"고 보장해도 배포판 저장소에는 왜 새 버전이 안 올라오는가.
3. 커널은 누가 어떤 과정으로 개발하고, 우분투는 그것을 언제 어떻게 가져가는가.
4. 소스가 여러 갈래로 분기된다면 사람들은 어디에 기여하는가. SSOT(단일 진실 원천)는 어디인가.
5. 구버전 브랜치에 수정을 cherry-pick할 때, 그 사이 버전에서 개발된 것들이 딸려 들어가지 않는다고 누가·무엇이 보장하는가.

짧게 답하면 이렇습니다. **개발이 일어나는 라인은 업스트림 프로젝트의 mainline 하나뿐이고, 나머지(stable 브랜치, LTS, 배포판 저장소)는 전부 그 라인의 특정 시점 스냅샷에 "수정만" 선별해 내려보내는 소비 채널입니다.** 기여는 위(mainline)로만 올라가고 코드는 아래로만 흘러내립니다. "딸려 들어감"은 사람의 기억이 아니라 커밋 메타데이터(`Fixes:`, `Cc: stable`, `Stable-dep-of:`)와 자동 적용·빌드 검사, 공개 리뷰, CI가 겹겹이 막습니다. 그리고 이 구조는 커널만의 것이 아니라 nginx를 포함한 리눅스 패키지 생태계 전반의 공통 패턴입니다.

## 왜 이런 구조가 생겼나

| 1993 | Ian Murdock이 Debian 창립. "커널 + 유저랜드를 패키징하고 저장소로 배포"하는 배포판 모델의 원형 중 하나. |

| 2003 | Red Hat Linux가 단종되고 커뮤니티 배포판 Fedora로 분리 — "커뮤니티 버전이 먼저 검증하고 상용판이 그 스냅샷을 가져가는" 구조의 대표 사례. |
| 2004 | Mark Shuttleworth의 Canonical이 Ubuntu 4.10(Warty Warthog) 출시. Debian 스냅샷 + 6개월 시간 기반 릴리스. 같은 해 Igor Sysoev가 nginx 공개. |
| 2005 | BitKeeper 라이선스 분쟁으로 Linus가 git을 직접 작성(4월). 같은 해 Greg Kroah-Hartman과 Chris Wright가 -stable 팀을 시작해 2.6.11.1부터 점버전 릴리스 — stable 규칙이 이때 명문화됨. 별도 개발 브랜치(2.7)를 만들지 않는 새 개발 모델(머지 윈도우 체계)이 2.6 시대에 정착. |
| 2006 | Ubuntu 6.06(Dapper Drake) — 첫 LTS. |
| 2008 | Stephen Rothwell이 linux-next 통합 트리 시작 — 다음 머지 윈도우에 들어갈 내용의 사전 충돌 검증. |
| 2011 | Linux 3.0 — 2.6.x 번호 체계 종료(번호 단순화). |
| 2013 | Ubuntu 12.04.2 포인트 릴리스부터 HWE(Hardware Enablement) 커널 스택 도입 — 동결을 유지하면서 새 커널을 공식 제공하는 우회로. |
| 2017년경 | Sasha Levin이 AUTOSEL 도입 — 태그가 안 붙은 수정 커밋을 머신러닝으로 발굴해 stable 후보로 제안. (도입 연도는 추정) |
| 2018 | Spectre/Meltdown 공개 — 거대한 신규 완화 코드(retpoline 등)가 구버전 stable 전체에 백포트된, "수정만 받는다" 원칙의 의도적 예외 사례. |
| 2023 | Greg Kroah-Hartman이 LTS 기본 유지 기간을 6년에서 2년으로 줄이는 방침을 밝힘. 이후 실제 EOL은 수요에 따라 그보다 길게 운영되고 있음(아래 관측 표 참고). |
| 2026-06-10 관측 | mainline은 7.x 시대(7.0.12 릴리스 공지 확인). LTS는 6.18, 6.12(–2028), 6.6, 6.1(–2027), 5.15, 5.10(–2026)이 병행 유지 중. |

### 설계자들이 버린 가정과 택한 해법

짝수/홀수 이중 브랜치 모델이 실패한 지점이 현재 구조의 출발점입니다.

1. 인식한 문제

    개발 브랜치(2.5)와 안정 브랜치(2.4)의 괴리가 수년 단위로 벌어졌습니다. 안정 브랜치 사용자는 새 하드웨어 지원을 못 받고, 배포판 벤더들은 각자 거대한 사설 패치셋(사실상의 포크)을 유지하며 비용을 떠안았습니다. Debian도 같은 시기 릴리스 예측이 무너져 있었습니다 — woody(2002)에서 sarge(2005)까지 3년이 걸렸고, 이 예측 불가능성이 2004년 "정해진 날짜에 무조건 낸다"는 우분투의 시간 기반 릴리스를 낳았습니다.

2. 버린 가정

    "안정성은 별도의 긴 숙성 브랜치에서 나온다"는 가정을 버렸습니다. 안정 브랜치를 오래 둘수록 mainline과의 거리가 벌어져 백포트가 더 어려워지는 악순환이 확인됐기 때문입니다.

3. 검토된 대안과 선택

    영구 개발 브랜치 유지(기존 모델), rolling release처럼 모두가 최신을 따라가게 하기(검증 부담을 사용자에게 전가), 벤더별 포크 방치(생태계 분열) 대신, **단일 mainline에서 상시 개발 + 릴리스 직후 짧은 stable 점버전(2.6.x.y)에 수정만 cherry-pick**하는 모델을 택했습니다.

4. 예측한 부작용과 대비

    수정 선별이 잘못될 위험은 규칙 명문화(2005년 stable rules), "mainline에 먼저 있어야 한다"는 원칙, 공개 리뷰 기간으로 대비했습니다.

5. 정당화

    분기 수를 늘리지 않고도 "빠른 개발"과 "안정 소비"를 동시에 얻습니다. 비용은 커밋 메타데이터와 자동화가 부담합니다. 기업들이 이 원칙을 자발적으로 지키는 이유는 경제성입니다 — 자기 분기에만 패치를 들고 있으면 영원히 자기가 rebase 비용을 내지만, mainline에 머지시키면 이후 모든 버전과 모든 배포판에 무상으로 실려 갑니다.

## 지형 — 하나의 개발 라인과 여러 소비 채널

용어를 먼저 고정합니다. **mainline**은 Linus Torvalds가 유지하는 유일한 개발 라인입니다. **머지 윈도우**는 릴리스 직후 2주간 서브시스템 메인테이너들의 신기능 pull request를 받는 기간이고, 그 뒤 7~8주간 rc(release candidate)를 매주 내며 수정만 받다가 릴리스합니다(전체 주기 9~10주 — kernel.org 명시). **stable 브랜치**는 릴리스된 버전 태그에서 갈라져 나와 수정 커밋의 cherry-pick만 받는 유지 브랜치(6.12.y)이며, 그중 수년간 유지하기로 지정된 것이 **LTS**입니다. **cherry-pick**은 특정 커밋 하나의 diff만 다른 브랜치에 적용하는 git 연산입니다 — 히스토리(중간 커밋들)는 따라오지 않습니다. **백포트**는 새 코드의 수정을 구버전 코드에 맞게 옮겨 심는 행위의 총칭입니다.

```text
            모든 기여 (개인 · Canonical · Red Hat · Intel · AMD …)
                              │  패치 → 메일링리스트 → 서브시스템 메인테이너 트리 → linux-next
                              ▼
  mainline ── 6.11 ── 6.12 ── 6.13 ─ … ─ 7.0 ──▶     ← 유일한 개발 라인 (SSOT)
                        │
                        │ 릴리스 태그 v6.12에서 분기
                        ▼
              stable 6.12.y (6.12.1 … 6.12.93 …)      ← mainline 커밋의 cherry-pick 전용
                        │
                        │ Canonical이 주기적으로 머지 + 자체 검증(SRU)
                        ▼
              Ubuntu 커널 트리 (6.12.y + SAUCE 델타 + config/패키징)   ※ 예시 구조
                        │
                        ▼
              우분투 아카이브 (-proposed → -updates / -security) → 사용자 apt
```

화살표 방향이 이 문서 전체의 요지입니다. **기여는 위로만, 코드는 아래로만.** 분기들은 개발 장소가 아니라 배포 채널입니다. stable 트리의 제1규칙이 "mainline에 이미 존재하는 패치만 받는다"이므로, stable에 직접 기여한다는 개념 자체가 성립하지 않습니다.

출판에 비유하면 mainline은 저자가 계속 고쳐 쓰는 원고이고, stable 점버전은 이미 인쇄된 판본에 정오표만 반영한 새 쇄(printing)이며, 배포판은 서점이 "이 판본은 우리가 보증한다"고 정한 특정 쇄입니다. 2쇄에 새 챕터가 끼어들 수 없듯 stable에 신기능이 들어가지 않습니다. 다만 이 비유는 정오표끼리 의존성이 생기는 지점(어떤 수정이 다른 선행 수정을 전제하는 경우)에서 깨집니다 — 그 경우의 처리가 별도 절차로 존재하며, 아래 "의존성이 정말 필요할 때"에서 다룹니다.

## 실물 해부 — 수정 하나가 6.12.93에 실리기까지

추상 설명 대신, 2026-06-10에 linux-6.12.y 브랜치(kernel.org의 Google 미러)에서 직접 확인한 실제 커밋을 해부합니다. 같은 "수정 하나"가 시점마다 다른 모습으로 관측됩니다.

```text
commit b7b72e88046328c9fdc638fe887d4240257dd5dc          # linux-6.12.y 브랜치, 6.12.93에 포함
KVM: arm64: vgic-its: Drop the translation cache reference
only for the erased entry                                 # use-after-free(해제 후 사용) 버그 수정

    commit 13031fb6b8357fbbcded2a7f4cba73e4781ee594 upstream
        # mainline 원본 커밋의 전체 해시.
        # stable 커밋마다 이 표기가 박혀 있어 "어느 mainline 커밋의 복사본인지"가
        # 기계적으로 추적된다. 이 표기가 없는 코드는 stable에 들어올 수 없다.

    Fixes: 8201d1028caa ("KVM: arm64: vgic-its: Maintain a translation cache per ITS")
        # 버그를 '만든' 원인 커밋의 지목.
        # "6.12.y에 이 수정이 필요한가?"는 "8201d1028caa가 v6.12에 포함돼 있는가?"로
        # 환원되고, 그 판정은 git이 즉답한다(아래 직접 확인해 보기 참고).
        # 원인 커밋이 6.13에서 처음 들어갔다면 6.12.y는 백포트 대상 자체가 아니다.

    Cc: stable@vger.kernel.org
        # 저자/메인테이너가 "이 수정은 stable 후보"라고 선언하는 태그.
        # 'Cc: stable # 6.1+'처럼 적용 하한 버전을 지정하는 문법도 있다.

    Signed-off-by: Marc Zyngier <maz@kernel.org>          # 해당 서브시스템(KVM/arm64) 메인테이너
    Signed-off-by: Greg Kroah-Hartman <gregkh@...>        # stable 트리 유지자 — 사람 책임의 사슬
```

이 수정의 일생을 시간 순서로 다시 놓으면 다음과 같습니다. 1~6은 이 커밋에서 직접 관측된 사실이고, 7은 절차상의 일반 흐름입니다(이 특정 커밋의 우분투 도달 여부는 관측하지 않음).

1. 과거 어느 시점, mainline 커밋 `8201d1028caa`가 기능(번역 캐시)을 넣으면서 버그도 함께 태어났습니다.
2. Hyunwoo Kim이 수정 패치를 작성해 메일링리스트로 보내고, 리뷰(Oliver Upton) 후 서브시스템 메인테이너(Marc Zyngier)가 받아들였습니다.
3. 머지 윈도우에 mainline으로 들어가 커밋 `13031fb6...`이 됐습니다. 이 시점의 모습은 "mainline 커밋"입니다.
4. `Cc: stable` 태그를 본 stable 팀이 6.12.y에 cherry-pick했습니다. 이 시점의 모습은 "upstream 참조가 박힌 stable 커밋"(`b7b72e8`)입니다.
5. 릴리스 전 공개 리뷰를 거쳤습니다 — 2026-06-10 메일링리스트(미러)에서 `[PATCH 6.12 000/307] 6.12.93-rc1 review`라는 제목, 즉 6.12.93에 들어갈 307개 패치의 공개 리뷰 스레드가 관측됩니다. 해당 서브시스템 메인테이너가 여기서 NAK할 수 있고, 그 기간 동안 CI(KernelCI 등)가 다중 아키텍처 빌드·부트 테스트를 돌립니다.
6. `Linux 6.12.93` 릴리스 공지와 함께 점버전으로 출고됐습니다. kernel.org 기준 stable 점버전은 "보통 주 1회" 나옵니다.
7. (절차상 흐름) Canonical 커널팀이 6.12.y를 자기 트리에 머지하고, SRU 절차(-proposed에서 검증·숙성 후 -updates 이동)를 거쳐 사용자의 `apt upgrade`에 도달합니다.

## 무엇이 "딸려 들어감"을 막는가 — 다섯 겹의 방어선

"6.12 기반 브랜치에 7.0 시점의 수정을 가져오면, 그 사이(6.13~6.18)에 개발된 것들이 묻어 들어오지 않나?"라는 의문에 대한 답입니다. 먼저 구조적 사실 하나 — cherry-pick은 커밋 하나의 diff만 가져오므로 **중간 커밋들이 커밋 단위로 동반 입장하는 경로 자체가 없습니다.** 위험은 다른 형태, 즉 "그 diff의 내용이 중간 작업을 전제하는 경우"로 존재하며, 이것을 다음 방어선들이 단계적으로 걸러냅니다.

1. 규칙 — 후보 자격의 제한

    stable 규칙(공식 문서) 인용: "mainline에 이미 존재해야 한다", "문맥 포함 100줄을 넘을 수 없다", "oops·행·데이터 손상·실제 보안 문제·빌드 에러 같은 실제 버그를 고치는 것이어야 한다". 신기능·리팩터링·이론적 문제는 애초에 후보가 아닙니다.

2. 메타데이터 — 혈통의 선언

    `Fixes:` 태그가 버그의 출생지를, `Cc: stable # 버전`이 적용 범위를 선언합니다. "이 브랜치가 대상인가"를 사람의 전역 기억이 아니라 커밋 자신의 선언 + git 조상 질의로 판정합니다.

3. 물리 검출 — 적용·빌드 실패의 표면화

    새 코드에 의존하는 패치는 구버전에 적용할 때 충돌하거나(주변 문맥이 다름) 빌드가 깨집니다(없는 함수 호출). 이때 조용히 우겨넣지 않고 자동 통지가 원저자에게 갑니다. 2026-06-10 메일링리스트 관측 실물: `FAILED: patch "[PATCH] KVM: arm64: Reassign nested_mmus array behind mmu_lock" failed to apply to 6.12-stable tree`. 저자는 필요하면 수동 백포트를 다시 보냅니다 — 의존성을 가장 잘 아는 사람에게 공이 돌아가는 구조입니다.

4. 공개 리뷰 — 사람의 거부권

    모든 stable 점버전은 rc로 메일링리스트에 올라가 48시간의 ACK/NAK 기간을 거칩니다(규칙 문서 명시). 들어갈 패치 전체 목록이 공개되고, 해당 서브시스템 메인테이너가 "이건 이 브랜치에 안 맞다"고 거부할 수 있습니다.

5. 자동 테스트와 사후 추적

    stable-rc를 KernelCI, Linaro LKFT, Intel 0-day, syzbot 같은 테스트 인프라가 상시로 돌리고, 릴리스 후 회귀가 신고되면 신속히 revert합니다(회귀 추적 체계 regzbot).

태그를 깜빡한 수정을 머신러닝으로 발굴해 제안하는 AUTOSEL도 이 파이프라인에 후보를 공급하는데, 이것 역시 바로 머지가 아니라 리뷰 기간을 거칩니다.

## 의존성이 정말 필요할 때 — 네 가지 갈래

수정이 중간 버전의 작업을 정말로 필요로 하는 경우는 발생하며, 그때마다 사람이 다음 중 하나를 명시적으로 선택합니다.

1. "같이 쓰면 좋은" 수준 — 단칼에 거절

    stable의 기준은 "필요한가"이지 "좋아지는가"가 아닙니다. 더 우아한 수정이 가능해도 의존성 없는 형태로만 들어갑니다.

2. 꼭 필요한 작은 의존성 — 지목해서 같이 입장

    선행 커밋을 태그로 지목하면(`Cc: stable # 6.12.x: <해시>: <제목>` 문법 — 규칙 문서 명시) 그 커밋만, 별도의 cherry-pick으로, 같은 리뷰를 거쳐 들어옵니다. 2026-06-10 관측 실물 — 6.12.93에 포함된 다음 커밋은 내용상 리팩터링인데도 들어와 있고, 그 이유가 커밋 메시지에 선언돼 있습니다.

    ```text
    commit 1285e83... (linux-6.12.y)
    mm: perform all memfd seal checks in a single place    # seal 검사를 한 곳으로 모으는 리팩터링

        Stable-dep-of: 3b041514cb6e ("memfd: deny writeable
        mappings when implying SEAL_WRITE")
            # "나는 저 보안 수정이 6.12.y에 적용되기 위한 선행 의존성으로 들어왔다"는 선언.
            # 중간 작업이 몰래 딸려 오는 것이 아니라, 꼬리표를 달고 선별 입장한다는 실물 증거.
    ```

3. 의존성이 큰 경우 — 구버전용 수정을 새로 작성

    1.0 시대 코드만으로 같은 버그를 잡는 별도 패치를 저자가 만들어 보냅니다. mainline 커밋과 내용이 달라도 되고, "mainline 어느 커밋에 해당하는 백포트"라는 추적 표기만 유지하면 됩니다.

4. 그래도 안 되면 — 그 브랜치는 포기

    "이 수정은 이 브랜치에 백포트하기엔 너무 침습적"이라는 판단은 흔하고 정당한 결말입니다. 그 버그가 아쉬운 사용자에게 주어지는 답은 수정의 하향이 아니라 **브랜치의 상향**("더 새 커널로 올라와라")이며, 우분투 HWE 커널 같은 공식 상향 경로가 그래서 존재합니다.

이 갈래들을 가르는 가치 서열이 stable의 본질입니다. stable의 약속은 "모든 버그가 고쳐진다"가 아니라 **"예상 밖의 변화가 없다"**이고, 둘이 충돌하면 항상 후자가 이깁니다. 극히 드문 예외가 2018년 Spectre/Meltdown입니다 — 사용자를 취약하게 두는 것보다 나쁜 선택지가 없어서, 거대한 신규 완화 코드를 구버전 stable 전체에 작정하고 백포트했습니다. 우발적 혼입이 아니라 비용을 치른 의도적 결정이라는 점에서 오히려 원칙을 증명하는 예외입니다.

## 배포판 쪽 절반 — 우분투의 동결과 공식 우회로

업스트림이 위쪽 절반이라면, 아래쪽 절반은 배포판의 동결(freeze) 모델입니다.

우분투는 기능이 아니라 **시간 기반**으로 릴리스합니다. 매년 4월·10월로 날짜가 먼저 정해져 있고, 사이클이 진행되며 단계적으로 동결됩니다 — Debian unstable 자동 동기화로 시작해, Feature Freeze(릴리스 약 2달 전, 이후 버그 수정만), Beta Freeze, Kernel Freeze(릴리스 약 2~3주 전, 커널 버전 최종 고정), Final Freeze, 릴리스 순서입니다. 커널 버전 선택은 사이클 중반에 Canonical 커널팀이 "릴리스 날짜까지 안정화 가능한 가장 새로운 업스트림"을 기준으로 결정합니다. 22.04(jammy)는 5.15, 24.04(noble)는 6.8을 실었고, 2026-06-10 관측 기준 kernel.ubuntu.com은 26.04가 업스트림 7.0 기반이라고 안내합니다.

릴리스 후에는 그 버전에 고정됩니다. nginx든 커널이든 같은 원리입니다.

- 패키지(nginx 예): noble은 수명 내내 1.24 기반. 보안 결함은 새 버전 도입이 아니라 해당 수정의 백포트로 처리되며, 버전 문자열 대신 패키지 리비전(`1.24.0-2ubuntu7.3`의 뒷부분)이 올라갑니다. "버전이 낮다 ≠ 취약하다"인 이유.
- 커널: noble은 수명 내내 6.8 기반이고 `6.8.0-XX-generic`의 ABI 번호만 올라갑니다. 업스트림의 6.8.y 유지는 2024년에 끝났으므로(6.8은 비LTS), 그 이후의 백포트는 Canonical이 직접 수행합니다 — 배포판이 "버전 동결 + 백포트"의 최종 책임자가 되는 지점입니다.
- 업데이트 절차: 수정은 곧바로 사용자에게 가지 않고 SRU(Stable Release Update) 절차를 거칩니다 — `-proposed` 포켓에서 검증·숙성 후 `-updates`로, 보안 수정은 보안팀이 `-security`로 내보냅니다. 2026-06-10 관측 기준 커널 SRU는 4~5주 주기(패치 마감 → 1~2주 준비·테스트 → 릴리스), 보안 사이클은 2주 주기로 공개 운영됩니다.

저장소(아카이브) 자체도 배포판이 운영합니다. 업스트림 프로젝트는 우분투 아카이브에 아무 권한이 없습니다. 업로드 권한을 가진 우분투 개발자가 소스 패키지를 Launchpad에 올리면 공식 빌드 팜이 전 아키텍처용 바이너리를 빌드·서명해 archive.ubuntu.com과 미러에 게시합니다. 내부는 라이선스·지원 수준에 따른 컴포넌트(main/restricted/universe/multiverse)와 용도별 포켓(release, -updates, -security, -proposed, -backports)으로 나뉩니다. 유저랜드 패키지 대부분의 1차 출처는 Debian이며(사이클 초 Debian unstable 자동 동기화 + 필요한 패키지만 델타 유지), 커널은 우분투가 처음부터 자체 패키징하는 대표 예외입니다. 우분투 커널 트리에는 업스트림에 (아직) 못 들어간 우분투 고유 패치(SAUCE — 예: 라이선스 충돌로 머지가 불가능한 ZFS)가 얹히는데, 이 델타는 매 사이클 새 베이스 위로 다시 얹어야 하는 빚이라 최소화가 원칙입니다.

동결이 기본값이되, 공식 우회로가 채널별로 마련돼 있습니다.

- 다음 릴리스: 새 버전은 개발 브랜치로 계속 유입되므로 다음 우분투 릴리스에는 들어갑니다. 막힌 것은 "이미 동결된 릴리스"뿐입니다.
- backports 포켓: stable 릴리스용으로 빌드한 더 새로운 버전의 옵트인 저장소. 단 패키지별 메인테이너가 올려야 존재합니다.
- HWE 커널: 이후 인터림 릴리스의 커널을 LTS에 공식 제공(2013년 12.04.2부터). 동결 원칙과 새 하드웨어 지원 요구를 한 저장소 안에서 양립시키는 장치입니다.
- 외부 공식 저장소: nginx.org가 직접 운영하는 apt/yum 저장소처럼 업스트림이 제공하는 채널을 추가할 수 있습니다. 이때 배포판이 해 주던 호환성 검증과 보안 추적 책임은 사용자에게 넘어옵니다.
- RHEL 계열의 모듈 스트림: 같은 OS 안에서 `dnf module`로 더 새로운 nginx 스트림(예: RHEL 9의 1.22, 1.24)을 선택하게 하는 방식 — 동결 유지와 신버전 제공을 공식 저장소 안에서 동시에 푸는 또 다른 답.

## 채널 한눈 비교

같은 "리눅스 소프트웨어"라도 어느 채널에서 받느냐에 따라 계약이 다릅니다. 칸의 일반 원리는 시간 무관, 구체 버전·주기는 2026-06-10 관측 기준입니다.

| 축 | mainline (kernel.org) | stable/LTS (kernel.org) | Ubuntu GA 커널 | Ubuntu HWE 커널 | RHEL 커널 | 배포판 아카이브의 일반 패키지 (예: nginx) |
| --- | --- | --- | --- | --- | --- | --- |
| 관리 주체 | Linus + 서브시스템 메인테이너 | Greg KH 등 stable 팀 | Canonical 커널팀 | Canonical 커널팀 | Red Hat 커널팀 | 배포판 패키지 메인테이너 (다수는 Debian 유래) |
| 들어오는 것 | 신기능 + 수정 전부 | mainline 수정의 cherry-pick만 | stable.y 머지 + SAUCE 델타 | 이후 릴리스의 커널 계열 통째 | 동결 베이스 + 대규모 선별 백포트(기능 포함) | 동결 버전 + 보안/중요 수정 백포트 |

| 검증 주체 | 메인테이너 리뷰 + linux-next + rc 테스트 | 48h 공개 리뷰 + CI 팜 | Canonical 자체 회귀 테스트 + SRU 숙성 | 동일 | Red Hat QA (패치 개별 선별) | 배포판 QA + SRU |

표에서 RHEL 열이 보여 주듯, "동결 베이스에 기능까지 백포트"하는 모델도 존재합니다 — RHEL 9의 커널은 버전 문자열만 5.14이고 내용물은 훨씬 새 커널의 기능을 조직적으로 백포트한 결과물이며, 그 공수가 곧 구독료의 실체입니다. 우발적 혼입과 제품 전략으로서의 백포트는 전혀 다른 것입니다.

## 혼동하기 쉬운 것들

대조쌍마다 양쪽을 같은 밀도로 적습니다.

- "버전이 낮다" vs "취약하다"

    배포판의 버전 문자열은 동결 시점을 가리킬 뿐입니다. 보안 수정은 백포트되어 패키지 리비전(`-2ubuntu7.3`)으로 표시됩니다. 그래서 버전 문자열만 보는 취약점 스캐너는 패치된 시스템을 취약하다고 오탐하기 쉽습니다. 판정은 버전이 아니라 changelog(CVE 항목)로 합니다.

- cherry-pick vs merge

    merge는 두 브랜치의 히스토리를 합치므로 중간 커밋이 전부 들어옵니다. cherry-pick은 커밋 하나의 diff만 복사합니다. stable이 cherry-pick 전용인 이유가 바로 "중간 작업의 커밋 단위 동반 입장"을 구조적으로 차단하기 위해서입니다. 남는 위험은 diff 내용이 중간 작업을 전제하는 경우뿐이고, 그것은 메타데이터·적용/빌드 실패·리뷰가 거릅니다.

- upstream의 "호환된다" vs 배포판의 "변하지 않는다"

    업스트림의 호환 보장은 자기 프로젝트 단독 기준입니다. 배포판의 약속은 더 강한 것 — 수만 패키지 조합 안에서 동작이 아예 변하지 않는다 — 이고, 검증 주체도 배포판 자신입니다. 그래서 업스트림이 호환을 보장해도 동결 릴리스의 기본 저장소에는 새 버전이 올라오지 않습니다.

- 배포판 트리 vs 기여 목적지

    배포판 커널 트리는 피드백의 수집·검증 창구이지 코드의 머지 목적지가 아닙니다. 우분투에서 발견한 일반 커널 버그의 수정은 mainline으로 올라갔다가(upstream first) stable을 거쳐 되돌아옵니다. 이 우회 덕분에 같은 수정이 우분투만이 아니라 모든 배포판에 도달합니다. 배포판 트리에서 직접 고치는 것은 배포판 고유 영역(config, SAUCE, 패키징)뿐입니다.

- "latest" vs "안정 최신"

    업스트림 프로젝트 자체도 채널이 둘인 경우가 많습니다. 커널의 mainline 최신(7.x)과 LTS 최신(6.12.93)은 다른 채널이고, nginx도 mainline(홀수 마이너)과 stable(짝수 마이너)을 나눠 릴리스하며 "latest"는 보통 mainline을 가리킵니다. 배포판 버전과 비교할 때는 어느 채널과 비교하는지부터 맞춰야 합니다.

## 한계와 실패 모드 — 이 시스템이 새는 곳

이 구조는 수학적 보장이 아니라 다층 방어 + 신속 롤백입니다. 어디서 새는지 알아야 신뢰 수준을 옳게 잡습니다.

- 의미 의존은 자동으로 못 잡습니다. 백포트 패치의 위험은 3단계입니다 — 텍스트 의존(적용 충돌로 자동 검출), 빌드 의존(컴파일 실패로 자동 검출), 의미 의존(깨끗하게 붙고 빌드되지만 구버전 문맥에서 오동작). 마지막 단계는 사람 리뷰·테스트·사후 revert만이 방어선이고, 실제로 stable 점버전이 회귀를 일으키는 사고가 간헐적으로 발생합니다.
- AUTOSEL 논쟁. 태그 누락 수정을 발굴하는 순기능과, 의미 문맥을 놓친 커밋을 끌어온다는 비판이 공존합니다(진행형 논쟁).
- 동결의 그늘. "백포트하기엔 너무 침습적"으로 분류된 버그는 그 브랜치에서 영영 고쳐지지 않을 수 있습니다. 동결 모델에서 사용자가 가진 답은 브랜치 상향뿐입니다.
- 신뢰 스펙트럼. stable.y를 통째로 따르지 않고 패치를 개별 선별하는 보수파(RHEL), 통째 머지 + 자체 검증의 중간파(Ubuntu), 검증을 사용자에게 넘기는 rolling(Arch 등)이 공존합니다. 어느 지점이든 공짜는 없고, 검증 비용을 누가 내느냐의 차이입니다.
- 시점 민감성. 이 문서의 버전·EOL·주기 수치는 2026-06-10 관측입니다. LTS 유지 기간 정책은 2023년에도 바뀌었을 만큼 유동적이므로, 수치를 인용할 일이 있으면 아래 확인 경로로 재관측한 뒤 씁니다.

## 직접 확인해 보기

각 항목은 "무엇을 실행하면, 무엇이 보여야 PASS인지"까지 적습니다. 1~2번은 우분투 머신, 3~4번은 리눅스 커널 git 클론이 필요합니다.

1. 버전 동결과 리비전 증가 관측

    ```sh
    apt policy nginx        # 후보 버전이 릴리스 베이스(noble이면 1.24.x)에 머물고
                            # '-2ubuntu7.3' 같은 리비전만 다르면 PASS.
                            # 베이스 마이너가 저절로 올라가 있으면 FAIL —
                            # 외부 저장소(PPA, nginx.org)가 추가됐다는 신호이므로
                            # /etc/apt/sources.list.d/ 를 확인한다.
    ```

2. "버전 동결 ≠ 보안 방치" 관측

    ```sh
    apt changelog nginx | head -40   # 버전은 그대로인데 CVE 번호가 박힌
                                     # 백포트 항목들이 보이면 PASS.
    ```

3. stable 브랜치의 정체 관측

    ```sh
    git log --oneline v6.12..stable/linux-6.12.y | wc -l   # 수천 개 — 전부 cherry-pick된 수정
    git show <위 범위의 아무 커밋>                           # 메시지에 'commit <해시> upstream'
                                                            # 표기가 있으면 PASS. 없으면 FAIL
                                                            # (mainline 출신이 아닌 코드).
    ```

4. "이 브랜치가 백포트 대상인가"의 기계 판정 재현

    ```sh
    git merge-base --is-ancestor 8201d1028caa v6.12; echo $?   # 0이면 원인 커밋이 v6.12에
                                                               # 존재 → 6.12.y는 수정 대상.
                                                               # 본문 실물 해부의 Fixes: 판정과
                                                               # 같은 질의다.
    ```

5. 방어선의 가동 관측

    lore.kernel.org/stable(차단 시 미러 spinics.net/lists/stable)에서 `FAILED: patch`와 `rc review`를 검색합니다. 최근 날짜의 자동 실패 통지와 수백 패치 규모의 공개 리뷰 스레드가 보이면 PASS — 절차가 문서상이 아니라 현재 가동 중이라는 뜻입니다.

6. 시점 민감 값 재관측

    kernel.org 릴리스 페이지에서 현행 LTS·EOL을, kernel.ubuntu.com에서 현행 SRU 주기를 확인해 본문 표와 대조합니다. 달라져 있으면 이 문서의 관측 표를 갱신합니다.
