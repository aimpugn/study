## `command`란?

`command`는 POSIX 셸 빌트인으로, 두 가지 일을 한다.

1. 주어진 이름을 **셸 함수·별칭(alias)을 거치지 않고** 명령어(외부 실행파일 또는 빌트인)로 실행한다.
2. `-v`/`-V` 옵션으로 그 이름이 **어떻게 해석되는지(경로·빌트인·별칭 여부)를 조회**한다.

다른 명령을 실행하거나 들여다보는 것이 일이라, 명령어를 다루는 명령어인 셈이다.

## 왜 존재하나?

- **동명 함수의 무한재귀 방지**: 외부 명령을 같은 이름의 함수로 감쌀 때, 함수 안에서 원래 명령을 부르려면 `command`가 필요하다.
- **이식성 있는 존재 검사**: 어떤 명령이 설치돼 있는지 스크립트에서 확인하는 표준 방법이 `command -v`다(아래 `which`와의 차이 참고).

## 어떤 경우에 사용되나?

1. 명령 존재 여부 검사 (스크립트 가드)
2. 동명 셸 함수/별칭을 우회해 원본 명령 실행
3. 표준 PATH로만 유틸 실행 (`command -p`)

## 어떻게 동작하는가?

- `command 이름 [인자]` : `이름`을 함수/별칭으로 보지 않고 실행한다(빌트인·외부 명령은 그대로 적용).
- `command -v 이름` : 실행될 대상을 한 줄로 출력. 외부 명령이면 경로, 빌트인이면 이름, 별칭이면 그 정의를 보여준다. **찾으면 종료코드 0, 없으면 0이 아님** → `if`/`||` 가드에 그대로 쓴다.
- `command -V 이름` : 사람이 읽기 좋은 자세한 설명("... is a shell builtin" 등).
- `command -p 이름` : 시스템 기본 PATH로 표준 유틸을 찾아 실행(PATH가 오염됐을 때 안전하게).

## `which`·`type`과의 차이

- `which`는 셸 빌트인이 아니라 **외부 프로그램**이라 OS·배포판마다 동작이 다르고, 셸 함수나 빌트인은 보지 못한다.
- `command -v`는 **POSIX 표준 빌트인**이라 이식성이 높고 함수·별칭·빌트인까지 인식한다. 그래서 스크립트의 존재 검사에는 `which`보다 `command -v`를 쓰는 편이 안전하다.
- `type`도 비슷한 정보를 주지만, 종료코드와 한 줄 출력으로 깔끔하게 떨어지는 쪽은 `command -v`다.

## 사용 예시

```sh
# 1) 존재 검사 — 예: Nexus 기동 전 Java(JRE)가 있는지 확인
command -v java >/dev/null 2>&1 || { echo "Java(JRE)가 필요합니다"; exit 1; }

# 2) 동명 함수 안에서 원본 명령 호출 (무한재귀 방지)
ls() { command ls --color=auto "$@"; }

# 3) 무엇으로 실행되는지 조회
command -v docker     # → /usr/bin/docker
command -V cd         # → cd is a shell builtin
```

## 주의사항

- `command -v`는 별칭·함수도 보고하므로, "외부 실행파일이 있는가"만 보려면 출력 형태를 함께 확인한다.
- `command`는 함수·별칭은 우회하지만 **빌트인은 우회하지 않는다**.
- 빌트인이라 `if`·서브셸 조건에서 가볍고 안전하게 쓸 수 있다(외부 `which` 호출보다 빠르고 이식성↑).
