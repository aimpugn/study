# 정규식 — "원하는 패턴만 남기기"는 왜 두 갈래가 필요한가

10,000자가 넘는 한 줄짜리 문자열에서 `title="L1PVWEB01(10.11.22.33)"` 같은 서버 정보만 남기고 나머지를 전부 지우려 하면, 정규식에는 "이 패턴 빼고 전부"를 직접 말하는 문법이 없다는 벽에 부딪힙니다. 이 문서는 그 벽을 넘는 표준 기법(두 갈래 교대 매칭 + 그룹 참조 치환)을 출발점으로 삼아, 백트래킹 정규식 엔진이 문자열 위를 걷는 모델 전체를 추적하고, 그 모델이 갈라져 나온 두 엔진 계보(백트래킹 vs 오토마타)까지 내려갑니다.

기준 환경은 VS Code 찾기/바꾸기(JavaScript 정규식)이되, 동작 원리는 Perl·PCRE·Java·Python `re` 등 모든 백트래킹 엔진에 공통입니다. 본문의 모든 관측값은 "직접 확인해 보기" 절의 스크립트로 재현했습니다(Node v24.15.0, Windows 11, 2026-06-12 실행).

## 1. 짧게 답하면

남길 패턴을 P라고 할 때, **(P)|(P가 시작되지 않는 구간)** 두 갈래로 문자열 전체를 빈틈없이 덮고, 치환식의 그룹 참조 `$1`로 P만 되살리는 방식을 씁니다. 서버 정보 사례의 실제 레시피는 다음과 같습니다.

찾기 (VS Code `Ctrl+H`, 정규식 모드 `.*` 켜기):

```
(title="[^"]*\(\d{1,3}(?:\.\d{1,3}){3}\)")|(?:(?!title="[^"]*\(\d{1,3}(?:\.\d{1,3}){3}\)").)+
```

바꾸기:

```
$1\n
```

쓰레기 구간 하나당 빈 줄 하나가 흔적으로 남으므로, 2차 패스로 찾기 `^\n` → 바꾸기 (빈 값)을 한 번 더 돌리면 서버 정보만 줄 단위로 남습니다.

이 구조가 성립하는 이유는 세 가지입니다.

1. 두 갈래가 "P가 매치되는 위치"와 "P가 매치되지 않는 위치"로 모든 위치를 정확히 양분하므로, 엔진이 글자를 건너뛰는 일이 없습니다.
2. 부정 lookahead `(?!P)`가 쓰레기 구간의 경계를 P 직전에 정확히 세워 줍니다.
3. 갈래1이 이기면 `$1` = P, 갈래2가 이기면 `$1` = 빈 문자열이므로, 치환식 하나로 조건부 치환 효과가 납니다.

쓰레기 구간 갈래만 단독으로 쓰면 안 되는 이유(각 P의 첫 글자만 살아남는 기현상)는 8절의 반례 trace에서 실측과 함께 확인합니다.

## 2. 출발점 — 실제 문제와 레시피

실제 입력은 모니터링 화면 등에서 복사한 한 줄짜리 HTML성 문자열이고, 그 안에 서버 이름과 IP가 박힌 `title` 속성이 산재합니다. 다이어그램 정렬을 위해 축약한 대표 입력은 다음과 같습니다.

```
입력:  ab title="L1PVA(1.2.3.4)" cd
목표:  title="L1PVA(1.2.3.4)"      <- 이것만 남기고 ab, cd 는 제거
```

남길 패턴 P를 토막 내면 다음과 같습니다.

```
title="          리터럴. 닻 역할 — 대부분의 위치에서 첫 글자 비교만으로 매치가
                 실패하므로, 6절의 lookahead 비용을 낮추는 효과까지 겸합니다
[^"]*            서버 이름부. 큰따옴표만 아니면 무엇이든(한글 포함) — 탐욕적으로
                 끝까지 먹은 뒤 다음 토큰 \( 를 위해 되돌아옵니다
\(  \)           괄호 리터럴. \ 를 빼면 캡처 그룹 문법으로 해석되어 다른 패턴이 됩니다
\d{1,3}          IP 첫 옥텟. 999 도 통과하는 느슨한 검사 — 추출 용도로는 충분하고,
                 엄밀히 하려면 옥텟마다 (?:25[0-5]|2[0-4]\d|1?\d?\d) 로 조입니다
(?:\.\d{1,3}){3} ".옥텟" 3회 반복. (?:...) 는 캡처하지 않는 묶음 — $1 번호를
                 어지럽히지 않으려고 씁니다
"                닫는 따옴표. 속성 경계를 고정합니다
```

`L1PV` 접두사가 붙은 서버만 남기려면 패턴 안의 `[^"]*` 를 `L1PV[^"]*` 로 바꾸되, 찾기 문자열에 P가 두 번(갈래1, lookahead 안) 등장하므로 **양쪽을 똑같이** 바꿔야 합니다. 한쪽만 바꾸면 어떻게 부서지는지는 8절에서 관측합니다.

## 3. 정규식에는 왜 "이것만 빼고"가 없나

정규식은 "존재하는 문자열의 모양"을 기술하는 언어입니다. 부정을 표현하는 자리는 딱 두 군데뿐입니다.

- 문자 클래스 부정 `[^"]` — 글자 **1개** 단위의 제외
- 부정 lookaround `(?!P)`, `(?<!P)` — 폭 0의 **검사**만 하고 소비는 못 함

즉 "임의 길이 패턴 P의 여집합을 매치해서 소비하라"는 연산자는 실무 정규식 문법에 없습니다. 이론적으로는 정규언어가 여집합에 닫혀 있어서(오토마타의 수용 상태를 반전하면 여집합 DFA가 나옵니다) "P를 포함하지 않는 문자열"의 정규식이 항상 존재하지만, 그것을 사람 손으로 쓰면 패턴이 감당 못 할 크기로 불어납니다. 그래서 실무에서는 lookahead로 같은 효과를 **합성**합니다 — `(?!P).` 즉 "P가 시작되지 않는 자리의 글자 하나"를 반복하는 방식이며, 이 관용구를 tempered greedy token이라고 부릅니다.

## 4. 두 계보의 역사 — 신경망 논문에서 전역 장애까지

정규식은 프로그래밍 도구가 아니라 신경생리학 모델에서 출발했고, 1968년을 기점으로 두 구현 계보로 갈라졌습니다. 오늘 쓰는 VS Code의 두 검색 기능에 두 계보가 하나씩 들어 있습니다(11절).

1. **1943** — 신경생리학자 Warren McCulloch와 논리학자 Walter Pitts가 신경 활동을 논리식으로 모델링한 논문을 발표합니다. 정규식의 가장 먼 조상입니다.
2. **1951/1956** — 수학자 Stephen Kleene이 위 신경망 모델이 표현할 수 있는 사건을 "regular events"로 정식화합니다(1951년 RAND 메모, 1956년 Automata Studies 수록). 반복 연산자 `*`의 이름(Kleene star)이 여기서 나왔습니다.
3. **1959** — Michael Rabin과 Dana Scott이 비결정적 유한 오토마톤(NFA)을 도입하고, NFA가 DFA와 표현력이 같음(부분집합 구성)을 증명합니다. 두 사람은 이 업적으로 1976년 튜링상을 받습니다.
4. **1967년경~1968** — Ken Thompson이 편집기 QED에 정규식 검색을 탑재하고(연도는 회고 자료 기반 근사), 1968년 CACM 논문으로 알고리즘을 공개합니다. 정규식을 NFA로 컴파일해 가능한 모든 상태를 동시에 전진시키는 방식 — **오토마타 계보**의 시작입니다.
5. **1973** — Thompson이 ed의 `g/re/p` 명령을 독립 도구 grep으로 분리합니다(Kernighan의 회고 기준).
6. **1986** — Henry Spencer가 자유 배포 C 정규식 라이브러리를 공개합니다. 재귀 백트래킹 구현 — **백트래킹 계보**가 업계로 퍼지는 기점입니다.
7. **1987** — Larry Wall이 Perl 1.0에서 정규식을 언어의 1급 기능으로 끌어올립니다.
8. **1992** — POSIX.2(IEEE 1003.2)가 BRE/ERE 문법과 leftmost-longest 매치 의미론을 표준화합니다.
9. **1994/1998** — Perl 5가 lookahead `(?=)` `(?!)`, 비캡처 그룹 `(?:)`, 비탐욕 수량자를 도입하고(1994), lookbehind는 Perl 5.005(1998)에서 추가됩니다. 이 문서의 핵심 도구가 태어난 시점입니다.
10. **1997** — Philip Hazel이 Exim 메일 서버를 위해 PCRE를 공개합니다. 이후 PHP, nginx, `grep -P` 등 도처에 이식됩니다.
11. **1999** — ECMA-262 3판이 JavaScript RegExp를 표준화합니다(lookahead 포함, lookbehind는 ES2018에서). VS Code 편집기 찾기가 쓰는 바로 그 문법입니다.
12. **2007** — Russ Cox가 "Regular Expression Matching Can Be Simple And Fast" 연재로 Thompson NFA를 재조명하며, 백트래킹 엔진의 지수 폭발을 실측으로 고발합니다.
13. **2010** — Google이 RE2를 공개합니다. 역참조·lookaround를 포기하는 대신 선형 시간을 보장합니다.
14. **2016/2017** — Andrew Gallant(BurntSushi)가 오토마타 기반 Rust regex 위에 ripgrep을 공개하고(2016), VS Code 1.11이 파일 검색을 ripgrep으로 교체합니다(2017).
15. **2016-07-20** — Stack Overflow가 34분간 전역 장애를 겪습니다. 공백 약 2만 개가 연속된 게시물에 공백 트리밍 정규식이 걸려 백트래킹이 곱으로 불어난 것이 원인이었습니다.
16. **2019-07-02** — Cloudflare가 약 27분간 전역 장애를 겪습니다. WAF 규칙 속 `.*(?:.*=.*)` 패턴의 catastrophic backtracking이 전 서버 CPU를 소모했습니다. 두 사건 모두 백트래킹 계보의 최악 시간이 실서비스에서 터진 사례입니다.

### 세 번의 설계 결정

**Thompson, 1968.** 풀려던 문제는 느린 기계(IBM 7094)에서 편집기 검색을 즉답 속도로 만드는 것이었습니다. "갈래를 하나씩 시도하고 실패하면 되돌아간다"는 소박한 탐색을 버리고, 가능한 모든 NFA 상태를 글자당 한 번에 전진시키는 길(너비 우선)을 택했습니다. 입력을 정확히 한 번만 훑으므로 시간은 입력×패턴 크기에 비례하고, 동시 상태 집합의 크기가 패턴 길이로 유계라 메모리도 예측 가능합니다. 정당화는 Kleene과 Rabin-Scott의 등가성 이론이었습니다.

**Spencer → Wall → Hazel, 1986~1997.** 프로그래머들이 원한 것은 이론적 순수성이 아니라 표현력(캡처, 역참조, lookaround)이었습니다. 이 계보는 선형 시간 보장이라는 가정을 버리고 재귀 백트래킹을 택했습니다 — 구현이 단순하고 새 기능이 합성하기 쉬웠기 때문입니다. 부작용(최악 지수 시간)은 수십 년 잠복하다 2016년 Stack Overflow와 2019년 Cloudflare에서 폭발했습니다. 당시의 정당화는 "일상 입력에서는 충분히 빠르다"였고, 적대적 입력이 드물던 시대에는 합리적인 베팅이었습니다.

**Cox → RE2, 2007~2010.** Google Code Search는 신뢰할 수 없는 사용자 입력(임의 정규식)을 받아야 했습니다. RE2는 "역참조와 lookaround가 필수"라는 가정을 버리고 Thompson NFA + 필요할 때만 구축하는(lazy) DFA로 돌아갔습니다. 기능 제한을 명시적 계약으로 문서화했고, 그 대가로 입력이 무엇이든 선형 시간을 증명할 수 있게 됐습니다.

## 5. 엔진의 보행 모델

백트래킹 엔진의 전역 치환(replace all)은 다음 뼈대로 움직입니다. 이 의사코드가 이후 모든 trace의 기준입니다.

```
위치 = 0
결과 = ""
while 위치 <= 입력 길이:
    매치 = 없음
    for 갈래 in 패턴의 갈래들(왼쪽 우선):        # (P) 먼저, 그다음 쓰레기 갈래
        매치 = 그 갈래를 현재 위치에서 시도
        if 매치: break
    if 매치:
        결과 += 치환식 적용(매치)                # $1 = 이 매치의 캡처 그룹 1
        위치 = 매치의 끝                          # 매치 끝에서 재개
    else:
        결과 += 입력[위치]                        # bump-along: 한 글자를 그대로 통과
        위치 += 1
```

(길이 0 매치의 무한 루프 방지 같은 세부는 생략했습니다. 이 문서의 패턴은 두 갈래 모두 최소 1글자를 소비하므로 해당하지 않습니다.)

여기서 두 가지가 모든 것을 결정합니다.

- **갈래는 왼쪽 우선입니다.** `A|B`는 매 위치에서 A를 먼저 시도하고, A가 실패해야 B로 넘어갑니다.
- **bump-along이 함정입니다.** 어떤 갈래도 매치되지 않는 위치를 만나면, 엔진은 그 글자를 결과에 **그대로 흘려보내고** 한 칸 전진합니다. 치환되지 않고 살아남는 글자는 전부 여기서 나옵니다. 8절의 기현상이 정확히 이 줄에서 발생합니다.

## 6. lookahead — 폭 0의 문지기

찾기 패턴의 전체 구조를 토막 내면 다음과 같습니다.

```
( P ) | (?: (?!P) . )+
  |         |     |
  |         |     +-- 실제 소비 담당: '.' 이 한 글자를 먹습니다
  |         +-- 문지기 (폭 0): "이 자리에서 P가 시작되나?" 검사만 합니다
  +-- 갈래1: P를 통째로 소비하고 $1 에 캡처합니다
```

`(?!P)`는 폭이 0인 단언(assertion)입니다. 현재 위치에서 P 매칭을 **시도해 보고**, 성공하면 단언이 실패, 실패하면 단언이 성공합니다. 어느 쪽이든 글자는 한 글자도 소비하지 않습니다. 클럽 입구에서 자격만 확인하고 입장은 시키지 않는 문지기에 비유할 수 있습니다 — 단, 이 비유는 두 군데서 깨집니다. 문지기는 한눈에 훑고 끝나는 게 아니라 P 매칭 전체를 실제로 시도해 보므로 비용이 0이 아니고, 한 번 통과시킨 손님을 기억하지 못해 모든 글자마다 검사를 처음부터 다시 합니다.

갈래2 `(?:(?!P).)+`의 내부는 "검사 → 한 글자 소비"의 루프입니다.

```
[ (?!P) 검사 ] --통과--> [ '.' 이 1글자 소비 ] --> 다시 [ (?!P) 검사 ] --> ...
      |
    실패 (= 바로 여기서 P가 시작된다는 뜻)
      |
      v
  루프 종료: 쓰레기 갈래가 P 직전 경계에서 정확히 멈춥니다
```

그냥 `.+`였다면 P까지 통째로 삼켰을 것을, lookahead가 매 글자 직전에 "여기서 P가 시작되면 멈춰"라고 제동을 거는 구조입니다. 이 관용구(tempered greedy token)가 "P가 아닌 구간 전체"를 한 덩어리 매치로 만들어 줍니다 — 구간당 한 번만 치환식이 적용되게 하려면 이 덩어리화가 필수입니다(10절 방법 1과의 차이).

## 7. 전체 trace — 정상 동작

입력 `ab title="L1PVA(1.2.3.4)" cd` 위에서 5절의 보행 모델을 그대로 굴립니다.

### 매치 #1 — 앞쪽 쓰레기 구간

```
ab title="L1PVA(1.2.3.4)" cd
^
위치 0 'a' : 갈래1 P    -> 'a' != 't' 이므로 실패
             갈래2 (?!P) -> 여기서 P 시작 안 함 -> 통과 -> '.' 이 'a' 소비
```

갈래2의 루프가 글자 단위로 전진합니다.

```
ab title="L1PVA(1.2.3.4)" cd
^^^
||+-- 위치 2 ' ' : (?!P) 통과 -> 소비
|+--- 위치 1 'b' : (?!P) 통과 -> 소비
+---- 위치 0 'a' : (?!P) 통과 -> 소비

ab title="L1PVA(1.2.3.4)" cd
   ^
   위치 3 : 여기서부터 P가 매치됨 -> (?!P) 실패 -> 루프 정지

==> 매치 #1 = "ab "   (갈래2 승)   $1 = ""
```

쓰레기 갈래는 P를 한 글자도 침범하지 않고 P 직전 경계에서 멈췄습니다.

### 매치 #2 — 서버 정보 본체

매치 #1이 끝난 위치 3에서 다음 매치를 시도합니다.

```
ab title="L1PVA(1.2.3.4)" cd
   ^
   위치 3 : 갈래1 P 시도 -> title="L1PVA(1.2.3.4)" 통째로 매치 성공

ab title="L1PVA(1.2.3.4)" cd
   [======== P =========]
==> 매치 #2 = P 전체   (갈래1 승)   $1 = title="L1PVA(1.2.3.4)"
```

여기가 두 갈래 교대 구조의 존재 이유입니다. 쓰레기 갈래가 멈춘 바로 그 자리에서 갈래1이 **출구**를 제공해 P를 통째로 소비합니다.

### 매치 #3 — 뒤쪽 쓰레기 구간

```
ab title="L1PVA(1.2.3.4)" cd
                         ^
위치 25 ' ' : 갈래1 실패 -> 갈래2 가동 -> ' ', 'c', 'd' 소비 -> 문자열 끝에서 정지

==> 매치 #3 = " cd"   (갈래2 승)   $1 = ""
```

### 치환 조립 — `$1\n`이 만들어내는 것

세 매치가 문자열 전체를 빈틈없이 덮었고, 각 매치가 치환식 `$1\n`으로 교체됩니다.

```
       매치        갈래    $1                          치환 결과
 #1    "ab "        2      ""                          "\n"
 #2    title="…"    1      title="L1PVA(1.2.3.4)"      title="L1PVA(1.2.3.4)"\n
 #3    " cd"        2      ""                          "\n"

최종 문서:
  (빈 줄)                      <- #1 의 흔적
  title="L1PVA(1.2.3.4)"
  (빈 줄)                      <- #3 의 흔적
```

같은 치환식 하나인데 갈래1이 이기면 `$1`이 차 있고 갈래2가 이기면 비어 있으므로, 결과적으로 조건 분기가 됩니다. 빈 줄은 쓰레기 구간 하나당 하나씩 남는 흔적이고, 2차 패스 `^\n` → 빈 값으로 청소합니다. P가 두 개인 입력에 대한 실측은 14절 관측 [1]·[2]에 있습니다.

## 8. 반례 trace — naive 접근이 부서지는 지점

### 반례 A — 쓰레기 갈래 단독이면 P의 첫 글자만 살아남는다

"P가 아닌 구간을 매치하는 패턴을 만들었으니, 그것만 단독으로 찾아 빈 값으로 치환하면 되지 않나"가 자연스러운 첫 시도입니다. 패턴 `(?:(?!P).)+`만 단독으로 굴려 보면 이렇게 됩니다.

```
ab title="L1PVA(1.2.3.4)" cd
   ^
위치 3 : (?!P) 가 첫 반복부터 실패 -> 이 위치에서는 어떤 매치도 성립 불가

+------------------------------------------------------------+
| 엔진의 기본 규칙 (bump-along):                              |
| 어떤 위치에서 매치가 안 되면, 그 글자를 결과에 그대로       |
| 흘려보내고 한 칸 전진해 재시도한다.                         |
| 흘려보낸 글자는 어떤 매치에도 속하지 않음 = 치환되지 않음   |
+------------------------------------------------------------+

ab title="L1PVA(1.2.3.4)" cd
    ^
위치 4 'i' : "itle=..." 에서는 P가 시작되지 않음 -> (?!P) 통과(!)
             쓰레기 갈래가 P의 나머지 전부를 삼키기 시작
```

결과를 한 줄로 겹쳐 보면 다음과 같습니다.

```
ab title="L1PVA(1.2.3.4)" cd
xxx^xxxxxxxxxxxxxxxxxxxxxxxx
   |
   x = 쓰레기로 매치되어 삭제된 글자
   ^ = 매치 불가로 건너뛴 글자 -> 생존

빈 값 치환 결과: "t"
```

P 직전까지는 잘 멈춰 놓고, 정작 P 시작 위치에서는 갈 곳이 없어 한 칸 밀린 뒤 P의 몸통을 쓰레기로 먹어 버리는 구조입니다. P가 등장할 때마다 첫 글자 하나씩 살아남으므로, P 두 개짜리 입력에 대한 실측 결과는 `"tt"`입니다(14절 관측 [3]).

### 반례 B — lookahead를 축약하면 외톨이 글자가 샌다

"lookahead는 P의 시작만 알아채면 되니 `(?!title=")`로 줄여도 되겠지"가 두 번째로 흔한 시도입니다. 이렇게 줄이면 IP가 없는 일반 속성 `title="noip"` 앞에서 쓰레기 갈래가 멈추는데, 갈래1(IP 필수)은 거기서 매치되지 않으므로 반례 A의 bump-along이 그대로 재현됩니다. 실측(14절 관측 [4])에서 출력 한가운데에 외톨이 `t` 줄이 생기는 것을 확인할 수 있습니다.

```
... title="L1PVB(5.6.7.8)"   <- 정상 추출
t                            <- title="noip" 의 잔해
...
```

교훈은 하나입니다. **갈래1의 P와 lookahead 안의 P는 한 글자도 다르지 않게 유지합니다.** 두 P가 다르면 "갈래1도 실패하고 (?!P)도 실패하는" 제3의 위치가 생기고, 그 자리마다 글자가 샙니다.

## 9. 왜 이 콤보는 무손실인가

`(?!P)`의 실패 조건과 갈래1 `P`의 성공 조건이 **정확히 같은 사건**이기 때문입니다.

```
          문자열의 임의 위치 i
                  |
        P 가 위치 i 에서 매치되는가?
           |              |
          YES             NO
           |              |
           v              v
    갈래1 이 P 통째    (?!P) 통과가 논리적으로 보장됨
    소비, $1 = P      -> 갈래2 가 최소 1글자 소비, $1 = ""
           |              |
           +------+-------+
                  |
                  v
        모든 위치에서 반드시 둘 중 하나가 성립
        -> bump-along 이 일어날 자리가 없음
        -> 건너뛰는 글자 없음 = 글자 유실/잔류 없음
```

두 갈래가 모든 위치를 남김없이 양분하므로, 5절 보행 모델의 `else` 가지(bump-along)가 실행될 일이 없습니다. 8절의 두 반례는 모두 이 양분을 깨뜨려서(갈래1 부재, 두 P 불일치) 제3의 위치를 만든 경우입니다.

## 10. 같은 일을 하는 네 가지 방법

"원하는 패턴만 남기기"는 본 레시피 말고도 여러 방법으로 같은 결과를 냅니다. 우열이 아니라 트레이드오프의 문제이므로 나란히 둡니다. (1)~(3)의 실측 출력은 14절 관측 [5]·[1]·[6]입니다.

```
(1) 글자 단위 교대:      (P)|.                    치환 $1
(2) 구간 단위 교대(본문): (P)|(?:(?!P).)+          치환 $1\n
(3) 게으른 선두:          .*?(P)                   치환 $1\n
(4) 멀티 커서 추출:       Ctrl+F 에 P 만 -> Alt+Enter -> 복사 -> 새 파일에 붙여넣기
```

- (1)은 lookahead가 아예 없어 가장 단순하고, 삭제(빈 값 이어 붙이기) 용도로는 본 레시피와 결과가 똑같습니다. 약점은 쓰레기 **글자마다** 별개 매치라는 점입니다. 치환식을 `$1\n`으로 바꾸는 순간 쓰레기 글자 수만큼 줄바꿈이 폭발합니다(관측 [5]). "구간당 한 번"의 치환이 필요하면 (2)의 덩어리화가 필수입니다.
- (2)는 쓰레기 구간당 매치가 하나라 치환식이 구간 단위로 적용되고, 그 대신 글자마다 lookahead 비용을 냅니다(13절에서 실측으로 무해함을 확인).
- (3)은 `.*?`(비탐욕)가 다음 P 직전까지 최소한으로 먹는 방식이라 lookahead가 필요 없고 빠릅니다. 대신 마지막 P **뒤의** 꼬리 쓰레기는 어떤 매치에도 안 걸려 그대로 남으므로(관측 [6]에서 `" ef"` 잔존), 꼬리 청소 2차 패스가 따로 필요합니다.
- (4)는 정규식 치환이 아니라 편집기 기능입니다. P만 찾고 `Alt+Enter`로 모든 매치에 멀티 커서를 건 뒤 복사하면, 새 문서에 매치들이 줄 단위로 붙습니다. 원본이 보존되므로 실무에서 가장 안전하지만, 일회성 추출이라 자동화·반복 작업에는 (1)~(3)이 맞습니다.

## 11. 혼동하기 쉬운 것

### `[^x]` vs `(?!P)` — 두 가지 부정

| 축 | 문자 클래스 부정 `[^"]` | 부정 lookahead `(?!P)` |
|---|---|---|
| 부정의 단위 | 글자 1개 | 이 위치에서 시작하는 임의 길이 패턴 |
| 소비 | 글자 1개를 소비 | 0글자 (검사만) |
| 단독 사용 | 가능 — 그 자체로 글자 하나를 매치 | 불가능에 가까움 — 소비할 짝(`.` 등)과 결합해야 의미 |
| 다중 글자 제외 | 불가 — `[^ab]`는 "a도 b도 아닌 한 글자"일 뿐 | `(?!title=").` 처럼 시퀀스 단위 제외 가능 |
| 전형적 용도 | 따옴표 안쪽 내용 `"[^"]*"` | tempered greedy token, 경계 조건 합성 |

### lookahead vs lookbehind

- `(?=P)` / `(?!P)` — 현재 위치의 **오른쪽**이 P인지/아닌지 검사합니다.
- `(?<=P)` / `(?<!P)` — 현재 위치의 **왼쪽**이 P인지/아닌지 검사합니다.
- 네 가지 모두 폭 0입니다. JavaScript는 lookbehind를 ES2018부터 지원하므로 VS Code 편집기 찾기에서 모두 사용할 수 있습니다.

### VS Code 안의 두 엔진 — 같은 앱, 다른 계보

| 축 | 편집기 찾기/바꾸기 (Ctrl+F/H) | 파일 검색 패널 (Ctrl+Shift+F) |
|---|---|---|
| 엔진 | JavaScript RegExp — 백트래킹 계보 | ripgrep(Rust regex) — 오토마타 계보 |
| lookaround·역참조 | 지원 | 기본 엔진은 미지원 → 자동으로 PCRE2(백트래킹)로 폴백 |
| 시간 보장 | 최악 지수 (12절) | 기본 엔진은 선형 보장, PCRE2 폴백 시 보장 소멸 |

1차 출처 인용 두 건. VS Code 1.11 릴리스 노트(2017): "Search is now powered by the excellent tool ripgrep, from Andrew Gallant". VS Code 소스의 `search.usePCRE2` 설정 폐기 메시지: "Deprecated. PCRE2 will be used automatically when using regex features that are only supported by PCRE2." — 즉 본 레시피처럼 lookahead가 들어간 패턴을 검색 패널에 넣으면, 사용자가 모르는 사이 백트래킹 엔진으로 갈아탑니다.

## 12. 두 엔진 계보 비교 — 백트래킹 vs 오토마타

| 축 | 백트래킹 계보 | 오토마타 계보 |
|---|---|---|
| 대표 구현 | Perl, PCRE2, JavaScript(V8 Irregexp), Java, Python re | POSIX grep/awk, RE2, Go regexp, Rust regex(ripgrep) |
| 매칭 전략 | 갈래를 하나씩 시도, 실패하면 직전 선택 지점으로 되돌아옴 | 가능한 모든 상태를 글자당 한 번에 동시 전진 |
| 시간 보장 | 최악 O(2^n) — 입력에 따라 폭발 | O(입력×패턴) 선형 보장 |
| 역참조 `\1` | 지원 | 미지원 — 정규언어의 표현력을 벗어나므로 원리상 불가 |
| lookaround | 지원 | RE2·Rust regex 미지원 (이론상 정규성은 유지되나 구현 비용으로 배제) |
| 매치 의미론 | leftmost-first — 갈래 순서에 의존 | POSIX 도구는 leftmost-longest, RE2·Rust는 Perl 호환 leftmost-first |
| catastrophic backtracking | 있음 (아래 실측) | 없음 |
| 신뢰 불가 입력(ReDoS) | 위험 — 입력 검증 필수 | 안전 — RE2의 존재 이유 |
| 이 문서와의 접점 | VS Code 편집기 찾기/바꾸기 = 본 레시피의 실행 환경 | VS Code 검색 패널 (lookaround 시 PCRE2로 이탈) |

### catastrophic backtracking 실측

중첩 수량자 `(a+)+$`에 매치가 **안 되는** 입력(`aaa...a!`)을 주면, 엔진이 a 묶음을 나누는 모든 경우의 수를 되짚느라 글자 하나가 늘 때마다 시간이 두 배로 뜁니다. 실측(Node v24.15.0, 워밍업 후):

| a의 개수 n | 시간 |
|---|---|
| 20 | 14.3 ms |
| 22 | 59.2 ms |
| 24 | 232.2 ms |
| 26 | 918.4 ms |
| 28 | 3,764.9 ms |

n이 2 늘 때마다 약 4배 — O(2^n)의 전형입니다. 28글자에 3.7초이니, Stack Overflow 사건의 "공백 2만 개"가 어떤 의미였는지 짐작할 수 있습니다(그 사건의 패턴은 지수형이 아닌 곱셈형 폭발이었는데도 서비스가 멈췄습니다).

한편 본 레시피의 lookahead는 중첩 수량자가 아니라서 이 폭발과 무관합니다. 글자마다 "P 시도 1회"가 더해질 뿐이고, P가 리터럴 `title="`로 시작해 대부분 첫 글자 비교에서 실패하므로 사실상 선형입니다. 실측: 212,530자 한 줄에서 서버 정보 50건 추출에 **0.6ms**(관측 [7]).

## 13. 한계와 실패 모드

1. **두 P 동일성 유지 부담.** 갈래1과 lookahead 안에 같은 패턴을 두 번 적어야 하므로 찾기 문자열이 길고, 한쪽만 고치는 실수에 취약합니다(8절 반례 B). 패턴을 수정할 일이 있으면 두 자리를 같이 고치는 것을 규칙으로 삼습니다.
2. **빈 줄 흔적.** 치환식이 갈래 구분 없이 일률 적용되므로 쓰레기 구간마다 `\n` 하나가 남습니다. `^\n` 2차 패스가 표준 마무리입니다.
3. **`.`은 줄바꿈을 매치하지 않습니다.** JavaScript에는 VS Code 찾기 위젯에서 쓸 수 있는 dotall 플래그가 없으므로, 여러 줄 파일에 적용할 때는 쓰레기 갈래의 `.`을 `[\s\S]`로 바꿉니다: `(P)|(?:(?!P)[\s\S])+`
4. **lookahead 비용의 이론적 상한.** 글자마다 P 매칭을 시도하므로 최악 O(입력 × P 실패 비용)입니다. `title="` 같은 접두가 대량으로 깔려 있고 `[^"]*`가 길게 달리는 병리적 입력에서는 느려질 수 있습니다(추론 — 실측은 정상 입력 기준 0.6ms/212k자). 일상 규모에서는 체감 불가입니다.
5. **역참조가 필요한 P는 오토마타 엔진으로 옮길 수 없습니다.** 역참조는 정규언어 밖이라 ripgrep 기본 엔진·RE2에서 원리적으로 지원이 불가능하고, PCRE2 폴백(=백트래킹)으로만 동작합니다. 폴백되는 순간 선형 시간 보장도 사라집니다.
6. **leftmost-first 의미론 의존.** 갈래1을 앞에 둔 것은 관례이고, 이 패턴에서는 갈래2가 P 시작 위치에서 아예 매치될 수 없으므로 순서를 바꿔도 결과가 같습니다. 다만 일반적으로 `A|B`는 순서 의존이므로, 두 갈래가 같은 위치에서 모두 매치 가능한 패턴을 설계할 때는 순서가 결과를 바꿉니다.

## 14. 직접 확인해 보기

아래 스크립트 하나가 본문의 모든 주장을 재현합니다. VS Code 찾기/바꾸기와 같은 JavaScript 엔진(Node)에서 실행합니다.

```js
// 검증 스크립트 — regex/regex.md 의 모든 관측값을 재현합니다.
const P = 'title="[^"]*\\(\\d{1,3}(?:\\.\\d{1,3}){3}\\)"';
const input = 'ab title="L1PVA(1.2.3.4)" cd title="L1PVB(5.6.7.8)" ef';

// [1] 본 레시피: (P)|(?:(?!P).)+  +  $1\n
const reCombo = new RegExp(`(${P})|(?:(?!${P}).)+`, 'g');
console.log('[1] combo  :', JSON.stringify(input.replace(reCombo, '$1\n')));

// [2] 빈 줄 청소 2차 패스: ^\n -> (빈 값)
console.log('[2] cleanup:', JSON.stringify(input.replace(reCombo, '$1\n').replace(/^\n/gm, '')));

// [3] 반례 A: 쓰레기 갈래 단독 -> 각 P 의 첫 글자만 생존
const reEater = new RegExp(`(?:(?!${P}).)+`, 'g');
console.log('[3] eater  :', JSON.stringify(input.replace(reEater, '')));

// [4] 반례 B: lookahead 를 (?!title=") 로 축약 -> IP 없는 title 에서 외톨이 글자
const input2 = input + ' title="noip" tail';
const reWrong = new RegExp(`(${P})|(?:(?!title=").)+`, 'g');
console.log('[4] abbrev :', JSON.stringify(input2.replace(reWrong, '$1\n')));

// [5] 등가 구현 (1): (P)|.
const reDot = new RegExp(`(${P})|.`, 'g');
console.log('[5] dot+$1 :', JSON.stringify(input.replace(reDot, '$1')));
console.log('[5] dot+nl :', JSON.stringify(input.replace(reDot, '$1\n')));

// [6] 등가 구현 (3): .*?(P) -> 꼬리 쓰레기 잔존
const reLazy = new RegExp(`.*?(${P})`, 'g');
console.log('[6] lazy   :', JSON.stringify(input.replace(reLazy, '$1\n')));

// [7] 규모: 약 21만 자 한 줄, P 50개 산재
const junk = 'x<a href="#" class="q">'.repeat(180);
let big = '';
for (let i = 0; i < 50; i++) big += junk + `title="L1PVN${String(i).padStart(2, '0')}(10.11.22.${i})"`;
big += junk;
const t0 = process.hrtime.bigint();
const out = big.replace(reCombo, '$1\n');
const t1 = process.hrtime.bigint();
console.log(`[7] ${big.length}자 한 줄: ${(Number(t1 - t0) / 1e6).toFixed(1)} ms, 추출 ${(out.match(/title=/g) || []).length}건`);

// [8] catastrophic backtracking 고전: (a+)+$ (워밍업 후 측정)
/(a+)+$/.test('aaa!');
for (const n of [20, 22, 24, 26, 28]) {
  const s = 'a'.repeat(n) + '!';
  const s0 = process.hrtime.bigint();
  /(a+)+$/.test(s);
  const s1 = process.hrtime.bigint();
  console.log(`[8] (a+)+$ vs 'a'x${n}+'!': ${(Number(s1 - s0) / 1e6).toFixed(1)} ms`);
}
```

PASS/FAIL 기준:

- [1] 두 서버 정보가 각자 `\n`으로 닫히고 쓰레기 구간 자리마다 `\n` 하나 — 다르면 갈래 구조가 깨진 것
- [2] 빈 줄 없이 두 줄 — `^\n` 청소가 동작한다는 뜻
- [3] 정확히 `"tt"` (P 개수만큼의 t) — 다른 글자가 남으면 bump-along 이해가 틀린 것
- [4] 출력 꼬리에 외톨이 `"t"` 줄 — 두 P 불일치의 증거가 보여야 정상
- [5] `$1`은 이어 붙고, `$1\n`은 쓰레기 글자 수만큼 `\n` 폭발
- [6] 꼬리 `" ef"` 잔존
- [7] 1ms 안팎(수십 ms 이내), 추출 50건 — 50건이 아니면 P가 잘못 세어진 것
- [8] n이 2 늘 때마다 약 4배 — 비율이 평평하면 엔진이 백트래킹이 아니라는 뜻

타이밍 수치는 실행마다 ±수십 % 변동하며, 비율(4배 추세)이 검증 대상입니다.

표현력 이론(촘스키 위계·유한 오토마타·"정규식은 어디까지 표현하나"·HTML 파싱 불가 정리)은 [regex/formal_languages.md](formal_languages.md)로 분리해 적재했습니다 — 아래 후보 중 펌핑 보조정리·부분집합 구성·정규 언어 닫힘 성질은 그쪽에서 다룹니다. 이 문서(엔진 동작론)와 그 문서(표현력 이론)가 정규식 학습의 두 축이고 상호 링크됩니다.

엔진 구현론 쪽으로 아직 열려 있는 후보:

- Thompson construction — 정규식을 NFA로 바꾸는 기계적 규칙 (1968 논문의 핵심)
- lazy DFA — RE2가 상태 폭발을 피하면서 DFA 속도를 내는 방법
- flavor 비교 — BRE/ERE/PCRE/JavaScript/Vim 문법 차이
- 유니코드 모드 — JavaScript `u`/`v` 플래그, `\p{...}` 속성 클래스
- lookbehind의 구현 제약 — 엔진별 고정폭/가변폭 지원 차이
- ReDoS 정적 탐지 — 취약 패턴을 코드 리뷰 단계에서 걸러내는 도구들
