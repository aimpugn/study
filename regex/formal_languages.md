# 정규식은 어디까지 할 수 있나 — 촘스키 위계, 유한 오토마타, 그리고 "HTML은 정규식으로 파싱 못 한다"

"HTML을 정규식으로 파싱하려 하지 마라"는 말은 단순한 실무 경고가 아니라 증명된 정리입니다. 그 증명은 1956년 촘스키(Noam Chomsky)가 세운 언어 위계(Chomsky hierarchy)에서 나옵니다 — 정규식이 표현하는 언어(정규 언어, Type 3)는 위계의 맨 아래 칸이고, HTML의 임의 깊이 중첩은 그 한 칸 위(문맥 자유, Type 2)의 능력을 요구하기 때문입니다.

그런데 여기엔 거의 항상 빠지는 반전이 있습니다. 이 정리는 **형식적(Type 3) 정규식**에 관한 것이고, 오늘 우리가 쓰는 PCRE·.NET·Perl 같은 엔진의 "정규식"은 역참조·재귀·균형 그룹으로 그 경계를 넘습니다. 즉 ".NET 정규식으로 균형 괄호를 매치할 수 있는가"의 답은 "그렇다"이고, 그럼에도 "실제 HTML을 정규식으로 파싱해야 하는가"의 답은 여전히 "아니다"입니다. 이 문서는 이 세 층 — 이론적 불가, 실무 엔진의 초과 능력, 그럼에도 잘못된 도구 — 을 분리해서 봅니다.

이 문서는 [regex/regex.md](regex.md)의 §15에서 예고한 이론 편이며, 그쪽의 엔진 동작론(백트래킹 보행 모델, tempered greedy token, catastrophic backtracking)과 짝을 이룹니다. 역사·엔진 계보가 겹치는 부분은 그쪽을 가리킵니다. 본문의 코드 실증값은 모두 "직접 확인해 보기" 절에서 재현했습니다(2026-06-12 실행, 환경은 해당 절에 명시).

## 1. 짧게 답하면

정규식(형식적 의미)이 표현하는 언어는 **정규 언어(regular language)**이고, 이는 **유한 오토마타(finite automaton)**가 인식하는 언어와 정확히 같습니다(Kleene 정리). 유한 오토마타는 유한 개의 상태만 갖고 보조 메모리가 없어서, 임의로 깊어지는 중첩의 깊이를 끝까지 셀 수 없습니다.

HTML의 태그 중첩 `<a><b>...</b></a>`은 여는/닫는 태그를 짝지으려면 깊이를 세야 하고, 이는 균형 괄호(Dyck 언어)를 인식하는 것과 같은 문제입니다. 균형 괄호는 정규 언어가 아니라 한 칸 위인 **문맥 자유 언어(context-free language)**이고, 이를 인식하려면 스택을 가진 **푸시다운 오토마타(pushdown automaton)**가 필요합니다. 따라서 형식적 정규식으로는 원리적으로 불가능합니다 — 펌핑 보조정리로 증명됩니다(8절).

반전은 두 가지입니다.

1. **현대 엔진의 "정규식"은 형식적 정규식이 아닙니다.** 역참조는 `(.+)\1`로 제곱 언어 `ww`(문맥 자유조차 아님)를 매치하고, .NET 균형 그룹과 PCRE 재귀는 균형 괄호(문맥 자유)를 실제로 매치합니다. 14절에서 .NET이 균형 괄호를 정확히 판정하는 것을 실행으로 보입니다. "정규식으로 중첩을 매치할 수 있다"는 이 확장 엔진에 대해서는 **참**입니다.
2. **그럼에도 실제 HTML에는 잘못된 도구입니다.** 실세계 HTML은 깨끗한 문맥 자유 문법으로도 기술되지 않습니다. 생략 가능 태그, 임의 순서 속성, 주석, CDATA, `<script>` 원시 텍스트, 오류 복구까지 얽혀 있어, 표준(WHATWG)이 정의하는 것은 문법이 아니라 상태 기반 토크나이저 + 트리 구축 **알고리즘**입니다. 그래서 전용 파서를 씁니다.

세 층을 한 줄로 줄이면: **형식 정규식으로는 못 한다(정리). 확장 엔진으로는 중첩을 매치할 수는 있다(사실). 그래도 HTML 파싱엔 쓰면 안 된다(실무).**

## 2. 왜 이 질문이 생기나 — 출발점

[regex.md](regex.md)에서 한 줄짜리 문자열에서 `title="..."` 서버 정보를 발라내는 작업을 했습니다. 그건 잘 됐습니다 — 추출 대상이 **중첩되지 않은 평면 패턴**이었기 때문입니다. 하지만 한 걸음 더 나가 "이 HTML에서 가장 바깥 `<div>`의 내용만 통째로 뽑아줘" 같은 요구로 가면, 정규식이 갑자기 무력해집니다. 바깥 `<div>`의 짝이 되는 `</div>`를 찾으려면 그 사이에 등장하는 안쪽 `<div>`/`</div>` 쌍을 세어 깊이 0으로 돌아오는 지점을 찾아야 하는데, 바로 이 "세기"가 정규식의 표현력 밖입니다.

이건 정규식을 잘 쓰고 못 쓰고의 문제가 아니라, **표현력의 경계**의 문제입니다. 경계가 어디인지, 왜 거기인지, 그리고 현대 엔진이 그 경계를 어떻게 넘는지를 알면 "이 작업에 정규식을 써도 되나"를 매번 추측하지 않고 판정할 수 있습니다. 그 지도가 촘스키 위계입니다.

### 촘스키는 왜 이 위계를 만들었나

촘스키의 동기는 컴퓨터가 아니라 **자연어 문법**이었습니다. 1956년 논문 "Three Models for the Description of Language"에서 그는 당시 유력하던 모델 — 단어가 확률적으로 이어진다고 보는 유한 상태 마르코프 과정(finite-state Markov source) — 이 영어 문장 구조를 포착할 수 없음을 논증했습니다. 영어에는 "if ... then ...", "either ... or ..." 처럼 임의로 멀리 떨어진 요소가 짝을 이루고 그 사이에 또 다른 짝이 중첩될 수 있는데(이를 자기 임베딩, self-embedding이라 부릅니다), 유한 상태 기계는 유한한 기억만으로 그 임의 깊이의 짝을 추적할 수 없다는 것이었습니다.

그가 밟은 사고를 풀면 이렇습니다.

1. 인식한 문제는 "유한 상태 모델이 자연어의 중첩 의존을 설명 못 한다"였습니다.
2. 그래서 "언어 = 단일한 한 종류의 기계"라는 가정을 버리고, 제약의 강도에 따라 **여러 등급의 문법**이 있다고 봤습니다.
3. 대안으로는 생성 규칙(production rule)의 형태에 제약을 거는 방식을 택했습니다. 규칙이 단순할수록 약한 기계에, 자유로울수록 강한 기계에 대응합니다.
4. 그렇게 나눈 각 등급이 정확히 어떤 추상 기계와 대응하는지를 1959년 논문에서 형식화했습니다.
5. 정당화는 수학적이었습니다. 각 등급의 언어 집합이 바로 아래 등급의 **진부분집합(proper subset)**임을, 즉 위 칸이 엄밀히 더 강함을 증명했습니다.

정규식이 이 그림에서 맨 아래 칸을 차지하게 된 것은 이 위계의 부산물이었습니다.

## 3. 가장 작은 기계 — 유한 오토마타와 "기억 없음"

추상론으로 가기 전에 손에 잡히는 기계 하나를 봅니다. 이진 문자열에서 **1의 개수가 짝수인지** 판정하는 유한 오토마타(DFA)입니다.

```
          ┌──0──┐              ┌──0──┐
          ▼     │              ▼     │
   ──▶ (( q0 )) ──────1──────▶ ( q1 )
       짝수 상태  ◀─────1──────  홀수 상태
       (수용)

   - q0 (이중 동그라미) = 시작이자 수용 상태(accept)
   - 1을 볼 때마다 q0↔q1 토글, 0은 제자리(self-loop)
   - 입력을 다 읽었을 때 q0에 있으면 "1의 개수 짝수" 판정
```

이 기계가 기억하는 것은 "지금까지 본 1의 개수가 짝수냐 홀수냐" 단 한 비트입니다. **1을 몇 개 봤는지는 기억하지 못합니다.** 상태가 두 개뿐이기 때문입니다. 이게 유한 오토마타의 본질이자 한계입니다 — 상태 수가 유한하므로, 구별해야 할 상황의 가짓수가 무한히 늘어나는 문제는 풀 수 없습니다.

이 한계를 정확히 박아두면: 어떤 유한 오토마타든 상태가 $k$개라면, 길이 $k$를 넘는 입력을 읽는 동안 반드시 **같은 상태를 두 번 밟습니다**(비둘기집 원리). 이 사실 하나가 8절 펌핑 보조정리의 씨앗이고, "정규식은 셀 수 없다"의 기계적 근거입니다.

대응 관계를 미리 박아두면: 이 기계가 인식하는 언어 "1이 짝수 개인 이진 문자열"은 정규식 `(0*10*1)*0*`로도 똑같이 쓸 수 있습니다. 같은 언어를 기계로 보면 DFA, 식으로 보면 정규식 — 이 둘이 같다는 것이 6절의 Kleene 정리입니다.

## 4. 촘스키 위계 — 네 칸의 지도

위계는 네 등급으로 이뤄지고, 각 등급은 바로 위 등급의 진부분집합입니다. 안쪽 칸일수록 약한 기계, 바깥 칸일수록 강한 기계에 대응합니다.

```
┌──────────────────────────────────────────────────────────────┐
│ Type 0 — 재귀 열거 가능 (recursively enumerable)               │
│ 기계: 튜링 기계 (Turing machine)  ·  메모리: 무한 테이프       │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ Type 1 — 문맥 의존 (context-sensitive)                   │ │
│  │ 기계: 선형 구속 오토마타 (LBA) · 메모리: 입력 길이 비례  │ │
│  │  ┌──────────────────────────────────────────────────┐   │ │
│  │  │ Type 2 — 문맥 자유 (context-free)                 │   │ │
│  │  │ 기계: 푸시다운 오토마타 (PDA) · 메모리: 스택 1개  │   │ │
│  │  │  ┌───────────────────────────────────────────┐   │   │ │
│  │  │  │ Type 3 — 정규 (regular)                    │   │   │ │
│  │  │  │ 기계: 유한 오토마타 (DFA/NFA)              │   │   │ │
│  │  │  │ 메모리: 없음 (유한 상태뿐)                 │   │   │ │
│  │  │  │ ◀── 형식적 정규식이 사는 칸               │   │   │ │
│  │  │  └───────────────────────────────────────────┘   │   │ │
│  │  │   { aⁿbⁿ }, 균형 괄호 = 여기 (Type2 \ Type3)      │   │ │
│  │  │   ◀── HTML 태그 중첩이 요구하는 능력              │   │ │
│  │  └──────────────────────────────────────────────────┘   │ │
│  │   { aⁿbⁿcⁿ } = 여기 (Type1 \ Type2)                     │ │
│  └─────────────────────────────────────────────────────────┘ │
│   정지 문제(halting) 류 = 여기 (Type0 \ Type1)                │
└──────────────────────────────────────────────────────────────┘
```

핵심은 **메모리 모델의 사다리**입니다. 메모리가 없으면(유한 상태뿐) 정규, 스택 하나가 붙으면 문맥 자유, 입력 길이만큼의 테이프면 문맥 의존, 무한 테이프면 튜링 기계입니다. 그리고 각 칸을 가르는 "이건 못 한다"의 대표 증인이 칸 사이에 적혀 있습니다 — `aⁿbⁿ`은 정규로는 못 하지만 문맥 자유로는 되고(스택으로 a를 쌓았다 b에서 꺼내면 됨), `aⁿbⁿcⁿ`은 문맥 자유로도 안 됩니다(스택 하나로는 두 개의 카운트를 못 맞춤).

각 등급이 "정확히" 그 기계와 대응한다는 것은 양방향입니다. 그 문법이 생성하는 모든 언어를 그 기계가 인식하고, 그 기계가 인식하는 모든 언어를 그 문법이 생성합니다. 이 대응이 위계를 단순한 분류가 아니라 정리로 만듭니다.

## 5. 문법 생성 규칙 해부 — 왜 칸이 갈리나

위계의 칸을 가르는 것은 **생성 규칙(production rule)의 좌변·우변에 거는 제약**입니다. 문법은 4-튜플 $G = (V, \Sigma, R, S)$로 정의됩니다.

```
V       비단말 기호(nonterminal/variable) 집합 — 아직 펼쳐질 "변수". 대문자 A,B
Σ       단말 기호(terminal) 집합 — 최종 출력에 남는 글자. 소문자 a,b
R       생성 규칙(production rule) 집합 — "좌변 → 우변", 좌변을 우변으로 치환
S       시작 기호(start symbol) — V 의 원소, 유도(derivation)의 출발점
α,β,γ   (V ∪ Σ)* 의 임의 문자열 (비단말·단말 섞인 열)
```

규칙 제약을 강한 칸(Type 3)부터 풀면 다음과 같습니다.

**Type 3 (정규).** 우선형(right-linear) 기준으로 `A → aB` 또는 `A → a`(공집합 허용 시 `A → ε`).

```
A → aB    좌변은 비단말 하나, 우변은 "단말 하나 + 비단말 하나"
A → a     또는 "단말 하나"로 종료
```

이 형태가 결정적입니다. 규칙 한 번 적용 = 단말 하나 출력 + 다음 비단말 하나로 이동. 비단말을 "현재 상태", 규칙 적용을 "상태 전이"로 읽으면 이건 정확히 유한 오토마타의 한 걸음입니다. 우변에 비단말이 **최대 하나**라서 "지금 어디에 있나"가 항상 단일 상태로 표현되고, 그래서 보조 메모리가 필요 없습니다. 이것이 Type 3 = 유한 오토마타인 구조적 이유입니다.

**Type 2 (문맥 자유).** `A → γ` — 좌변이 **비단말 하나**, 우변은 임의의 문자열.

```
A → γ     좌변은 반드시 단일 비단말 A. 우변 γ 는 무엇이든.
          예: S → aSb   (우변에 비단말 S 가 다시 등장 — 재귀!)
```

좌변이 비단말 하나라는 제약은 "A는 주변 문맥과 무관하게(context-free) 언제나 γ로 펼칠 수 있다"는 뜻입니다. 결정적 차이는 우변에 비단말이 **여럿 등장하거나 자기 자신이 재등장**할 수 있다는 점입니다. `S → aSb`는 S 안에 S가 다시 들어가므로, 유도하면 `S ⇒ aSb ⇒ aaSbb ⇒ aaabbb...` 처럼 깊이가 자라납니다. 이 재귀를 기계로 구현하려면 "지금 a를 몇 개 펼쳤는지"를 쌓아둘 스택이 필요합니다 — 그래서 Type 2 = 푸시다운 오토마타입니다.

`aⁿbⁿ`을 생성하는 완전한 문맥 자유 문법은 이 두 줄입니다.

```
S → a S b      n 을 하나 늘리며 a 와 b 를 바깥에서 안으로 짝지어 두름
S → ε          바닥(n=0)에서 종료
```

`S → aSb`가 한 번 적용될 때마다 a와 b가 정확히 하나씩, 그것도 **짝으로** 추가됩니다. 개수가 어긋날 도리가 없습니다. 정규식이 흉내 낼 수 없는 게 바로 이 "바깥 a와 안쪽 b를 한 규칙이 동시에 책임지는" 구조입니다 — 정규 문법의 `A → aB`는 a를 내보낸 뒤 미래의 b와의 연결을 기억할 자리가 없습니다.

**Type 1 (문맥 의존).** `αAβ → αγβ`, 단 `γ ≠ ε`(비축약, non-contracting).

```
αAβ → αγβ   A 를 γ 로 바꾸되, 좌우 문맥 α_β 가 있을 때만. γ 는 공백 불가.
            (예외: S → ε 은 S 가 어떤 우변에도 안 나올 때만 허용)
```

A를 펼치는 게 **주변 문맥 α, β에 의존**합니다(context-sensitive). "γ가 공백이 아니다"는 비축약 조건 때문에 유도가 진행될수록 문장이 짧아지지 않고, 그래서 필요한 작업 공간이 입력 길이로 묶입니다(선형 구속). `aⁿbⁿcⁿ`이 이 칸의 대표입니다.

**Type 0 (무제약).** `γ → α`, 좌변에 비단말이 하나라도 있으면 우변은 무엇이든(공백 포함). 제약이 사실상 없고, 튜링 기계와 대응합니다. 정지 문제처럼 "답이 있으면 찾지만 없으면 영원히 도는" 언어가 여기 삽니다.

규칙 제약이 풀릴수록(좌변에 문맥이 붙고, 우변이 자유로워지고, 축약이 허용될수록) 대응 기계의 메모리가 늘어난다 — 이것이 위계를 관통하는 한 줄입니다.

## 6. 정규식 = 유한 오토마타 = 정규 언어 (Kleene 정리)

세 가지가 정확히 같은 언어 집합을 정의합니다. 시점과 무관한 구조적 사실입니다.

```
정규 언어  ⟺  유한 오토마타(DFA/NFA)가 인식하는 언어  ⟺  (형식) 정규식이 나타내는 언어
```

이 등가성이 **Kleene 정리(Kleene's theorem)**입니다. Stephen Kleene이 1951년 RAND 기술보고서 RM-704(출판본은 1956년 *Automata Studies*)에서, 정규식에 해당하는 "정규 사건(regular events)" 개념과 유한 오토마타의 등가를 보였습니다. 반복 연산자 `*`를 Kleene star라 부르는 것이 여기서 나왔습니다.

실무적 함의가 큽니다. 정규식으로 쓸 수 있는 모든 패턴은 어떤 유한 오토마타로 변환되고, 그 역도 성립합니다. 그래서 "이 패턴을 형식 정규식으로 쓸 수 있나"라는 질문은 "이 언어가 유한 오토마타로 인식되나", 다시 "이 언어가 정규 언어인가"와 **같은 질문**입니다. 셋 중 어느 관점에서 따져도 답이 같으므로, 어려운 쪽을 쉬운 쪽으로 바꿔 증명할 수 있습니다. 8절에서 "HTML 중첩은 정규식으로 안 된다"를 "균형 괄호는 정규 언어가 아니다"로 바꿔 증명하는 것이 바로 이 등가성을 쓰는 것입니다.

정규 언어는 합집합·연결·Kleene star뿐 아니라 **여집합(complement)·교집합·반전에도 닫혀 있습니다**. 여집합 닫힘은 구조적으로 자명합니다 — DFA의 수용 상태 집합 $F$를 $Q \setminus F$로 뒤집으면(전이는 그대로) 즉시 여집합의 DFA가 나옵니다. 이론적으로 "부정"은 정규성을 유지합니다.

이 사실은 regex.md §3에서 "정규식에는 P의 여집합을 직접 매치하는 연산자가 없어 lookahead로 합성한다"고 한 것과 묘하게 겹칩니다. 정규 언어가 여집합에 닫혀 있다는 **이론**과, 실무 정규식 문법에 "여집합 연산자"가 **없다**는 것은 별개입니다. 이론은 "여집합 DFA가 존재한다"만 보장하고, 그 DFA를 사람이 읽을 정규식으로 옮기면 감당 못 할 크기로 불어나기 때문에, 실무에서는 lookahead로 우회하는 것입니다. 이론적 닫힘과 실무 표기 가능성을 같은 것으로 착각하면 안 됩니다.

## 7. DFA vs NFA — 같은 능력, 다른 크기

유한 오토마타에는 두 변종이 있습니다. 같은 언어 부류(정규 언어)를 인식하므로 **표현력은 동일**하지만, 상태 수와 구현이 다릅니다.

| 축 | DFA (결정적) | NFA (비결정적) |
|---|---|---|
| 전이 함수 | $\delta: Q \times \Sigma \to Q$ — 다음 상태가 정확히 하나 | $\delta: Q \times \Sigma_\varepsilon \to \mathcal{P}(Q)$ — 다음 상태가 상태 집합(0개 이상) |
| ε-전이 | 없음 | 있음 (입력 소비 없이 상태 이동) |
| 한 입력에서 갈 곳 | 항상 하나 (결정적) | 여럿 가능 ("동시에 여러 갈래") |
| 인식 능력 | 정규 언어 | 정규 언어 (동일) |
| 상태 수 | 최악 NFA의 $2^n$ | 더 작을 수 있음 |
| 실행 모델 | 한 경로만 따라감 | 모든 가능 경로를 동시에(또는 백트래킹) |

둘이 같은 능력임을 보이는 것이 **부분집합 구성(subset/powerset construction)**입니다. NFA가 "동시에 여러 상태에 있을 수 있다"면, 그 "현재 상태들의 집합"을 하나의 DFA 상태로 묶으면 됩니다. NFA 상태가 $n$개면 그 부분집합은 $2^n$개이므로, $n$-상태 NFA는 최대 $2^n$-상태 DFA로 변환됩니다. 이를 Rabin과 Scott이 1959년 논문 "Finite Automata and Their Decision Problems"(IBM J. Res. Dev.)에서 정립했고, 두 사람은 이 업적으로 1976년 튜링상을 공동 수상했습니다.

이 $2^n$이 빈말이 아닌 예: 알파벳 $\{0,1\}$에서 **뒤에서 $k$번째 글자가 1인** 문자열의 언어. NFA는 약 $k+1$개 상태면 됩니다 — "여기가 뒤에서 $k$번째다"라고 비결정적으로 추측한 뒤 나머지 $k-1$글자를 세기만 하면 되니까요. 그러나 DFA는 **$2^k$개 상태**가 필요합니다. 결정적 기계는 추측할 수 없으니, 마지막 $k$글자(가능한 비트열 $2^k$가지)를 전부 구별해 기억해야 합니다. 비결정성이 인식 능력은 안 늘리지만 상태 수는 지수적으로 줄이는 것입니다.

이 대비가 regex.md §12의 엔진 계보와 직접 이어집니다. 백트래킹 엔진(JavaScript·PCRE)은 NFA의 갈래를 하나씩 시도하다 막히면 되돌아가는 방식이라 최악에 지수 시간이 걸리고(catastrophic backtracking), Thompson/RE2 계열은 NFA의 모든 갈래를 글자당 한 번에 전진시켜(부분집합 구성을 실시간으로 하는 셈) 선형 시간을 보장합니다. 같은 NFA를 "한 경로씩 백트래킹"으로 도느냐 "모든 경로 동시 전진"으로 도느냐의 차이가 그 성능 격차의 뿌리입니다.

## 8. 펌핑 보조정리 — 정규 언어가 "셀 수 없음"의 증명

"정규식으로 `aⁿbⁿ`을 못 한다"를 감이 아니라 증명으로 닫는 도구가 **펌핑 보조정리(pumping lemma)**입니다. 3절에서 본 "상태가 유한하면 긴 입력에서 반드시 상태를 두 번 밟는다"를 형식화한 것입니다.

### 정리의 진술

```
L 이 정규 언어이면, L 에만 의존하는 정수 p ≥ 1 (펌핑 길이) 가 존재하여,
|w| ≥ p 인 모든 w ∈ L 은 w = xyz 로 분해되어 다음을 만족한다:
   (1) |y| ≥ 1          y 는 비지 않음 (펌핑할 실체가 있음)
   (2) |xy| ≤ p          x 와 y 는 앞쪽 p 글자 안에 들어감
   (3) 모든 i ≥ 0 에 대해 xyⁱz ∈ L
                         y 를 0번·1번·여러 번 반복해도 여전히 L 의 원소
```

기호를 자연어로 풀면: 정규 언어의 충분히 긴 문자열은 항상 "앞부분 $x$ + 펌핑 가능한 중간 $y$ + 나머지 $z$"로 쪼개지고, 그 중간 $y$를 **몇 번을 반복하든** 결과가 여전히 그 언어에 속합니다. 왜냐하면 $y$는 유한 오토마타가 같은 상태로 돌아오는 루프 구간이고(3절), 루프는 0번이든 100번이든 돌아도 시작·끝 상태가 같으니 수용 여부가 안 바뀌기 때문입니다.

```
   상태를 다시 밟는 그 사이 구간이 y (루프)
        ┌──────── y ────────┐
   ──▶ q_start ─...─▶ q_loop ─...─▶ q_loop ─...─▶ q_accept
        └─ x ─┘     (같은 상태로 복귀)    └──── z ────┘
   y 를 통째로 빼도(i=0), 두 번 돌려도(i=2) q_loop→q_loop 이므로 수용 불변
```

이 정리는 "정규 언어라면 반드시 이 성질을 가진다"는 필요조건입니다. 그래서 **대우**로 씁니다 — 어떤 언어가 이 펌핑 성질을 어기면, 그 언어는 정규가 아닙니다.

### `aⁿbⁿ`은 정규가 아니다 — 단계별 추적

귀류법으로, `L = { aⁿbⁿ : n ≥ 0 }`이 정규라고 가정합니다.

```
1단계. L 이 정규라 가정 → 펌핑 길이 p 가 존재한다.

2단계. w = aᵖbᵖ 를 고른다.  w ∈ L 이고 |w| = 2p ≥ p 이므로 정리 적용 가능.
       w = aaa...aaa bbb...bbb   (a 가 p 개, b 가 p 개)

3단계. 정리가 보장하는 분해 w = xyz 를 받는다. 단 |xy| ≤ p, |y| ≥ 1.

4단계. w 의 앞 p 글자는 전부 a 다. |xy| ≤ p 이므로 xy 는 a 영역 안에만 있다.
       → y 는 a 들로만 구성된다.  y = aᵏ  (k ≥ 1)
       w = [ x ][ y=aᵏ ][ z ]
            └──── 전부 앞쪽 a 구간 ────┘    z = (남은 a들) bᵖ

5단계. 조건 (3)에서 i=2 를 취한다 (y 를 한 번 더 반복):
       xy²z = aᵖ⁺ᵏ bᵖ        a 가 p+k 개, b 는 여전히 p 개
       k ≥ 1 이므로 a 의 수 ≠ b 의 수  →  xy²z ∉ L

6단계. 그런데 (3)은 xy²z ∈ L 이어야 한다고 말한다. 모순.
       → 가정이 거짓. L 은 정규가 아니다. ∎
```

핵심은 4단계입니다. $y$가 앞쪽 $a$ 구간에 갇히는 바람에, $y$를 펌핑하면 $a$만 늘고 $b$는 그대로라 균형이 깨집니다. 정규 언어의 "루프는 한 군데서만 돈다"는 성질이 "두 곳의 개수를 맞춰 세기"와 양립할 수 없는 것입니다. 이게 3절에서 본 "유한 상태 기계는 셀 수 없다"의 정확한 형식화입니다.

`aⁿbⁿ`을 균형 괄호로 바꿔 읽으면(`a`=`(`, `b`=`)`) 그대로 "여는 괄호 $n$개 뒤 닫는 괄호 $n$개"이고, 일반 균형 괄호(Dyck 언어)는 이보다 더 복잡하니 역시 정규가 아닙니다. HTML 태그 중첩이 정확히 이 구조입니다.

### 14절 실증과의 연결

펌핑 보조정리는 증명 도구라 "실행"이 아니라 논증이지만, 그 결론은 코드로 체감됩니다. 순수 정규식 `a*b*`는 `aaabb`(a 3개, b 2개)를 **매치합니다** — 개수를 맞추라고 강제할 방법이 정규식에 없기 때문입니다(14절 관측 [B2]). 정규식은 "a들 다음에 b들"이라는 모양은 강제해도 "같은 개수"는 강제하지 못합니다. 펌핑 보조정리가 증명한 바로 그 무능이 한 줄 실행으로 드러납니다.

## 9. 그래서 HTML — 형식 논증과 bobince

이제 사슬을 잇습니다.

```
HTML 태그의 임의 깊이 중첩  <a><b><c>...</c></b></a>
   = 여는/닫는 태그를 깊이 세어 짝짓기
   = 균형 괄호(Dyck 언어) 인식
   = 펌핑 보조정리로 "정규 언어 아님"이 증명됨 (8절)
   = 유한 오토마타로 인식 불가 (6절, Kleene)
   = 형식적 정규식으로 표현 불가  ∎
```

중요한 정밀화: 문제는 중첩이라는 현상 자체가 아니라 **무한 중첩**입니다. 깊이를 $N$으로 고정하면(예: "최대 3겹까지만") 정규식으로도 됩니다 — 단, 깊이마다 별도 패턴이 필요하고 $N$을 늘릴 때마다 패턴이 커집니다. whatwg 메일링 리스트의 한 정리가 이를 그대로 말합니다: "Regular expressions can parse strings written in a regular language. That means no nested elements. And if you want to do a finite number of nested elements, you have to write one regex for each level." 즉 유한 상태 기계가 셀 수 있는 깊이는 상태 수에 박혀 있고, 임의 깊이는 무한한 상태를 요구합니다.

이 명제의 가장 유명한 출처가 2009년 Stack Overflow 답변입니다. "RegEx match open tags except XHTML self-contained tags"라는 질문(id 1732348)에 사용자 bobince가 단 답변으로, 거의 밈이 됐습니다. 미러 교차검증으로 확인된 핵심 문장들(verbatim):

> "You can't parse [X]HTML with regex. Because HTML can't be parsed by regex."
>
> "Regular expressions are a tool that is insufficiently sophisticated to understand the constructs employed by HTML."
>
> "Even Jon Skeet cannot parse HTML using regular expressions."
>
> "Have you tried using an XML parser instead?"

원문 중간에는 정규식으로 HTML을 파싱하려는 시도가 광기로 치닫는 과정을 손상된 유니코드(Zalgo 텍스트, "He comes" / "the pony he comes")로 묘사한 악명 높은 구절이 있습니다. 그 구절의 글자 단위 표기는 이 환경에서 확정하지 못해 그대로 옮기지 않습니다(원문은 출처의 SO 링크 참조). 답변의 진지한 알맹이는 마지막 줄입니다 — 정규식 대신 파서를 쓰라.

### 실세계 HTML은 문맥 자유 문법으로도 부족하다

여기서 한 단계 더 내려가야 정확합니다. "HTML = 문맥 자유 언어"라고 단정하면 절반만 맞습니다. 이상적인 well-formed XML이라면 깨끗한 문맥 자유에 가깝지만, 실세계 HTML은 그보다 지저분합니다.

- 생략 가능 태그: `<li>` 다음 `<li>`가 오면 앞의 것이 자동으로 닫히고, `</p>`는 생략 가능합니다.
- 임의 순서의 속성, 따옴표 없는 속성값.
- 주석 `<!-- -->`, CDATA, `<script>`/`<style>` 안의 원시 텍스트(그 안에서는 `<`가 태그 시작이 아님).
- 깨진 마크업에 대한 **오류 복구** 규칙.

그래서 표준(WHATWG HTML)이 정의하는 것은 생성 문법이 아니라 **알고리즘**입니다. 스펙 원문: "The input to the HTML parsing process consists of a stream of code points, which is passed through a tokenization stage followed by a tree construction stage." 즉 코드포인트 스트림을 토큰화 단계 → 트리 구축 단계로 흘려보내는, 오류 처리까지 명시된 상태 기반 알고리즘입니다. "HTML = 어떤 CFG" 라는 등식이 표준 차원에서 성립하지 않으므로, 정규식이든 단순 CFG 파서든 실세계 HTML에는 부족하고, 스펙을 구현한 전용 파서(예: WHATWG 알고리즘을 그대로 구현한 parse5)가 정답입니다.

## 10. 결정적 반전 — 실무 정규식은 경계를 넘는다

지금까지의 불가능성은 전부 **형식적(Type 3) 정규식**에 관한 것입니다. 현대 엔진의 "정규식"은 형식 정규식이 아니고, 다음 기능들로 정규성을 초과합니다. 능력의 종류가 둘로 갈립니다 — 하나는 "같은 것을 다시"(역참조), 다른 하나는 "균형 잡힌 중첩"(재귀·균형 그룹).

### 역참조 — 정규성을 넘지만 다른 축으로

역참조 `\1`은 "앞서 매치한 그 문자열과 똑같은 것"을 요구합니다. `(.+)\1`은 `ww` 꼴 — 같은 문자열이 두 번 반복되는 제곱 언어(squares)를 매치하는데, 이 언어는 정규가 아닐 뿐 아니라 **문맥 자유조차 아닙니다**. 14절 실측: `(.+)\1`이 `abab`는 매치(`w=ab`), `abcb`는 불매치.

게다가 역참조를 포함한 정규식의 매칭 판정은 **NP-완전**입니다(Aho, "Algorithms for Finding Patterns in Strings", Handbook of Theoretical Computer Science Vol. A, 1990). 즉 역참조는 정규식을 정규 언어 밖으로 끌어낼 뿐 아니라 계산 복잡도까지 바꿉니다.

다만 역참조의 능력은 위계를 "위로" 한 칸 올라가는 것과는 결이 다릅니다. "같은 부분 문자열의 반복"이라는 직교적 능력이지, "균형 잡힌 임의 깊이 중첩"이 아닙니다. 그래서 역참조만으로 HTML 중첩을 제대로 매치하지는 못합니다 — 14절 관측 [A2]에서 `<(\w+)>.*?</\1>`이 `<a><b>x</a></b>`(교차된 잘못된 중첩)에도 매치되는 것이 그 증거입니다. 역참조는 "여는 태그와 닫는 태그의 이름이 같다"는 것은 보장해도, 중첩이 올바른지(스택처럼 짝이 맞는지)는 추적하지 못합니다.

### 균형 그룹·재귀 — 진짜로 문맥 자유를 매치

스택을 흉내 내는 기능이 따로 있습니다.

- **.NET 균형 그룹(balancing group)** `(?<name1-name2>...)`: 매치한 것을 스택처럼 쌓고(여는 괄호에서 push) 꺼냅니다(닫는 괄호에서 pop). Microsoft 문서: "The balancing group definition uses *name2* as a stack. The beginning character of each nested construct is placed in the group ... When the closing character is matched, its corresponding opening character is removed."
- **Perl/PCRE 재귀** `(?R)`, `(?1)`: 패턴이 자기 자신을(또는 특정 그룹을) 재귀 호출합니다. perlre: "`(?0)` ... Recurse to the beginning of the whole pattern."

이 기능들은 푸시다운 오토마타의 스택을 정규식 문법 안으로 끌어들인 것이라, 균형 괄호(문맥 자유 언어)를 실제로 매치합니다. 14절에서 .NET 균형 그룹이 `(())()`·`a(b(c)d)e`는 매치(True), `(()`·`())`는 불매치(False)로 균형 괄호를 정확히 판정하는 것을 실행으로 확인했습니다. **이 순간 "정규식"은 더 이상 Type 3이 아닙니다.**

세 층을 코드 실측으로 겹쳐 보면(14절):

```
순수 정규식    a*b*            'aaabb' → 매치   (개수 못 셈 = Type 3 한계)
역참조         (.+)\1          'abab'  → 매치   (제곱 ww = 문맥 자유 밖, 다른 축)
균형 그룹      (?<o>\()...     '(())' → 매치, '(()' → 불매치  (균형 괄호 = Type 2)
```

그러므로 "정규식으로 HTML 못 한다"는 명제는 **형식 정규식**에 대해서만 참이고, 확장 엔진에서는 중첩을 매치할 수 있습니다. 많은 엔지니어가 이 둘을 뭉뚱그립니다. 정확히는 "**형식 정규식**으로는 표현력 자체가 불가능하고, **확장 정규식**으로는 가능하지만 실세계 HTML의 지저분함(9절) 때문에 여전히 부적합하다"입니다.

## 11. 혼동하기 쉬운 것

### 형식 정규식 vs 엔진 정규식

| 축 | 형식 정규식 (formal regex) | 엔진 정규식 (PCRE/.NET/Perl 등) |
|---|---|---|
| 표현력 | 정확히 정규 언어 (Type 3) | 역참조·재귀·균형 그룹으로 Type 3 초과 |
| 역참조 | 없음 (정의에 없음) | 있음 — 제곱 언어 `ww` 매치, 매칭 NP-완전 |
| 재귀/균형 그룹 | 없음 | 있음 — 균형 괄호(Type 2) 매치 |
| 시간 보장 | 선형(오토마타로 컴파일 시) | 백트래킹은 최악 지수 (regex.md §12) |
| "HTML 파싱" | 원리적으로 불가 | 일부 가능하나 실세계 HTML엔 부적합 |

"정규식으로 X 못 한다"는 주장을 만나면 항상 어느 쪽 "정규식"인지부터 가려야 합니다.

### 정규 vs 문맥 자유 — 무엇이 경계를 가르나

- 정규(Type 3): 유한 상태만. "지금까지의 요약"을 유한 비트로 압축할 수 있는 언어. 평면 패턴, 고정 깊이.
- 문맥 자유(Type 2): 스택 하나. "임의 깊이로 열고 역순으로 닫기"가 되는 언어. 균형 괄호, 산술식, 태그 중첩.
- 가르는 질문: **임의 깊이로 짝을 맞춰 세야 하는가?** 그렇다면 정규 밖입니다.

### "매치할 수 있다" vs "파싱해야 한다"

표현력의 문제와 도구 적합성의 문제는 별개입니다. 확장 정규식이 균형 괄호를 매치할 수 있다는 것이 "HTML을 정규식으로 파싱하라"는 허가가 아닙니다. 깨진 마크업·오류 복구·주석·인코딩 엔티티 앞에서 정규식은 부서지고, 전용 파서가 그 모든 경우를 표준대로 처리합니다.

### DFA vs NFA

둘은 표현력이 같습니다(7절). "NFA가 더 강하다"는 흔한 오해인데, 비결정성은 인식 능력이 아니라 상태 수 효율과 구현 방식만 바꿉니다.

## 12. 네 기계의 비교 — 위계 종합표

촘스키 위계 자체가 이 문서의 마스터 비교표입니다. 여러 축으로 쪼개면 다음과 같습니다.

| 축 | Type 3 정규 | Type 2 문맥 자유 | Type 1 문맥 의존 | Type 0 무제약 |
|---|---|---|---|---|
| 문법 규칙 좌변 | 비단말 1 | 비단말 1 | 문맥 포함 `αAβ` | 비단말 포함 임의 |
| 문법 규칙 우변 | `aB` 또는 `a` | 임의 문자열 | `αγβ`, γ≠ε | 임의(공백 가능) |
| 대응 기계 | 유한 오토마타 (DFA/NFA) | 푸시다운 오토마타 (PDA) | 선형 구속 오토마타 (LBA) | 튜링 기계 (TM) |
| 메모리 모델 | 없음 (유한 상태) | 스택 1개 | 입력 길이 비례 테이프 | 무한 테이프 |
| 대표 가능 언어 | `a*b*`, 짝수 개수 | `aⁿbⁿ`, 균형 괄호 | `aⁿbⁿcⁿ` | 정지 문제류 |
| 못 하는 대표 | `aⁿbⁿ` (셀 수 없음) | `aⁿbⁿcⁿ` (두 카운트) | 일부 결정 불가 | (결정 불가만 남음) |
| 여집합 닫힘 | 닫힘 | 안 닫힘 | 닫힘 | 안 닫힘 |
| 교집합 닫힘 | 닫힘 | 안 닫힘 | 닫힘 | 닫힘 |
| 멤버십 판정 | 선형 | 다항 (CYK 등) | PSPACE | 결정 불가능 가능 |
| 형식 정규식 | = 이 칸 | 초과 (역참조/재귀로 엔진이 진입) | — | — |
| 이 문서와의 접점 | 정규식이 사는 칸 | HTML 중첩이 요구하는 칸 | `aⁿbⁿcⁿ` 반례 | 이론적 상한 |

### naive vs 올바른 도구 — HTML 처리

같은 목표("HTML에서 데이터 뽑기")의 두 접근을 대비하면 "왜 파서인가"가 드러납니다.

| | naive: 정규식 | 올바름: 전용 파서 |
|---|---|---|
| 평면 추출(중첩 없음) | 잘 됨 (regex.md의 title 추출) | 과함 |
| 임의 깊이 중첩 | 형식 정규식 불가, 확장도 취약 | 스택으로 정확 |
| 깨진 마크업 | 부서짐 | 표준 오류 복구 |
| 주석/CDATA/script | 오탐 | 컨텍스트별 처리 |
| 적정 용도 | 단순·평면·신뢰된 입력 | 실세계 HTML 전반 |

경계는 "중첩을 임의 깊이로 따라가야 하는가"입니다. 아니라면(평면 패턴) 정규식이 가볍고 적절하고 — regex.md의 작업이 그랬습니다 — 맞다면 파서로 넘어갑니다.

## 13. 한계와 실패 모드

1. **펌핑 보조정리는 필요조건일 뿐입니다.** "펌핑 성질을 만족한다"가 "정규다"를 보장하지 않습니다(역은 거짓). 비정규성 증명에는 강력하지만, 정규성 증명에는 더 강한 도구(Myhill–Nerode 정리: 동치류가 유한 개면 정규)가 필요합니다.
2. **"HTML은 문맥 자유"는 근사입니다.** 9절에서 봤듯 실세계 HTML은 깨끗한 CFG로도 부족합니다. 위계는 이상화된 모델이고, 실제 파싱 표준은 그 위에 오류 복구 등을 얹은 알고리즘입니다.
3. **확장 정규식의 표현력 초과는 비용을 동반합니다.** 역참조 매칭은 NP-완전이고, 재귀·균형 그룹은 백트래킹 위에서 동작하므로 catastrophic backtracking(regex.md §12)에 노출됩니다. "할 수 있다"와 "해도 안전하다"는 다릅니다.
4. **고정 깊이 우회는 패턴이 커집니다.** 중첩 깊이를 $N$으로 한정하면 형식 정규식으로도 되지만, $N$마다 패턴이 부풀고 $N+1$겹은 조용히 놓칩니다. "깊이 제한"을 명시하지 않으면 침묵 실패가 됩니다.
5. **위계의 "정확한 대응"은 비결정 기계 기준인 칸이 있습니다.** Type 2는 비결정적 PDA와 대응하고, 결정적 PDA(DPDA)는 더 좁은 부류(결정적 문맥 자유)만 인식합니다. "PDA = 문맥 자유"는 비결정 PDA에 한한 진술입니다.
6. **여집합/교집합 닫힘은 칸마다 다릅니다.** 정규·문맥 의존은 여집합에 닫혀 있지만 문맥 자유는 아닙니다(12절). "상위 칸일수록 모든 게 더 된다"는 직관은 닫힘 성질에서는 어긋납니다.

## 14. 직접 확인해 보기

본문의 표현력 주장을 두 엔진으로 실증합니다. (A) .NET 정규식은 PowerShell에서 네이티브로 돌고, (B) 순수 정규식·역참조는 Python `re`로 돕니다.

### A. .NET 정규식 — 균형 그룹은 문맥 자유를 매치한다 (PowerShell)

```powershell
# 균형 괄호(Dyck 언어, 문맥 자유) — .NET 균형 그룹
$balanced = '^(?:[^()]|(?<o>\()|(?<-o>\)))*(?(o)(?!))$'
foreach ($s in '(())()','()()','a(b(c)d)e','(()','())','(((') {
  "{0,-12} => {1}" -f $s, ([regex]::IsMatch($s, $balanced))
}

# 역참조로 태그 쌍은 매치하나 올바른 중첩은 보장 못 함
$tag = '<(\w+)>.*?</\1>'
foreach ($s in '<a>x</a>','<a><a>x</a></a>','<a><b>x</a></b>') {
  "{0,-22} => {1}" -f $s, ([regex]::IsMatch($s, $tag))
}
```

2026-06-12 실제 출력 (.NET 10.0.0.0, PowerShell 7.6.2, Windows 11):

```
== [A1] 균형 괄호 ==
(())()       => True        # 균형 → 매치
()()         => True
a(b(c)d)e    => True        # 다른 글자 섞여도 균형이면 매치
(()          => False       # 여는 괄호 1개 안 닫힘 → 불매치
())          => False       # 닫는 괄호 초과 → 불매치
(((          => False

== [A2] 역참조 태그 ==
<a>x</a>               => True   # 정상 쌍
<a><a>x</a></a>        => True   # 안쪽 <a>x</a> 매치
<a><b>x</a></b>        => True   # 교차된 잘못된 중첩에도 매치(!) — 역참조는 중첩 검증 못 함
```

PASS/FAIL:
- [A1] 균형 괄호가 True/True/True/False/False/False면 .NET 정규식이 Dyck 언어(문맥 자유)를 정확히 판정한다는 증거 — "정규식은 Type 3에 갇힌다"가 형식 정규식 한정 명제임을 보임.
- [A2] 마지막 `<a><b>x</a></b> => True`는 역참조가 "이름 같은 쌍"은 잡아도 "올바른 중첩"은 못 잡는다는 증거(10절).

### B. 순수 정규식과 역참조 — 셀 수 없음과 제곱 언어 (Python)

```python
import re
# [B1] 역참조: 제곱 언어 ww (문맥 자유 밖)
print('square (.+)\\1 abab:', bool(re.fullmatch(r'(.+)\1', 'abab')))   # True (w=ab)
print('square (.+)\\1 abcb:', bool(re.fullmatch(r'(.+)\1', 'abcb')))   # False
# [B2] 순수 정규식은 개수를 못 센다
print('a*b* on aaabb (a3 b2):', bool(re.fullmatch(r'a*b*', 'aaabb')))  # True(!) — 불일치도 통과
```

2026-06-12 실제 출력 (Python 3.14.4):

```
square (.+)\1 abab: True
square (.+)\1 abcb: False
a*b* on aaabb (a3 b2): True
```

PASS/FAIL:
- [B1]이 True/False면 역참조가 `ww`를 매치 — 정규를 넘어 문맥 자유 밖까지 감.
- [B2]가 True면 순수 정규식 `a*b*`가 개수 불일치(`aaabb`)도 받아들임 — 8절 펌핑 보조정리가 증명한 "정규 언어는 `aⁿbⁿ`을 셀 수 없음"의 실측. `a*b*`로는 a와 b의 개수를 같게 강제할 방법이 없음.

(참고: Python `regex` 서드파티 모듈을 설치하면 `(?R)` 재귀로 균형 괄호 매치도 재현할 수 있으나, 본 환경엔 미설치라 A의 .NET 균형 그룹으로 대신 실증함.)

타이밍이 아니라 매치 여부(True/False)가 검증 대상이므로 실행 환경이 달라도 결과는 동일해야 합니다.
