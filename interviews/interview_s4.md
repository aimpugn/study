### **심층 기술 면접 시나리오**

**면접관**: 안녕하세요. 지원서 잘 봤습니다. 여러 프로젝트에서 다양한 기술을 다루신 것 같네요. 레거시 서비스를 Golang으로 포팅하셨다고 했는데, 이 과정에서 동시성 문제를 다루셨을 것 같습니다. Go에서의 동시성 모델과 이를 지원하는 메커니즘에 대해 자세히 설명해 주시겠어요?

**지원자**: 안녕하세요. 네, Go에서의 동시성은 언어 차원에서 지원되는 중요한 특징입니다. Go는 동시성 모델로 CSP(Communicating Sequential Processes)를 기반으로 합니다. 이는 프로세스들이 메시지 패싱을 통해 통신하는 모델입니다.

Go에서는 **고루틴(goroutine)**과 **채널(channel)**을 통해 동시성을 구현합니다.

- **고루틴**: 고루틴은 Go 런타임에 의해 관리되는 경량 스레드입니다. `go` 키워드를 사용하여 함수를 고루틴으로 실행할 수 있습니다. 고루틴은 매우 적은 메모리 오버헤드로 수천 개를 생성할 수 있으며, Go 런타임 스케줄러가 이를 관리합니다.

- **채널**: 채널은 고루틴 간에 타입 안전한 방식으로 데이터를 전달하는 동기화 메커니즘입니다. 채널을 통해 데이터를 송수신함으로써 고루틴 간의 통신과 동기화를 동시에 해결할 수 있습니다. 채널은 버퍼링 여부에 따라 동기적 또는 비동기적으로 동작할 수 있습니다.

Go의 동시성 모델은 "메모리를 공유하여 통신하지 말고, 통신을 통해 메모리를 공유하라"는 철학을 따릅니다. 이는 공유 메모리에 대한 접근을 최소화하고, 경쟁 상태(race condition)를 방지하는 데 도움이 됩니다.

**면접관**: 그렇다면 Go에서의 메모리 모델에 대해 말씀해 주실 수 있을까요? 특히, 메모리 접근의 일관성과 동시성 안전성을 어떻게 보장하는지 궁금합니다.

**지원자**: 네, Go의 메모리 모델은 프로그램의 동시성 안전성을 위해 **happens-before 관계**를 정의하고 있습니다. 이는 어떤 메모리 연산이 다른 메모리 연산보다 먼저 일어났음을 보장하는 관계입니다.

- **happens-before 관계**는 고루틴 간의 동기화 이벤트를 통해 형성됩니다. 예를 들어, 채널을 통한 송신과 수신, `sync` 패키지의 `Mutex`, `WaitGroup` 등의 동기화 프리미티브, 그리고 `atomic` 연산 등이 있습니다.

- 이러한 동기화 이벤트를 사용하지 않고는 고루틴 간의 메모리 쓰기와 읽기의 순서를 보장할 수 없으며, 이는 동시성 버그로 이어질 수 있습니다.

따라서, 동시성 프로그래밍을 할 때는 반드시 동기화 메커니즘을 사용하여 고루틴 간의 메모리 접근을 제어해야 합니다.

**면접관**: 좋습니다. 그럼 채널을 사용할 때 발생할 수 있는 데드락(deadlock) 상황은 어떻게 예방하셨나요?

**지원자**: 채널 사용 시 데드락은 주로 송신자와 수신자가 서로를 기다릴 때 발생합니다. 이를 예방하기 위해 다음과 같은 방법을 사용했습니다.

1. **채널의 버퍼링**: 버퍼가 없는 채널은 송신과 수신이 동시에 이루어져야 하지만, 버퍼가 있는 채널은 버퍼 크기만큼 송신자가 블로킹 없이 보낼 수 있습니다. 이를 통해 일시적인 비동기 통신을 가능하게 했습니다.

2. **`select` 문 사용**: `select` 문을 사용하여 여러 채널의 송수신을 대기하거나, 타임아웃 및 디폴트 케이스를 활용하여 블로킹을 방지했습니다.

3. **채널 닫기 및 신호 전송**: 작업이 완료되었음을 알리기 위해 채널을 닫거나, 특별한 신호 값을 보내어 고루틴이 종료될 수 있도록 했습니다.

4. **설계 단계에서의 검토**: 고루틴과 채널의 상호 작용을 도식화하여 가능한 데드락 시나리오를 미리 파악하고 구조를 개선했습니다.

또한, Go의 런타임은 프로그램 종료 시 데드락 감지를 제공하기 때문에 개발 과정에서 문제를 빠르게 발견하고 수정할 수 있었습니다.

**면접관**: 아주 잘 설명해 주셨습니다. 이제 가비지 컬렉션에 대해 이야기해 보죠. Go의 가비지 컬렉터는 어떤 알고리즘을 사용하며, 이로 인한 성능 영향은 어떻게 관리하셨나요?

**지원자**: Go의 가비지 컬렉터는 **동시 병행 마크 앤 스위프(concurrent mark-and-sweep)** 알고리즘을 사용합니다.

- **마크 단계**: 가비지 컬렉터는 루트 집합에서 시작하여 도달 가능한 객체를 마킹합니다. 이 과정은 애플리케이션 스레드와 병행하여 진행됩니다.

- **스위프 단계**: 마킹되지 않은 객체들을 해제하여 메모리를 반환합니다.

Go의 가비지 컬렉터는 **STW(Stop-The-World)** 시간이 짧도록 설계되어 있습니다. 즉, 프로그램의 실행을 멈추는 시간이 최소화되어 애플리케이션의 응답성에 미치는 영향을 줄입니다.

성능 관리는 다음과 같이 했습니다.

- **메모리 할당 최소화**: 불필요한 객체 생성을 피하고, 재사용 가능한 객체는 풀링하여 메모리 할당과 가비지 컬렉션 부담을 줄였습니다.

- **메모리 프로파일링**: `pprof`와 같은 도구를 사용하여 메모리 사용 패턴을 분석하고, 메모리 누수나 과도한 할당 지점을 찾아 최적화했습니다.

- **가비지 컬렉션 튜닝**: `GOGC` 환경 변수를 조정하여 가비지 컬렉션의 빈도를 제어했습니다. 예를 들어, `GOGC` 값을 높이면 가비지 컬렉션이 덜 자주 발생하지만, 메모리 사용량이 늘어납니다.

**지원자**: 네, CAP 정리는 분산 시스템에서 동시에 세 가지 특성을 모두 만족시킬 수 없다는 이론입니다.

- **일관성(Consistency, C)**: 모든 노드가 같은 시점에 동일한 데이터를 볼 수 있음.

- **가용성(Availability, A)**: 모든 요청이 항상 성공적으로 응답을 받을 수 있음.

- **파티션 내성(Partition Tolerance, P)**: 네트워크 분할이 발생하더라도 시스템이 계속 동작함.

CAP 정리에 따르면 분산 시스템에서 네트워크 분할은 피할 수 없는 상황이므로, 설계자는 일관성과 가용성 중 하나를 선택해야 합니다.

- **CP 시스템**: 일관성을 우선시하여, 네트워크 분할 시 일부 서비스가 응답하지 않을 수 있지만, 데이터의 일관성을 보장합니다. 예: 분산 데이터베이스에서의 강한 일관성 모드.

- **AP 시스템**: 가용성을 우선시하여, 네트워크 분할 시에도 서비스는 계속 응답하지만, 데이터 일관성이 약해질 수 있습니다. 예: NoSQL 데이터베이스에서의 eventual consistency 모드.

분산 시스템 설계에서는 애플리케이션의 요구사항에 따라 일관성과 가용성의 균형을 맞춰야 합니다. 예를 들어, 금융 거래 시스템에서는 일관성이 중요하므로 CP를 선택하고, 소셜 미디어 피드 같은 경우 가용성이 중요하므로 AP를 선택할 수 있습니다.

**면접관**: 그렇다면 여러분이 작업하신 마이크로서비스 아키텍처에서 데이터 일관성을 어떻게 유지하셨나요? 특히 사가 패턴을 사용하셨다고 했는데, 그 작동 원리와 적용 시 고려사항을 설명해 주세요.

**지원자**: 네, 마이크로서비스 아키텍처에서는 각 서비스가 독립된 데이터베이스를 가지므로, 전통적인 ACID 트랜잭션을 사용하기 어렵습니다. 이를 해결하기 위해 **사가 패턴(Saga Pattern)**을 적용했습니다.

- **사가 패턴의 작동 원리**:

    - 비즈니스 로직을 여러 개의 로컬 트랜잭션으로 분할합니다.
    - 각 로컬 트랜잭션은 하나의 서비스 내에서 실행되며, 성공하면 다음 트랜잭션을 호출합니다.
    - 전체 사가가 성공하면 모든 트랜잭션이 커밋되고, 실패하면 이미 완료된 트랜잭션들을 보상하는 보상 트랜잭션을 실행합니다.

- **적용 시 고려사항**:

    - **보상 트랜잭션 설계**: 모든 로컬 트랜잭션에 대해 취소 작업을 정의해야 합니다. 이는 일부 작업이 취소 불가능한 경우 어려울 수 있습니다.

    - **오류 처리 및 재시도 로직**: 네트워크 장애나 서비스 다운 등으로 인해 트랜잭션이 실패할 수 있으므로, 재시도 메커니즘과 오류 처리 로직을 구현해야 합니다.

    - **데이터 일관성 지연**: 최종적 일관성을 보장하므로, 중간 상태에서 데이터 불일치가 발생할 수 있습니다. 이를 사용자에게 적절히 표시하거나, 일관성 지연을 최소화하는 방법이 필요합니다.

    - **사가 조정 방식**: 중앙에서 사가를 관리하는 오케스트레이션 방식과, 각 서비스가 다음 트랜잭션을 호출하는 코레오그래피 방식이 있습니다. 각 방식의 장단점을 고려하여 선택해야 합니다.

저는 주로 코레오그래피 방식을 사용하여 서비스 간의 결합도를 낮추고, 이벤트 기반 아키텍처를 활용했습니다.

**지원자**: 네, HTTP/1.1과 HTTP/2의 주요 차이점은 다음과 같습니다.

1. **멀티플렉싱(Multiplexing)**:

   - **HTTP/1.1**: 한 연결에서 하나의 요청-응답 사이클만 처리합니다. 이를 해결하기 위해 여러 개의 연결을 열거나, 파이프라이닝을 사용하지만, 여전히 제한적입니다.
   - **HTTP/2**: 단일 연결에서 여러 개의 스트림을 통해 동시에 다중 요청과 응답을 주고받을 수 있습니다. 이를 통해 연결의 효율성을 높이고 지연 시간을 줄입니다.

2. **헤더 압축(Header Compression)**:

   - **HTTP/1.1**: 헤더가 텍스트로 전송되며, 중복된 헤더 정보가 많습니다.
   - **HTTP/2**: HPACK 알고리즘을 사용하여 헤더를 이진 형태로 압축하고, 헤더 테이블을 통해 중복을 제거합니다.

3. **서버 푸시(Server Push)**:

   - **HTTP/2**: 서버가 클라이언트의 요청에 응답하기 전에 관련 리소스를 미리 푸시할 수 있습니다. 이를 통해 페이지 로딩 시간을 단축할 수 있습니다.

4. **이진 프레이밍(Binary Framing)**:

   - **HTTP/2**: 모든 통신이 이진 프레임으로 이루어져 프로토콜 파싱이 효율적입니다.

**HTTP/2의 이점**:

- **성능 향상**: 멀티플렉싱과 헤더 압축으로 대역폭 사용량과 지연 시간을 줄입니다.

- **리소스 최적화**: 단일 연결로 다중 요청을 처리하므로, 서버와 클라이언트의 리소스 사용이 효율적입니다.

- **애플리케이션 계층 프로토콜의 발전**: 이진 프레이밍으로 인해 새로운 프로토콜 설계가 용이해집니다.

gRPC는 이러한 HTTP/2의 이점을 활용하여 고성능의 RPC 통신을 제공합니다.

**면접관**: 그렇다면 프로토콜 버퍼(Protocol Buffers)가 무엇이고, gRPC에서 어떻게 사용되는지 설명해 주세요.

    - **다양한 언어 지원**: 다양한 프로그래밍 언어에서 사용할 수 있도록 코드 생성기를 제공합니다.

- **gRPC에서의 사용**:

    - **서비스 정의**: `.proto` 파일에서 RPC 서비스와 메시지 형식을 정의합니다.

    - **코드 생성**: gRPC 플러그인을 사용하여 서버와 클라이언트용 스텁 코드를 자동 생성합니다.

    - **직렬화 및 역직렬화**: 프로토콜 버퍼를 통해 메시지를 직렬화하여 HTTP/2를 통해 전송하고, 수신 측에서 역직렬화합니다.

이를 통해 gRPC는 효율적이고 타입 안전한 RPC 통신을 구현할 수 있습니다.

- **B-트리**:

    - 각 노드가 키와 데이터를 모두 저장합니다.

    - 리프 노드와 내부 노드 모두 데이터 레코드를 가질 수 있습니다.

    - 노드 간에 포인터를 통해 자식을 가리킵니다.

- **B+트리**:

    - 데이터는 리프 노드에만 저장되고, 내부 노드는 키와 자식에 대한 포인터만 가집니다.

    - 리프 노드들이 양쪽으로 연결 리스트 형태로 연결되어 있습니다.

**B+트리를 사용하는 이유**:

1. **효율적인 범위 검색**:

   - 리프 노드의 연결 리스트를 통해 순차적인 데이터 접근이 용이하며, 범위 쿼리에 최적화되어 있습니다.

2. **예측 가능한 I/O 성능**:

   - 데이터가 리프 노드에만 존재하므로, 검색 경로가 일정하고, 디스크 I/O 패턴이 예측 가능합니다.

3. **높은 팬 아웃(Fan-out)**:

   - 노드에 더 많은 키를 저장할 수 있어 트리의 높이가 낮아지고, 디스크 액세스 횟수가 줄어듭니다.

4. **인덱스의 안정성**:

   - 데이터 삽입과 삭제 시 트리의 균형을 유지하기 위한 재구성이 최소화됩니다.

따라서, 데이터베이스에서는 B+트리를 인덱스로 사용하여 대용량 데이터에서도 효율적인 검색 성능을 제공합니다.

**빅오 표기법(Big O Notation)**은 알고리즘의 복잡도를 상한선으로 나타내는 표기법입니다.

- **이유**:

    - **단순화**: 입력 크기가 충분히 클 때 지배적인 항만을 고려하여 알고리즘의 성능을 단순화합니다.

    - **비교 가능성**: 알고리즘 간의 성능을 쉽게 비교할 수 있습니다.

    - **환경 독립성**: 하드웨어나 상수 시간 등의 환경적인 요소를 배제하고, 알고리즘 자체의 성능을 평가합니다.

예를 들어, O(n), O(n log n), O(n²) 등의 표기법을 사용하여 알고리즘의 시간 복잡도를 표현합니다.
