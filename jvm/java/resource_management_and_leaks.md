# 자원과 누수 — JVM이 알아서 치우는 것과 못 치우는 것

`Validation.buildDefaultValidatorFactory()`를 테스트 클래스 필드로 두면 IntelliJ가 "`ValidatorFactory` used without try-with-resources"라고 경고한다. 자연스러운 반문은 *"테스트라 어차피 프로세스 끝나면 다 회수될 텐데 굳이?"*다. 결론부터: **이 자원에 한해선 맞는 직관이다. 그런데 "테스트라서 괜찮다"는 *이유*는 틀렸다.** 맞는 이유는 "이 자원이 힙 안에만 있고 유한하기 때문"이다. 그 둘을 가르려면 JVM이 자원을 어떻게 다루는지를 알아야 한다.

## 1. 두 종류의 자원 — 힙과 외부

Java 프로그램이 쓰는 자원은 성격이 전혀 다른 둘로 나뉜다.

1. **힙 메모리** — `new`로 만든 객체. JVM 안 힙에 산다.
2. **외부(네이티브) 자원** — JVM *밖*, OS나 다른 시스템이 소유하는 것: 열린 파일 핸들(file descriptor), 네트워크 소켓, OS 스레드, DB 커넥션, off-heap(네이티브) 메모리. Java 쪽엔 그것을 가리키는 작은 *핸들 객체*만 있고, 진짜 자원은 힙 밖에 있다.

| | 힙 메모리 | 외부 자원 |
|---|---|---|
| 예 | `new`로 만든 객체, 배열, 캐시 | 파일 핸들, 소켓, 스레드, DB 커넥션, 네이티브 메모리 |
| 누가 회수하나 | GC가 자동으로 (도달 불가능해지면) | 내가 직접 `close()` (또는 풀에 반납) |
| GC가 알아서 해주나 | 그렇다 | **아니다** |
| 한도 | 힙 크기 (보통 수백 MB~수 GB) | OS/풀의 **하드 한도** (프로세스당 fd 수, 풀 커넥션 수 등) |
| 안 풀면 생기는 일 | 도달 가능한 채 쌓이면 *메모리 누수* | *자원 누수* — 한도에 부딪혀 실행 중 고갈 |

핵심 비대칭: **GC는 1번은 책임지지만 2번은 책임지지 않는다.**

## 2. GC가 하는 일 vs close()가 하는 일

**GC(가비지 컬렉터)** 는 힙 객체가 *도달 불가능*(어떤 살아있는 참조로도 닿을 수 없음)해지면 자동으로 회수한다. 힙은 손으로 free하지 않는다 — GC가 한다.

그런데 외부 자원은 GC로 풀리지 않는다. 이유가 둘이다.

- **(a) GC는 *메모리* 압박에만 반응하지 *핸들* 압박은 모른다.** JVM은 힙이 찰 때 GC를 돌린다. 네가 파일 1만 개를 열어둔 건 JVM의 관심사가 아니다. 그래서 힙은 텅 비었는데 OS의 "열 수 있는 파일 수" 한도부터 먼저 친다 → GC는 돌 이유가 없어 안 돌고 → 핸들은 안 풀리고 → `Too many open files`로 죽는다.
- **(b) GC가 핸들 객체를 수거하더라도, 그 밑의 fd를 닫아주는 정리 훅(finalizer/Cleaner)이 *언제* 도는지는 보장이 없다.** 명세는 "언젠가, 어쩌면"만 약속한다. 프로그램 끝나기 전에 돈다는 보장조차 없다. 그래서 즉시 반납을 GC에 기댈 수 없다.

그래서 **`close()` 와 `AutoCloseable`이 존재한다.** `close()`는 외부 자원을 *지금, 확정적으로* 돌려주는 행위다(fd를 OS로, 커넥션을 풀로). `try-with-resources`는 블록을 벗어날 때 `close()`가 반드시 불리도록 해주는 문법 설탕일 뿐이다. 순수 힙 객체는 `close()`가 필요 없다(GC가 처리) — `AutoCloseable`이라는 표식 자체가 "나는 GC가 즉시 못 푸는 걸 감싸고 있으니 닫아 달라"는 신호다.

## 3. "누수"의 두 가지 뜻

사람들이 "누수(leak)"라고 부르는 건 사실 둘이고, 고치는 법도 다르다.

- **메모리 누수 (힙 쪽):** 다 썼는데도 객체가 *도달 가능*한 채 남아 GC가 못 거두고 힙이 자란다. 전형: 계속 add만 하고 비우지 않는 `static` 리스트. 고치는 법은 `close()`가 아니라 *참조를 끊는 것*.
- **자원 누수 (외부 쪽):** fd/소켓/커넥션을 열고 `close()`하지 않는다. 핸들 객체가 (불확실하게) GC될 때까지, 혹은 프로세스가 죽을 때까지 OS 쪽 자원이 잡혀 있다. OS/풀엔 **하드 한도**가 있으므로 누수된 핸들은 *천장을 향해 쌓인다*. (실전 사례 — HTTP 클라이언트를 요청마다 새로 열고 `use { }`로 닫기를 반복해 그 안의 커넥션 엔진·이벤트 루프 스레드를 누수한 케이스: [../thread_leak.md](../thread_leak.md).)

```java
// 안티패턴: 루프마다 스트림을 열고 안 닫음
for (Path p : thousandsOfFiles) {
    InputStream in = Files.newInputStream(p);  // fd 하나 점유
    read(in);
    // close() 없음 → fd가 계속 쌓임
}
// 힙은 넉넉한데 OS의 fd 한도(예: 1024)에 먼저 부딪혀 "Too many open files"로 터진다.
// 메모리 압박이 없으니 GC는 돌지 않고, 따라서 핸들도 안 풀린다.
```

```java
// 제대로: 블록 끝에서 즉시 close()
for (Path p : thousandsOfFiles) {
    try (InputStream in = Files.newInputStream(p)) {
        read(in);
    }
}
```

## 4. "프로세스 끝나면 회수된다" — 참이지만 다른 질문에 답한 것

이 문장은 *프로세스 종료 시점*에선 참이다. OS *프로세스*가 끝나면 OS가 그 프로세스의 자원을 한꺼번에 전부 거둔다 — 힙, fd, 소켓, 스레드, 네이티브 메모리. 프로세스를 넘겨 살아남는 건 없다. 그러니 *맨 끝 순간*만 보면 안 닫은 무엇이든 무의미해진다.

문제는, 진짜 위험은 "종료 후에 뭐가 남나"(아무것도 안 남는다)가 아니라 **"실행 *도중*에 그게 쌓이거나 바닥나는가"**라는 점이다. 이 질문에 "프로세스 끝나면 회수"는 답을 주지 못한다.

테스트 *스위트*는 수천 개 테스트를 한 줄로 이어 도는 **하나의 장수(長壽) JVM**이다. 각 테스트가 커넥션을 하나씩 누수하면, 커넥션이 실행 내내 쌓여 50번째쯤 풀이 고갈되고 — 그 테스트들은 JVM이 "정리"하러 종료하기 *한참 전에* 먼저 깨진다. 종료가 치워주는 것은 *그 전에 이미 죽은* 테스트를 구하지 못한다.

## 5. 그래서 판단 기준 — 테스트/프로덕션이 아니라 자원의 본질

정석 판단은 *"테스트인가?"*가 아니라 이것이다:

> **이 `close()` 대상이 외부·고갈성 자원(fd, 소켓, 스레드, 커넥션, 네이티브 메모리)을 감싸는가? 그리고 그 인스턴스가 실행 중 쌓일 수 있는가?**

- **그렇다** (DB 커넥션, `FileInputStream`, 소켓, 스레드풀) → 테스트에서도 *항상* `close()`. 누수하면 *뒤 테스트*가 고갈로 깨진다.
- **아니다** (힙 안에만 있고 유한한 양성 자원) → 안 닫아도 된다. 도중에 바닥날 게 없으니 "프로세스 종료가 거둔다"가 *정당한 보조 근거*가 된다.

즉 면죄부의 근거는 "테스트라서"가 아니라 "**이 자원이 힙-only·유한·프로세스 스코프라서**"여야 한다. 같은 면죄부를 커넥션이나 핸들에 주면, 그건 테스트에서도 틀린다.

## 6. ValidatorFactory에 적용

기본 Hibernate `ValidatorFactory`는 사실상 *힙*이다 — 제약 메타데이터 캐시와 EL(Expression Language) 팩토리 객체. 기본 구성에선 파일도 소켓도 열지 않고 스레드도 만들지 않는다. OS가 한도를 두는 게 없다.

- 따라서 안 닫는 것은 **외부 자원 누수가 아니다.** 기껏해야 가벼운 *힙* 비용이다. 테스트 클래스마다 `static` 필드의 팩토리가 실행 내내 도달 가능하게 남는다 — 클래스가 로드돼 있는 한 그 static 필드는 도달 가능하고, 테스트 클래스는 보통 실행 내내 로드된 채다. 그래서 클래스당 팩토리 하나가 쌓이긴 하지만, 그 수는 유한하고(큰 스위트라도 다 합쳐 수십 MB 수준) 종료 때 회수된다. 바닥날 게 없다.
- 그래서 실무적으로 괜찮다. `close()`는 그 캐시를 조금 일찍 비우는 정도이고, 어차피 곧 끝날 테스트 JVM에선 의미가 작다.
- IntelliJ가 잡는 이유는, 그 인스펙션이 *모든* `AutoCloseable`을 일률적으로 본다는 데 있다 — "이건 양성 힙-only"인지 "이건 소켓"인지 구별하지 못한다. 경고는 *기술적으로는 맞지만*(닫을 수 있는데 안 닫음) *실제 위험은 ~0*이라 무시할 만하다.

남는 실익은 누수가 아니라 **반복**이다(클래스마다 팩토리를 새로 빌드). 굳이 손본다면 명분은 "누수 위험"이 아니라 "반복 제거·빌드 1회"여야 한다. 선택지:

- **현행 유지** — 이 자원 한정으로 정당(경고를 양성으로 수용).
- **공유 static 홀더** — 스위트 통틀어 한 번만 빌드, 반복 제거. 단 팩토리 하나는 여전히 안 닫혀 경고가 *한 곳*에 남음.
- **JUnit5 확장**(`ValidatorExtension`) — 루트 스토어의 `CloseableResource`로 등록해 스위트 끝에서 한 번 정상 `close()`되게 하고, 파라미터로 주입받는다. 경고가 *제대로* 사라진다. 대신 손이 더 들고 클래스마다 `@ExtendWith`를 붙여야 한다.

참고로 *try-with-resources를 필드 초기화에 그대로 쓸 수는 없다*: 블록을 벗어나면 팩토리가 닫히는데, 여러 테스트가 공유하려고 오래 살려둔 `Validator`는 *닫힌 팩토리* 위에 남아 이후 `validate()` 동작이 미정의가 된다. 그래서 "한 번 빌드해 오래 공유"하려면 홀더나 확장처럼 *수명을 따로 관리*해야 한다.

## 7. 스스로 확인하기

- 어떤 `XxxFactory`/스트림/커넥션을 안 닫았을 때, *프로세스가 끝나기 전에* 뭔가 바닥날 수 있는가? 없으면 양성, 있으면 반드시 닫는다.
- 힙은 멀쩡한데 `Too many open files`로 죽는 시나리오를 한 문장으로 설명할 수 있는가? (GC는 메모리 압박에만 반응하므로.)
- "테스트라서 괜찮다"고 말하고 싶을 때, 그 자리에 "이 자원이 힙-only·유한이라서"를 넣어도 말이 되는가? 안 되면 닫아야 하는 자원이다.
