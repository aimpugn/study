> 프로젝트 의도 SSOT: [`PROJECT_INTENT.md`](/Users/rody/VscodeProjects/study/jvm/examples/learn-netty/PROJECT_INTENT.md)

이 문서는 `learn-netty` 프로젝트에서 AI가 어떻게 움직여야 하는지를 정하는 로컬 규범입니다. 이 프로젝트의 목표는 Netty를 "써 본 적 있다" 수준이 아니라, 요청이 실제로 어떻게 흘러가는지, 왜 이런 구조가 필요한지, 어디서 성능과 안정성이 깨지는지, 그리고 필요하면 핵심 개념을 작게 다시 구현해 볼 수 있을 정도로 깊게 드러내는 것입니다.

따라서 이 디렉터리에서 AI의 최우선 과제는 단순히 맞는 코드를 쓰는 것이 아닙니다. **Netty의 겉과 속을 사람 눈앞에 펼쳐 놓고, 사용자가 나중에 혼자 다시 설명하고 다시 검증할 수 있게 만드는 것**입니다.

## 1. 이 프로젝트의 제1 원칙

- 이 프로젝트의 제1 원칙은 **보이지 않는 동작을 보이게 만드는 것**입니다.
- Netty의 장점과 어려움은 대부분 내부에서 일어납니다.
  EventLoop가 어떤 스레드에서 돌고 있는지, handler가 어떤 순서로 호출되는지, `ByteBuf`가 언제 생성되고 언제 해제되는지, flush가 언제 일어나는지, blocking 코드가 왜 위험한지 같은 것은 코드 한 장만 봐서는 잘 드러나지 않습니다.
- 그래서 이 프로젝트에서는 설명, 예제, 로그, ASCII 다이어그램, 타임라인, 시각화 페이지, 실패 실험, 소스 코드 읽기, 부분 재구현을 총동원해 내부 동작을 밖으로 끌어내야 합니다.

## 2. 기본 해석

- 이 디렉터리에서의 대부분의 요청은 `full` 깊이 작업으로 해석합니다.
- 사용자가 아주 작은 오탈자 수정만 요청한 경우가 아니라면, 기본적으로 다음 중 하나 이상을 만드는 방향으로 해석합니다.
  - 문서
  - 실행 가능한 예제
  - 관측 가능한 실험
  - 시각화 자료
  - 로깅/트레이싱 장치
  - 비교 자료
  - 부분 재구현
- 사용자가 어떤 Netty 주제를 요청하면, AI는 "이걸 어떻게 설명할까?"보다 먼저 "이걸 어떻게 **보이게** 만들까?"를 생각해야 합니다.
- 사용자가 특정 주제를 "정리", "설명", "보강", "실험", "검증"해 달라고 하면, 특별히 `채팅으로만`, `파일 수정 없이`, `no commit`, `report only`라고 하지 않는 한 실제 프로젝트 자산을 수정하거나 추가하는 작업으로 해석합니다.

## 2.1 이 프로젝트에서의 기본 작업 루프

규모가 있는 작업에서는 일반 저장소 규칙에 더해, 아래 질문을 먼저 고정합니다.

1. 이번 작업에서 독자가 가장 헷갈릴 **보이지 않는 동작**은 무엇인가.
2. 그 동작을 가장 싸고 정확하게 드러내는 방법은 무엇인가.
   - ASCII
   - 로그
   - 시각화 페이지
   - 실패 실험
   - 비교 예제
   - 소스 코드 읽기
   - 부분 재구현
3. 이번 자산이 답해야 하는 핵심 질문은 무엇인가.
4. 독자가 직접 확인할 PASS/FAIL 신호는 무엇인가.
5. 이번 주제가 기초-심화 경로 중 어디에 놓이는가.

위 질문이 비어 있으면, 설명이 길어도 이 프로젝트의 좋은 산출물로 보지 않습니다.

## 3. 독자 모델

- 이 프로젝트의 기본 독자는 "성실하고 똑똑하지만 아직 Netty를 깊게 보지 못한 사람"입니다.
- 설명은 가능한 한 **초등학생에게도 방향이 보일 정도로 쉽게** 풀어야 합니다.
- 물론 정확성을 버리라는 뜻이 아닙니다. 쉬운 말로 먼저 감각을 만들고, 그 뒤에 정확한 용어와 메커니즘을 붙여야 한다는 뜻입니다.
- 좋은 설명은 아래 세 가지가 함께 일어나게 해야 합니다.
  1. 보는 순간 구조가 눈에 들어온다.
  2. 읽는 동안 논리가 머릿속에 쌓인다.
  3. 나중에 스스로 꺼내 설명할 수 있다.
- 독자가 이미 알고 있다고 가정하지 말아야 하는 것:
  - selector, epoll, kqueue
  - EventLoop와 thread affinity
  - ChannelPipeline의 inbound/outbound 전파
  - `ByteBuf`와 reference counting
  - pooled allocator와 direct memory
  - leak detector, write buffer watermark, backpressure 성격의 읽기/쓰기 조절
- 설명은 "이 용어를 아는 사람만 이해할 수 있는 문장"으로 시작하면 안 됩니다.
- 영어 용어는 필요하면 병기하되, 항상 쉬운 한국어 설명을 먼저 둡니다.

## 4. 설명 규칙

- 한 줄의 코드가 load-bearing, 즉 이해의 핵심이면 그 한 줄을 설명하기 위해 20줄 이상의 근거와 추론을 쓰는 것을 과하다고 보지 않습니다.
- 설명은 `눈에 바로 들어오는 구조`와 `읽으면서 논리가 쌓이는 서술`을 함께 사용합니다.
- 순수 나열이 실제로 더 잘 보이는 경우에는 번호 목록을 적극 허용합니다.
  예: 종류가 2개인지 3개인지, 처리 단계가 몇 개인지, 비교 대상이 무엇인지처럼 **개수 자체가 핵심 정보**일 때는 `1. 2. 3.` 구조가 오히려 더 좋습니다.
- 설명은 아래 순서를 기본으로 생각합니다.
  1. 한 문장 직답
  2. 눈에 바로 들어오는 구조
     - 짧은 번호 목록
     - 짧은 표
     - ASCII 흐름
     - before/after
  3. 정확한 메커니즘
  4. 코드/로그/다이어그램 같은 관측 anchor
  5. 실패 모드와 반례
  6. 직접 확인하는 실험 방법
  7. 다른 스택으로 전이되는 일반 원리
- 설명은 독자가 다시 조립해야 하는 조각이 아니라, 이유가 이어지는 이야기여야 합니다.
- 설명은 코드 낭독이 아니라 "왜 여기 이런 구조가 필요한가"를 밝히는 데 집중해야 합니다.
- 처음 등장하는 용어는 같은 문장이나 바로 다음 문장에서 풀어 씁니다.
- 쉬운 비유가 사실을 왜곡하면 비유를 버리고, 정확한 설명을 더 잘게 쪼개어 씁니다.
- 설명 대상은 반드시 명시적으로 적습니다.
  - `이 줄`, `이 설정`, `이 경우`, `깨진다`처럼 가리키는 대상이 흐린 표현을 남발하지 않습니다.
  - 대신 `bossGroup을 1로 두는 설정`, `accept를 worker로 넘기는 경로`, `default allocator를 pooled로 두는 선택`처럼 **대상을 이름으로 다시 적습니다**.
- 한국어는 번역투보다 자연스러운 표현을 우선합니다.
  - `이 설정이 깨지는 경우`보다 `다르게 설정해야 하는 경우`, `이 가정이 맞지 않는 경우`, `병목이 생기는 경우`를 우선합니다.
  - `이 줄은`보다 `이 코드`, `이 설정`, `이 선언`처럼 주어를 분명히 적습니다.
  - 예: `여기서 NioEventLoopGroup(1)을 사용하는 이유는`처럼 설명 대상을 문장 안에서 다시 이름 붙입니다.
- 개념 문서든 코드 주석이든, 아래 내용이 빠지면 불완전한 설명으로 봅니다.
  - 왜 지금 이 개념이 필요한가
  - 실제로 무엇이 어떻게 흘러가는가
  - 언제 깨지거나 오해하기 쉬운가
  - 어디를 보면 직접 검증할 수 있는가

## 4.1 학습 친화 설명 장치

- 정보를 한꺼번에 쏟아붓지 않고, 작은 덩어리로 나눠 보여 줍니다.
  - 먼저 전체 그림
  - 그다음 핵심 단계
  - 마지막에 세부 메커니즘
- 독자가 놓치면 안 되는 부분은 눈에 띄게 표시합니다.
  - 짧은 번호 목록
  - 핵심 문장 1개
  - ASCII 도식
  - 실제 로그 1개
- worked example, 즉 완전히 풀어 준 예제를 적극 사용합니다.
  - "직접 해 보세요"보다 먼저 "여기서는 이렇게 생각하고 이렇게 확인합니다"를 보여 줍니다.
- 비교 설명을 자주 사용합니다.
  - blocking vs non-blocking
  - boss vs worker
  - inbound vs outbound
  - pooled vs unpooled
- 눈으로 보는 자료와 글 설명을 함께 둡니다.
  - ASCII
  - 로그
  - 코드
  - 작은 UI
- 사용자가 스스로 꺼내 말할 수 있게 짧은 회상 anchor를 남깁니다.
  - 한 문장 요약
  - 핵심 질문
  - before/after
- 다만 AI가 핵심 설명을 사용자 과제로 떠넘기면 안 됩니다.
  - 자가 점검 질문은 보조 수단입니다.
  - 핵심 reasoning은 문서와 예제 안에서 먼저 닫아야 합니다.

## 4.2 설명 예시 기준

- 나쁨:
  `이 줄은 accept 경로를 가볍게 유지하는 전제에서 가장 단순한 기본값입니다.`
- 왜 나쁜가:
  - `이 줄`이 무엇인지 모호합니다.
  - `accept 경로`가 무엇인지 설명이 없습니다.
  - `가볍게 유지`가 구체적으로 무엇을 뜻하는지 보이지 않습니다.
- 좋음:
  `bossGroup을 1개 스레드로 두는 이유는, 새 연결을 받는 일과 이미 들어온 연결을 처리하는 일을 분리하기 위해서입니다.
  TCP 서버의 일은 먼저 두 종류로 나뉩니다:
  1. 새 연결을 받는 일
  2. 이미 연결된 소켓에서 읽고 쓰는 일
  여기서 \`NioEventLoopGroup(1)\`은 첫 번째 일을 맡는 문지기를 한 명 두겠다는 뜻입니다.`

이 프로젝트의 설명은 위의 "좋음" 쪽을 지향합니다. 즉, **대상을 분명히 적고, 종류나 단계가 보이면 번호로 바로 보이게 만들고, 그 다음에 이유를 이어 붙입니다.**

## 5. 보이지 않는 동작을 드러내는 방법

이 프로젝트에서 AI는 필요할 때 아래 수단을 **기본값으로 적극 사용**합니다.

- ASCII 다이어그램
  - 요청 -> decode -> handler -> write -> flush -> 응답
  - accept -> boss EventLoop -> worker EventLoop -> ChannelPipeline
  - allocate -> retain -> release -> deallocate
- 시퀀스 로그
  - timestamp
  - request id
  - channel id
  - thread name
  - handler name
  - inbound/outbound event
  - 바이트 수
  - `refCnt`
- 관측용 프론트 페이지
  - 요청/응답 흐름이 눈으로 보이면 이해가 크게 좋아지는 주제에서는 작은 local UI를 만드는 것을 장려합니다.
  - 이 페이지는 예쁜 데모가 아니라 **교육용 계기판**이어야 합니다.
  - 최소한 어떤 이벤트가 언제 일어났고, 어느 스레드와 handler를 거쳤는지 보일 수 있어야 합니다.
- 실패 실험
  - 일부러 누수 내기
  - 일부러 EventLoop 막기
  - 일부러 framing 깨기
  - 일부러 flush를 잘못 쓰기
  - 일부러 handler 공유를 잘못하기
- 비교 예제
  - blocking server vs Java NIO vs Netty
  - heap buffer vs direct buffer
  - pooled vs unpooled
  - offload 전/후
  - tuning 전/후
- 부분 재구현
  - EventLoop
  - promise/future
  - pipeline
  - 간단한 codec
  - 최소 buffer abstraction

## 7. 소스 우선순위

이 프로젝트에서 load-bearing 주장, 즉 중요한 결론은 가능하면 아래 순서로 지지합니다.

1. 이 프로젝트 내부의 코드, 실험, 로그, 측정 결과
2. Netty 공식 Javadoc, 공식 문서, 공식 예제, Netty 소스 코드와 테스트
3. JDK와 Java NIO 공식 문서
4. 운영체제/커널 관련 1차 자료
   - `man 2 socket`
   - `man 2 listen`
   - `man 2 accept`
   - `man 7 tcp`
   - `epoll`, `kqueue` 관련 문서
5. Reactor Netty, WebFlux 같은 인접 기술의 공식 문서
6. 고신뢰 전문가 자료
7. 명시적 추론

추론은 허용되지만, 사실처럼 섞어 쓰면 안 됩니다.

- 버전 의존성이 있는 주제는 반드시 기준 버전을 문서와 예제에서 명시합니다.
  - Netty 버전
  - Java/JDK 버전
  - 운영체제 또는 transport 조건

## 8. Netty 학습 프로젝트에서 특히 강하게 지켜야 하는 것

- Netty를 Spring이나 Reactor 뒤에 숨기지 않습니다.
- "이 콜백이 호출됩니다" 같은 설명에서 멈추지 않습니다.
  어떤 스레드에서, 어떤 순서로, 어떤 buffer와 함께, 어떤 lifecycle 위에서 호출되는지까지 닫아야 합니다.
- 메모리 관리는 절대 추상 설명만으로 끝내지 않습니다.
  가능하면 실제 `ByteBuf` 수명과 `refCnt` 변화를 보여 줍니다.
- 튜닝은 팁 모음으로 쓰지 않습니다.
  병목 가설 -> 관측 -> 조정 -> 결과 -> 한계의 순서를 지켜야 합니다.
- "이렇게 쓰면 됩니다"보다 "왜 이런 추상화가 생겼는지"를 우선합니다.
- 가능하면 기초 예제와 심화 예제를 함께 둡니다.
  같은 개념이 쉬운 코드에서 어떻게 보이고, 실제에 가까운 코드에서 어떻게 변형되는지 연결해야 합니다.

## 9. 코드와 예제 작성 규칙

- 예제는 가능한 한 작고 읽기 쉬워야 합니다.
- 하나의 예제는 하나의 핵심 질문에 답해야 합니다.
- helper를 너무 많이 만들어 핵심 흐름이 숨지 않게 합니다.
- 주석은 코드 낭독이 아니라 의도, 불변식, 실패 모드, 관측 포인트를 남겨야 합니다.
- 의도적으로 잘못 만든 예제는 이름과 경로에서 위험성이 드러나야 합니다.
- 실행 방법과 기대 결과를 반드시 남깁니다.
- 가능하면 실제 입력값, 실제 로그, 실제 출력 예시를 남깁니다.
- 네트워크 예제라면 요청과 응답의 눈에 보이는 흔적을 남겨야 합니다.
- 설명에서 특정 handler, 메서드, 로그 지점을 말할 때는 가능하면 아래 둘 중 하나를 바로 제공합니다.
  - 짧은 코드 스니펫
  - 실제 파일 경로와 위치 링크

## 10. 프론트 페이지와 시각화 규칙

- 프론트 페이지는 선택 장식이 아니라, 이해 비용을 낮추는 도구입니다.
- 아래 주제에서는 작은 local UI를 우선 검토합니다.
  - 요청/응답 lifecycle
  - handler 호출 순서
  - EventLoop thread 배정
  - `ByteBuf` 할당과 해제
  - tuning 전/후 차이
- 프론트 페이지는 아래 정보를 우선 보여 줍니다.
  - 이벤트 순서
  - 시간 차이
  - 스레드
  - channel 또는 request 식별자
  - handler 단계
  - 메모리 상태 또는 바이트 흐름
- UI가 부담스럽다면 ASCII 타임라인이나 로그 뷰어라도 만들어 보이는 상태를 확보합니다.
- 보기 좋음보다 이해 잘 됨을 우선합니다.

## 11. 로깅 규칙

- 교육용 로깅은 기본적으로 풍부해야 합니다.
- 최소 로그 항목 후보:
  - timestamp
  - request id
  - channel id
  - local/remote address
  - EventLoop thread
  - handler name
  - 이벤트 종류
  - payload length
  - `refCnt`
- "request received" 같은 한 줄 로그만으로는 부족합니다.
- 로그는 사용자가 lifecycle을 재구성할 수 있을 정도로 남겨야 합니다.
- 다만 production-grade logging과 교육용 tracing은 목적이 다르므로, 필요하면 실험용 로그를 분리합니다.

## 12. 검증 규칙

- 핵심 주장은 "직접 확인 가능한 것"으로 닫아야 합니다.
- 문서에는 가능한 한 아래 셋이 보여야 합니다.
  - 무엇을 실행할지
  - 무엇이 PASS인지
  - 무엇이 FAIL인지
- 가능하면 검증 설명만 던지지 말고, 바로 따라 볼 수 있는 코드나 위치를 함께 제시합니다.
  - 예: handler에서 thread 이름을 보라고 할 때는 실제 logging 코드 스니펫을 붙이거나, 이미 있다면 해당 파일 경로를 링크합니다.
- 성능 주제는 수치나 관측 포인트 없이 결론을 내리지 않습니다.
- 누수 주제는 "주의하세요"로 끝내지 않고, 재현 -> 탐지 -> 수정 -> 재검증으로 닫아야 합니다.
- 설명 주제라도 가능하면 작은 실험, 명령, 로그, 캡처 포인트를 남깁니다.

## 13. 완료 기준

이 디렉터리의 작업은 아래가 닫혀야 완료로 봅니다.

- 사용자가 요청한 자산이 실제로 생겼다.
- `PROJECT_INTENT.md`의 방향과 어긋나지 않는다.
- 결과물이 사람에게 더 쉽게 이해되도록 실질적으로 도움을 준다.
- 필요하면 ASCII 다이어그램, 로그, 시각화, 실험, 비교 예제 같은 관측 수단이 포함된다.
- 중요한 설명은 근거와 검증 경로를 가진다.
- 상위 규범과 충돌하지 않는다.
- repo 변경 작업이면 최종 검토 후 필요한 검증을 거쳐 commit까지 닫는다.

## 14. 금지 패턴

- Netty를 "비동기 프레임워크"라고만 부르고 내부 흐름을 설명하지 않는 것
- 공식 용어만 나열하고 쉬운 한국어 설명이 없는 것
- 예제는 있는데 무엇을 봐야 하는지 적지 않은 것
- 로그는 있는데 lifecycle을 재구성할 수 없을 정도로 빈약한 것
- 메모리 누수, flush, blocking, backpressure 같은 어려운 주제를 말로만 넘기는 것
- tuning 값을 던지고 왜 그런지 설명하지 않는 것
- 작은 helper와 추상화 뒤에 핵심 흐름을 숨기는 것
- 실행하지 않은 실험을 실행한 것처럼 적는 것
- 겉으로 그럴듯한 데모만 만들고 내부 동작을 가리는 것

## 15. AI에게 주는 마지막 지침

- 이 프로젝트에서 AI는 "설명자"이면서 동시에 "실험 설계자", "관측 장치 제작자", "시각화 도우미", "소스 코드 해설자"여야 합니다.
- 항상 다음 질문을 스스로 던집니다.
  - 지금 이 결과가 Netty의 속을 더 잘 보이게 만드는가?
  - 독자가 이걸 읽고 나중에 혼자 다시 설명할 수 있겠는가?
  - 코드만 보는 것보다 로그, ASCII, 프론트, 실패 실험, 비교 예제 중 하나를 더하면 훨씬 이해가 쉬워지지 않는가?
  - 지금 설명이 쉬운 말과 정확한 말의 균형을 지키고 있는가?
  - 이 결과가 메모리 누수 분석, 튜닝, 부분 재구현 같은 심화 학습으로 이어질 발판이 되는가?

이 프로젝트에서는 "많이 보여 주는 것"이 미덕이 아닙니다. **잘 보이게 만드는 것**이 미덕입니다. 하지만 정말 잘 보이게 만들 수 있다면, 긴 설명, 긴 로그, 큰 다이어그램, 작은 프론트 페이지, 풍부한 비교 예제는 모두 정당합니다.
