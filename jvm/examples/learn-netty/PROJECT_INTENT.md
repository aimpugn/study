# PROJECT_INTENT

이 프로젝트의 목적은 Netty를 "사용할 줄 아는 수준"에서 멈추지 않고, 왜 이런 구조가 필요한지, 실제 요청이 어떤 경로로 흘러가는지, 어디서 성능과 안정성이 깨지는지, 그리고 필요하다면 핵심 개념을 간소하게라도 스스로 다시 구현해 볼 수 있을 정도로 깊게 이해하는 데 있습니다.

즉, 이 프로젝트는 단순 예제 모음이 아니라 단계적으로 난도를 올리며 Netty의 개념, 코드, 런타임 동작, 운영상 문제를 함께 익히는 학습용 실험실을 지향합니다. 처음에는 아주 작은 코드와 눈에 보이는 실행 결과로 감각을 만들고, 이후에는 EventLoop(채널 이벤트를 순차적으로 처리하는 실행 루프), Channel(연결 추상화), ChannelPipeline(핸들러 체인), ByteBuf(바이트 버퍼), reference counting(참조 수 기반 수명 관리), native transport(운영체제별 전송 구현), allocator(버퍼 할당기), leak detection(누수 탐지), tuning(튜닝) 같은 주제로 내려가며 "겉으로는 잘 돌아가지만 내부에서는 무엇이 일어나는가"를 계속 추적합니다.

## 왜 이 프로젝트를 따로 만드는가

Netty는 겉으로 보면 "비동기 서버 프레임워크" 정도로 요약되기 쉽지만, 실제로는 커널의 소켓과 selector(여러 소켓 이벤트를 한 번에 감시하는 I/O 다중화 도구), JVM의 메모리 모델, direct memory(힙 밖 메모리), thread model(스레드 배치 구조), protocol framing(메시지 경계 규칙), backpressure에 가까운 흐름 제어(처리 속도에 맞춰 읽기와 쓰기를 조절하는 방식), pipeline 설계, reference counting(참조 수로 메모리 생명주기를 관리하는 방식) 같은 개념이 얽혀 있습니다. 이 층위를 연결해서 이해하지 못하면 예제는 따라 칠 수 있어도, 메모리 누수나 느린 flush, 이벤트 루프 블로킹, 잘못된 handler 설계, 과한 튜닝 같은 문제를 스스로 찾아내기 어렵습니다.

이 프로젝트는 그런 단절을 줄이기 위해 만들어집니다. 한 주제를 문서로만 설명하지 않고, 작은 실행 코드, 단계별 실험, 시각화 가능한 화면이나 로그, 실패 사례, 비교 예제까지 함께 두어 "읽고 아는 것"을 "직접 보고 설명하고 다시 검증할 수 있는 것"으로 바꾸는 것이 목표입니다.

## 최종적으로 도달하고 싶은 수준

이 프로젝트가 잘 만들어졌다면, 학습자는 아래 질문들에 스스로 답하고 필요한 실험까지 설계할 수 있어야 합니다.

- Netty 서버가 요청을 받을 때 커널, selector, EventLoop, ChannelPipeline, handler가 어떤 순서로 관여하는가.
- 왜 EventLoop는 단일 스레드 실행 모델을 선호하고, 어떤 경우에 그 가정이 깨지거나 비용이 커지는가.
- `ByteBuf`와 reference counting이 왜 필요한지, `retain()`과 `release()`를 잘못 다루면 어떤 누수나 use-after-release 성격의 문제가 생기는가.
- 단순 echo 서버와 HTTP 서버, WebSocket 서버, custom protocol 서버가 Pipeline 관점에서 어떻게 달라지는가.
- `AUTO_READ`(읽기 이벤트를 자동으로 계속 당길지), write buffer watermark(쓰기 버퍼 수위), allocator, pooled/direct buffer, native transport 같은 설정이 어떤 상황에서 도움이 되고 언제 오히려 독이 되는가.
- leak detector, 로그, 메트릭, 실험 코드를 이용해 실제 메모리 누수나 이벤트 루프 정체를 어떻게 좁혀 들어갈 수 있는가.
- 아주 간소한 형태로라도 EventLoop, future/promise(비동기 완료 약속), pipeline, buffer abstraction(버퍼 추상화)을 직접 만들어 보며 Netty가 해결하는 문제를 몸으로 이해할 수 있는가.

## 학습 범위

이 프로젝트는 아래 흐름을 따라 기초에서 심화로 올라갑니다.

### 1. 감각 만들기

가장 단순한 blocking server(연결마다 기다리며 처리하는 서버), Java NIO selector 예제, 그리고 최소 Netty echo server를 나란히 보며 "왜 Netty가 필요한가"를 먼저 감각적으로 이해합니다. 이 단계의 핵심은 추상화 이름을 외우는 것이 아니라, 연결 하나와 요청 하나가 실제로 어떻게 흘러가는지 눈으로 따라갈 수 있게 만드는 것입니다.

### 2. Netty 핵심 추상화 이해

`ServerBootstrap`, `EventLoopGroup`, `Channel`, `ChannelPipeline`, `ChannelHandler`, `ChannelFuture`, `ByteBuf` 같은 핵심 추상화를 작은 코드와 로그로 분해해서 이해합니다. 각 추상화는 "무슨 API가 있는가"보다 "어떤 책임을 가지고, 빠지면 무엇이 불편해지는가"를 중심으로 다룹니다.

### 3. 요청 처리 흐름 가시화

요청이 들어와 응답이 나갈 때까지의 흐름을 가능한 한 눈에 보이게 만듭니다. 예를 들어 handler 진입 순서, inbound/outbound 전파, decode/encode 과정, flush 시점, EventLoop thread 이름, channel lifecycle을 화면이나 로그로 보여 주는 식입니다. 목표는 "추상 설명"을 "관측 가능한 흐름"으로 바꾸는 것입니다.

### 4. 프로토콜과 경계 이해

TCP 기반 메시지 경계, framing, partial read/write, sticky packet처럼 초보자가 자주 놓치는 문제를 실험으로 확인합니다. 이후 HTTP, WebSocket, custom binary/text protocol로 확장하면서 codec과 pipeline 설계가 어떻게 달라지는지 비교합니다.

### 5. 메모리와 안정성 이해

`ByteBuf`의 heap/direct 차이, pooled allocator, reference counting, zero-copy 성격의 최적화, leak detection을 다룹니다. 단순히 "주의하세요"라고 끝내지 않고, 실제로 누수를 만들어 보고, 탐지하고, 고치는 실험까지 포함합니다.

### 6. 성능과 튜닝 이해

thread 수, EventLoop 배치, blocking task offload, socket option, allocator, write/read 정책, native transport, batching, flush 전략 같은 주제를 다룹니다. 여기서는 무조건 "빠르게 만드는 법"보다, 어떤 병목을 어떤 관측으로 확인하고 왜 그 설정이 그 병목에만 유효한지를 설명하는 데 초점을 둡니다.

### 7. 내부 개념 재구현

학습의 마지막 단계에서는 EventLoop, future/promise, 간단한 pipeline, 최소 buffer abstraction 같은 요소를 축약해서 직접 구현해 봅니다. 목표는 Netty를 대체하는 프레임워크를 만드는 것이 아니라, Netty의 설계가 어떤 문제를 풀기 위해 어떤 대가를 치르는지 몸으로 이해하는 것입니다.

## 이 프로젝트에서 반드시 동원할 학습 방식

이 프로젝트는 한 가지 방식으로만 설명하지 않습니다. 같은 주제를 여러 각도에서 반복해 이해가 단단해지도록 구성합니다.

- 짧고 실행 가능한 최소 예제
- 단계별로 난도를 올리는 확장 예제
- 요청 흐름을 보여 주는 화면, 로그, 타임라인, 시각화 자료
- 일부러 실패하게 만든 실험과 그 복구 과정
- Java NIO, Reactor Netty, Spring WebFlux 같은 인접 기술과의 비교
- 운영 관점의 문제 재현: 메모리 누수, 느린 응답, EventLoop 블로킹, flush 오용, 잘못된 handler 공유
- 개념을 몸에 붙이기 위한 간단한 재구현 과제

## 산출물의 품질 기준

이 프로젝트에 들어가는 각 자산은 "설명", "코드", "실험", "관측", "검증"이 끊기지 않아야 합니다. 좋은 산출물은 단순히 정답을 말하는 문서가 아니라, 왜 그런 결론이 나오는지와 어떻게 직접 확인할 수 있는지까지 닫혀 있어야 합니다.

앞으로 추가되는 문서나 예제는 가능하면 아래 조건을 만족해야 합니다.

1. 무엇을 이해시키려는지 질문 형태로 분명해야 한다.
2. 작은 실행 코드나 재현 가능한 실험이 있어야 한다.
3. PASS 신호와 FAIL 신호가 보여야 한다.
4. "왜 중요한가"와 "언제 깨지는가"가 함께 설명되어야 한다.
5. 다른 주제와 연결되는 다음 학습 경로가 보여야 한다.

## 비목표

이 프로젝트는 아래 방향을 지향하지 않습니다.

- Netty API를 훑는 얕은 치트시트 모음
- Spring이나 Reactor 뒤에 숨은 Netty를 가린 채 결과만 쓰는 예제 모음
- 근거 없이 "이 설정이 빠르다" 같은 튜닝 팁만 나열하는 문서
- 실행하지 않은 예제를 실행한 것처럼 적는 정리 문서
- 초급과 고급 내용을 섞어 두어 학습 경로가 보이지 않는 자료 더미

## 앞으로의 판단 기준

새 문서, 새 예제, 새 실험을 추가할 때는 항상 이 질문으로 돌아옵니다.

- 이 자산이 Netty를 더 깊이 이해하게 만드는가, 아니면 표면 사용법만 늘리는가.
- 학습자가 요청 처리 흐름을 실제로 관측할 수 있게 만드는가.
- 문제가 났을 때 원인을 좁혀 들어가는 능력을 키워 주는가.
- 기초에서 심화로 이어지는 단계 속 어디에 놓이는지 분명한가.
- 나중에 메모리 누수 분석, 성능 튜닝, 내부 구조 이해, 부분 재구현으로 이어질 발판을 만드는가.

이 문서는 단순 소개문이 아니라, 앞으로 이 프로젝트에 무엇을 넣고 무엇을 넣지 않을지를 판단하는 기준 문서입니다. 모든 예제와 문서는 최종적으로 "Netty를 깊게 이해한 사람만 할 수 있는 질문과 실험"에 가까워지도록 설계합니다.
