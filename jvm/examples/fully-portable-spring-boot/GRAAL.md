# GraalVM와 JVM을 직관적으로 이해하기

> 이 문서는 특정 예제(fully-portable-spring-boot)를 토대로 libjvm·glibc·musl 정적 링킹·배포 전략까지 OS·툴체인 관점에서 깊게 파는 심화 노트다. native-image의 **개념 요약**(closed-world·AOT·빌드타임 초기화·대가)은 [../../java/graalvm_native_image.md](../../java/graalvm_native_image.md)에 따로 정리돼 있다.

## JDK와 JVM: 기본 구조를 먼저 잡기
- JDK = 언어 도구 모음집입니다. `javac`(컴파일러), 표준 라이브러리, 그리고 실행기의 핵심인 JVM(HotSpot)까지 한 꾸러미로 제공합니다.
- JVM(HotSpot) = 바이트코드 실행기입니다. C/C++로 작성된 `libjvm`이 OS 위에서 동작하며, 클래스 로딩 → 검증 → 초기화 → JIT 컴파일 → GC 같은 일을 수행합니다.
- 실행 흐름을 한 번에 그려보면: `javac`가 `.java`를 `.class`(바이트코드)로 → JVM이 클래스를 필요할 때마다 동적으로 로드하고 링크(심볼을 실제 구현으로 연결) → JIT가 자주 쓰이는 바이트코드를 네이티브 코드로 최적화 → GC가 메모리를 관리.

조금 더 낮은 수준에서 보면, 이 모든 것은 OS 위의 “일반적인 네이티브 프로세스”로 돌아갑니다.
- 리눅스 기준으로 `java` 실행 파일은 ELF 바이너리이고, 실행되면 커널이 이 바이너리를 메모리에 올린 뒤, 동적 링커(`ld-linux-x86-64.so.2` 같은 파일)를 통해 필요한 공유 라이브러리(`libjvm.so`, `libc.so.6`, `libpthread.so.0`, `libz.so.1` 등)를 로드합니다.
- 즉, JVM 자체도 결국 C로 작성된 네이티브 프로그램이며, 다른 C 프로그램과 똑같이 glibc 같은 C 표준 라이브러리에 의존합니다. 단지 내부적으로는 바이트코드라는 “가상 CPU용 기계어”를 한 번 더 해석·컴파일한다는 점이 다를 뿐입니다.

여기서 중요한 키워드는 `libc`, 특히 리눅스에서 가장 흔한 구현체인 `glibc`입니다.
- `libc`는 C 언어 표준 라이브러리이면서, 동시에 시스템 콜을 감싸는 얇은 추상화 레이어입니다. 파일 I/O, 스레드, 네트워크, 메모리 할당 같은 저수준 기능을 제공합니다.
- JVM(HotSpot/GraalVM 포함)은 이런 기능을 직접 커널 시스템 콜로 날로 호출하지 않고, 대부분은 `glibc`가 제공하는 API를 통해 사용합니다. 그래서 “JVM을 실행하려면 호환되는 glibc가 필요하다”는 말이 나옵니다.

## GraalVM은 무엇인가
- 하나의 “JDK 배포판”입니다. OpenJDK 위에 Graal 컴파일러와 다국어(Polyglot) 엔진, 그리고 native-image 같은 툴을 더한 번들입니다.
- Graal 컴파일러는 기존 HotSpot의 C2 JIT을 대체하거나 보완하는 고성능/고유연성 JIT입니다.
- Truffle 언어 런타임 덕에 Java 외에 JavaScript, Python, Ruby, R 등도 같은 VM 위에서 돌아갈 수 있습니다.
- GraalVM이 곧 “JDK 25” 같은 버전이 아니라, “JDK 17/21/25 기반으로 빌드된 GraalVM 배포판”이라는 점이 핵심입니다.

## JDK 버전과 GraalVM의 관계
- JDK 버전 번호(17, 21, 25…)는 언어 기능과 표준 라이브러리 세대, 그리고 그 세대에 맞는 HotSpot을 의미합니다.
- GraalVM은 그중 하나의 JDK 기반을 선택해 “GraalVM for JDK 21” 식으로 배포합니다. 즉, GraalVM은 독립적인 JDK가 아니라 특정 JDK를 품고 확장한 배포판입니다.
- 결과적으로 `java -version`을 찍으면 “GraalVM”과 함께 기반 JDK 버전이 함께 표시됩니다.

## JVM이 런타임에 하는 일 (동적 링킹의 감각 잡기)
- 클래스 로딩: 바이트코드가 필요해지는 시점에 클래스 로더가 `.class`나 JAR에서 바이트코드를 읽습니다. 읽은 순간에야 클래스가 JVM 메타데이터로 “현실화”됩니다.
- 검증과 초기화: 바이트코드가 규칙을 어기지 않는지 검증하고, 정적 초기화 블록을 실행하며, 필드에 초기값을 세팅합니다.
- 동적 링킹: 메서드 호출, 필드 참조 같은 심볼을 실제 메모리상의 구현체로 연결합니다. 이 연결은 처음 사용할 때 이루어지고, 이후엔 캐시되어 빠르게 실행됩니다.
- JIT 컴파일: 인터프리터로 먼저 실행하다가, 자주 호출되는 바이트코드 영역을 네이티브 코드로 바꿔 성능을 끌어올립니다.
- 네이티브 라이브러리(JNI): 표준 라이브러리에도 C/C++로 작성된 부분이 있습니다. 예를 들어 압축/해제는 내부적으로 zlib을 사용하며, 이런 라이브러리는 OS의 동적 링커를 통해 로드됩니다 (`libz`, `libc` 등).
- 요약: JVM은 “필요할 때마다 불러와서 바로 묶고, 뜨거워지면 JIT로 바꿔치기”하는 동적 시스템입니다. OS의 동적 링커와도 맞물려 네이티브 라이브러리를 끌어다 씁니다.

OS 입장에서 이 과정을 다시 보면:
- 애플리케이션의 바이트코드를 실행하는 주체가 `libjvm.so`이고, 이 `libjvm.so`는 다른 공유 라이브러리들(`libc`, `libpthread`, `libz` 등)에 동적으로 의존합니다.
- 프로세스가 시작될 때 동적 링커(`ld-linux-...`)는 ELF 바이너리 안의 “이 프로그램은 어떤 라이브러리를 필요로 한다(예: NEEDED: libjvm.so, libc.so.6 …)” 정보를 읽고, 각각을 찾아 메모리에 매핑(mmap)합니다.
- 그런 뒤 각 라이브러리 내의 심볼(함수/전역변수 주소)을 재배치(relocation)해서, 코드 안에서의 함수 호출이 실제 메모리 주소를 가리키도록 “동적 링킹” 과정을 수행합니다.

JVM이 말하는 “동적 링크”(클래스/메서드 해석)와 OS 수준의 “동적 링킹”(공유 라이브러리 로드)은 레이어가 다릅니다.
- OS 동적 링킹: ELF/DLL 수준에서 `libjvm`, `libc`, `libz` 같은 네이티브 라이브러리를 런타임에 올리고, 심볼을 연결합니다.
- JVM 동적 링크: 로딩된 네이티브 JVM 위에서 `.class` 파일들의 메서드/필드 참조를 해석하고, 호출점(call site)을 실제 구현체에 바인딩합니다.
- native-image는 이 두 세계(네이티브 라이브러리 레벨의 링크 + 자바 클래스 레벨의 링크)를 가능한 한 빌드 시점으로 끌어당겨 “정적인 그림”으로 만들어 버리려는 시도라고 볼 수 있습니다.

## Native Image가 하는 일 (동적 세계를 정적으로 얼리기)
- 네이티브 이미지 빌더는 애플리케이션과 필요한 JDK 부분을 AOT(사전 컴파일)로 분석해 닫힌 세계(closed world) 모델을 만듭니다.
- 리플렉션, 동적 프록시, 리소스 로딩 등 “런타임에 결정될 것들”을 빌드 시점에 최대한 확정하려고 합니다. 이때 힌트를 주기 위한 설정 파일(리플렉션 설정 등)이 필요할 수 있습니다.
- 초기 힙과 메타데이터를 빌드 단계에서 만들어 실행 파일에 패킹합니다. 실행 시에는 “이미 준비된 세계”를 바로 펼치는 느낌이라 기동이 매우 빠릅니다.
- 결과물은 OS가 바로 실행할 수 있는 ELF/Mach-O/PE 바이너리이며, JVM을 통째로 띄우지 않고도 GC, 스레딩, 런타임 지원 코드가 함께 들어 있습니다.

툴체인 관점에서 native-image 안에서 실제로 무슨 일이 일어나는지 단계별로 보면:
- 1단계(분석): Graal 컴파일러가 애플리케이션의 바이트코드를 훑으면서 “이 함수는 실제로 호출되는가?”, “이 클래스의 어떤 메서드들이 살아남는가?”를 분석합니다. 이때 리플렉션/동적 프록시/서비스 로딩 같은 “간접 호출”은 사용자가 제공한 설정 파일을 통해 힌트를 줘야 정확히 파악할 수 있습니다.
- 2단계(코드 생성): 살아남은 메서드들에 대해 네이티브 코드를 생성합니다. 여기서 나오는 결과물은 종종 중간 형태(예: 오브젝트 파일, LLVM 비트코드 등)로 만들어지고, 이후 시스템 툴체인(`gcc`, `clang`, `ld`)이 이해할 수 있는 형식으로 변환됩니다.
- 3단계(링킹): 마지막에 OS의 링크 도구(`ld`)를 호출해 하나의 실행 파일로 묶습니다. 이때 `-static`, `-static-libgcc`, `-Wl,--gc-sections` 같은 옵션이 사용될 수 있고, zlib이 필요하다면 `-lz` 같은 플래그가 함께 전달됩니다. 즉, native-image의 “마지막 한 방”은 결국 OS의 일반적인 C/C++ 빌드 파이프라인과 동일합니다.
- 4단계(이미지 내 초기화): 일부 정적인 초기화 코드(예: Spring 컨텍스트 생성)를 빌드 시점에 미리 실행해 그 결과를 힙 스냅샷 형태로 실행 파일에 저장하기도 합니다. 이렇게 하면 실행 시에 다시 계산할 필요가 없어서 시작 시간이 극적으로 짧아집니다.

## Native Image와 리플렉션/동적 클래스 로딩

여기서 자주 헷갈리는 지점이 “native-image로 빌드하면 리플렉션은 다 죽는가?”, “JAR/WAR를 나중에 읽어서 클래스를 로딩하는 건 완전히 불가능한가?”입니다.

### 리플렉션은 “제한된 형태로” 계속 가능하다

- native-image의 핵심 가정은 **closed world**입니다.  
  → “이미지 빌드 시점에 도달 가능한(reachable) 모든 코드와 타입을 다 알고 있다”는 전제 위에서 동작합니다.
- `Class.forName`, `SomeType.class.getDeclaredMethods()` 같은 리플렉션 호출 자체는 막히지 않습니다. 다만:
  - **그 대상 타입/멤버가 이미지 빌드 시점에 이미 존재하고**,  
  - 그 타입/멤버를 리플렉션으로 쓸 것이라는 사실을 **리플렉션 설정(`reflect-config.json` 등)으로 미리 알려줘야** 합니다.
- Graal은 이 설정을 보고:
  - 해당 클래스를 이미지 안에 포함시키고,
  - 필드/메서드/생성자 메타데이터를 버리지 않으며,
  - 필요하면 `Reflection.register(...)` 등을 통해 리플렉션 API가 사용할 수 있게 준비합니다.
- 반대로, 빌드 시점에 전혀 도달 불가능한 타입(예: 플러그인 JAR에만 들어 있고, classpath에도 없던 클래스)을 런타임에 갑자기 문자열로 만들어 `Class.forName("com.foo.Plugin" + userInput)` 하는 식의 패턴은 설계상 지원되지 않습니다.

정리하면:
- “리플렉션이 아예 안 된다”가 아니라,  
  → **“미리 선언된 범위 안에서, 빌드 시점에 알려진 타입/멤버에 대해서만 리플렉션이 가능하다”**가 정확한 표현입니다.

### JAR/WAR를 읽어서 새로운 클래스를 로딩하는 것은?

- JAR/WAR 파일을 **바이트 수준으로 읽는 것 자체**는 당연히 가능합니다. 그냥 파일 I/O일 뿐이니까요.
- 문제는 그 안에서 **새로운 바이트코드를 꺼내서 JVM처럼 `defineClass`로 올려 실행하는 패턴**입니다.
  - 일반 HotSpot JVM에서는 `URLClassLoader`, `ClassLoader.defineClass` 등을 써서, 실행 중에 새로운 JAR/WAR를 추가하고 그 안의 클래스를 로딩하는 것이 가능합니다(플러그인, 앱 서버의 동적 배포 등).
  - native-image 안에서는 바이트코드를 해석/컴파일해 실행하는 JVM이 더 이상 존재하지 않습니다. 컴파일된 것은 **이미지 안에 들어 있는 코드**뿐입니다.
- 결과적으로:
  - **이미지 빌드 시 classpath에 없던 JAR/WAR**에서 클래스를 꺼내서 실행하는 모델은 사실상 안 된다고 보면 됩니다.
  - 빌드 시 classpath에 있었고, native-image가 reachability 분석에서 “이 클래스는 쓸 수 있다”고 판단한 것들만 실제 바이너리에 들어갑니다. “나중에 들어올지도 모르는 플러그인 JAR”는 이 세계관에 맞지 않습니다.

그래서:
- “실행 중에 새로운 JAR를 떨궈 넣으면 그걸 classpath에 추가해서 즉시 기능이 늘어난다”는 전통적인 JVM 스타일은 native-image에서는 직접 재현하기 어렵습니다.
- 이 지점이 바로 “정적 세계(native-image)”와 “동적 세계(일반 JVM)”의 가장 큰 구조적 차이입니다.

## 정적 링킹과 이식성 이야기
- 동적 링킹: 실행 시 OS가 필요한 공유 라이브러리를 찾습니다. 호스트 환경(glibc 버전, 시스템 zlib 등)에 따라 달라지므로 배포 대상마다 “호환 여부”를 신경 써야 합니다.
- 정적 링킹: 필요한 라이브러리를 빌드 결과물에 포함시켜 실행 파일 하나로 끝내는 방식입니다. 네이티브 이미지가 정적 링킹을 시도하면, glibc나 zlib 같은 필수 라이브러리를 바이너리에 묶어 OS 의존도를 줄입니다.
- 실제 제약:
  - Linux: glibc는 완전 정적 링킹이 까다롭습니다. GraalVM은 musl 기반 이미지를 지원해 “정적에 가까운” 결과물을 만들 수 있고, 이를 통해 다른 리눅스 배포판에서도 잘 동작하도록 합니다.
  - macOS: 완전 정적 링킹이 제한적이므로 시스템 라이브러리에 일부는 의존합니다. 그래도 JVM 전체를 요구하지 않기 때문에 배포 부담이 크게 줄어듭니다.
  - Windows: PE 포맷으로 빌드하며 MSVC 런타임 등에 대한 의존성을 최소화하려고 하지만 완전 정적은 아닐 수 있습니다.
- zlib 같은 라이브러리는 JAR 압축 해제, HTTP 압축 응답 처리 등에서 필요합니다. 네이티브 이미지로 빌드하면 이 라이브러리도 바이너리에 포함하거나, 정적/동적으로 링크합니다. 정적으로 묶이면 대상 환경에 zlib이 없어도 실행됩니다.
- 핵심 메시지: 동적 세계(JVM + OS 링커)를 빌드 타임에 최대한 얼려 넣어, 실행 시 “한 덩어리”로 가져가면 배포와 기동이 쉬워지고, 호환성 걱정이 줄어듭니다.

여기서 “왜 동적 링킹이 처음에 존재했는가?”를 짚고 넘어가면 이해가 더 잘 됩니다.
- 초창기 유닉스/리눅스 환경에서 모든 프로그램이 필요한 라이브러리를 정적으로 포함하면, 디스크 용량과 메모리 사용량이 폭발적으로 늘어납니다. 모든 바이너리가 자기만의 `printf`, 자기만의 `malloc`을 들고 다니는 꼴이기 때문입니다.
- 그래서 `libc`, `libm`, `libz` 같은 공용 라이브러리를 공유 라이브러리(.so)로 만들어 놓고, 여러 프로세스가 이를 함께 사용하도록 한 것이 동적 링킹입니다. 이 방식은 메모리와 디스크 측면에서 매우 효율적입니다.
- 또 다른 이유는 유지보수입니다. 보안 버그가 발견된 `libc`를 하나 고치면, 그 라이브러리에 의존하는 모든 프로그램이 다음 실행 시 자동으로 새 버전을 사용하게 됩니다. 다시 재컴파일할 필요가 없습니다.

하지만 이런 동적 링킹에는 “환경에 따라 미묘하게 다르게 동작한다”는 그림자가 있습니다.
- 어떤 서버에는 glibc 2.17, 다른 서버에는 2.31이 설치되어 있을 수 있습니다. 바이너리가 특정 버전에만 존재하는 심볼(함수)을 사용한다면, 낮은 버전의 glibc에서는 실행이 실패합니다.
- zlib도 마찬가지입니다. 빌드할 때는 `libz.so.1.2.13`에 맞춰 링크했는데, 런타임 환경에는 오래된 `libz.so.1.2.7`만 있다면, 심볼 버전 불일치로 실행이 실패할 수 있습니다.

Graal native-image가 정적 링킹(또는 정적에 가깝게)을 하고 싶어하는 이유는 바로 이 “환경 의존성”을 줄이고 싶어서입니다.
- 가능한 많은 라이브러리를 바이너리 안에 같이 넣어 버리면 “이 바이너리는 그 자체로 완결된 세계”에 가까워집니다.
- 특히 리눅스에서 musl 같은 C 라이브러리를 사용해 완전 정적 링크를 하면, 호스트 시스템의 glibc 버전과 상관없이 동작하는 “portable Linux 바이너리”에 가까운 결과를 얻을 수 있습니다.
- 현실적으로는 100% 정적이 아닌 경우도 많지만, 최소한 “JVM 설치 필요”와 “정확한 JDK 버전 맞추기”라는 큰 짐을 벗겨 준다는 점에서 의미가 큽니다.

### glibc와 libc의 역할을 조금 더 깊게
- glibc는 GNU C Library의 줄임말로, 리눅스에서 가장 널리 쓰이는 `libc` 구현입니다. POSIX에서 정의하는 함수들(`open`, `read`, `write`, `pthread_create` 등)을 제공하고, DNS/NSS/로케일 같은 부가 기능도 담당합니다.
- C/C++로 작성된 대부분의 프로그램(당연히 JVM도 포함)은 glibc에 링크되어 빌드됩니다. 정적 링크를 하든 동적 링크를 하든, “이 프로그램이 호출하는 시스템 콜과 런타임 지원 코드를 누가 제공하냐?”의 답이 glibc입니다.
- static 링크의 경우 `libc.a` 같은 정적 라이브러리에서 필요한 오브젝트만 골라 바이너리에 복사해서 넣습니다. dynamic 링크의 경우 `libc.so.6` 같은 공유 라이브러리에 대한 “참조”만 남고, 실행 시 동적 링커가 그 라이브러리를 찾아 메모리에 올립니다.
- glibc는 DNS/NSS 같은 부분에서 플러그인 구조(dlopen 기반)를 많이 사용하기 때문에 “완전한 정적 링크”가 어려운 편입니다. 이 때문에 리눅스에서는 musl 같이 “정적 링크 친화적인 libc”가 등장했고, GraalVM도 이런 라이브러리와 잘 맞물리도록 발전해 왔습니다.

### 동적 링커(ld-linux)와 ld의 차이
- `ld`는 링크 시점에 실행되는 정적 도구입니다. 오브젝트 파일(.o)들과 라이브러리(.a/.so)를 입력으로 받아 하나의 실행 파일 또는 공유 라이브러리를 만듭니다.
- `ld-linux-*.so`(혹은 `ld.so`)는 실행 시점에 동작하는 “동적 링커/로더”입니다. ELF 바이너리의 인터프리터 필드(`PT_INTERP`)에 경로가 적혀 있고, 커널이 프로그램을 시작할 때 이 동적 링커부터 실행합니다.
- native-image로 빌드할 때는 Graal이 내부적으로 `ld`를 호출해 하나의 바이너리를 만들고, 완성된 바이너리는 실행될 때 다시 `ld-linux`에 의해 필요한 공유 라이브러리를 로드하게 됩니다. 완전 정적일 경우에는 `ld-linux`의 역할이 최소화됩니다(필수적이진 않아질 수 있음).

## 공유 라이브러리(glibc, libpthread, libz)의 구체적인 역할

## GraalVM Native Image를 설명할 때의 한 문장 버전
- “자바 앱과 필요한 JDK 조각, GC, 스레드, 압축 라이브러리(zlib)까지 한 덩어리로 사전 컴파일해 두면, 실행 시 OS가 추가로 찾을 게 거의 없어서 빠르고 이식성 높은 단일 실행 파일이 된다.”

이 한 문장 속에 담긴 의미를 OS 관점에서 풀면:
- “자바 앱과 필요한 JDK 조각” = 바이트코드와, 그 바이트코드를 실행하던 `libjvm`의 상당 부분을 네이티브 코드로 뽑아낸 결과물입니다.
- “GC, 스레드” = 원래 `libjvm` 안에 있던 가비지 컬렉터와 스레드 스케줄링 관련 런타임 코드가, 이제는 하나의 실행 파일 안에 정적으로 포함되어 있다는 뜻입니다.
- “압축 라이브러리(zlib)” = JAR/ZIP/HTTP 압축에 필요했던 zlib도 더 이상 호스트 시스템에서 찾지 않고, (가능하다면) 함께 링크해 들고 다닌다는 의미입니다.

## 필수 개념 정리 (말로 풀어보기)
- GraalVM은 “JDK 확장 배포판”이다: 특정 JDK(17/21/25 등)를 품고, Graal JIT·Polyglot·native-image 도구를 추가한 번들.
- JVM의 동적 본질: 클래스와 네이티브 라이브러리를 필요할 때 로드·링크하고, 뜨거운 코드를 JIT로 최적화한다.
- native-image의 정적 본질: 닫힌 세계 분석으로 동적 결정을 빌드 시점에 확정하고, 필요한 라이브러리를 (가능하면 정적으로) 바이너리에 묶어 배포한다.
- 이식성의 핵심: glibc 같은 시스템 의존성을 줄여야 진짜 “한 파일로 어디서나”가 가까워진다. Linux에서는 musl, macOS/Windows에서는 일부 시스템 의존성이 남지만 JVM 설치 의존성은 사라진다.

여기에 OS/툴체인 레벨의 한 줄 요약을 더 붙이면 다음과 같습니다.
- “JVM 모드”에서는: `java`(ELF) → 커널 → `ld-linux` → `libjvm.so`, `libc.so.6`, `libz.so.1` … → 그 위에서 바이트코드 실행.
- “native-image 모드”에서는: `myapp`(ELF, Graal native image) → 커널 → (가능하면 최소한의 공유 라이브러리만) → 바로 애플리케이션 메인 진입.  
  둘 다 결국은 ELF + glibc 위에 서 있지만, native-image는 그 위에 있던 “바이트코드 VM” 층을 빌드 타임에 접어서 바이너리 안에 넣어버렸다고 볼 수 있습니다.

## Native Image와 플러그인/확장 구조

여기서 자연스럽게 나오는 질문이 “그럼 plug & play, 플러그인 아키텍처는 완전히 불가능한가?”입니다.  
답은 “예전 JVM 스타일 그대로는 어렵지만, **설계를 조금 바꾸면 다른 형태의 plug & play는 가능**하다”에 가깝습니다.

### 1) 완전 동적 플러그인(JAR drop-in)은 사실상 불가능

- 전통적인 JVM 플러그인 모델을 떠올려 보면:
  - `plugins/` 디렉터리에 JAR를 하나 놓고,
  - 부팅 시 또는 런타임에 해당 JAR를 스캔해서,
  - `ServiceLoader`/리플렉션/클래스 로더로 플러그인을 찾아 올린 뒤,
  - 인터페이스(예: `Plugin`)를 구현한 클래스를 실행합니다.
- 이 모델의 전제는 “JVM이 언제든지 새로운 JAR를 classpath에 추가하고, 그 안의 바이트코드를 로딩해 실행할 수 있다”는 것입니다.
- native-image는 **이미지 빌드 시점에 classpath가 고정**되고, 그때 분석된 클래스들만 바이너리에 들어갑니다.  
  → 따라서 “나중에 등장할지도 모르는 JAR의 바이트코드”를 로딩하는 방식의 플러그인은 같은 프로세스 안에서는 재현하기 어렵습니다.

### 2) 그래도 “유연성”을 얻는 몇 가지 패턴

완전히 동일한 경험은 아니지만, 실무에서 자주 쓰는 타협안/패턴들을 정리해 보면:

1. **모든 플러그인 코드를 이미지 안에 넣되, 설정으로 on/off**
   - 플러그인 후보들을 하나의 모듈 집합으로 보고, 모두 classpath에 올려 둔 채 native-image를 빌드합니다.
   - 런타임에는 설정 파일/DB/플래그를 보고 “이 플러그인은 활성화/비활성화”를 결정합니다.
   - 장점: 단일 바이너리, 실행 구조 단순, Graal 친화적.
   - 단점: “플러그인을 나중에 JAR로 배포해 추가”하는 경험은 아니라는 점, 필요 없는 플러그인 코드도 바이너리 안에 들어간다는 점(다만 코드 제거 최적화로 일부는 날아갈 수 있음).

2. **플러그인을 별도 프로세스로 분리**
   - 코어 로직을 native-image로 빌드된 바이너리로 제공하고,
   - 플러그인들은 **별도의 JVM 프로세스**(혹은 또 다른 native-image)로 띄워서 IPC로 통신합니다(HTTP/gRPC/STDIN/STDOUT, Unix socket 등).
   - 사용자는 겉으로 보기엔 “플러그인 JAR를 하나 추가하면 기능이 늘어난다”는 경험을 유지할 수 있습니다.
     - 코어 바이너리는 “`plugins/*.jar`를 찾아서 `java -cp plugin.jar com.example.PluginMain`을 실행”하거나,
     - 또는 “플러그인 서버가 제공하는 HTTP/gRPC 엔드포인트를 호출”하는 식으로 동작합니다.
   - 장점: 플러그인 쪽은 여전히 완전한 JVM 세계(동적 클래스 로딩, 리플렉션 자유롭고, 기존 라이브러리 그대로 사용).
   - 단점: 프로세스 경계(직렬화/통신 비용, 배포 복잡도).

3. **플러그인을 “다른 native-image 바이너리”로 취급**
   - 각 플러그인을 별도 native-image로 빌드해, `plugins/`에 바이너리 형태로 둡니다.
   - 코어 바이너리는 사용자가 특정 기능을 요청하면 해당 플러그인 바이너리를 프로세스로 실행합니다.
   - 장점: 모든 코드가 native-image의 이점을 누립니다(빠른 기동, 작은 footprint).
   - 단점: 플러그인 간 코드/GC 공유가 안 되고, 프로세스 수가 늘어납니다. JVM 스타일의 “같은 프로세스 안에서 동적 확장”과는 느낌이 다릅니다.

4. **스크립트/DSL 기반 플러그인**
   - 코어를 native-image로 만들고, 플러그인 부분을 **스크립트 언어(예: JavaScript, Lua, Kotlin script)**로 두는 방식입니다.
   - core 바이너리 안에 해당 스크립트 인터프리터를 포함시키고(또는 GraalVM polyglot 기능을 사용해서),
   - 플러그인 배포는 “스크립트 파일”로 합니다.
   - 장점: 코어는 AOT의 이점을 모두 누리면서, 플러그인은 상당히 동적으로 추가/수정 가능.
   - 단점: 플러그인이 자바 코드처럼 정적 타입/IDE 지원을 얻기 어렵고, 성능·디버깅 모델이 달라집니다.

### 3) “core는 native-image, 플러그인은 JAR”라는 그림에 대한 현실적인 답

질문에서 고민하신 그림을 그대로 적어 보면:
- “core 기능은 native-image 바이너리로 제공하고,  
   특정 케이스별 추가 모듈은 JAR 등으로 제공해서  
   프로세스 실행 중에 추가/제거가 간편했으면 좋겠다.”

이 요구를 **완전히 같은 프로세스 안에서, 기존 JVM 플러그인처럼** 만족시키긴 어렵습니다.  
이유는:
- native-image 바이너리 안에는 “새로운 JAR의 바이트코드를 읽어들여 곧바로 실행할 수 있는 JVM”이 더 이상 들어 있지 않기 때문입니다.

대신 선택지는 다음 둘 중 하나로 수렴합니다.
- **(A) 기능은 모두 이미지 안에 넣되, 사용 여부만 동적으로 제어한다.**
  - 코드 관점에서는 정적이지만, 비즈니스 관점에서는 “설정으로 플러그인 on/off”가 가능합니다.
  - 배포 관점에서 “하나의 바이너리만 배포하면 된다”는 단순함을 얻습니다.
- **(B) 플러그인은 별도 프로세스로 분리해서 동적으로 추가한다.**
  - core는 native-image로, 플러그인은 JVM(JAR) 또는 또 다른 native-image로 두고,
  - core가 런타임에 플러그인 프로세스를 실행/종료하면서 기능을 확장합니다.
  - 사용자는 “플러그인 아카이브를 추가하면 기능이 늘어난다”는 plug & play 경험을 유지합니다.

요약하면:
- **native-image 내부에서 “완전 동적 JAR 플러그인”은 포기해야 할 것**이고,
- 대신 “AOT된 코드 안에서 설정 기반 유연성” 또는 “프로세스 경계를 활용한 플러그인” 중 하나(또는 적절한 혼합)를 선택하는 것이 현실적인 설계 방향입니다.

## 공유 라이브러리(glibc, libpthread, libz)의 구체적인 역할

여기까지는 “glibc가 필요하다”, “libpthread에 의존한다” 수준에서 이야기했습니다. 이제는 각각이 **무엇을 언제, 왜** 하는지를 `java -jar SpringApp.jar`라는 구체적인 시나리오에 붙여서 보겠습니다.

### glibc: 시스템 콜 위의 사용자 공간 런타임

- 커널과 직접 대화하는 최소 단위는 시스템 콜입니다(`syscall`/`int 0x80` 등). 하지만 직접 시스템 콜 번호를 넣고 호출하는 코드를 쓰는 것은 번거롭고, 커널 버전·아키텍처마다 차이가 있습니다.
- glibc는 이 시스템 콜들을 함수 형태로 감싼 레이어입니다. 예를 들어:
  - 파일 I/O: `open`, `read`, `write`, `close`, `stat` …
  - 메모리: `malloc`, `free`, `mmap`, `munmap`, `brk` …
  - 시간: `clock_gettime`, `gettimeofday` …
  - 프로세스: `fork`, `execve`, `waitpid`, `getpid` …
  - 네트워크: `socket`, `connect`, `accept`, `send`, `recv` …
- 이 함수들은 내부에서 **적절한 시스템 콜을 호출**하거나, vDSO 같은 커널 제공 최적화 경로를 사용합니다. 즉, 대부분의 C 코드(그리고 그 위에 있는 JVM)는 시스템 콜을 거의 직접 쓰지 않고 glibc를 거칩니다.

프로세스 시작 시점에도 glibc는 깊게 관여합니다.
- `execve("/usr/bin/java", ...)`가 호출되면 커널이 ELF 헤더를 읽고, 프로그램의 엔트리 포인트(사실상 glibc가 제공하는 스타트업 코드 `_start`)로 점프합니다.
- `_start`는 스택을 정리하고, TLS(Thread-Local Storage)와 런타임 환경을 초기화한 뒤, `__libc_start_main(main, ...)`을 호출합니다.
- `__libc_start_main`이 실제 `main(int argc, char** argv)`를 호출하기 전에, 전역/정적 초기화자, `atexit` 처리, 로케일 설정 등 glibc가 맡은 일을 마무리합니다.
- `java` 실행 파일의 `main` 함수(실제론 JDK의 런처 코드)는 바로 이 glibc의 도움을 받은 뒤에야 실행을 시작합니다. 즉, `java -jar`를 치는 순간부터 glibc는 이미 일을 하고 있습니다.

JVM 내부에서도 glibc는 계속 사용됩니다.
- JVM은 큰 힙 영역을 확보하기 위해 `mmap`/`munmap` 같은 함수를 사용합니다(대체로 glibc 래퍼를 통해). HotSpot/GraalVM이 자체 메모리 관리 정책을 갖고 있더라도, OS 레벨 메모리 확보는 glibc 경유입니다.
- 클래스 파일이나 JAR를 읽을 때는, 결국 `open`/`read`/`close`를 사용합니다. 로깅 출력도 `write`/`fwrite` 등으로 흘러갑니다.
- 소켓 열기, DNS 조회, 호스트 이름 해석 같은 네트워크 관련 작업도 큰 틀에서 glibc가 담당합니다(NSS, resolver 코드 포함).

정리하면:
- glibc는 **프로세스가 시작되는 순간부터 종료할 때까지 계속 쓰이는 공기 같은 존재**입니다.
- `java -jar SpringApp.jar`를 실행할 때도, JVM이 뜨기 **이전**부터 glibc가 스타트업 코드와 시스템 콜 래퍼로 프로세스를支え주고, JVM이 뜬 뒤에도 파일/네트워크/시간/메모리 관련 거의 모든 OS 상호작용에 등장합니다.

### libpthread: POSIX 스레드와 자바 스레드

전통적인 리눅스/유닉스에서 스레드는 `libpthread`가 제공하는 POSIX Threads(API 이름은 `pthread_*`)로 표현됩니다.
- `pthread_create`, `pthread_join`, `pthread_mutex_lock`, `pthread_cond_wait`, `pthread_rwlock_*`, `pthread_key_create`(TLS) 같은 함수들이 여기에 속합니다.
- 이 함수들은 내부에서 커널의 `clone`(또는 `clone3`) 시스템 콜을 사용해 새로운 스레드를 만들고, `futex` 시스템 콜을 사용해 뮤텍스/조건 변수 대기·깨우기를 구현합니다.
- 현대 glibc에서는 `libpthread`와 glibc의 경계가 많이 섞여 있지만, 개념적으로는 “스레드 지원”을 담당하는 별도 라이브러리로 보는 것이 이해에 좋습니다.

JVM에서의 쓰임새를 단계적으로 보면:
- JVM이 초기화될 때부터, GC 스레드, JIT 컴파일 스레드, 신호 처리 스레드 등 여러 **VM 내부용 스레드**를 만듭니다. 이때 HotSpot/GraalVM의 C++ 코드 안에서 `pthread_create`를 호출합니다(보통 `os::create_thread` 같은 추상화를 거쳐서).
- 애플리케이션 코드에서 `new Thread().start()`를 호출하면:
  - 자바 레벨에서는 `java.lang.Thread`가 만들어지고,
  - VM 내부 구현에서 이 자바 스레드를 담을 **OS 스레드**가 필요해지며,
  - 그 순간에 다시 `pthread_create`가 호출되어 커널 레벨 스레드가 하나 더 생성됩니다.
- 즉, **자바의 “플랫폼 스레드” 하나가 곧 커널 스레드(pthread) 하나**라고 생각하면 크게 틀리지 않습니다(HotSpot은 1:1 매핑).
- 자바 “가상 스레드”(Project Loom)는 수십만 개를 만들 수 있지만, 결국 이들은 훨씬 소수의 “캐리어 스레드” 위에서 스케줄됩니다. 이 캐리어 스레드 역시 `pthread_create`로 만들어진 진짜 커널 스레드입니다.

언제 libpthread가 필요하냐고 물으면:
- 프로세스 시작 직후에는 단 하나의 메인 스레드만 존재합니다. 이 시점에는 `libpthread` 기능이 아직 거의 쓰이지 않습니다.
- JVM 런타임을 초기화하면서 첫 GC 스레드/컴파일 스레드 등을 만들 때부터 `libpthread`가 본격적으로 사용됩니다.
- 애플리케이션이 스레드를 만들고 동기화할 때마다, 그 밑바닥에는 `pthread_*` 호출과 `futex` 시스템 콜이 깔려 있다고 보면 됩니다.

### libz: JAR/HTTP 압축에서의 역할

zlib 자체는 “deflate 압축 알고리즘을 구현한 C 라이브러리”입니다. `libz.so`/`libz.a` 형태로 배포되고, `deflate`, `inflate` 같은 함수들을 제공합니다.

JVM/JDK와 Spring 앱 관점에서 “언제” 쓰이는지 살펴보면:
- `java -jar SpringApp.jar`를 실행하면, 런처는 먼저 **JAR 파일을 glibc의 `open`/`read`로 읽어들인 뒤**, JDK의 `java.util.jar`/`java.util.zip` 코드가 중앙 디렉터리와 엔트리를 파싱합니다.
- JAR의 각 엔트리는 deflate로 압축되어 있을 수 있고, 이때 자바 코드에서 실제 바이트를 풀어내는 단계에 들어가면, JNI를 통해 네이티브 zlib의 `inflate` 계열 함수가 호출됩니다.
- Spring Boot fat JAR의 경우, `BOOT-INF/lib/*.jar`처럼 “JAR 안에 JAR” 구조가 있기 때문에, 부트스트랩 로더가 이 중첩된 JAR들을 읽고 풀어내는 과정에서 zlib이 반복적으로 사용됩니다.
- HTTP 클라이언트/서버에서도 `Content-Encoding: gzip` 등의 헤더를 처리할 때 zlib을 사용합니다. 이 역시 자바 코드 위에서 동작하지만, 실질적인 압축/해제는 libz에 위임되는 경우가 많습니다.

따라서:
- zlib은 JVM이 뜨는 그 순간에 **필수는 아니지만**, JAR/ZIP/HTTP 같은 압축 데이터를 다루는 첫 순간부터 본격적으로 참여합니다.
- Spring 애플리케이션에서는 “JAR에서 클래스를 읽는 거의 모든 순간”에 zlib이 관여한다고 생각해도 무방합니다. 클래스 로딩, 리소스 로딩, HTTP 통신 등.

### `java -jar SpringApp.jar` 시나리오로 일련의 흐름 정리

이제 glibc/libpthread/libz가 **언제, 어떻게** 쓰이는지를 하나의 타임라인으로 정리해 보면:
- 1단계(쉘 → 커널):
  - 사용자가 `java -jar SpringApp.jar`를 입력합니다.
  - 쉘이 `execve("/usr/bin/java", ["java","-jar","SpringApp.jar"], env)`를 호출합니다.
- 2단계(커널 → 동적 링커 → glibc):
  - 커널이 `/usr/bin/java` ELF를 로드하고, 인터프리터 필드(동적 링커 경로)를 읽어 해당 `ld-linux-*.so`를 메모리에 올립니다.
  - 동적 링커가 `NEEDED` 항목을 보고 `libc.so.6`, `libpthread.so.0`, `libjli.so`, 필요하다면 `libz.so.1` 등을 로드합니다.
  - glibc 스타트업 코드 `_start` → `__libc_start_main`이 실행되며, 최종적으로 JDK 런처의 `main` 함수가 호출됩니다. 이 시점부터 glibc는 적극적으로 사용 중입니다.
- 3단계(Java 런처 → JVM 초기화):
  - 런처는 인자/환경을 파싱하고, `dlopen("libjvm.so")`를 호출해 JVM을 동적으로 로드합니다(`libdl`/glibc 사용).
  - JVM 초기화 코드가 실행되면서, 힙을 위해 `mmap`, 스레드 스택을 위해 추가 메모리 확보, 시그널 핸들러 설치 등 다양한 OS 호출을 glibc 함수를 통해 수행합니다.
  - GC 스레드, 컴파일 스레드 등 JVM 내부 스레드를 만들기 위해 `pthread_create`가 호출되기 시작합니다. 이때부터 libpthread가 실전 투입입니다.
- 4단계(JAR 로딩 및 Spring 부트스트랩):
  - JVM이 `-jar SpringApp.jar` 옵션을 이해하고, 해당 JAR를 열어 `MANIFEST.MF`와 엔트리들을 읽습니다. 여기까지는 `open`/`read` 등 glibc I/O가 담당합니다.
  - 압축된 엔트리(클래스/리소스)를 읽을 때, `java.util.zip.*` 코드가 내부에서 네이티브 zlib 함수(`inflate` 등)를 호출합니다. 이때 libz가 사용됩니다.
  - Spring Boot 로더가 fat JAR 구조를 해석하면서, 중첩된 JAR들을 다시 열고 해제하는 과정에서도 libz가 계속 등장합니다.
- 5단계(애플리케이션 런타임):
  - 사용자가 `new Thread()`나 `Executors.newFixedThreadPool`로 스레드를 만들면, 결국 JVM 내부에서 다시 `pthread_create`가 호출되어 커널 스레드가 생깁니다.
  - HTTP 요청/응답, 파일/DB I/O, 로그 출력 등은 모두 glibc의 네트워크/파일/버퍼링 API를 타고 커널 시스템 콜로 이어집니다.
  - gzip/zip 같은 압축 데이터가 오고 가는 순간마다 libz가 압축/해제를 담당합니다.

질문으로 돌아가면:
- “glibc, libpthread, libz는 JVM이 뜰 때 필요한가요, 아니면 런타임에 필요한가요?”에 대한 답은:
  - glibc: **프로세스 시작부터 종료까지 내내** 필요합니다.
  - libpthread: JVM이 내부/사용자 스레드를 만들기 시작하는 **초기 런타임 시점부터** 종료할 때까지 필요합니다.
  - libz: 압축된 데이터를 다루기 시작하는 순간(대개 JAR 로딩 시점부터), 그리고 그 이후의 모든 압축 관련 작업에 필요합니다.

## zlib과 `ld -lz`를 자세히 보기
- zlib은 deflate 알고리즘을 구현한 범용 압축 라이브러리입니다. gzip/zip의 핵심 알고리즘이기도 하고, HTTP 콘텐츠 압축에도 널리 사용됩니다.
- Java 세계에서는 `java.util.zip` 패키지, JAR/ZIP 파일 압축 해제, HTTP 클라이언트/서버의 압축 응답 처리 등 엄청나게 많은 곳에서 zlib 기능이 필요합니다.
- JVM/JDK 구현 입장에서는 zlib을 순수 자바로만 구현하기보다는, 이미 검증된 C 라이브러리 zlib에 JNI로 연결하는 것이 성능·안정성 측면에서 유리합니다. 그래서 `libzip` 또는 `libz`에 의존하는 네이티브 코드가 들어갑니다.

링크 타임에 `-lz` 옵션을 주는 것은:
- `ld` 입장에서 `-l<name>`은 “lib<name>.so 또는 lib<name>.a를 찾아 링크해라”라는 뜻입니다.
- `-lz`는 “`libz.so` 또는 `libz.a`를 찾아서, 이 오브젝트/라이브러리가 필요로 하는 심볼(예: `deflate`, `inflate`)을 거기서 해결하라”는 의미입니다.
- 동적 링크 시(`libz.so`): 최종 바이너리에는 “이 프로그램은 libz.so.1에 의존한다”는 정보만 기록됩니다. 실행할 때 동적 링커가 이 라이브러리를 찾아 메모리에 올립니다.
- 정적 링크 시(`libz.a`): `libz.a` 안에 있는 오브젝트 파일들이 실제 코드/데이터 형태로 최종 바이너리에 복사·합쳐집니다. 실행 시 추가로 libz.so를 찾을 필요가 없어집니다(그만큼 바이너리 크기는 커집니다).

Graal native-image가 zlib을 다루는 방식은:
- “이 애플리케이션이 JAR/HTTP 압축을 쓸 것인가?” “네이티브 이미지 안에서 zlib을 어떻게 연결할 것인가?”를 분석합니다.
- 필요하다면 native-image가 내부적으로 `-lz` 또는 zlib의 정적 라이브러리를 링크 플래그에 추가하여, 최종 바이너리에 zlib 의존성을 녹여 넣습니다.
- 정적 링크를 선택하면 호스트 시스템에 zlib이 설치되어 있는지 여부와 상관없이 압축 기능을 사용할 수 있고, 동적 링크를 선택하면 시스템 zlib을 재사용해 바이너리 크기를 줄일 수 있습니다.

## 배포 전략 비교: musl 정적 native-image vs `java -jar`

이제까지는 “어떻게 돌아가는가”를 봤다면, 여기서는 “어떻게 배포할 것인가”를 관점으로 **두 가지 전략**을 비교해 보겠습니다.

- 전략 A: Graal native-image + musl로 **정적 링킹된 단일 바이너리**를 만들고, 이를 Docker 이미지나 직접 배포.
- 전략 B: 특정 JDK 버전(예: 8, 11, 17…)을 기준으로 만든 **`java -jar` 애플리케이션**을 배포(필요하면 JRE/JDK를 함께 패키징).

당신의 목표는 “OS 환경, libc, 설치된 Java 버전에 최대한 영향받지 않는 배포”이기 때문에, 각 전략이 이 목표와 어떻게 맞고 어긋나는지에 특히 초점을 맞춥니다.

### 전략 A: musl 정적 native-image 단일 바이너리

**장점(목표에 잘 맞는 부분)**  
- OS/라이브러리 의존성 최소화:
  - musl 기반 정적 링크를 하면 glibc, 시스템 zlib 같은 의존성을 거의 제거할 수 있어, **리눅스 배포판/버전 차이에 영향받지 않는** 실행 파일에 가깝게 됩니다.
  - Docker 이미지에 이 바이너리와 필요한 설정 파일만 넣으면, 호스트 커널 버전만 맞으면 거의 어디서나 동일하게 동작합니다(물론 커널 ABI 최소 버전은 필요).
- Java 설치 필요 없음:
  - JVM/JDK를 설치할 필요가 없고, 컨테이너 안에 JDK를 넣어 줄 필요도 없습니다. **“Java 언어로 만들었지만, 배포는 네이티브 프로그램처럼”** 할 수 있습니다.
- 빠른 기동, 적은 메모리:
  - 자바 VM을 띄우는 오버헤드가 없고, 미리 컴파일된 코드만 있으므로 cold start가 매우 빠릅니다.
  - 불필요한 JDK 부분이 제거되기 때문에, 같은 기능 대비 메모리 사용량이 줄어드는 경우가 많습니다(물론 GC/스레드/프레임워크에 따라 다릅니다).
- 보안/공격면:
  - 호스트 시스템의 JDK, glibc, zlib 업그레이드에 따라 동작이 바뀌지 않는 대신, **검증된 바이너리와 컨테이너 이미지를 그대로 재배포**하는 식으로 관리할 수 있습니다.
  - 공격면이 줄어든다고 볼 수도 있고, 반대로 “라이브러리 업데이트에 자동으로 따라가지 않는다”는 점에서 관리 책임이 증가한다고도 볼 수 있습니다.

**단점(특히 유연성과 운영 측면에서)**  
- 동적 기능 제약:
  - 앞서 정리한 것처럼, 리플렉션/동적 클래스 로딩/플러그인 JAR 로딩 같은 **“실행 중에 새로운 바이트코드를 끼워 넣는”** 패턴이 사실상 막힙니다.
  - 플러그인, 스크립팅, 런타임에 모듈 추가/제거 같은 요구가 크다면, 설계를 많이 바꿔야 합니다(별도 프로세스, 스크립트 언어, 설정 기반 on/off 등).
- 업데이트 그라뉼러리티(배포 단위가 거칠다):
  - 기능을 조금만 추가해도 **새 native-image를 빌드해서 모든 배포 대상에 다시 배포**해야 합니다.
  - plug & play 방식으로 “플러그인 JAR 하나만 추가” 같은 업데이트는 할 수 없습니다.
- 빌드 복잡도:
  - musl 정적 링크 환경을 맞추는 것이 생각보다 까다로울 수 있습니다(툴체인, C 라이브러리 버전, OS 이미지 등).
  - CI/CD 파이프라인에서 각 아키텍처/플랫폼(musl용 Alpine, glibc용 Ubuntu 등)에 맞는 이미지를 따로 빌드·테스트해야 할 수 있습니다.
- 디버깅/프로파일링:
  - 표준 JVM 도구(jstack, jmap, Flight Recorder 등)를 쓸 수 없고, native-image용 별도 도구 또는 OS 레벨 디버거(gdb, perf)를 사용해야 합니다.
  - Java 개발자에게는 접근성이 낮게 느껴질 수 있습니다.

**당신의 목표와의 정합성**  
- “OS, libc, Java 버전에 영향받지 않게”라는 관점에서는 **전략 A가 가장 이상적인 그림**에 가깝습니다.
- 하지만 “plug & play로 특정 고객에게만 모듈을 추가/제거하고 싶다”라는 요구와는 정면으로 충돌합니다. 이 요구를 수용하려면:
  - 플러그인을 별도 프로세스로 분리하거나,
  - 기능은 전부 이미지 안에 넣고 설정으로만 on/off하는 식의 타협이 필요합니다.

### 전략 B: 특정 JDK 버전 기준 `java -jar` 배포

이 전략은 크게 두 가지 변형이 있습니다.
- B-1: “사용자 시스템에 이미 깔려 있는 JDK/JRE에 기대고, 우리는 JAR만 배포한다.”
- B-2: “우리만의 JDK 이미지를 함께 패키징해서 Docker 이미지나 zip/tar로 통째로 배포한다(jlink, custom runtime).”

당신이 언급하신 “Java 8 같은 널리 쓰이는 낮은 버전으로 배포하자”는 이야기는 B-1에 가깝습니다.

**장점**  
- 동적 기능, 유연성:
  - 일반 JVM이기 때문에, 리플렉션/동적 클래스 로딩/플러그인/OSGi/스크립팅 등 **자바 세계의 모든 동적 기능을 그대로 누릴 수 있습니다.**
  - `plugins/*.jar`를 runtime에 추가하고, 재기동/리로드로 기능을 늘리는 **plug & play 모델**이 자연스럽습니다.
- 개발 생산성:
  - 디버깅, 프로파일링, 모니터링 도구가 풍부합니다(jstack, jmap, jfr, VisualVM, async-profiler, Java agent 등).
  - 대부분의 라이브러리가 JVM을 기본 타겟으로 설계되어 있어, native-image보다 호환성 이슈가 적습니다.
- 언어/표준 라이브러리 독립성:
  - Java 8로 타겟팅하면, “Java 8 이상 JVM”에서 실행 가능합니다. 즉, 서버에 Java 11/17만 설치되어 있어도, **bytecode 타겟을 8로 맞추면 그대로 돌아갑니다.**
  - 이 점에서 “Java 버전” 문제는 생각보다 덜 심각합니다: 타겟은 8, 런타임은 11/17/21 등으로 꽤 폭넓게 호환됩니다(물론 deprecated API 사용, 모듈화 이슈에는 주의).

**단점**  
- 호스트 환경 의존성:
  - B-1 방식(사용자 JVM에 기대기)은 **설치된 JDK/JRE 버전, glibc 버전, OS 설정**에 매우 민감합니다.
    - 너무 낮은 JDK: 새 바이트코드/라이브러리 기능을 못 쓰거나, 아예 실행이 안 됨.
    - 너무 높은 JDK: 모듈 시스템, 보안 정책, 제거된 JDK 컴포넌트 등으로 인해 예전 코드나 리플렉션이 깨지는 경우가 있음.
  - B-2 방식(우리가 JDK까지 같이 배포)은 이 문제를 많이 줄이지만, 그만큼 이미지가 무겁고, “OS 환경 의존성 제거”라는 관점에서는 A와 큰 차이가 줄어듭니다(어차피 Docker 안에 우리 JDK를 넣어야 해서).
- 기동 시간, 메모리:
  - JVM을 띄우는 비용 때문에 cold start가 느리고, 작은 컨테이너 환경에서 오버헤드가 체감됩니다.
  - 특히 서버리스/짧은-lived 프로세스 환경에서는 native-image 대비 불리합니다.
- 보안/운영 관리:
  - 호스트의 JDK 또는 우리가 패키징한 JDK에 취약점이 발견되면, **JDK 자체를 업데이트한 이미지를 다시 배포**해야 합니다.
  - 이 점은 전략 A와 완전히 동일한 문제이기도 합니다(거기선 glibc/musl/zlib, 여기선 JDK).

**“그냥 Java 8로 하면 되지 않을까?”에 대한 현실적인 시각**  
- 많은 환경에 Java 8이 아직도 설치되어 있거나, 설치 가능하다는 건 사실입니다.
- 하지만:
  - 서버/클라우드 벤더들은 이미 11, 17, 21 같은 LTS 버전 중심으로 움직이고 있고,
  - 운영팀이나 보안팀 입장에서는 EOL된 Java 8을 새로 설치하거나 계속 유지하는 것을 꺼립니다.
- 더 중요한 포인트는:
  - **Java 8 타겟으로 컴파일한 JAR는 Java 11/17/21에서도 잘 돌아갈 수 있다**는 점입니다(특정 내부 API에 의존하지만 않으면).
  - 즉, “Java 8을 타겟으로 하는 것”과 “모든 곳에 Java 8만 설치되어 있다고 가정하는 것”은 다른 이야기입니다.
- 현실적인 접근은:
  - **bytecode 타겟을 8로 유지**하면서,
  - 배포 쪽에서는 “해당 서버/컨테이너에 최소한 LTS JDK(11/17)를 설치해 달라” 또는 “우리가 Docker 이미지로 JDK를 같이 가져가겠다” 쪽으로 가는 것입니다.

### 목표 관점에서 두 전략 정리

당신의 목표:  
> “OS 환경과 libc 등 라이브러리와 Java 언어에 최대한 영향받지 않게 프로그램을 배포”

이를 기준으로 보면:

- **환경 독립성(호스트에 대한 의존도)**:
  - A(musl native-image 정적): 가장 강력함. 호스트에는 커널만 맞으면 되고, glibc/JDK 설치 여부에 크게 의존하지 않습니다.
  - B-1(java -jar, 호스트 JDK 의존): 가장 약함. 호스트 JDK 버전/설정, glibc에 민감합니다.
  - B-2(java -jar + 우리 JDK 패키징): 중간. Docker 안에서는 독립적이지만, 여전히 glibc 등 OS 라이브러리에 의존합니다.

- **유연성/플러그인/plug & play**:
  - A: 같은 프로세스 안에서 **완전 동적 JAR 플러그인**은 사실상 불가능. 대신 별도 프로세스/스크립트/설정 on-off 등 우회 설계 필요.
  - B: JVM이 살아 있으므로, 전통적인 JAR 플러그인, ServiceLoader, 동적 모듈 시스템을 그대로 사용할 수 있음.

- **배포/업데이트 비용**:
  - A: 기능이 조금만 바뀌어도 **모든 대상에 새 바이너리 배포** 필요.
  - B-1: 플러그인은 JAR만 추가/교체하면 되고, core도 JAR 교체로 끝낼 수 있음(단, 호스트 JVM 의존).
  - B-2: core는 새 Docker 이미지 배포 필요, 플러그인은 여전히 JAR drop-in 가능.

### 실무적인 결론/권장 패턴

모든 요구를 한 번에 만족시키는 “마법의 하나”는 없습니다. 대신 다음과 같은 **하이브리드 구조**를 고려해 볼 수 있습니다.

- **코어는 native-image, 플러그인은 JVM/스크립트/외부 프로세스**:
  - 코어 로직(성능·cold start가 중요한 부분)은 전략 A로 빌드해 musl 정적 바이너리/컨테이너로 배포합니다.
  - 플러그인/고객별 커스터마이징이 필요한 부분은:
    - 별도 JVM 프로세스(JAR)로 두거나,
    - 스크립트 언어(Graal polyglot, Lua, JS 등)로 구현해 코어에서 해석 실행하거나,
    - REST/gRPC 같은 네트워크 인터페이스 뒤에 숨겨둡니다.
  - 이렇게 하면 “코어 배포는 환경 독립성 + 빠른 기동”, “플러그인 배포는 plug & play”라는 두 세계를 적당히 섞을 수 있습니다.

- **혹은, 전부 JVM으로 두되 배포는 B-2 패턴으로 통제**:
  - 환경 독립성이 “glibc, Java 버전까지 완전 제거” 수준까지는 필요 없고,
  - plug & play가 훨씬 더 중요하다면,
  - 전략 B-2(우리가 관리하는 JDK를 포함한 Docker 이미지 + JAR 플러그인)가 더 현실적인 선택입니다.

이 문서 전체를 기준으로 보면:
- **native-image + musl**은 “OS/라이브러리/Java에 덜 휘둘리는, 정적·예측 가능한 세계”를 줍니다.
- **java -jar**는 “동적, 확장 가능, 플러그인 친화적인 세계”를 줍니다.

최종 선택은 “환경 독립성 vs 동적 확장성” 중 어디에 더 무게를 둘지, 그리고 이를 위해 얼마만큼의 설계 복잡도를 받아들일지에 대한 팀의 판단입니다.

## 자주 받는 질문에 대비하는 짧은 답변
- Q: GraalVM이 곧 JDK 25인가요?  
  A: 아니요. “JDK 25 기반 GraalVM”처럼 특정 JDK를 품은 확장 배포판입니다.
- Q: 왜 native-image가 빠른가요?  
  A: JIT 없이 미리 컴파일된 네이티브 코드와 준비된 힙/메타데이터로 바로 실행하기 때문입니다.
- Q: 진짜 완전 독립적인가요?  
  A: JVM 설치는 필요 없지만, OS별 필수 라이브러리 제약은 남습니다. Linux는 musl 기반으로 거의 정적에 가깝게 만들 수 있습니다.
- Q: zlib 같은 건 어디 쓰이나요?  
  A: JAR/HTTP 압축 등에서 사용되며, native-image 시점에 정적/동적으로 바이너리에 포함됩니다.
