# IDE에선 되는데 CLI에선 안 된다 — IDE와 빌드 도구의 동작 차이

"IDE에선 컴파일·테스트가 다 통과하는데 `mvn`(또는 CI)에선 깨진다", 또는 그 반대. 자바 개발에서 가장 흔한 혼란 중 하나다. 원인은 거의 항상 같다 — IDE는 빌드·실행·테스트를 *자기 모델과 실행 구성*으로 돌리고, 그 모델이 빌드 도구(Maven/Gradle)와 미묘하게 다르기 때문이다. 이 노트는 특정 IDE 제품에 매이지 않고, IDE의 빌드·테스트 동작이 CLI 빌드 도구와 *어디서* 갈리는지를 일반 원리로 정리한다.

읽고 나면 다음을 스스로 설명할 수 있어야 한다 — IDE가 의존성을 아는 경로와 실제로 컴파일·실행하는 엔진이 왜 다를 수 있는지, green이 갈리는 구체적 축(클래스패스·JDK·인코딩·작업 디렉터리·프로파일·테스트 탐색), 그리고 무엇을 "진실의 기준"으로 삼아야 하는지.

## 1. 직답

대부분의 IDE는 빌드 도구의 프로젝트 정의(`pom.xml`/`build.gradle`)를 **임포트**해 의존성과 소스 루트를 가져온다. 하지만 실제 컴파일·실행·테스트는 IDE의 *자체 엔진과 실행 구성(run configuration)*으로 돌린다. 그래서 의존성 *정보*는 빌드 도구와 공유돼도, 실제 *실행 환경*(클래스패스 구성 방식, 작업 디렉터리, JVM/JDK, 인코딩, 어떤 테스트를 도는지)은 IDE와 CLI가 다를 수 있다. green이 갈리는 건 거의 다 이 "실행 환경 차이"다.

## 2. IDE가 의존성을 아는 법

IDE는 `pom.xml`/`build.gradle`을 읽어 의존성 그래프를 자기 프로젝트 모델(모듈·라이브러리·클래스패스)로 변환한다. 여기서 두 가지가 따라온다.

- **임포트/싱크가 필요하다** — `pom.xml`을 바꿔도 IDE 모델이 자동으로 따라오지 않을 수 있다. 의존성을 추가했는데 IDE에서 빨간 줄이 그대로면, 대개 "리임포트/싱크"가 안 된 것이다. 반대로 IDE 모델엔 있는데 `pom.xml`엔 안 적힌 의존성을 코드가 쓰면, IDE에선 되고 CLI에선 깨진다.
- **IDE 인덱스/캐시** — IDE는 자체 인덱스를 둔다. 가끔 이 캐시가 stale해서 실제와 어긋난다(캐시 무효화로 해결되는 부류의 문제).

즉 IDE의 "의존성 인식"은 빌드 도구의 정의를 *복제한 스냅샷*이지, 빌드 도구 그 자체가 아니다.

## 3. 컴파일: 자체 엔진 vs 빌드 도구 위임

많은 IDE는 빠른 피드백을 위해 *자체 증분 컴파일러*로 컴파일한다. 이건 `javac`/Maven compiler 플러그인과 설정·동작이 다를 수 있다(경고 수준, 애너테이션 처리, 컴파일 순서 등). 그래서 "IDE는 컴파일됐다는데 `mvn`은 `cannot find symbol`로 실패", 또는 그 반대가 가능하다.

대부분의 IDE는 이 차이를 줄이는 옵션 — *빌드를 Maven/Gradle에 위임(delegate)* — 을 제공한다. 위임 모드로 두면 IDE도 실제 빌드 도구로 컴파일·테스트하므로 CLI와 결과가 거의 일치한다. green 차이가 잦다면 이 설정부터 본다.

## 4. 실행/테스트 구성(run configuration)

IDE는 main·테스트를 **run configuration**으로 실행한다. 거기엔 다음이 들어간다 — 작업 디렉터리(working directory), VM 옵션(`-D…`, 인코딩), 환경 변수, 클래스패스(어느 모듈 기준), 활성 프로파일, 사용할 JDK(프로젝트 SDK). 이 구성이 `mvn`이 만드는 실행 환경과 다르면 결과가 갈린다.

이게 5절의 모든 차이의 공통 뿌리다. CLI(`mvn`)도 결국 자기 방식의 "실행 구성"을 만드는데, 두 구성이 어긋나는 것이다.

## 5. green이 갈리는 구체적 축

- **작업 디렉터리** — IDE는 보통 모듈 루트를, CLI는 `mvn`을 호출한 위치를 작업 디렉터리로 둘 수 있다. `Path.of("src/test/resources/x")`처럼 *상대경로*로 파일을 읽는 코드는 작업 디렉터리가 다르면 한쪽에서만 깨진다.
- **인코딩** — IDE 전역/프로젝트 인코딩 설정, JVM `file.encoding`, Maven `project.build.sourceEncoding`, surefire `argLine`이 제각각이면 한쪽에서만 한글 등이 깨진다([`../encoding/uhc.md`](../encoding/uhc.md)).
- **JDK** — IDE의 프로젝트 SDK와 셸의 `JAVA_HOME`이 다르면, 한쪽은 컴파일/실행되고 다른 쪽은 `release version N not supported`나 `UnsupportedClassVersionError`가 난다([`./javac_classpath_and_release.md`](./javac_classpath_and_release.md)).
- **활성 프로파일** — IDE의 Maven 프로파일 토글과 CLI의 `-P`가 다르면, 소스 루트·의존성·플러그인이 달라져 컴파일 대상부터 갈린다.
- **어떤 테스트를 도나** — IDE는 클릭한 클래스/메서드/디렉터리를 돌리지만, surefire는 이름 패턴(`**/*Test.java` 등)으로 고른다. 이름 규칙이 안 맞는 테스트는 IDE에선 돌고 CLI에선 누락된다([`./maven_build_lifecycle.md`](./maven_build_lifecycle.md)).
- **클래스패스 구성** — 2·3절. 임포트 stale, 위임 여부, 모듈 경계 차이.

## 6. 테스트 결과 리포트의 차이

IDE는 테스트 결과를 자체 트리 UI로 보여주고, "결과 내보내기" 기능으로 HTML/XML을 만든다. 이 포맷은 surefire가 남기는 `target/surefire-reports/*.xml`과 **다른 포맷**이다. 둘 다 "테스트 결과"지만, 대량 실패를 기계로 집계할 때 권위 있는 원천은 빌드 도구의 surefire XML이다.

또 JUnit 5의 **디스플레이 네임 제너레이터**나 `@DisplayName`을 쓰면, 리포트에 보이는 표시 이름이 실제 메서드명·클래스명과 다를 수 있다. 그래서 "리포트의 이 실패가 어느 클래스·메서드인지"를 표시 이름만으로 못 찾는 경우가 생긴다 — 이때는 스택트레이스나 원본 식별자(FQN)를 봐야 한다([`../thinking/failure_triage.md`](../thinking/failure_triage.md)).

## 7. 무엇을 진실의 기준으로 둘 것인가

CI와 배포는 결국 빌드 도구(CLI)로 돈다. 그래서 **빌드 도구의 결과(`mvn ...`)를 진실의 기준**으로 삼는다. IDE green은 빠른 피드백이지 최종 판정이 아니다. 차이를 줄이는 표준 절차는 다음과 같다.

- IDE 빌드를 Maven/Gradle에 위임하도록 설정한다(3절).
- run configuration의 작업 디렉터리·VM 옵션(인코딩)·JDK·활성 프로파일을 CLI와 일치시킨다.
- 최종 확인은 항상 깨끗한 셸에서 `mvn ...`으로, 결과는 surefire XML로 교차 확인한다.
- 의존성·구조를 바꾸면 IDE를 리임포트하고, 의심되면 캐시를 무효화한다.

## 8. 스스로 확인하기

- IDE가 의존성을 "안다"는 것과 빌드 도구가 의존성을 해석하는 것은 어떻게 다른가? 둘이 어긋나는 흔한 상황은?
- 상대경로로 파일을 읽는 테스트가 IDE에선 통과하고 CLI에선 실패한다. 어떤 축의 차이를 먼저 의심하나?
- "IDE는 컴파일되는데 mvn은 안 된다"를 가장 확실히 줄이는 IDE 설정은?
- IDE의 테스트 결과 HTML과 surefire XML 중, 대량 실패 집계에 무엇을 쓰고 왜?
- 최종 green의 기준을 IDE가 아니라 CLI로 두는 이유는?

## 9. 확인한 사실과 추론

사실: IDE가 빌드 도구 프로젝트를 임포트해 자체 모델로 다루는 것, 빌드 위임 옵션의 존재, run configuration이 작업 디렉터리·VM 옵션·JDK·프로파일을 담는 것, surefire XML과 IDE 내보내기 포맷이 다른 것, JUnit 5 디스플레이 네임 제너레이터로 표시 이름이 메서드명과 달라질 수 있는 것은 일반적으로 확인되는 동작이다.

추론·주의: 구체 메뉴 이름·동작은 IDE 제품과 버전마다 다르므로(증분 컴파일러 종류, 위임 설정 위치 등), 실제 적용은 사용하는 IDE의 공식 문서로 확인해야 한다. "IDE는 자체 컴파일러를 쓴다"도 제품·설정에 따라 다르며, 위임 모드면 빌드 도구 컴파일러를 쓴다.
